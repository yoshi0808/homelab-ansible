## 実装概要

`authy.internal` 上の FreeRADIUS 3.2.8 が正常稼働しているかを確認する、read-only healthcheck role。

shell (`radius-healthcheck.sh`) が対象ホスト上で収集と JSON 整形のみを担当し、判定・分類・fail 制御・report 保存はすべて Ansible tasks 側で行う。`core.md` の責務分離方針に準拠している。

## 各ファイルの処理概要

### radius-healthcheck.sh

**収集項目**

| 項目 | 収集内容 |
|---|---|
| freeradius.service_active | `systemctl is-active freeradius` の出力 |
| freeradius.version | FreeRADIUS バージョン文字列 |
| ports.udp_1812 | 1812/udp が Listen しているか（yes/no） |
| ports.udp_1813 | 1813/udp が Listen しているか（yes/no） |
| journal.error_count_1h | 直近1時間の ERROR/FATAL ログ件数 |
| journal.errors | 直近1時間の ERROR/FATAL ログ本文（複数行文字列） |
| chrony.tracking | `chronyc tracking` の出力全体 |

**使用コマンド**

| コマンド | 用途 |
|---|---|
| `systemctl is-active freeradius` | サービス稼働確認 |
| `ss -H -lun` | UDP ポート Listen 確認 |
| `freeradius -v` | バージョン取得 |
| `journalctl -u freeradius --since "1 hour ago" --no-pager -o short-iso` | journal エラー収集 |
| `chronyc tracking` | 時刻同期状態収集 |
| `python3` (標準ライブラリ `json`, `os`, `datetime`) | JSON 生成 |

**JSON 出力構造**

```json
{
  "collected_at": "2026-05-06T21:29:29Z",
  "freeradius": {
    "service_active": "active",
    "version": "3.2.8"
  },
  "ports": {
    "udp_1812": "yes",
    "udp_1813": "yes"
  },
  "journal": {
    "error_count_1h": 0,
    "errors": ""
  },
  "chrony": {
    "tracking": "..."
  }
}
```

### tasks/main.yml

**処理フロー**

1. `ansible.builtin.copy` — `radius-healthcheck.sh` を `/usr/local/sbin/` に配置（owner: root, mode: 0750）
2. `ansible.builtin.command` — スクリプト実行、結果を `healthcheck_raw` に格納（`changed_when: false`）
3. `ansible.builtin.fail` — `healthcheck_raw.stdout` が空の場合に fail（診断メッセージ付き）
4. `ansible.builtin.set_fact` — `from_json` で JSON をパースし `radius_health` へ格納
5. `ansible.builtin.set_fact` — `radius_criticals` / `radius_warnings` リストを組み立て
6. `ansible.builtin.set_fact` — `radius_report` を組み立て（観測値 + 判定結果）
7. `ansible.builtin.file` — report ディレクトリ作成（`delegate_to: localhost`, `become: false`）
8. `ansible.builtin.copy` — JSON report を `reports/radius-health/` に保存（`delegate_to: localhost`, `become: false`）
9. `ansible.builtin.debug` — サマリーを標準出力に表示
10. `ansible.builtin.fail` — `radius_criticals` が空でない場合に fail

**判定ロジック**

`radius_criticals` に積む条件:
- `freeradius.service_active != "active"` → サービス停止
- `ports.udp_1812 != "yes"` → 1812/udp 未 Listen
- `ports.udp_1813 != "yes"` → 1813/udp 未 Listen

`radius_warnings` に積む条件:
- `journal.error_count_1h > 0` → journal エラーあり

最終ステータス:
- `CRITICAL`: criticals に1件以上
- `WARNING`: criticals 0件、warnings に1件以上
- `OK`: criticals・warnings ともに0件

**report 保存**

- 保存先: `{{ radius_healthcheck_report_dir }}/{{ inventory_hostname }}_{{ ansible_facts['date_time']['iso8601_basic_short'] }}.json`
- 保存処理: `delegate_to: localhost` / `become: false` で制御ノード側に保存

## 実装上の判断・経緯

### jq → Python3 に変更した理由

初回実装では shell 内の JSON 生成に `jq -n` を使用していた。Codex レビュー（`2026-05-06_002_codex_review.md`）で「`jq` が対象ホストにない場合、標準出力に valid JSON が出ず `from_json` で失敗する」と指摘を受けた。

再実装では `jq` の事前確認 task を Ansible 側に追加したが、shell 単体で `jq` なしの場合に valid JSON を返せない問題は残存していた。さらに次の改修で `jq` 依存を完全排除し、Python 標準ライブラリ（`json`, `os`, `datetime`）を使うヒアドキュメント方式に切り替えた。

Python3 を採用した理由:
- `authy` は Ansible 管理対象のため Python 3 が前提として妥当
- `json.dumps` が複数行・ダブルクォート・バックスラッシュ・改行を正しくエスケープする
- `journalctl` や `chronyc tracking` の複数行出力でも JSON 構造が壊れない
- `jq` インストールの前提を authy に課さずに済む

環境変数経由でシェル変数を Python に渡し、`json.dumps` で一括シリアライズする実装になっている。

### ssのawkカラム修正（$5→$4）の経緯

初期実装では `ss -H -lun` の出力に対して `awk '{print $5}'` でカラムを抽出していた。実際の `ss -H -lun` 出力では local address/port は4列目（`$4`）に現れるため、`$5` では正しくポートを取得できず、Listen 判定が常に `no` になる問題が発生した。

実機テスト時に問題が顕在化し、`$4` に修正することで 1812/udp・1813/udp の判定が正しく動作するようになった（`Status: OK` を確認）。

### バージョン二重出力の修正内容

`freeradius -v` は複数行を出力し、`grep -oE 'Version [0-9][0-9.]+'` だけでは version 文字列が複数行にマッチして二重出力になるケースがあった。`grep -m1` を追加して最初のマッチ1件のみを取得する形に修正し、`awk '{print $2}'` でバージョン番号部分だけを抽出するようにした。

修正前: `grep -oE 'Version [0-9][0-9.]+'`
修正後: `grep -m1 -oE 'Version [0-9][0-9.]+' | awk '{print $2}'`

## 注意点

- `systemctl is-active freeradius || echo "inactive"` は、サービスが inactive / failed の場合に `inactive\ninactive` や `failed\ninactive` のような二重値になる可能性がある。テスト環境では問題化していないが、将来的に `|| true` に変更して出力をそのまま収集する方式に整理することを推奨。
- report ファイル（`reports/radius-health/*.json`）は実行生成物であり、`.gitignore` で除外されている。commit 時に意図せず混入しないよう注意。
- Python3 の存在が前提。Ansible 管理対象として `interpreter_python = /usr/bin/python3` を `ansible.cfg` で指定しているため妥当だが、shell 内では `python3` コマンドを呼ぶ形になっている。
