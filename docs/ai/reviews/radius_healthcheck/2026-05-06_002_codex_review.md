# radius_healthcheck Codex Review

## 総評

Claude Code の実装は、RADIUS healthcheck の初回要求に対して大筋では妥当です。

shell 側を収集と JSON 整形に寄せ、warning / critical 分類と fail 制御を Ansible tasks 側に置いている点は `core.md` の責務分離に沿っています。restart / reload / reboot / patch / upgrade などの変更操作も含まれていません。

ただし、新方針では `inventories/homelab/hosts.yml` を使う想定であるため、`inventories/lab.ini` に `radius_servers` を追加した点は方針違反の可能性が高く、実行前に修正必須です。また、shell の JSON 出力は `jq` が存在しない環境では壊れるため、依存関係の扱いを明確にする必要があります。

## 良い点

- `roles/radius_healthcheck/files/radius-healthcheck.sh` は、おおむね収集と JSON 整形のみを行っており、正常 / 異常判定や warning / critical 分類を shell 内に持ち込んでいない。
- `systemctl is-active`, `ss`, `journalctl`, `freeradius -v`, `chronyc tracking` は read-only な確認コマンドであり、変更操作は含まれていない。
- Ansible tasks 側で `radius_criticals` / `radius_warnings` を組み立て、最後に `ansible.builtin.fail` しているため、`core.md` の「判断と fail 制御は Ansible 側」に合っている。
- `Run radius healthcheck` に `changed_when: false` が付いており、確認コマンドの実行が changed 扱いにならない。
- レポート保存先の default は `{{ playbook_dir }}/../reports/radius-health` で、要求仕様の `reports/radius-health/` に合っている。
- report directory 作成と report 保存は `delegate_to: localhost` / `become: false` になっており、対象ホストではなく実行元へ保存する意図として妥当。
- 初回除外の `radtest`、証明書確認、設定構文チェックは実装されておらず、要求仕様に沿っている。
- 秘密鍵や証明書秘密鍵の中身を読む処理はない。

## 問題点

1. `inventories/lab.ini` への `radius_servers` 追加は新 inventory 方針とずれている

   - 該当箇所: `inventories/lab.ini:8-9`
   - `core.md` と依頼文では、新方針として `inventories/homelab/hosts.yml` を使う想定が示されている。
   - そのため、`radius_servers` は `inventories/homelab/hosts.yml` に YAML inventory として定義するのが自然。
   - `authy.internal` という名前ベースになっている点自体は良いが、追加先が `lab.ini` である点が方針違反になり得る。

2. shell の JSON 出力が `jq` 未導入時に壊れる

   - 該当箇所: `roles/radius_healthcheck/files/radius-healthcheck.sh:37-62`
   - JSON 生成を `jq -n` に依存しているが、`jq` が対象ホストにない場合、標準出力に valid JSON が出ず、`roles/radius_healthcheck/tasks/main.yml:18` の `from_json` で失敗する。
   - check shell の責務として JSON を返す設計なら、`jq` の存在確認をする、Ansible 側で事前確認する、または標準寄りの手段で JSON を生成する方針を明確にした方がよい。

3. port listen 判定がやや粗い

   - 該当箇所: `roles/radius_healthcheck/files/radius-healthcheck.sh:10-20`
   - `ss -ulnp | grep -q ':1812'` / `':1813'` は、表示形式によっては別の文字列にも一致し得る。
   - 要求は `1812/udp` と `1813/udp` の listen 確認なので、`ss -H -lun` の local address/port をより厳密に見る方が安全。

4. `Copy radius healthcheck script` は changed になり得る

   - 該当箇所: `roles/radius_healthcheck/tasks/main.yml:2-8`
   - check 実行そのものには `changed_when: false` が付いているが、script 配置は初回や差分発生時に changed になる。
   - これは Ansible の通常動作として許容できる一方、「healthcheck playbook は確認のみで changed を出したくない」という運用方針なら、script の一時実行方式や配置専用 role との分離を検討する余地がある。
   - ただし、`core.md` では copy して `/usr/local/sbin/` に配置する方式も許容しているため、即時の重大問題ではない。

5. `chronyc tracking` は収集のみで、同期異常の warning 判定はない

   - 該当箇所: `roles/radius_healthcheck/tasks/main.yml:30-33`, `roles/radius_healthcheck/tasks/main.yml:49-50`
   - 要求仕様は「chronyc tracking による時刻同期状態収集」であり、必ず warning 判定せよとは書かれていない。
   - 現状は仕様違反ではないが、「同期状態確認」という言葉を health 判定まで含む意味で扱うなら、Ansible 側に warning 条件を追加する必要がある。

## 修正必須

1. inventory の追加先を新方針に合わせる

   - `inventories/lab.ini` への `radius_servers` 追加は戻す。
   - `inventories/homelab/hosts.yml` に `radius_servers` / `authy` を定義する。
   - `ansible_host` は IP 直書きではなく `authy.internal` にする。
   - SSH ユーザーや鍵の扱いは既存の homelab inventory 方針に合わせる。

2. `jq` 依存の扱いを明確化する

   - `jq` を対象ホストの前提条件として docs / requirements に明記する、または Ansible task で `command -v jq` を read-only に確認して分かりやすく fail する。
   - より堅くするなら、shell で `jq` がない場合も valid JSON のエラーオブジェクトを返せる実装にする。
   - ただし、その場合も warning / critical 分類は shell ではなく Ansible 側で行う。

## 修正推奨

1. UDP port 判定を厳密化する

   - `ss -H -lun` などを使い、local address の port が `:1812` / `:1813` で終わることを確認する。
   - 可能なら process 情報 `-p` に依存せず、listen 状態だけを見る。`-p` は root 権限がないと情報が欠けることがある。

2. JSON parse 失敗時の扱いを改善する

   - `from_json` の前に `healthcheck_raw.rc` と `healthcheck_raw.stdout` を確認し、空 stdout や非 JSON の場合に原因が分かる fail メッセージを出す。
   - 現状でも失敗はするが、`jq` 未導入などの原因が利用者に伝わりにくい。

3. script 配置による changed の扱いを運用方針として決める

   - `/usr/local/sbin/` に配置する方式を続けるなら、初回 changed は許容する。
   - 完全 read-only 実行結果を重視するなら、`ansible.builtin.script` による一時実行、または script 配置 role と healthcheck role の分離を検討する。

4. journal の検索条件を必要に応じて調整する

   - 現状は `grep -E 'ERROR|FATAL'` で大文字のみ対象。
   - FreeRADIUS / journald の実ログ表記に合わせ、必要なら case-insensitive にする。
   - ただし、閾値や分類を細かくしすぎないという今回の方針から、現時点では最小限でよい。

## 今回は修正不要だが将来検討

- `chronyc tracking` の出力から `Leap status`, `System time`, `Last offset` などを Ansible 側で warning 判定する。
- FreeRADIUS の service 名が環境によって `freeradius` 以外になる場合に備え、default 変数化する。
- report に `ansible_host` や実行 playbook 名を含め、後から監査しやすくする。
- 将来拡張として `radtest`、証明書確認、設定構文チェックを追加する場合は、今回の role に混ぜすぎず、要求仕様を更新してから Ansible 側で判定を追加する。

## Claude Code への再実装依頼が必要か

必要です。

主理由は inventory 方針違反の可能性が高いことと、`jq` 依存により JSON 出力が常に valid とは言い切れないことです。

## 再実装が必要な場合の指示案

以下の方針で再実装してください。

- `inventories/lab.ini` に追加した `radius_servers` は削除してください。
- 新方針に従い、`inventories/homelab/hosts.yml` に `radius_servers` グループと `authy` ホストを YAML inventory 形式で追加してください。
- `authy` の `ansible_host` は IP 直書きではなく `authy.internal` にしてください。
- `radius-healthcheck.sh` は引き続き収集と JSON 整形のみにしてください。warning / critical / fail 判定は Ansible tasks 側に置いてください。
- `jq` を使う場合は、対象ホストに `jq` がない場合でも利用者が原因を理解できるよう、Ansible 側で事前確認するか、shell が valid JSON のエラー情報を返すようにしてください。
- UDP 1812 / 1813 の listen 判定は、`ss` の出力に対する単純な `grep ':1812'` より厳密にしてください。
- restart / reload / reboot / patch / upgrade などの変更操作は追加しないでください。
- `radtest`、証明書確認、設定構文チェックは今回も追加しないでください。
- report 保存先は `reports/radius-health/` のままにし、localhost への保存は `delegate_to: localhost` / `become: false` を維持してください。

## このまま実行してよいか

このまま実行するのは推奨しません。

特に、新方針の inventory が `inventories/homelab/hosts.yml` であるなら、`inventories/lab.ini` への追加を前提に実行すると運用方針とずれます。また、対象ホストに `jq` がない場合は JSON parse で失敗します。

## 修正後に実行すべきか

修正後に実行すべきです。

最低限、inventory の追加先を修正し、`jq` 依存の扱いを明確化してから実行してください。

## 実行前にユーザーが確認すべきコマンド

```bash
ansible-inventory -i inventories/homelab/hosts.yml --graph
```

```bash
ansible -i inventories/homelab/hosts.yml radius_servers -m ping
```

```bash
ansible -i inventories/homelab/hosts.yml radius_servers -b -m command -a 'command -v jq'
```

```bash
ansible-playbook -i inventories/homelab/hosts.yml playbooks/radius_healthcheck.yml --check
```

`--check` では script 配置の copy タスクが changed 予定として出る可能性があります。これは現実装の構造上あり得るため、完全に changed を避けたい場合は実装方針の見直しが必要です。
