## Findings

### 1. 中: OK の healthcheck でも mail.yml が必須になっている

対象: `roles/radius_healthcheck/tasks/main.yml:2`

`Load mail variables` が role 先頭で無条件に実行される。`radius_criticals` / `radius_warnings` が空で通知しないケースでも `inventories/vars/mail.yml` が存在しなければ playbook 全体が失敗する。

`ubuntu_vm_patch_policy.md` では `healthcheck: OK` は通知なしと定義されているため、OK 時に SMTP 設定が必須になるのは運用上の結合が強い。

ただし、今回の要求仕様では「vars 読み込みを tasks の先頭に追加」と明記されているため、実装は仕様どおりである。`mail.yml` を実行環境に常備する運用なら許容可能。手動実行環境や通知未設定環境でも healthcheck とレポート保存だけは動かしたい場合は、後続で通知 task の直前に `when: radius_criticals | length > 0 or radius_warnings | length > 0` 付きで `include_vars` を移動することを検討したい。

## 改善確認

### 1. `ignore_errors: true` 追加は有効

`roles/radius_healthcheck/tasks/main.yml:115` に `ignore_errors: true` が追加されている。

これにより SMTP 認証失敗・ネットワーク不通・STARTTLS エラーが起きても、CRITICAL 時に `Fail if critical` へ到達できる。`2026-05-09_011_implement.md` の説明どおり、前回指摘された「メール障害が RADIUS 判定を隠す」問題は解消されている。

### 2. 挿入位置は要件どおり

通知 task は `Print health summary` の直後、`Fail if critical` の直前にある。通知後に critical fail する構造になっている。

### 3. 通知条件は方針どおり

```yaml
when: radius_criticals | length > 0 or radius_warnings | length > 0
```

OK 時は送信しない。`ubuntu_vm_patch_policy.md` の通知方針と一致している。

### 4. 件名・本文・送信方式は要件どおり

- 件名: `[RADIUS Healthcheck] {{ radius_report.result.status }} - {{ inventory_hostname }}`
- 本文: host / status / criticals / warnings を含む
- 添付なし
- `secure: starttls`
- `delegate_to: localhost`
- `become: false`

### 5. shell 側と既存判定ロジックは変更なし

`roles/radius_healthcheck/files/radius-healthcheck.sh` は変更されていない。判定ロジック、レポート保存、`Fail if critical` の配置も維持されている。

## 結論

`2026-05-09_011_implement.md` の変更内容は妥当。`ignore_errors: true` の追加により、通知失敗で RADIUS の CRITICAL 判定が隠れる問題は解消されている。

今回の要求仕様に対しては、実装内容はおおむね適合しており、syntax-check も通過している。

残る `include_vars` の位置は仕様どおりではあるが、OK 時にも mail.yml が必須になる運用リスクがある。通知設定を常に用意する前提なら許容、そうでなければ後続で見直したい。
