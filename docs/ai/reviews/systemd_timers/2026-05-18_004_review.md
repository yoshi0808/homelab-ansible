## 結論

`playbooks/systemd_timers.yml` から `connection: local` が削除され、前回レビューの直接の指摘には対応されている。

ただし、`ansible_connection: local` を `control_nodes` / `dev_nodes` の group_vars に追加したことで、
`target_hosts` が示すホストと実際に task が実行されるホストがずれる問題は、実行場所によって依然として発生する。

このままでは、ansy からデフォルト実行した場合に quory ではなく ansy へ systemd unit を配置するリスクが残るため、修正を推奨する。

## 指摘事項

### 1. `control_nodes` に `ansible_connection: local` を設定しており、ansy からのデフォルト実行で quory に配置されない

**重要度**: High

該当箇所:

- `playbooks/systemd_timers.yml:3`
- `inventories/homelab/group_vars/control_nodes.yml:2`
- `inventories/homelab/hosts.yml:16`

現在の playbook は以下の通り、デフォルト対象を `control_nodes` にしている。

```yaml
hosts: "{{ target_hosts | default('control_nodes') }}"
```

一方で、`control_nodes` の group_vars には以下が追加されている。

```yaml
ansible_connection: local
```

この組み合わせでは、ansy から以下を実行した場合:

```bash
ansible-playbook playbooks/systemd_timers.yml -i inventories/homelab/hosts.yml
```

inventory 上の対象ホストは `quory` になるが、接続方式は local なので、実際の task は ansy 上で実行される。
結果として、quory に配置したい systemd unit が ansy の `/etc/systemd/system/` に配置される。

`2026-05-18_003_implement.md` では「タイマーを配置したいホスト自身から実行すること」が前提として追記されている。
しかし要求仕様では、playbook の hosts 変数化について以下のように定義されている。

```yaml
hosts: "{{ target_hosts | default('control_nodes') }}"
```

- デフォルトは `control_nodes`（quory）
- ansy で実行する場合は `-e "target_hosts=dev_nodes"` で指定

この要求から見ると、`target_hosts` は「unit を配置する対象ホスト」を表すべきであり、
実行元ホストと一致していることを暗黙の前提にすると誤操作しやすい。

推奨修正:

- `inventories/homelab/group_vars/control_nodes.yml` から `ansible_connection: local` を削除する。
- quory へ配置する場合は通常の inventory 接続で quory に接続する。
- quory 自身でローカル適用したい場合は、実行時に `-e ansible_connection=local` を明示するか、別途 `local` グループ / host_vars で限定する。

少なくとも、`control_nodes` 全体に `local` を固定するのは避けた方がよい。

### 2. group_vars への `ansible_connection: local` 追加は、この role 以外にも波及する

**重要度**: Medium

該当箇所:

- `inventories/homelab/group_vars/control_nodes.yml:2`
- `inventories/homelab/group_vars/dev_nodes.yml:2`

今回の要求スコープは、`ansy_systemd_timers` から `systemd_timers` へのリネームと変数化に限定されている。

しかし `group_vars/control_nodes.yml` と `group_vars/dev_nodes.yml` に `ansible_connection: local` を追加すると、
今後これらのグループを対象にするすべての playbook に影響する。

現時点で `control_nodes` / `dev_nodes` を対象にする playbook は `systemd_timers.yml` のみだったが、
`core.md` では `control_nodes` は quory 管理用、`dev_nodes` は ansy 管理用のグループとして定義されている。
将来 `quory_setup.yml` / `quory_update.yml` / `semaphore_setup.yml` などが追加された場合、
意図せず実行元ローカルに適用される可能性がある。

推奨修正:

- 接続方式は playbook / 実行時オプション / host_vars など、用途が限定される場所に寄せる。
- 少なくとも group_vars に置く場合は、「そのグループは常に対象ホスト自身で実行する」という運用制約を正本ドキュメントに明記する。

## 確認できたこと

- `playbooks/systemd_timers.yml` から playbook レベルの `connection: local` は削除されている。
- role 名は `systemd_timers` に変更されている。
- `roles/systemd_timers/defaults/main.yml` で以下の変数が定義されている。
  - `systemd_timers_working_dir`
  - `systemd_timers_run_user`
  - `systemd_timers_playbook_bin`
- service template 内の変数参照は `systemd_timers_*` に更新されている。
- timer のスケジュール・playbook 名・unit 名は変更されていない。
- 秘密情報の追加は見当たらない。
