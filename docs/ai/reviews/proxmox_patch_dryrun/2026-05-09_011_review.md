# Review: proxmox_patch_dryrun apt-get update 追加

レビュー日: 2026-05-09

## 結論

このまま commit は非推奨です。

`apt-get update` を追加する方針自体は、修正済み policy と整合しています。  
ただし、現在の shell 実装では `apt-get update` の出力が JSON stdout を壊す可能性があり、core.md の shell 責務ルールにも抵触します。

## 指摘事項

### 1. `apt-get update` の出力で JSON parse が壊れる可能性がある

対象:

- `roles/proxmox_patch_dryrun/files/proxmox-dryrun-collect.sh:11-12`
- `roles/proxmox_patch_dryrun/tasks/main.yml:34-36`

現在の実装:

```bash
apt-get update -qq 2>&1
```

このコマンドは stdout に出力し得ます。  
一方、Ansible 側は `dryrun_raw.stdout | from_json` で、shell stdout 全体が JSON である前提です。

そのため、`apt-get update` が warning / error / repository message を出した場合、stdout が以下のようになり得ます。

```text
W: ...
{"collected_at": "...", ...}
```

この場合、`from_json` が失敗し、構造化された `BLOCKED` 判定や report 生成に進めません。

これは core.md の以下の方針に反します。

- check 系 shell は結果を JSON に整形して標準出力へ返す
- shell は対象ホスト上の情報収集センサーとして扱う

shell stdout は JSON のみにすべきです。

### 2. `apt-get update` 失敗が Status 判定に入っていない

対象:

- `roles/proxmox_patch_dryrun/files/proxmox-dryrun-collect.sh:12`
- `roles/proxmox_patch_dryrun/tasks/main.yml:89-96`

現在は `apt-get update` の成否を変数化していません。  
また `set -e` もないため、`apt-get update` が失敗しても後続の `apt-get check` / `apt-get -s dist-upgrade` が実行されます。

その結果、古い package list に基づいた simulation 結果で `PATCH_READY` などに進む可能性があります。

推奨:

- `apt_update_output=$(apt-get update -qq 2>&1) && apt_update_ok=true || apt_update_ok=false`
- JSON に `apt_update_ok` / `apt_update_output` を含める
- merge 側で `node_summaries` に含める
- Ansible の `_pre_status` で `apt_update_ok == false` を `BLOCKED` にする

## shell / Ansible 責務分離

方針としては概ね維持されていますが、現状の `apt-get update -qq 2>&1` は stdout に JSON 以外を出し得るため、責務分離以前に「shell stdout は JSON」という契約が崩れます。

`apt-get update` の実行自体は policy で許容された semi-safe 操作として扱えます。  
ただし、その成否と出力は JSON の観測値として返し、判定は Ansible 側で行うべきです。

## 検証

- `bash -n roles/proxmox_patch_dryrun/files/proxmox-dryrun-collect.sh`: 成功

## commit 可否

commit 不可です。

理由:

- shell stdout が JSON 以外で汚染される可能性がある
- `apt-get update` 失敗が `BLOCKED` 判定に入らない
- 古い package list のまま `PATCH_READY` 判定へ進み得る
