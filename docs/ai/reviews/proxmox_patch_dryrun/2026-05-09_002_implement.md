# proxmox_patch_dryrun 実装内容

実装日: 2026-05-09

## 実装概要

`proxmox_patch_dryrun` role および `playbooks/proxmox_patch_dryrun.yml` を実装した。

pve1/pve2 上で `apt-get -s dist-upgrade` を実行し、更新候補・changelog diff を収集。
ansy 上でマージ・Codex 分類・Status/Urgency 判定・レポート保存・メール送信を行う。

## 処理フロー詳細

### Phase 1: 収集（pve1/pve2 で並列実行）

- `proxmox-dryrun-collect.sh` を `/usr/local/sbin/` に配置して実行
- `LC_ALL=C apt-get -s dist-upgrade` でシミュレーション
- Python で `Inst` / `Remv` 行を解析
- セキュリティリポジトリ由来の判定: `security.debian.org` / `-security` / `Debian-Security` をパターンマッチ
- `apt changelog <pkg>` で changelog 取得
  - 更新パッケージ: `apt_pkg.version_compare` でインストール済みより新しいエントリのみ抽出
  - 新規パッケージ: 最新エントリ 1 件のみ
- `/var/run/reboot-required` の有無を確認
- JSON で stdout に出力

### Phase 2: マージ（localhost）

- `proxmox-dryrun-merge.py` に両ノードのデータを渡す
- パッケージ名でデデュープ（同名パッケージは 1 エントリに統合し `nodes: ["pve1","pve2"]` を付与）
- `IMPORTANT_COMPONENTS` / `IMPORTANT_COMPONENT_PREFIXES` 環境変数でリスト照合
- `important_component` フラグを各パッケージに付与
- `classification.common` / `pve1_only` / `pve2_only` に分類
- `unified_dryrun.json` を生成

### Phase 2b: 事前ステータス判定

- sim_ok が False のノードがある → `BLOCKED`
- updates も removes も 0 → `NO_UPDATES`
- それ以外 → `NEEDS_CLASSIFICATION` (Codex へ)

### Phase 3: Codex CLI 呼び出し（`NEEDS_CLASSIFICATION` 時のみ）

- `scripts/codex-classify.sh unified_dryrun.json codex_output.json` を実行
- 出力: `package_classifications`, `status_inputs`, `mail_subject`, `mail_body`, `report_md`

### Phase 4: 最終判定

Codex 出力と機械的フラグから以下を決定:

**Status 判定順序:**
1. `BLOCKED` (sim_ok 失敗)
2. `NO_UPDATES`
3. `MAJOR_UPGRADE_DETECTED` (Codex: major_upgrade_suspected)
4. `MAINTENANCE_REQUIRED` (重要コンポーネント更新あり or removes あり)
5. `PATCH_READY`

**Urgency 判定順序:**
1. `cve_types_detected` が URGENT 条件に合致 (RCE / LPE / VM escape / 認証バイパス等) → `URGENT`
2. Codex urgency_candidate に `URGENT` が含まれる → `URGENT`
3. `HIGH` が含まれる → `HIGH`
4. security_repo フラグ付きパッケージあり → `HIGH`
5. `NORMAL` が含まれる → `NORMAL`
6. それ以外 → `LOW`

### Phase 5: レポート保存

- `reports/proxmox-dryrun/{{ timestamp }}_unified_dryrun.json`
- `reports/proxmox-dryrun/{{ timestamp }}_codex_output.json` (NEEDS_CLASSIFICATION 時)
- `reports/proxmox-dryrun/{{ timestamp }}_final.json`
- `reports/proxmox-dryrun/{{ timestamp }}_report.md` (Codex 生成 or 最小テンプレート)

### Phase 6: メール送信

- `inventories/vars/mail.yml` から認証情報を読み込み
- `community.general.mail` モジュールで送信 (STARTTLS)
- NEEDS_CLASSIFICATION: Codex 生成の件名・本文・MD 添付
- NO_UPDATES / BLOCKED: 定型の件名・本文・添付なし

## 設計上の判断

- `apt-get update` は実行しない（要求仕様の「変更操作なし（apt-get -s のみ）」制約に従う）
- changelog 取得に `apt_pkg` モジュールを使用（Proxmox/Debian 標準で利用可能）
- パッケージのデデュープは merge.py 側で行い、tasks は機械的な照合のみ
- `hostvars[groups['proxmox'][0]]` / `[1]` でノードを順序付きで参照（hosts.yml の記述順）
- BLOCKED / NO_UPDATES では Codex を呼ばない（codex-classify.sh が空リストを拒否するため）

## 注意点

- `groups['proxmox'][0]` / `[1]` はインベントリ順（pve1, pve2）に依存する。順序が変わると参照先が変わる
- `apt changelog` はネットワークアクセスが必要。タイムアウトは 60 秒に設定
- reports/*.md は `.gitignore` で除外済み。MD レポートはコミット対象外
