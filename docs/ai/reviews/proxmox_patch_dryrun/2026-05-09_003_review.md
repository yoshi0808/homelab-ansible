# Review: proxmox_patch_dryrun

レビュー日: 2026-05-09

- `docs/ai/prompts/core.md`
- `docs/ai/prompts/proxmox_patch_policy.md`

- `roles/proxmox_patch_dryrun/tasks/main.yml`
- `roles/proxmox_patch_dryrun/files/proxmox-dryrun-collect.sh`
- `roles/proxmox_patch_dryrun/files/proxmox-dryrun-merge.py`

## 前提整理

`pve1` / `pve2` の固定参照は、以下のユーザー希望に基づく設計前提として扱う。

- `pve1`: 主系
- `pve2`: 従系 / 先行検証ノード
- 主要 VM は `pve1` に配置する

したがって、今回のレビューでは `pve1` / `pve2` 固定参照自体は指摘対象外とする。

## 結論

このまま commit する前に、policy と要求仕様の不一致を整理した方がよい。

実装そのものは、shell / Ansible の責務分離を大きく崩していない。一方で、`proxmox_patch_policy.md` の `proxmox_patch_dryrun.yml` 定義と、今回の要求仕様・実装の間に差分が残っている。

## 指摘事項

### 1. `PATCH_READY` 判定が policy の条件を満たしていない

`proxmox_patch_policy.md` の `PATCH_READY` 条件には以下が含まれている。

- healthcheck が OK
- `apt-get check` 成功
- `apt-get -s dist-upgrade` 成功
- remove なし
- major upgrade 疑いなし
- 重要コンポーネント更新なし

一方、実装では `roles/proxmox_patch_dryrun/tasks/main.yml` の最終判定で、主に以下を見て `PATCH_READY` に到達する。

- `apt-get -s dist-upgrade` の `sim_ok`
- remove の有無
- 重要コンポーネント更新の有無
- Codex 出力の `major_upgrade_suspected`

healthcheck OK と `apt-get check` 成功を確認していないため、policy 上の `PATCH_READY` 条件を完全には満たしていない。

判断:

- `PATCH_READY` を後続の自動適用判断に使うなら、commit 前に修正推奨。
- dry-run 通知専用であり、apply 側で必ず再判定する運用なら、policy または要求仕様にその前提を明記する必要がある。

### 2. `apt-get update` の扱いが policy と要求仕様で衝突している

`proxmox_patch_policy.md` では `proxmox_patch_dryrun.yml` の処理概要に `apt-get update` が含まれている。

一方、要求仕様では制約として「変更操作なし（`apt-get -s` のみ）」が明記され、実装メモでも `apt-get update` は実行しない判断になっている。

このため、実装が悪いというより、policy と要求仕様の正本関係が曖昧になっている。

判断:

- `apt-get update` を dry-run の一部とするなら実装不足。
- 「変更操作なし」を優先するなら、policy 側の `apt-get update` 記載を改訂した方がよい。

### 3. `NO_UPDATES` の通知方針が policy と要求仕様で衝突している

`proxmox_patch_policy.md` では `NO_UPDATES` の行動が「通知しない」となっている。

一方、要求仕様では「NO_UPDATES でもメール送信するが件名に明示」となっており、実装もその方針に従っている。

判断:

- ユーザー運用として NO_UPDATES 通知が必要なら、実装は要求仕様に沿っている。
- ただし core / policy 準拠レビューとしては、policy 側の記述更新が必要。

### 4. changelog 全文保存の扱いが policy とずれている可能性がある

`proxmox_patch_policy.md` では changelog 全文を report に保存する運用が記載されている。

実装では各 package の `changelog_diff` を unified JSON に含め、Codex 生成の `report_md` にも含める想定になっている。

ただし、パッケージごとの changelog text file を個別保存する実装ではない。

判断:

- unified JSON / MD report に含める方針で十分なら問題なし。
- policy が「changelog text files」を個別成果物として求めるなら実装不足。

## shell / Ansible 責務分離

概ね守られている。

`roles/proxmox_patch_dryrun/files/proxmox-dryrun-collect.sh` は以下に限定されている。

- `apt-get -s dist-upgrade` の実行
- `Inst` / `Remv` 行の抽出
- security repo 由来らしさの観測値付与
- changelog diff の取得
- `/var/run/reboot-required` の有無確認
- JSON 出力

shell 側で `PATCH_READY` / `MAINTENANCE_REQUIRED` / `BLOCKED` などの最終判断はしていない。

Ansible 側は以下を担当している。

- shell 配置・実行
- JSON 読み込み
- pve1 / pve2 の結果マージ
- Codex CLI 呼び出し
- Status / Urgency 判定
- report 保存
- mail 送信

これは `core.md` の責務分離に沿っている。

## PVE1/PVE2 固定について

今回のユーザー希望により、`pve1` を主系、`pve2` を従系・先行検証ノードとして固定扱いする前提は妥当とする。

そのため、以下は今回の commit 阻害要因とはしない。

- `hostvars['pve1']`
- `hostvars['pve2']`
- `reboot_required_pve1`
- `reboot_required_pve2`
- メール本文内の `pve1/pve2` 固定表現

ただし、将来 `--limit` 単一ノード実行を正式サポートする場合は、別途設計変更が必要。

- `ansible-playbook -i inventories/homelab/hosts.yml playbooks/proxmox_patch_dryrun.yml --syntax-check`: 成功

## commit 可否

現時点では、そのまま commit は非推奨。

理由:

- `PATCH_READY` 判定が policy の healthcheck / `apt-get check` 条件を満たしていない
- `apt-get update` の扱いが policy と要求仕様で衝突している
- `NO_UPDATES` 通知方針が policy と要求仕様で衝突している

ただし、PVE1/PVE2 固定はユーザー希望に基づく前提として許容する。

## 推奨対応

1. dry-run で `PATCH_READY` を確定するなら、healthcheck OK と `apt-get check` 成功を判定に入れる。
2. dry-run は通知専用で、apply 側で再判定する方針なら、policy にその責務境界を明記する。
3. `apt-get update` と `NO_UPDATES` 通知について、policy と要求仕様のどちらを正とするか決めて文書を揃える。
