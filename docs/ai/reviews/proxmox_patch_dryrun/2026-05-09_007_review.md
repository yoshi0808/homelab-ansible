## 結論

このまま commit はまだ非推奨。

前回指摘した policy の自己矛盾は概ね解消されている。ただし、healthcheck gate の実装が Ansible のホスト単位 fail 動作を考慮できておらず、「pve1 / pve2 のいずれかが WARNING / CRITICAL の場合 dry-run を実行しない」という意図を保証できていない。

## 指摘事項

### 1. 片系 healthcheck NG 時に、もう片系で dry-run が進む可能性がある

`playbooks/proxmox_patch_dryrun.yml:7-9` で `proxmox_healthcheck` の後に `proxmox_patch_dryrun` を実行している。

`roles/proxmox_patch_dryrun/tasks/main.yml:6-9` には healthcheck gate が追加されている。

```yaml
- name: Fail if healthcheck is not OK
  ansible.builtin.fail:
    msg: "Healthcheck is {{ proxmox_report.result.status }} on {{ inventory_hostname }}, dryrun aborted."
  when: proxmox_report.result.status != 'OK'
```

しかし、Ansible の通常動作では、あるホストで task が fail しても、他ホストは play を継続する。

そのため、たとえば `pve2` が `WARNING` でこの gate に失敗しても、`pve1` は `roles/proxmox_patch_dryrun/tasks/main.yml:15-36` の dry-run collect まで進む可能性がある。

さらに `proxmox_healthcheck` role 側の `CRITICAL` fail もホスト単位である。

- `roles/proxmox_healthcheck/tasks/main.yml:150-153`

この場合も、片方が `CRITICAL` で失敗しても、もう片方が後続 role へ進む可能性がある。

これは `2026-05-09_006_implement.md` の以下の説明と一致しない。

> pve1 / pve2 のいずれかで healthcheck が WARNING / CRITICAL の場合、dry-run は実行されない。

影響:

- healthcheck NG の片系がある状態で、もう片系だけ `apt-get check` / `apt-get -s dist-upgrade` / `apt changelog` を実行し得る。
- その後 `hostvars['pve1'].node_dryrun` / `hostvars['pve2'].node_dryrun` の片方が未定義になり、merge 前後で失敗する可能性がある。

推奨:

- `playbooks/proxmox_patch_dryrun.yml` に `any_errors_fatal: true` を追加し、片系の healthcheck fail で全ホストの後続処理を止める。
- もしくは healthcheck role が fail する前提を見直し、run_once のクラスタ単位 gate で pve1 / pve2 の status をまとめて評価してから Phase 1 に進む。

今回の構成では、`pve1` / `pve2` 固定前提なので `any_errors_fatal: true` を playbook に入れるのが最小修正に見える。

### 2. policy はまだ `--limit` 単一ノード実行可と書いている

`docs/ai/prompts/proxmox_patch_policy.md:416-421` では、`proxmox_patch_dryrun.yml` の対象に以下が残っている。

- `--limit` による単一ノード実行も可

一方、実装は以下のように `pve1` / `pve2` の両方が存在する前提になっている。

- `roles/proxmox_patch_dryrun/tasks/main.yml:58-60`
- `roles/proxmox_patch_dryrun/tasks/main.yml:92-95`

ユーザー希望により `pve1` 主系 / `pve2` 従系の固定前提は妥当と扱う。ただし、その前提を採用するなら policy 側の `--limit` 単一ノード実行可は削除または「非対応」と明記した方がよい。

判断:

- `--limit` 単一ノード実行をサポートしない方針なら、policy 修正で足りる。
- サポートする方針なら、現在の固定参照実装では不足。

前回レビューで確認済みの通り、以下は対応済み。
