## 修正内容

### P1: Webhook URL と `token` の不一致を修正（notify.yml）

`community.general.slack` の `token` は `https://hooks.slack.com/services/` 以降のパス部分（`T.../B.../...`）を要求する。
URL をそのまま渡すと送信先が二重になり、全通知が失敗する。

対応: `_slack_webhook`（URL）から接頭辞を除去した `_slack_token` を生成し、`token` へ渡す。
`no_log: true` を付与して URL が誤ってログへ出ないようにした。

### P1: `Load Slack variables` に `no_log: true` を追加（notify.yml）

`include_vars` は復号済み変数を変数スペースに書き込む。
verbosity 上昇時や callback plugin によって Webhook URL がログへ露出し得るため、`no_log: true` を追加した。

### P1/P2: 通知準備失敗を best-effort 化（notify.yml）

Vault 復号失敗・変数未定義・Webhook 選択失敗など、`include_vars` や `set_fact` の失敗が play 全体を止めていた。
notify.yml 全体を `block/rescue` で囲み、準備含めた全失敗を rescue で吸収した。
rescue ではデバッグメッセージを出力するのみで、呼び出し元 play には影響しない。

### P2: post-healthcheck 異常時の通知先を `#alerts` に修正（proxmox_patch_apply_node）

`_post_hc_status != 'OK'`（パッチ適用後の healthcheck 異常）でも `_notify_channel` が `patches` になっていた。
要求仕様では proxmox_patch_apply_node の異常はすべて `#alerts` のため、条件を修正した。

変更前: `_notify_channel: "{{ 'alerts' if (_apply_abort_type != 'none' or _apply_failed | bool) else 'patches' }}"`
変更後: `_notify_channel: "{{ 'alerts' if (_apply_abort_type != 'none' or _apply_failed | bool or _post_hc_status != 'OK') else 'patches' }}"`

### P2: reboot 失敗時の `#alerts` 通知を追加（ubuntu_nightly.yml）

`ansible.builtin.reboot` タイムアウト時、play がその場で失敗し `#alerts` に何も届かなかった。
reboot + サービス確認 + 結果通知を `block/rescue` で囲み、reboot 失敗時も rescue で `#alerts` へ通知する。

- block 内の `Fail if critical` を削除し、block/rescue の外へ移動
  - rescueでキャッチされると reboot 失敗通知と混在するため
- rescue でエラー通知（critical/#alerts）を送信し、その後 `ansible.builtin.fail` で play を失敗させる
- block 外の `Fail if critical` は `nightly_criticals | default([])` でガードし、rescue 後の undefined を回避

2 プレイ（`radius_servers` / `monitoring_servers`）とも同じ構成で修正した。
