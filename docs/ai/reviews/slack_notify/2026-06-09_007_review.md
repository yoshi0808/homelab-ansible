## Findings

### P2: ポリシーの「mail.yml は削除済み」という記述が実際のリポジトリ状態と一致しない

- 該当箇所:
  - `docs/ai/prompts/ubuntu_vm_patch_policy.md:212`
  - `inventories/vars/mail.yml`

更新後のポリシーには、`inventories/vars/mail.yml` は不要になったため「削除済み」と記載されている。しかし同ファイルは現在も存在し、`git ls-files` でも Git 管理対象であることを確認した。現在の git diff に削除差分もない。

Slack移行対象の実装ファイルから `mail.yml` の読込みと `community.general.mail` は削除されているため、コード移行自体には影響しない。ただし運用ポリシーが存在しないファイルとして扱っていると、秘密情報の棚卸しや将来の整理時に誤認を招く。

対応は次のどちらかに統一する必要がある。

- `mail.yml` を残す場合は、「Slack移行対象では使用しない」など実態に合う記述へ修正する。
- リポジトリ全体で不要と判断する場合は、他のポリシーや README に残る SMTP / `mail.yml` の記述も整理した上で、実ファイルを削除する。

要求仕様は移行対象タスクからメール通知と不要なロードを削除することを求めているが、`mail.yml` 自体の削除を明示的には要求していない。そのため、文書を実態に合わせて修正する方が今回の範囲では小さい。

### P3: ポリシーの更新日と版が今回の改訂を反映していない

- 該当箇所:
  - `docs/ai/prompts/ubuntu_vm_patch_policy.md:3-5`
  - `docs/ai/prompts/ubuntu_vm_patch_policy.md:175-212`

通知方式を SMTP から Slack Incoming Webhook へ変更する運用上の改訂が入った一方、文書先頭は `更新日: 2026-05-27`、`版: v1.2` のままである。

この文書を運用方針の正本として扱うなら、少なくとも更新日を今回の変更日である `2026-06-09` に更新し、既存の版管理ルールに従って版番号も更新した方がよい。

## 前回指摘の確認

`2026-06-09_005_review.md` の指摘であった、Ubuntu運用ポリシーの通知方式と実装の不一致は解消されている。

- 通知方式が `roles/common_slack/tasks/notify.yml` による Slack Incoming Webhook と明記された。
- Vault暗号化済みの `inventories/vars/slack.yml` を使用することが記載された。
- `ubuntu_nightly` と healthcheck の通知先・status が実装と一致している。
- 通知失敗を best-effort として扱う方針が記載された。

## 確認結果

実行した確認:

```bash
git diff --check

ANSIBLE_LOCAL_TEMP=/tmp/ansible-local ANSIBLE_REMOTE_TEMP=/tmp/ansible-remote \
  ansible-playbook --syntax-check -i inventories/homelab/hosts.yml <対象 playbook>

git ls-files --stage inventories/vars/mail.yml inventories/vars/slack.yml
```

結果:

- `git diff --check` は成功。
- 変更対象を呼び出す playbook 10本の syntax-check はすべて成功。
- `inventories/vars/slack.yml` は Vault 暗号化形式であることを確認した。
- `inventories/vars/mail.yml` は現在も存在し、Git 管理対象である。
- Slackへの実送信は行っていない。

## 判定

前回指摘された通知方式のポリシー不整合は解消され、Slack通知コードにも新たな重大な問題は見つからなかった。

ただし、運用ポリシーに事実と異なる「mail.yml は削除済み」という記述が残っている。この記述と文書メタデータを修正してから確定・commitすることを推奨する。
