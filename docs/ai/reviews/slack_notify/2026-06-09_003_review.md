`inventories/vars/slack.yml` は Vault 暗号化済みのため、内容を読まずにレビューした。

## Findings

### P1: Webhook URL 全体を `token` に渡しているため、Slack送信が失敗する

- 該当箇所:
  - `docs/ai/reviews/slack_notify/2026-06-08_001_requirement.md:7-14`
  - `roles/common_slack/tasks/notify.yml:25-36`

要求仕様では `slack_webhook_info` / `slack_webhook_alerts` / `slack_webhook_patches` は Webhook URL と定義されているが、共通タスクはその値をそのまま `community.general.slack.token` に渡している。

`community.general.slack` の `token` は、Incoming Webhook URL 全体ではなく、`https://hooks.slack.com/services/` より後ろの `T.../B.../...` 部分を受け取る仕様である。現在の実装で URL 全体を渡すと、モジュール内部では概ね次のような送信先が組み立てられる。

```text
https://hooks.slack.com/services/https://hooks.slack.com/services/T.../B.../...
```

送信タスクには `ignore_errors: true` と `no_log: true` があるため、全通知が失敗しても play は継続し、通常出力から原因を確認しにくい。今回の移行目的そのものが達成されないため blocking。

推奨:

- Vault 側を要求仕様どおり URL のまま維持する場合は、共通タスクで `https://hooks.slack.com/services/` の接頭辞を除去してから `token` に渡す。
- または変数を token 部分だけに変更し、変数名・要求仕様も `slack_token_*` に合わせる。
- 修正後、テスト用チャンネルへの実送信確認を行う。

### P1: Slack変数の読込タスクに `no_log` がなく、Webhook URLがログへ露出し得る

- 該当箇所:
  - `docs/ai/reviews/slack_notify/2026-06-08_001_requirement.md:55`
  - `roles/common_slack/tasks/notify.yml:8-12`

要求仕様は「Webhook URL を含むタスクには `no_log`」としている。`include_vars` の結果には読み込んだ変数が含まれるため、詳細 verbosity、callback plugin、実行基盤側のログ保存条件によっては Vault 復号後の Webhook URL が出力され得る。

`Set Slack webhook URL` と送信タスクには `no_log: true` があるが、秘密値を最初に読み込む `Load Slack variables` にはない。Webhook URL は漏えい時に第三者が投稿できる認証情報なので、同タスクにも `no_log: true` が必要。

### P2: post-healthcheck 異常時の通知先が `#patches` になっている

- 該当箇所:
  - `docs/ai/reviews/slack_notify/2026-06-08_001_requirement.md:46`
  - `roles/proxmox_patch_apply_node/tasks/main.yml:492-500`

要求仕様のチャンネル割り当てでは `proxmox_patch_apply_node` の異常は `#alerts` とされている。

しかし `_apply_failed == false`、`_apply_abort_type == none`、`_post_hc_status != OK` の場合、status は `warning` になる一方、channel は `patches` のままになる。パッチ適用後に Proxmox healthcheck が `CRITICAL` または `UNKNOWN` でも alerts に届かず、異常の見落としにつながる。

`_post_hc_status != 'OK'` も `_notify_channel` の alerts 条件へ含める必要がある。

### P2: 通知共通タスクの準備失敗は無視されず、通知対象の play 自体を失敗させる

- 該当箇所:
  - `roles/common_slack/tasks/notify.yml:8-32`
  - `docs/ai/reviews/slack_notify/2026-06-08_002_implement.md:119-121`

送信タスクだけは `ignore_errors: true` だが、次の失敗は無視されない。

- Vault password 未指定、Vault 復号失敗、ファイル不在
- 必須 webhook 変数の未定義
- Webhook 選択用 `set_fact` の評価失敗

そのため、たとえば healthcheck の WARNING 通知時に Slack 設定の読込が失敗すると、本来の healthcheck 判定より先に play が失敗する。`proxmox_patch_apply_node` では `always` 内の通知失敗により、後続の最終レポート保存と元の失敗の再送出にも到達しない可能性がある。

また implement 文書には「Vault ロード失敗時は `ignore_errors` で通知スキップ」とあるが、実装はその説明と一致していない。

通知を best-effort とする方針なら、共通タスク全体を block/rescue 化して準備失敗も吸収する必要がある。逆に Slack 設定不備を blocking とするなら、パッチ適用前など安全な位置で明示的に検証し、運用方針と implement 文書を修正する必要がある。

### P2: `ubuntu_nightly` の reboot 失敗時に `#alerts` 通知へ到達しない

- 該当箇所:
  - `docs/ai/prompts/ubuntu_vm_patch_policy.md:117-126`
  - `docs/ai/reviews/slack_notify/2026-06-08_001_requirement.md:44`
  - `playbooks/ubuntu_nightly.yml:66-79`
  - `playbooks/ubuntu_nightly.yml:148-174`
  - `playbooks/ubuntu_nightly.yml:240-253`
  - `playbooks/ubuntu_nightly.yml:379-407`

現在は reboot 開始を `#info` へ送った後、`ansible.builtin.reboot` がタイムアウトまたは失敗すると、その場で host の処理が終了する。異常結果通知は reboot 後のサービス確認より後ろにしかないため、`#alerts` には何も届かない。

要求仕様の `ubuntu_nightly` 異常時は `#alerts`、Ubuntu policy の「reboot 実行時・異常時の通知」という方針を満たすには、reboot と post-check を block/rescue で囲み、reboot 失敗も error/critical として alerts へ通知する必要がある。

## 良い点

- メール通知呼び出しは対象ファイルから削除され、共通 `include_tasks` に集約されている。
- healthcheck 系は OK の場合に通知せず、WARNING / CRITICAL のみ alerts に送る条件になっている。
- Slack送信タスク自体には `no_log: true` があり、送信失敗で play を止めない構成になっている。
- `proxmox_hw_check` は要求仕様どおり今回の変更対象外。
- Vault 暗号化済みの秘密ファイルを平文化・読取りしていない。

## 確認結果

実行した確認:

```bash
git diff --check

ANSIBLE_LOCAL_TEMP=/tmp/ansible-local ANSIBLE_REMOTE_TEMP=/tmp/ansible-remote \
  ansible-playbook --syntax-check -i inventories/homelab/hosts.yml <対象 playbook>

ANSIBLE_LOCAL_TEMP=/tmp/ansible-local ANSIBLE_REMOTE_TEMP=/tmp/ansible-remote \
  ansible-doc community.general.slack
```

結果:

- `git diff --check` は成功。
- 変更対象 role を呼び出す playbook 9本の syntax-check は成功。
- Ansible Core は 2.20.1。
- インストール済み `community.general.slack` の実装とドキュメントで、`token` は Webhook URL 末尾の token 部分を指定することを確認した。
- Slackへの実送信は行っていない。

## 判定

このまま commit することは推奨しない。

少なくとも Webhook URL と `token` の不一致を修正し、`Load Slack variables` の `no_log`、post-healthcheck 異常の alerts 振り分けを直した上で再レビューが必要。通知準備失敗と reboot 失敗を best-effort 通知としてどう扱うかも、要求仕様に沿って明確にする必要がある。
