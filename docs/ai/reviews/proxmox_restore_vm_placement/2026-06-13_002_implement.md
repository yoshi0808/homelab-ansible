## 実装概要

Phase 3 の「メンテナンス解除 → LRM 待機」の直後に、HA VM への `relocate` コマンド発行を追加した。

Proxmox HA はメンテナンス解除後に自動リロケートを行わないため、
明示的に `ha-manager crm-command relocate` を発行してから復帰ポーリングに進む設計に修正した。

## 変更内容の要約

### `roles/proxmox_restore_vm_placement/tasks/main.yml`

**変更箇所**: Phase 3 — 「Record maintenance mode disabled」の直後、「Get HA VMs that prefer target_node」の直前

**追加したタスク（2つ）**:

```yaml
- name: Find HA VMs that prefer target_node but are currently elsewhere
  ansible.builtin.set_fact:
    _ha_vms_to_relocate: >-
      {{ _restore_all_resources |
         selectattr('type', 'defined') |
         selectattr('type', 'in', ['qemu', 'lxc']) |
         selectattr('tags', 'defined') |
         selectattr('tags', 'search', '(^|;)hacritical(;|$)') |
         selectattr('tags', 'search', '(^|;)prefer' ~ inventory_hostname ~ '(;|$)') |
         rejectattr('node', 'equalto', inventory_hostname) |
         list }}

- name: Relocate HA VMs back to target_node
  ansible.builtin.command:
    cmd: ha-manager crm-command relocate vm:{{ item.vmid }} {{ inventory_hostname }}
  loop: "{{ _ha_vms_to_relocate }}"
  loop_control:
    label: "{{ item.type }}:{{ item.vmid }} {{ item.name | default('') }} from {{ item.node }}"
  changed_when: true
  when: _ha_vms_to_relocate | length > 0
```

**修正後の Phase 3 フロー**:

```
1. ha-manager crm-command node-maintenance disable <node>   ← 既存（変更なし）
2. ha-manager status ポーリング → lrm active/idle 待機        ← 既存（変更なし）
3. Record maintenance mode disabled                           ← 既存（変更なし）
4. [新規] prefer<target> かつ他ノードにいる HA VM を特定
5. [新規] 各 VM に ha-manager crm-command relocate vm:<id> <target_node> を発行
6. Get HA VMs that prefer target_node (_ha_prefer_target 構築) ← 既存（変更なし）
7. pvesh get /cluster/resources ポーリング → 全 HA VM が target_node で running になるまで待機  ← 既存（変更なし）
```

## 設計上の判断

### データソースに `_restore_all_resources` を使う理由

`_ha_vms_to_relocate` の構築には Phase 1 取得済みの `_restore_all_resources` を使用した。
LRM 待機後に再取得する案も検討したが、パッチ適用中に HA VM 構成が変わるケースは実運用上ほぼない。
既存の `_ha_prefer_target` タスクも同じデータソースを使っており、一貫性を保った。

### 既存タスクを変更しない理由

- `Get HA VMs that prefer target_node`（`_ha_prefer_target` を構築）
  `when: _ha_prefer_target | length > 0` の guard として引き続き機能するため、変更しない。
- `Wait for HA-managed VMs to return to target node`
  `until` 条件は調査結果で「論理的に正しい」と確認済みのため、変更しない。

### コメントを追記した理由

`relocate` 追加の WHY（Proxmox HA が自動リロケートしない理由）は非自明なため、
コメントブロック 2 行を追加した。

## 注意点

- `_ha_vms_to_relocate` は `_restore_all_resources`（Phase 1 取得）を参照する。
  メンテナンス解除後にクラスター構成が変わった場合は最新データを反映しないが、
  通常の Proxmox パッチ運用では問題にならない。
- `ha-manager crm-command relocate` は `vm:<vmid>` 形式でしか受け付けない。
  LXC コンテナにも `vm:` プレフィックスを使う（Proxmox HA の仕様）。
  既存コードも同様の前提で設計されているため問題なし。
