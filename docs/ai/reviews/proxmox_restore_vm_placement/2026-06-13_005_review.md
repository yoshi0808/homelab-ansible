## Findings

### Low: 正本ポリシーの Mode A 手順と role の Phase 見出しに旧フローが残っている

該当箇所:

- `docs/ai/prompts/proxmox_patch_policy.md:1253`
- `roles/proxmox_restore_vm_placement/tasks/main.yml:158`

`proxmox_patch_policy.md` の詳細な `6.7` と playbook 冒頭コメントは、maintenance disable 後に明示的な `ha-manager crm-command relocate` を発行する内容へ正しく更新されている。

一方、同じ正本ポリシーの Mode A 実行フローでは、restore の処理順が次のままになっている。

```text
maintenance disable → HA復帰待機
```

実際の処理は次の順序である。

```text
maintenance disable → LRM待機 → HA対象特定 → relocate発行 → HA復帰待機
```

また、role の Phase 3 見出しにある `Disable maintenance mode (triggers HA restore)` は、maintenance disable 自体が HA restore を開始するように読めるため、今回確認した Proxmox HA の挙動と一致しない。

修正案:

- Mode A の restore 手順へ `relocate` 発行を追記する
- Phase 3 見出しを、maintenance disable と明示 relocate の両方を示す表現へ変更する

例:

```text
pve2 を restore（non-HA migrate back → 停止VMを起動 → maintenance disable
→ LRM待機 → HA relocate → HA復帰待機 → post-restore healthcheck）
```

```yaml
# Phase 3: Disable maintenance mode and relocate HA VMs home
```
