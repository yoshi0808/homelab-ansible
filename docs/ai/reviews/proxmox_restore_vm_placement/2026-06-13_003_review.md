## Findings

### Medium: 運用ポリシーと playbook の処理説明が旧挙動のまま

該当箇所:

- `docs/ai/prompts/proxmox_patch_policy.md:243`
- `docs/ai/prompts/proxmox_patch_policy.md:250`
- `docs/ai/prompts/proxmox_patch_policy.md:812-814`
- `playbooks/proxmox_restore_vm_placement.yml:14-15`

現在の文書は、maintenance mode を disable すると HA VM / CT が自動復帰すると説明している。

```text
maintenance mode disable により HA が自動復帰
```

しかし、今回の調査結果と実装は、disable だけでは稼働中の HA VM / CT は home node に戻らず、各サービスに対する明示的な `ha-manager crm-command relocate` が必要という前提である。

`core.md` では `proxmox_patch_policy.md` を運用ポリシーの正本として扱っているため、この矛盾を残すと、将来の改修や障害調査で旧前提へ戻る可能性がある。また、playbook 冒頭の実行手順も実際の処理を正しく表していない。

修正案:

- タグ分類表の restore 動作を「maintenance mode disable 後、明示的な HA relocate で復帰」に更新する
- restore ロールの説明に `hacritical` + `prefer<target>` の対象抽出と `crm-command relocate` を追記する
- `6.7` の処理概要で、maintenance inactive 待機後に relocate 発行のステップを追加する
- playbook 冒頭コメントの「automatically」を削除し、明示 relocate を記載する

## 実装確認

追加された処理は要求仕様と整合している。

- `hacritical` と `prefer<target>` の両方を持つ QEMU / LXC のみを対象としている
- target node に既にいるリソースを除外している
- Proxmox HA のサービス ID として QEMU / LXC 共通の `vm:<vmid>` を使用している
- relocate 後は既存のポーリングで target node 上の配置と `running` 状態を確認している
- relocate 失敗時は既存の `block` / `rescue` により restore 全体が失敗扱いになる

上記 Findings 以外に、今回のコード差分に重大な問題は見つからなかった。
