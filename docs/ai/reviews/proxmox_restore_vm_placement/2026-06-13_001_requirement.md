# 調査報告: proxmox_restore_vm_placement HA VM 復帰タイムアウト問題

日付: 2026-06-13
対象: `roles/proxmox_restore_vm_placement/tasks/main.yml`
問題: HA 管理 VM の pve1 への復帰待ちが 30 回リトライ（約 5 分）でタイムアウト

---

## 調査対象タスク

```
roles/proxmox_restore_vm_placement/tasks/main.yml
  - Phase 3: Disable maintenance mode (line 161–219)
  - "Wait for HA-managed VMs to return to target node" (line 194–219)
```

デフォルト値（`defaults/main.yml`）:
- `proxmox_restore_ha_timeout`: 300 秒
- `proxmox_restore_ha_poll_interval`: 10 秒
- retries = 300 / 10 = **30 回**（約 5 分）

---

## 調査結果

### 1. 判定ロジックに問題はないか

判定ロジック自体（`until` 条件）は論理的に正しい。

```yaml
until: >-
  (条件A: prefer<target> タグを持つ HA VM が他ノードにいない == 0) and
  (条件B: target_node 上の prefer<target> HA VM で running でないものが == 0)
```

- 条件 A: HA VM が全て target_node に戻ったことを確認する
- 条件 B: target_node 上の HA VM が全て running であることを確認する

この 2 条件の AND は適切である。

**ただし、軽微な懸念点が 1 つある。**
`_ha_prefer_target` リスト（line 184–192）は Phase 3 開始前（メンテナンス解除前）に取得した
`_restore_all_resources` を使って構築される。この時点では VM はまだ pve2 にいる。
一方、ポーリングの `until` 条件は毎回 `pvesh get /cluster/resources` を新規取得して判定する。
この非対称性は実害にはならない（タグベースのフィルタなので VM の場所によらず正しく機能する）が、
`_ha_prefer_target` がリスト長チェック（`when: _ha_prefer_target | length > 0`）にしか使われていないことに注意が必要。

---

### 2. HA VM への relocate 自動実行はしているか

**していない。これが根本原因。**

現在の Phase 3 の処理順序：

```
1. ha-manager crm-command node-maintenance disable <node>   ← メンテナンス解除
2. ha-manager status をポーリング → lrm が active/idle になるまで待つ
3. pvesh get /cluster/resources をポーリング → HA VM が target_node に戻るまで待つ  ← ここでタイムアウト
```

**`relocate` コマンドは一切発行していない。**

---

### 3. タイムアウトの根本原因

Proxmox HA の `prefer` ポリシーは **初期配置（起動時の選好）** であり、
**メンテナンス解除後に自動でリロケートを起動する仕組みではない。**

Proxmox HA の動作仕様：

| 操作 | 動作 |
|---|---|
| メンテナンスモード有効 | pve1 上の HA VM を他ノードへ退避 |
| メンテナンスモード解除 | LRM が active/idle に戻る。ただし VM の自動リロケートは **発生しない** |
| `ha-manager crm-command relocate vm:<id> <node>` | 明示的に CRM へリロケート指示。数秒で実行される |

つまり、メンテナンスを解除しても、pve2 で稼働中の HA VM は
「pve2 で正常稼働中」として認識され、HA は自発的にリロケートしない。
HA の `prefer` タグは、次回起動時の配置先選好に影響するが、**起動中 VM の強制移動は行わない。**

手動で `ha-manager crm-command relocate` を実行したら数秒で戻った、という報告と完全に一致する。

---

## 修正案

Phase 3 の「メンテナンス解除 → LRM 待機」の直後に、
各 HA VM に対して `relocate` コマンドを発行するタスクを追加する。

### 追加するタスクのイメージ

```yaml
# メンテナンス解除・LRM 待機の直後に挿入

- name: Find HA VMs that prefer target_node but are currently elsewhere
  ansible.builtin.set_fact:
    _ha_vms_to_relocate: >-
      {{ _restore_all_resources |
         selectattr('type', 'defined') |
         selectattr('type', 'in', ['qemu', 'lxc']) |
         selectattr('tags', 'defined') |
         selectattr('tags', 'search', '(^|;)hacritical(;|$)') |
         selectattr('tags', 'search', '(^|;)prefer' ~ inventory_hostname ~ '(;|$)') |
         rejectattr('node', 'equalto', inventory_hostname) |
         list }}

- name: Relocate HA VMs back to target_node
  ansible.builtin.command:
    cmd: ha-manager crm-command relocate vm:{{ item.vmid }} {{ inventory_hostname }}
  loop: "{{ _ha_vms_to_relocate }}"
  loop_control:
    label: "{{ item.type }}:{{ item.vmid }} {{ item.name | default('') }} from {{ item.node }}"
  changed_when: true
  when: _ha_vms_to_relocate | length > 0
```

この後に既存の「Wait for HA-managed VMs to return to target node」ポーリングが続く。

### 修正後の Phase 3 フロー

```
1. ha-manager crm-command node-maintenance disable <node>
2. ha-manager status ポーリング → lrm active/idle 待機
3. [新規] prefer<target> かつ他ノードにいる HA VM を特定
4. [新規] 各 VM に ha-manager crm-command relocate vm:<id> <target_node>
5. pvesh get /cluster/resources ポーリング → 全 HA VM が target_node で running になるまで待機
```

---

## 補足: `_ha_vms_to_relocate` のデータソースについて

現在の `_ha_prefer_target` および修正案の `_ha_vms_to_relocate` は、
Phase 1 開始時に取得した `_restore_all_resources` を使用している。
メンテナンス解除後に状態が変わっている可能性がゼロではないため、
必要に応じて Phase 3 の LRM 待機後に `pvesh get /cluster/resources` を再取得し、
その結果から `_ha_vms_to_relocate` を構築する設計も検討できる。

ただし実害のあるケースは稀（パッチ適用中に HA VM の構成が変わることはない）なので、
実装の簡潔さを優先して既存の `_restore_all_resources` を再利用する方針でよい。

## まとめ

| 調査項目 | 結論 |
|---|---|
| 判定ロジックの問題 | `until` 条件は正しい。問題なし |
| relocate の自動実行 | **していない。これが根本原因** |
| 根本原因 | Proxmox HA はメンテナンス解除後に自動リロケートしない。`ha-manager crm-command relocate` の明示的な発行が必要 |
| 修正方針 | LRM 待機後、prefer タグを持つ HA VM に `relocate` を発行してから復帰ポーリングを行う |
