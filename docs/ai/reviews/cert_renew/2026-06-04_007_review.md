## Findings

### P1: quory 自身の deploy/restart 失敗時に CA 秘密鍵 cleanup が実行されない可能性がある

- 該当箇所:
  - `playbooks/cert_renew.yml:21`
  - `playbooks/cert_renew.yml:53`
  - `roles/homelab_cert_renew/tasks/deploy_semaphore.yml:45`
  - `roles/homelab_cert_renew/tasks/cleanup.yml:1`

要求仕様では、CA 秘密鍵は quory 上の `/run/semaphore-ca/` に一時展開し、処理後に必ず削除することになっている。

現在の cleanup は後続 Play の `hosts: quory` として実行される。しかし、Play 2 の `semaphore_servers` には `quory` 自身も含まれており、quory の証明書配置や `Restart semaphore` が失敗すると、quory は Ansible の failed host 扱いになる。その状態では後続の `hosts: quory` cleanup Play が quory に対して実行されず、`/run/semaphore-ca/ca.key` が残る可能性がある。

特に今回の修正で handler ではなく明示 restart に変更されたため、`roles/homelab_cert_renew/tasks/deploy_semaphore.yml:45` の restart 失敗はそのまま quory host の失敗になる。

推奨:

- cleanup を failed host 状態の影響を受けにくい形にする。
- 例: cleanup Play は `hosts: localhost` で実行し、cleanup task を `delegate_to: "{{ cert_renew_ca_host }}"` する。
- もしくは cleanup 前に `meta: clear_host_errors` を使う。ただし unreachable な quory には効かないため、可能な失敗範囲と運用上の限界を明記する。
- より堅牢にするなら、発行・配布・通知を `block` / `rescue` / `always` に再構成し、CA cleanup を `always` 相当に置く。

### P2: 証明書発行途中で失敗したホストが通知では `skipped` と表示される

- 該当箇所:
  - `roles/homelab_cert_renew/tasks/issue.yml:36`
  - `roles/homelab_cert_renew/tasks/issue.yml:72`
  - `roles/homelab_cert_renew/tasks/issue.yml:89`
  - `playbooks/cert_renew.yml:75`

通知は `hostvars[h].cert_renewed | default(false)` を見て、false または未設定なら `skipped` と表示する。

しかし `cert_renewed` は `issue.yml` の最後で設定されるため、以下のように発行途中で失敗した場合は未設定のままになる。

- OpenSSL config のテンプレート作成に失敗した。
- EC 秘密鍵生成に失敗した。
- CSR 生成に失敗した。
- CA 署名に失敗した。

この場合、実際には「更新不要で skipped」ではなく「更新対象だったが発行失敗」だが、最終通知では `skipped` と誤表示されうる。

推奨:

- `cert_needs_renewal` を通知で参照し、`cert_renewed` 未設定かつ `cert_needs_renewal: true` の場合は `issue FAILED` と表示する。
- あるいは `issue.yml` の更新要否判定直後に `cert_renew_attempted: true` を設定し、発行成功後に `cert_issued: true` を設定する。
- 通知の状態は `skipped` / `issued but deploy failed` / `issued and restarted` / `issue failed` のように、発行失敗と更新不要を分離する。

## Notes

- Play 2 / Play 3 に `serial: 1` が追加されており、quory 上で同時に複数の `openssl x509 -CAcreateserial` が動く前回 P1 の serial file 競合リスクは解消されている。
- `deploy_semaphore.yml` / `deploy_proxmox.yml` で handler を使わず明示的に `systemd` restart を実行し、その後に `cert_deployed: true` を設定する形になっている。前回 P2 の「restart 成功前に deployed 扱いになる」問題は概ね解消されている。
- PVE の一時ファイル削除は引き続き `block` / `always` で行われており、`pvenode cert set` や `pveproxy` restart 失敗時にも `/tmp/pve_*_{{ inventory_hostname }}.pem` は削除される構造になっている。
- `ansible-playbook -i inventories/homelab/hosts.yml playbooks/cert_renew.yml --syntax-check` は、`ANSIBLE_LOCAL_TEMP=/tmp/ansible-local` と `ANSIBLE_REMOTE_TEMP=/tmp/ansible-remote` を指定して成功した。

## Overall

前回レビューの serial 競合と restart 成功追跡は、今回の実装でよく改善されている。

ただし、CA 秘密鍵の cleanup が quory の failed host 状態に左右される点は、要求仕様の「処理後に必ず削除」に直接関わる。ここはマージ前に直した方がよい。
