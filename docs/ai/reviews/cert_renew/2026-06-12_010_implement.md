## 指摘対応の詳細

### [Medium] 確認手順をすべて quory からの実行に統一

**指摘内容**: 前回の確認手順では ansy / monnie 上で `openssl verify` を実行しており、
`/home/yoshi/.cert/ca/` の CA 証明書を参照していた。
しかし Policy では CA 証明書は quory のみに配置するため、
ansy / monnie ではそのパスが存在せず手順どおりに実行できない。

**方針**: 直接の `openssl verify` は quory のローカル証明書（quory 自身のもの）に限定し、
他の全サービスは quory からの `s_client -CAfile` による確認に統一する。
CA 証明書・秘密鍵を他ホストへ配置する必要はない。

---

#### 再訂正後の確認手順（運用者確認待ち）

**実行ホスト: すべて quory**

```sh
ROOT_CA=/home/yoshi/.cert/ca/home_radius_ca.crt
INT_CA=/home/yoshi/.cert/ca/home_tls_ca.crt
```

> **前提**: ルートCA証明書（Home-RADIUS-CA の公開証明書）を
> `/home/yoshi/.cert/ca/home_radius_ca.crt` として quory に配置しておくこと。
> 秘密鍵はオフライン保管のまま。証明書（公開情報）のみで検証可能。

---

##### (1) チェーン検証: quory 自身の Semaphore 証明書

quory のローカルに配布済みの fullchain から `openssl x509` でリーフを取り出し、verify する。
**（quory のローカルファイルのみ参照。他ホストへの接続不要）**

```sh
openssl x509 -in /etc/semaphore/tls/quory.internal.crt -out /tmp/leaf_quory.crt
openssl verify -CAfile "$ROOT_CA" -untrusted "$INT_CA" /tmp/leaf_quory.crt
rm -f /tmp/leaf_quory.crt
```

期待結果: `/tmp/leaf_quory.crt: OK`

---

##### (2) フルチェーン提示確認: 全サービス（quory から s_client で実行）

`-CAfile` でルートCA証明書を明示することで `Verify return code: 0 (ok)` が正確に得られる。

```sh
# Semaphore: ansy
echo | openssl s_client \
  -connect ansy.internal:3000 -servername ansy.internal \
  -showcerts -CAfile "$ROOT_CA" \
  2>&1 | grep -E "( s:| i:|Verify return code)"

# Semaphore: quory
echo | openssl s_client \
  -connect quory.internal:3000 -servername quory.internal \
  -showcerts -CAfile "$ROOT_CA" \
  2>&1 | grep -E "( s:| i:|Verify return code)"

# Grafana: monnie
echo | openssl s_client \
  -connect monnie.internal:3000 -servername monnie.internal \
  -showcerts -CAfile "$ROOT_CA" \
  2>&1 | grep -E "( s:| i:|Verify return code)"

# pveproxy: pve1
echo | openssl s_client \
  -connect pve1.internal:8006 -servername pve1.internal \
  -showcerts -CAfile "$ROOT_CA" \
  2>&1 | grep -E "( s:| i:|Verify return code)"

# pveproxy: pve2
echo | openssl s_client \
  -connect pve2.internal:8006 -servername pve2.internal \
  -showcerts -CAfile "$ROOT_CA" \
  2>&1 | grep -E "( s:| i:|Verify return code)"
```

期待結果の形式（例: quory）:

```text
 0 s:CN = quory.internal
   i:CN = Home-TLS-CA
 1 s:CN = Home-TLS-CA
   i:CN = Home-RADIUS-CA
Verify return code: 0 (ok)
```

確認ポイント:
- cert 0（depth=0）の issuer が `Home-TLS-CA` であること（中間CA署名）
- cert 1（depth=1）の subject が `Home-TLS-CA` であること（中間CA証明書が提示されている）
- `-CAfile` で指定した Home-RADIUS-CA に対して `Verify return code: 0 (ok)` であること

### [Low] 初回移行時の `force_renew: true` 必要性の明記

**指摘内容**: `cert_renew.yml` は既存証明書の残存日数が 15 日超の場合、
発行済みCA（旧ルートCA署名）の issuer を確認せず更新をスキップする。
中間CA移行の初回実行では全対象を確実に更新するために `force_renew: true` が必要。

なお `cert_renew_quory.yml` は play 内で `force_renew: true` が固定済みであり変更不要。

**対応**: `cert_renew_policy.md` §10 に初回移行実行手順を追記した（下記参照）。

---

#### 初回移行実行手順（要約）

**cert_renew_quory.yml（quory の Semaphore 証明書）**:
通常どおり実行。play 内で `force_renew: true` が固定済みのため追加操作不要。

```sh
ansible-playbook -i inventories/homelab/hosts.yml playbooks/cert_renew_quory.yml
```

**cert_renew.yml（ansy / pve1 / pve2 / monnie）**:
`force_renew: true` を明示して実行することで残存日数にかかわらず全対象を更新する。

```sh
ansible-playbook -i inventories/homelab/hosts.yml playbooks/cert_renew.yml \
  -e force_renew=true
```

Semaphore から実行する場合は、Task Template の Extra Variables に `force_renew: true` を設定する。
