- 該当:
  - `playbooks/cert_renew_quory.yml:1`
  - `playbooks/cert_renew.yml:5-7`

- 状況:
  - `playbooks/cert_renew_quory.yml` は `A` で staged されており、前回指摘の「未追跡」は解消されている。
  - しかし `playbooks/cert_renew.yml`, `.gitignore`, `roles/homelab_cert_renew/defaults/main.yml`, `roles/homelab_cert_renew/tasks/prepare_ca.yml`, `roles/homelab_cert_renew/files/fix_pem.py` は unstaged のまま。

  - 実装メモが説明している最終状態と、実際に commit される内容がズレる。
- 対応案:
  - 1 commit にまとめるなら、本体変更も `git add` して staged 内容を揃える。
  - 意図的に分割するなら、commit 単位ごとに動作する状態になるよう、`cert_renew_quory.yml` 追加と `cert_renew.yml` の quory 除外、prepare_ca 方式変更の境界を明確にする。

### Low: CA秘密鍵の owner 前提が実装メモにあるが、コードでは検査していない

  - `docs/ai/reviews/cert_renew/2026-06-05_028_implement.md:48-56`
  - `roles/homelab_cert_renew/tasks/prepare_ca.yml:12-28`
- 状況:

  - 実際の `prepare_ca.yml` は存在確認と mode 0600 のみを検査し、owner / uid は見ていない。
- 影響:
  - root 所有や別ユーザー所有でも mode 0600 なら通る可能性があり、運用前提と実チェックに差が出る。
  - `become: true` で実行される限り動作はし得るが、ドキュメント上の期待値とズレる。
- 対応案:
  - owner を本当に要件にするなら `ca_key_stat.stat.pw_name == 'yoshi'` などを検査する。
  - owner を問わず root で読めればよいなら、実装メモの「所有者: yoshi」を運用例として表現し直す。
