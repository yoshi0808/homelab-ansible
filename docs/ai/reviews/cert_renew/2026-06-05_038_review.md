# Review: cert_renew monnie (Grafana) 追加実装

日付: 2026-06-05

## 結論

修正推奨。`deploy_grafana.yml` が追加され、monnie でも issue -> deploy -> restart の流れは構成されている。
ただし、Grafana が実際に新しい証明書パスを読む保証が差分内にないため、成功通知になってもブラウザ警告が解消されない可能性がある。

## Findings

### High: Grafana の TLS 設定更新・検証がなく、新しい証明書を配置しても使われない可能性がある

- 対象:
  - `roles/homelab_cert_renew/defaults/main.yml:53`
  - `roles/homelab_cert_renew/defaults/main.yml:54`
  - `roles/homelab_cert_renew/tasks/deploy_grafana.yml:17`
  - `roles/homelab_cert_renew/tasks/deploy_grafana.yml:45`

`monnie` の証明書パスを `/etc/grafana/certs/monnie.crt` / `/etc/grafana/certs/monnie.key` に変更し、`deploy_grafana.yml` はそのパスへファイルを配置して `grafana-server` を再起動している。
しかし、Grafana 側の `grafana.ini` / systemd 環境設定に対して `protocol = https`, `cert_file`, `cert_key` をこの新パスへ向ける変更や検証がない。

既存の Grafana 設定が旧パス `/etc/grafana/tls/monnie.internal.crt` / `.key` を参照している場合、今回の playbook は新しい証明書を別ディレクトリに置くだけで、再起動後も旧証明書のままになる。
既存設定が HTTP のままの場合も、証明書更新は成功扱いになるが TLS 化されない。`cert_deployed: true` は `deploy_grafana.yml:51` で無条件に記録されるため、通知上も「renewed and deployed」と見えてしまう。

修正案:

- Grafana 設定を Ansible 管理して `cert_host_config.cert_path` / `key_path` と同期させる。
- もしくは既存の Grafana 設定が参照している確定パスに `cert_path` / `key_path` を戻す。
- restart 後に `grafana-server` の active 確認だけでなく、`openssl s_client` 等で `monnie.internal:3000` が更新後の証明書を返すことを検証するタスクを追加する。

### Medium: cert_renew の現行ポリシーでは monnie が未実装のままになっている

- 対象:
  - `playbooks/cert_renew.yml:73`
  - `playbooks/cert_renew.yml:116`
  - `docs/ai/prompts/cert_renew_policy.md:24`

今回の差分では `cert_renew.yml` に monnie play を追加し、通知サマリーにも `monnie` を含めている。
一方、要求仕様ファイルが参照する現行正本 `docs/ai/prompts/cert_renew_policy.md` では、`monnie | Grafana | Phase 2（未実装）` のまま。

コード上は本番 playbook の対象に入っているのに、運用ポリシー上は未実装扱いのままだと、Semaphore スケジュール対象・Grafana パス・検証手順の判断が分裂する。
monnie を今回から有効化するなら、同じ変更で `cert_renew_policy.md` の対象サービス、証明書パス、検証方針も更新するべき。

### Low: monnie の SAN に IP が入っていない

- 対象:
  - `roles/homelab_cert_renew/defaults/main.yml:56`
  - `roles/homelab_cert_renew/defaults/main.yml:60`

`monnie` は `cn: monnie.internal`、SAN DNS は `monnie.internal` / `monnie` だが、`san_ip: []` のまま。
初回要求仕様の Phase 1 対象ではないため当時の表には monnie がないが、証明書仕様としては「SAN: DNS + IP（hosts_config で定義）」が現行ポリシーにある。過去の monitoring 要件では monnie は `monnie.internal` とされている。

通常アクセスが DNS 名だけなら実害は小さい。ただし IP 直打ちや監視・疎通確認で IP を使う運用がある場合、ブラウザ警告や検証失敗が残る。Phase 2 として正式追加するなら `monnie.internal` を SAN IP に含めるか、DNS のみでよい理由をポリシーに明記したい。

## 確認できたこと

- `roles/homelab_cert_renew/tasks/deploy_grafana.yml` は秘密鍵の slurp / copy に `no_log: true` を付けている。
- `monnie` は inventory 上で `monnie.internal` として定義されており、名前解決方針には沿っている。
- Proxmox の `/tmp` 配置タスクへの `group: root` 追加は妥当。
- `ansible-playbook --syntax-check playbooks/cert_renew.yml` は、`ANSIBLE_LOCAL_TEMP=/tmp/ansible-local TMPDIR=/tmp` 指定時に成功した。

## 補足

最初の構文チェックは Ansible が `/home/yoshi/.ansible/tmp` に一時ディレクトリを作ろうとして、サンドボックスの読み取り専用制約で失敗した。`/tmp` を指定した再実行では構文エラーは出ていない。
