## Findings

### P2: remote check 前に到達不能・失敗したホストが通知では `skipped` と表示される

- 該当箇所:
  - `playbooks/cert_renew.yml:75`
  - `playbooks/cert_renew.yml:76`
  - `playbooks/cert_renew.yml:79`
  - `roles/homelab_cert_renew/tasks/issue.yml:6`
  - `roles/homelab_cert_renew/tasks/issue.yml:22`

今回の修正で、`cert_needs_renewal: true` かつ `cert_renewed` 未設定の場合は `issue FAILED` と表示されるようになり、発行途中失敗と更新不要は分離された。

ただし、対象ホストが `stat` 実行前に unreachable になった場合や、`cert_needs_renewal` を set_fact する前に失敗した場合、`hostvars[h].cert_needs_renewal` は未定義のままになる。現在の通知ロジックは `default(false)` を使うため、未定義は `false` 扱いとなり、通知では `skipped` と表示される。

月次自動実行ではメール通知が主な確認手段になるため、Ansible の実行ログに unreachable が出ていても、メール本文が `skipped` と表示するのは誤認リスクがある。

推奨:

- `cert_needs_renewal is not defined` の場合は `check FAILED` または `unreachable or precheck FAILED` と表示する。
- `skipped` は `cert_needs_renewal is defined` かつ `false` の場合だけに限定する。
- 例:

```jinja2
{% if hostvars[h].cert_needs_renewal is not defined %}
{{ h }}: check FAILED
{% elif not needs_renewal %}
{{ h }}: skipped
{% elif not renewed %}
{{ h }}: issue FAILED
...
{% endif %}
```

### P2: CA cleanup 失敗時に通知 Play が実行されず、失敗がメール通知されない

- 該当箇所:
  - `playbooks/cert_renew.yml:53`
  - `playbooks/cert_renew.yml:63`
  - `roles/homelab_cert_renew/tasks/cleanup.yml:2`
  - `roles/homelab_cert_renew/tasks/cleanup.yml:6`

cleanup Play は `hosts: localhost` になり、`delegate_to: "{{ cert_renew_ca_host }}"` で quory 上の `/run/semaphore-ca/` を削除する構造に変わった。これにより、quory が通常の failed host 状態になっても cleanup を起動できるようになっており、前回 P1 への対応としては改善されている。

一方、この delegated cleanup が失敗すると、Play の実行ホストである `localhost` が failed 状態になる。次の通知 Play も `hosts: localhost` なので、cleanup 失敗後は通知 Play が実行されず、CA cleanup 失敗という最重要の異常がメールで通知されない可能性がある。

要求仕様にはメール通知が含まれており、特に CA 秘密鍵削除失敗は人間が把握すべき状態なので、通知が消えるのは運用上つらい。

推奨:

- cleanup タスクは `register` + `failed_when: false` で結果を保持し、通知本文で `cleanup FAILED` を明示する。
- 通知後に Playbook 全体を失敗扱いにしたい場合は、通知送信後に cleanup 結果を見て `fail` する。
- もしくは cleanup と通知を同じ `localhost` Play 内の `block` / `always` にまとめ、cleanup 成否に関わらず通知処理へ進める。

## Notes

- cleanup Play を `hosts: localhost` にし、`cleanup.yml` 側で `delegate_to: "{{ cert_renew_ca_host }}"` と `become: true` を指定したことで、quory 自身の deploy/restart 失敗で cleanup が丸ごとスキップされる前回 P1 は概ね解消されている。
- 通知ロジックに `cert_needs_renewal` を追加したことで、発行途中失敗を `issue FAILED` と表示できるようになっている。
- Play 2 / Play 3 の `serial: 1`、明示 restart 後の `cert_deployed: true`、PVE の `block` / `always` cleanup は維持されている。
- `ansible-playbook -i inventories/homelab/hosts.yml playbooks/cert_renew.yml --syntax-check` は、`ANSIBLE_LOCAL_TEMP=/tmp/ansible-local` と `ANSIBLE_REMOTE_TEMP=/tmp/ansible-remote` を指定して成功した。

## Overall

前回レビューの主要指摘はよく改善されている。特に CA cleanup の起動条件と、発行途中失敗の通知分離は前進している。

残る論点は、失敗時のメール通知の正確性。自動運用では通知が人間の判断材料になるため、未定義状態を `skipped` に倒さないこと、cleanup 失敗をメールで知らせることを入れてからマージするのがよい。
