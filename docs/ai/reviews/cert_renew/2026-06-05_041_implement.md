## 変更内容の要約

### High 対応: IP SAN を /etc/hosts / DNS から動的解決する

**方針:**
- `defaults/main.yml` には IP を書かない（`san_ip: []` のまま）
- 証明書発行時に CA ホスト（quory）上で `getent hosts <cn>` を実行して IPv4 アドレスを取得する
- 取得した IP を `cert_resolved_san_ip` fact としてテンプレートに渡す
- 名前解決できない場合は `failed_when: rc != 0` で fail させる

**issue.yml に追加した 2 タスク**（Template OpenSSL config の直前に挿入）:

```yaml
- name: Resolve hostname IPv4 address from CA host
  ansible.builtin.command:
    cmd: getent hosts {{ cert_host_config.cn }}
  register: cert_hostname_resolve
  delegate_to: "{{ cert_renew_ca_host }}"
  changed_when: false
  when: cert_needs_renewal
  failed_when: cert_hostname_resolve.rc != 0

- name: Set resolved IP fact
  ansible.builtin.set_fact:
    cert_resolved_san_ip: "{{ cert_hostname_resolve.stdout.split()[0] }}"
  when: cert_needs_renewal
```

- `getent hosts <cn>` は quory の `/etc/hosts` を参照して解決する
- stdout の例: `ansy.internal    ansy.internal   ansy`
- `.split()[0]` で先頭フィールド（IPv4 アドレス）のみを取り出す
- `delegate_to: cert_renew_ca_host` により quory 上で実行される

**openssl-san.cnf.j2 の変更:**

```jinja2
{% for ip in (cert_host_config.san_ip | default([])) + ([cert_resolved_san_ip] if cert_resolved_san_ip is defined else []) %}
IP.{{ loop.index }} = {{ ip }}
{% endfor %}
```

- `cert_host_config.san_ip`（常に `[]`）と `cert_resolved_san_ip`（動的解決値）を結合する
- `cert_resolved_san_ip is defined` ガードにより、fact が未設定の場合は IP SAN なしで動作する
- テンプレートは `when: cert_needs_renewal` のタスクから呼ばれるため、通常は必ず defined になる

**発行される証明書の SAN（例: ansy）:**

```
DNS.1 = ansy.internal
DNS.2 = ansy
IP.1 = ansy.internal
```

IP アドレスはリポジトリ内に存在しないが、証明書には含まれる。

### Medium 対応: Grafana TLS 検証を SAN 確認に強化

変更前:
```yaml
- name: Verify Grafana is serving TLS
  ansible.builtin.shell:
    cmd: "openssl s_client -connect localhost:3000 < /dev/null"
  failed_when: grafana_tls_check.rc != 0
```

変更後:
```yaml
- name: Verify Grafana TLS certificate SAN contains expected hostname
  ansible.builtin.shell:
    cmd: "echo | openssl s_client -connect localhost:3000 -servername {{ cert_host_config.cn }} 2>/dev/null | openssl x509 -noout -ext subjectAltName 2>/dev/null"
  register: grafana_cert_check
  changed_when: false
  failed_when: cert_host_config.cn not in grafana_cert_check.stdout
```

- `-servername` で SNI を指定して接続する
- パイプで `openssl x509 -noout -ext subjectAltName` に渡し、返却証明書の SAN を抽出する
- `cert_host_config.cn`（= `monnie.internal`）が SAN に含まれない場合は fail
- TLS が有効でない場合は openssl の出力が空になり、SAN チェックが fail する
- `cert_host_config.cn` を使用することで、将来別のホストに展開した場合も汎用的に動作する

## 注意点

- `getent hosts` は `/etc/nsswitch.conf` の hosts エントリに従って解決する。quory の設定が `files dns` であれば `/etc/hosts` を優先する。
- IPv6 アドレスのみ解決される環境では `.split()[0]` が IPv6 アドレスになる。現状の homelab は IPv4 のみのため問題なし。
- `openssl s_client` + `openssl x509` パイプは Grafana が完全に起動して TLS を受け付けるまで待機しないため、直前の `wait_for` タスクが重要。
- Grafana TLS 検証の `2>/dev/null` はホームラボ CA が system trust store にないための証明書検証エラーを抑制している。SAN の存在確認は証明書チェーンの検証なしでも行える。
