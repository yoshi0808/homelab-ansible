## P2-1 対応詳細: unreachable ホストを `check FAILED` と表示

### 問題

変更前の通知ロジック:
```jinja2
{% set needs_renewal = hostvars[h].cert_needs_renewal | default(false) | bool %}
{% if not needs_renewal %}
{{ h }}: skipped
```

`cert_needs_renewal | default(false)` は未定義 (`cert_needs_renewal` が設定される前に失敗・unreachable) でも `false` を返す。結果として「更新不要」と「到達不能 / 事前チェック失敗」が区別できず、どちらも `skipped` と表示されていた。

### 対応

`is not defined` テストで未定義を先に判定する。

```jinja2
{% if hostvars[h].cert_needs_renewal is not defined %}
{{ h }}: check FAILED
{% else %}
{% set needs_renewal = hostvars[h].cert_needs_renewal | bool %}
...
{% endif %}
```

通知の状態判定フロー（最終版）:

| 条件 | 表示 |
|---|---|
| `cert_needs_renewal` 未定義 | `check FAILED` (unreachable または stat 前失敗) |
| `cert_needs_renewal` = false | `skipped` (証明書が有効) |
| `cert_needs_renewal` = true、`cert_renewed` 未設定 | `issue FAILED` (発行途中失敗) |
| `cert_needs_renewal` = true、`cert_renewed` = true、`cert_deployed` = true | `renewed and deployed` |
| `cert_needs_renewal` = true、`cert_renewed` = true、`cert_deployed` 未設定 | `renewed but deploy FAILED` |

## P2-2 対応詳細: CA cleanup 失敗時も通知が届く構造に変更

### 問題

変更前の `cleanup.yml`:
```yaml
- name: Remove CA staging directory from tmpfs
  ansible.builtin.file:
    path: "{{ cert_renew_ca_dir }}"
    state: absent
  delegate_to: "{{ cert_renew_ca_host }}"
  become: true
```

cleanup タスクが失敗すると cleanup Play の `localhost` が failed 状態になり、次の通知 Play (`hosts: localhost`) が実行されなかった。CA 秘密鍵削除失敗という最重要の異常がメールで通知されない問題があった。

### 対応

**cleanup.yml**:

```yaml
- name: Remove CA staging directory from tmpfs
  ansible.builtin.file:
    path: "{{ cert_renew_ca_dir }}"
    state: absent
  delegate_to: "{{ cert_renew_ca_host }}"
  become: true
  register: cleanup_result
  ignore_errors: true

- name: Record cleanup failure
  ansible.builtin.set_fact:
    cert_cleanup_failed: true
  when: cleanup_result is failed
```

`ignore_errors: true` により cleanup タスクが失敗しても cleanup Play は正常完了する。`localhost` は failed 状態にならないため、次の通知 Play が確実に実行される。

`cleanup_result is failed` は実際の失敗 (ansible.builtin.file が non-zero rc や module error を返した場合) を検出する。`cert_cleanup_failed: true` が localhost の変数空間に設定され、通知 Play から参照できる。

**cert_renew.yml** (通知 Play):

CA cleanup 状態を通知本文に追加:
```jinja2
CA cleanup: {{ 'FAILED' if cert_cleanup_failed | default(false) | bool else 'OK' }}
```

通知送信後に明示 fail:
```yaml
- name: Fail if CA cleanup failed
  ansible.builtin.fail:
    msg: "CA cleanup FAILED: /run/semaphore-ca/ may still exist on {{ cert_renew_ca_host }}"
  when: cert_cleanup_failed | default(false) | bool
```

`fail` を通知の後に置くことで:
1. メール通知が届く (cleanup 失敗の内容を含む)
2. Playbook が明示的に失敗する (CI / Semaphore でのエラー検知に貢献)

### 動作フロー（cleanup 失敗時）

```
Play 4 cleanup (hosts: localhost):
  - file state: absent → FAILED
  - ignore_errors: true → play は正常継続
  - set_fact cert_cleanup_failed: true
  → localhost は NOT failed → Play 5 が起動

Play 5 notification (hosts: localhost):
  - cert_cleanup_failed = true
  - Build summary → "CA cleanup: FAILED" を含む
  - Send mail → 通知届く
  - Fail if cleanup failed → Playbook が FAILED で終了
```

### 運用上の限界

quory が **unreachable** の場合、`delegate_to: quory` の接続自体が失敗する。`ignore_errors: true` は task failure は吸収するが、unreachable は吸収しない。その場合 `cert_cleanup_failed` は設定されず通知では `CA cleanup: OK` と誤表示になる。

ただし `/run/semaphore-ca/` は tmpfs のため quory の次回再起動で自動消去される。quory が unreachable な状況では CA 鍵へのアクセス自体も不可能であり、運用上の実害は軽微。
