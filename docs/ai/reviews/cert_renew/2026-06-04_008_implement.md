## P1 対応詳細: cleanup を quory の failed 状態に依存しない構造に変更

### 問題

変更前の cleanup Play は `hosts: quory` だった。Play 2 (`semaphore_servers`) には quory 自身が含まれており、quory での semaphore restart が失敗すると quory が failed host 扱いになる。Ansible はその後の `hosts: quory` Play を quory に対して実行しないため、`/run/semaphore-ca/ca.key` が残る可能性があった。

要求仕様の「処理後に必ず削除」に直接関わる。

### 対応

**cert_renew.yml**:

```yaml
# Before
- name: cert_renew - cleanup CA key from quory
  hosts: quory
  gather_facts: false
  become: true

# After
- name: cert_renew - cleanup CA key from quory
  hosts: localhost
  gather_facts: false
  become: false
```

**cleanup.yml**:

```yaml
# Before
- name: Remove CA staging directory from tmpfs
  ansible.builtin.file:
    path: "{{ cert_renew_ca_dir }}"
    state: absent

# After
- name: Remove CA staging directory from tmpfs
  ansible.builtin.file:
    path: "{{ cert_renew_ca_dir }}"
    state: absent
  delegate_to: "{{ cert_renew_ca_host }}"
  become: true
```

`hosts: localhost` にすることで cleanup Play は常に localhost で起動する。localhost は他 Play の failed host 状態の影響を受けない。

`delegate_to: "{{ cert_renew_ca_host }}"` (= quory) で実際の削除は quory 上で行われる。`become: true` を task レベルに移動し、quory 上での `/run/semaphore-ca/` 削除に root 権限を使う。

### 運用上の限界

quory が **unreachable** (SSH 接続不可) の場合、`delegate_to: quory` も失敗する。この場合は cleanup も実行できない。ただし quory が unreachable な状況では CA 秘密鍵ファイルへのアクセス自体も不可能であり、セキュリティ上の実害はない。また `/run/semaphore-ca/` は tmpfs のため、quory の次回再起動で自動的に消える。

## P2 対応詳細: 通知で `issue FAILED` を `skipped` と分離

### 問題

`cert_renewed` は `issue.yml` の最後 (Record renewal status) で設定される。openssl 操作 (鍵生成・CSR生成・CA署名) が途中で失敗すると `cert_renewed` は未設定のまま残る。

変更前の通知ロジック:
```jinja2
{% if not renewed %}
{{ h }}: skipped        ← 「更新不要」と「発行失敗」が区別できない
```

`cert_renewed | default(false)` が false になるのは:
- 証明書が有効で更新不要 (cert_needs_renewal = false → cert_renewed = false)
- 証明書が更新対象だったが発行途中で失敗 (cert_renewed 未設定)

どちらも `skipped` と表示されていた。

### 対応

`cert_needs_renewal` は `issue.yml` の早い段階 (stat + 残日数計算の後) で set_fact される。openssl 操作の前に設定されるため、操作途中で失敗しても残る。

```jinja2
{% set needs_renewal = hostvars[h].cert_needs_renewal | default(false) | bool %}
{% set renewed = hostvars[h].cert_renewed | default(false) | bool %}
{% set deployed = hostvars[h].cert_deployed | default(false) | bool %}
{% if not needs_renewal %}
{{ h }}: skipped
{% elif not renewed %}
{{ h }}: issue FAILED
{% elif deployed %}
{{ h }}: renewed and deployed
{% else %}
{{ h }}: renewed but deploy FAILED
{% endif %}
```

状態の判定フロー:

| cert_needs_renewal | cert_renewed | cert_deployed | 表示 |
|---|---|---|---|
| false | false | - | `skipped` (証明書が有効) |
| true | false/未設定 | - | `issue FAILED` (発行失敗) |
| true | true | true | `renewed and deployed` (全成功) |
| true | true | false/未設定 | `renewed but deploy FAILED` (deploy/restart失敗) |

### 注意点

`cert_needs_renewal` も issue.yml の中で設定されるため、ホストが stat 実行前に unreachable になった場合は未設定になる。その場合は `default(false)` により `skipped` と表示されるが、Ansible のプレイ出力でホストが unreachable/failed と明示されるため運用上の問題は軽微。
