## Findings

### P2: cleanup の delegated unreachable は `ignore_errors` では吸収されず、通知まで進まない可能性がある

- 該当箇所:
  - `roles/homelab_cert_renew/tasks/cleanup.yml:2`
  - `roles/homelab_cert_renew/tasks/cleanup.yml:6`
  - `roles/homelab_cert_renew/tasks/cleanup.yml:9`
  - `playbooks/cert_renew.yml:63`
  - `playbooks/cert_renew.yml:93`

今回の修正で cleanup task に `register: cleanup_result` と `ignore_errors: true` が追加され、通常の module failure であれば `cert_cleanup_failed: true` を立てて通知まで進めるようになった。この方向は前回指摘への対応としてよい。

ただし、`ignore_errors` は Ansible の unreachable を吸収しない。cleanup は `hosts: localhost` から `delegate_to: "{{ cert_renew_ca_host }}"` で quory に接続するため、quory への SSH が unreachable になった場合は delegated task が unreachable で止まり、`cleanup_result` / `cert_cleanup_failed` が期待通り設定されない可能性がある。

実装メモでは「quory が unreachable の場合は `cert_cleanup_failed` は設定されず通知では `CA cleanup: OK` と誤表示」と整理されているが、実際には `localhost -> quory` の delegated unreachable により通知 Play 自体へ進めない可能性もある。どちらにせよ、CA cleanup が未確認の状態を `OK` または無通知に倒すのは、要求仕様の「処理後に必ず削除」「メール通知」と相性が悪い。

推奨:

- cleanup task に `ignore_unreachable: true` も追加し、unreachable も通知へ反映できるようにする。
- `cleanup_result is unreachable` または `cleanup_result.unreachable | default(false)` を見て、`cert_cleanup_failed: true` を設定する。
- 通知本文は `CA cleanup: OK / FAILED / UNREACHABLE` のように、削除失敗と到達不能を分けると運用判断しやすい。
- cleanup 後の `fail` は、通常 failure と unreachable の両方を失敗扱いにする。

## Notes

- 通知ロジックで `hostvars[h].cert_needs_renewal is not defined` を先に判定するようになっており、remote check 前に失敗したホストを `skipped` と誤表示する前回 P2 は改善されている。
- cleanup の通常 failure については、`ignore_errors: true` + `register` + `cert_cleanup_failed` + 通知後 `fail` の流れになっており、前回 P2 の「cleanup 失敗で通知 Play が動かない」問題は一部改善されている。
- Play 2 / Play 3 の `serial: 1`、明示 restart 後の `cert_deployed: true`、PVE の `/tmp` 一時ファイル cleanup は維持されている。
- `ansible-playbook -i inventories/homelab/hosts.yml playbooks/cert_renew.yml --syntax-check` は、`ANSIBLE_LOCAL_TEMP=/tmp/ansible-local` と `ANSIBLE_REMOTE_TEMP=/tmp/ansible-remote` を指定して成功した。

## Overall

前回レビューの指摘には順当に対応されており、通常の失敗経路では通知の正確性が上がっている。

残る主な論点は、cleanup の delegated unreachable を失敗状態としてメール通知へ載せること。ここまで入れば、証明書更新の自動運用としてかなり安心して回せる状態になる。
