# Review: cert_renew git diff

日付: 2026-06-05
対象: cert_renew 関連の作業ツリー差分

## 指摘事項

### High: `cert_renew_quory.yml` が未追跡のため、このままでは quory の証明書更新経路が消える

- 該当:
  - `playbooks/cert_renew.yml:5-6`
  - `playbooks/cert_renew.yml:55-71`
  - `playbooks/cert_renew_quory.yml:1`
- 状況:
  - 既存の `cert_renew.yml` から quory の semaphore 証明書更新 play が削除されている。
  - 代替として `playbooks/cert_renew_quory.yml` が作成されているが、`git status --short` では未追跡。
- 影響:
  - 追跡済み差分だけを commit / deploy すると、quory の証明書更新処理がリポジトリに入らない。
  - Phase 1 要求の対象である quory の証明書が自動更新されなくなる。
- 対応案:
  - `playbooks/cert_renew_quory.yml` を Git 管理対象に追加する。
  - 併せて systemd timer 側の実行入口がこの playbook を参照することも別途確認する。

### High: monnie が本番 playbook に追加されているが、配布処理が未実装のため毎回「更新したが配布失敗」扱いになる

- 該当:
  - `playbooks/cert_renew.yml:5`
  - `playbooks/cert_renew.yml:73-84`
  - `playbooks/cert_renew.yml:112-126`
  - `roles/homelab_cert_renew/defaults/main.yml:52-61`
- 状況:
  - 要求仕様では monnie/Grafana は Phase 2 の初回除外スコープ。
  - 今回の差分では monnie を `cert_renew.yml` の対象と通知サマリーに追加している。
  - しかし `deploy_grafana` は未実装で、issue のみ実行して cleanup される。
- 影響:
  - monnie の証明書ファイルが存在しない、または更新閾値以下の場合、証明書は発行されても配布されない。
  - 発行済み成果物は cleanup で削除され、次回も同じ状態になりやすい。
  - 通知は `monnie: renewed but deploy FAILED` となる想定で、既知の未実装を定期実行の失敗として混入させる。
- 対応案:
  - Phase 2 の deploy 実装が完了するまで、monnie を `cert_renew.yml` と通知サマリーから外す。
  - もしくは同じ変更で `deploy_grafana.yml` まで実装し、実際に配布・Grafana reload/restart まで完結させる。

### Medium: quory 用 playbook の cleanup 失敗検知はあるが、通知経路がない

- 該当:
  - `playbooks/cert_renew_quory.yml:51-66`
- 状況:
  - 実装メモ通り `cert_renew_quory.yml` には通知ブロックがない。
  - cleanup 失敗時は fail するため systemd 側では検知可能だが、既存のメール通知運用とは別経路になる。
- 影響:
  - systemd timer の失敗監視や journal 確認が未整備だと、quory の証明書更新失敗や CA 一時ファイル削除失敗に気づきにくい。
- 対応案:
  - systemd timer 実装時に `OnFailure=`、journal 監視、またはメール通知 play を明示する。
  - 仕様として「quory 用 playbook はメール通知なしで systemd の失敗検知に寄せる」なら、その運用前提をドキュメントに残す。
