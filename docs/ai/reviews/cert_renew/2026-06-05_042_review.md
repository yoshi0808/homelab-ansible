# Review: cert_renew IP SAN 動的解決 + Grafana TLS 検証強化

日付: 2026-06-05

## 結論

大きな方向性は良い。`defaults/main.yml` に内部 IP を直書きせず、証明書発行時に CA ホスト側で名前解決して IP SAN を生成する実装になっている。これは公開リポジトリ化の意図と整合している。
ただし、現行ポリシー文書の SAN 説明が古いままなのと、`getent hosts` の結果から IPv4 を厳密に選別していない点は修正推奨。

## Findings

### Medium: cert_renew_policy.md の SAN 説明が動的名前解決方式に更新されていない

- 対象:
  - `docs/ai/prompts/cert_renew_policy.md:122`
  - `roles/homelab_cert_renew/tasks/issue.yml:36`
  - `roles/homelab_cert_renew/templates/openssl-san.cnf.j2:23`

今回の実装では、`issue.yml` が CA ホスト側で `getent hosts {{ cert_host_config.cn }}` を実行し、その先頭フィールドを `cert_resolved_san_ip` としてテンプレートへ渡している。
一方、現行ポリシーの証明書仕様は `SAN | DNS + IP（hosts_config で定義）` のまま。

公開リポジトリ化のために IP を `defaults/main.yml` / `hosts_config` に直書きしない方針へ変わっているので、正本側も「IP SAN は quory の `/etc/hosts` または DNS による名前解決結果から発行時に動的生成する」と明記した方がよい。
このままだと、次回レビューや実装で `san_ip: []` が欠落に見えたり、逆に IP 直書きが正しいと誤解される。

### Low: `getent hosts` の先頭フィールドをそのまま使っており、IPv4 解決の保証が弱い

- 対象:
  - `roles/homelab_cert_renew/tasks/issue.yml:36`
  - `roles/homelab_cert_renew/tasks/issue.yml:47`

タスク名は `Resolve hostname IPv4 address from CA host` だが、実際には `getent hosts <cn>` の `stdout.split()[0]` をそのまま採用している。
quory の `/etc/hosts` が IPv4 のみなら期待どおり動くが、将来 IPv6 行や DNS 応答が先に返る環境では、IPv4 ではない値が `cert_resolved_san_ip` に入る可能性がある。

修正案:

- IPv4 に限定するなら `getent ahostsv4 {{ cert_host_config.cn }}` を使う。
- あるいは `getent hosts` の出力から IPv4 形式だけを抽出し、見つからなければ fail させる。
- IPv6 SAN も許容する方針なら、タスク名・実装メモ・ポリシーの表現を「IP address」に寄せる。

### Low: Grafana の検証は DNS SAN の確認までで、動的生成した IP SAN までは見ていない

- 対象:
  - `roles/homelab_cert_renew/tasks/deploy_grafana.yml:73`

`openssl s_client ... | openssl x509 -noout -ext subjectAltName` の出力に `cert_host_config.cn` が含まれることを確認しているため、Grafana が `monnie.internal` の証明書を返しているかは以前より確認しやすくなった。
ただし、今回の主題である動的 IP SAN が実際に返却証明書に含まれているかまでは検証していない。

必須ではないが、IP SAN 生成を強く保証したいなら、発行後の CA ホスト上の証明書、または Grafana 返却証明書の `subjectAltName` に `IP Address:<resolved_ip>` が含まれることまで確認すると、今回の実装意図に対する検証としてより強くなる。

## 確認できたこと

- `defaults/main.yml` の `san_ip: []` は、公開リポジトリ化を前提に IP を直書きしない意図と整合している。
- `issue.yml` は `delegate_to: "{{ cert_renew_ca_host }}"` で名前解決を CA ホスト側に委譲している。`cert_renew.yml` では `cert_renew_ca_host: localhost` なので、quory 上で実行される想定に合っている。
- `openssl-san.cnf.j2` は `cert_host_config.san_ip` と `cert_resolved_san_ip` を結合して IP SAN を出力する形になっている。
- Grafana TLS 検証は、単なる TLS handshake から `subjectAltName` に期待 hostname が含まれる確認へ強化されている。
- `ANSIBLE_LOCAL_TEMP=/tmp/ansible-local TMPDIR=/tmp ansible-playbook --syntax-check playbooks/cert_renew.yml` は成功した。

## 補足

前回レビューの High finding「IP SAN が生成されない」は、実装としては概ね解消されている。残る主な作業は、動的名前解決方式を正本ドキュメントへ反映することと、IPv4 を厳密に扱うかどうかの方針整理。
