## Findings

### P1: CA秘密鍵の展開先が「quory」ではなく実行ノード依存になっている

- 該当箇所:
  - `playbooks/cert_renew.yml:9`
  - `playbooks/cert_renew.yml:63`
  - `roles/homelab_cert_renew/tasks/issue.yml:36`
  - `roles/homelab_cert_renew/tasks/issue.yml:44`
  - `roles/homelab_cert_renew/tasks/issue.yml:72`

要求仕様では「CA秘密鍵はansyに置かない（quoryのみ）」「quory上の `/run/semaphore-ca/` に一時展開」とされている。

しかし実装は `hosts: localhost` と `delegate_to: localhost` を使っているため、CA証明書・CA秘密鍵・発行済み秘密鍵・CSR・証明書は、Playbookを実行したノードの `/run/semaphore-ca/` に作成される。

Semaphoreをquory上で実行する運用に限定すれば結果的にquoryになるが、ansyから手動実行した場合はCA秘密鍵がansyに展開され、要求仕様の制約に反する。

推奨:

- CA処理用の実行ホストを明示的に `quory` に固定する。
- 例: `cert_renew_ca_host: quory` のような変数を用意し、prepare / issue / cleanup の委譲先を `{{ cert_renew_ca_host }}` にする。
- その場合、`copy src:` でcontrollerローカルファイルを読む箇所も、quory上の生成物を各対象ホストへ配る設計に合わせて調整する必要がある。

### P2: 失敗時にも「completed」通知が送られ、結果が成功に見える可能性がある

- 該当箇所:
  - `playbooks/cert_renew.yml:73`
  - `playbooks/cert_renew.yml:89`
  - `roles/homelab_cert_renew/tasks/issue.yml:89`

最後の通知Playは常に `Certificate renewal completed` を送る構造になっている。

一方、`cert_renewed` は `issue.yml` の最後で「発行が必要だったか」を記録しており、deploy成功やサービス再起動成功を表していない。

そのため、例えば以下のようなケースで通知が実態とずれる。

- 証明書発行は成功したが、対象ホストへのコピーに失敗した。
- `pvenode cert set` に失敗した。
- handler のサービス再起動に失敗した。
- 一部ホストが unreachable だった。

この場合でも最終通知では `renewed` または `skipped` のみが出るため、実行者が成功完了と誤認する恐れがある。

推奨:

- 通知文面を単純な `completed` にせず、失敗ホスト・失敗タスクがある場合は `FAILED` / `PARTIAL` を明示する。
- `cert_renewed` とは別に `cert_deployed` / `cert_apply_succeeded` のような状態を記録する。
- 失敗時にも cleanup と通知をしたい場合は、各Playを `block` / `rescue` / `always` で設計し、通知に失敗状態を反映する。

### P2: PVEノード上の一時秘密鍵が失敗経路で残る可能性がある

- 該当箇所:
  - `roles/homelab_cert_renew/tasks/deploy_proxmox.yml:10`
  - `roles/homelab_cert_renew/tasks/deploy_proxmox.yml:19`
  - `roles/homelab_cert_renew/tasks/deploy_proxmox.yml:33`
  - `roles/homelab_cert_renew/tasks/deploy_proxmox.yml:40`

要求仕様では「`/tmp` の一時ファイルは処理後に削除」とされている。

現在の削除タスクは通常経路の最後にあるため、以下のいずれかが失敗すると `/tmp/pve_key.pem` や `/tmp/pve_key_pkcs8.pem` が残る可能性がある。

- PKCS8変換
- `pvenode cert set`
- その前後のAnsibleタスク

特に秘密鍵ファイルは `mode: "0600"` ではあるが、要件上は残さない設計が必要。

推奨:

- PVE deploy処理を `block` / `always` にして、一時ファイル削除を `always` 側へ移す。
- 可能ならホストごとに一意な一時ファイル名を使い、並列実行や途中再実行時の衝突も避ける。

## Notes

- `ansible-playbook -i inventories/homelab/hosts.yml playbooks/cert_renew.yml --syntax-check` は、`ANSIBLE_LOCAL_TEMP=/tmp/ansible-local` を指定したうえで成功した。
- `.gitignore` に `*.crt` / `*.key` / `*.csr` / `*.pem` を追加している点は、証明書・秘密鍵をGit管理しない制約に沿っている。
- `no_log: true` はCA秘密鍵の書き込み、秘密鍵生成、署名、秘密鍵配布に付いており、主な秘密情報操作は隠蔽されている。
- `inventories/homelab/group_vars/semaphore_servers.yml` の `ansible_user: ann` / `ansible_ssh_private_key_file: ~/.ssh/id_ann` は `core.md` のAnsible管理ユーザー方針と整合している。

## Overall

構文上は通っており、Phase 1対象ホスト・SAN定義・EC secp384r1・45日証明書・15日閾値・Semaphore/PVEの反映処理という主要要件は概ね実装されている。

ただし、CA秘密鍵の配置先が実行ノード依存になっている点は要求仕様の安全制約に直接関わるため、マージ前に修正した方がよい。
