# 要求仕様: cert_renew role の Home-TLS-CA(中間CA)対応

作成日: 2026-06-12
版: 本文書は要求仕様（requirement）である

---

## 目的

TLS証明書の署名CAを、ルートCA(Home-RADIUS-CA)直接署名から
中間CA(Home-TLS-CA)署名に移行する。
ルートCA秘密鍵はオフライン保管に移行済みであり、quory上に存在しない。
このため現行実装は次回実行時に必ず失敗する。次回 systemd timer 発火
(2026-07-01頃)前に本改修を完了させる。

## 対象ホスト / 対象グループ

cert_renew Policy v1.0 と同一(ansy / quory / pve1 / pve2 / monnie)。
対象サービス・Playbook分離構成(cert_renew.yml / cert_renew_quory.yml)も変更しない。

## 作成・更新対象ファイル

- cert_renew role(roles/homelab_cert_renew/ 配下の該当ファイル)
- cert_renew Policy ドキュメント(v1.0 → v2.0 へ改版)
- 必要に応じて inventories の関連変数

## 実施項目

1. CA参照パスの変更
   旧: /home/yoshi/.cert/radius_ca.crt / radius_ca.key
   新: /home/yoshi/.cert/ca/home_tls_ca.crt / home_tls_ca.key
   - /run/semaphore-ca/(tmpfs)への一時展開・cleanup・mode 0600検証の
     既存方式は維持する
2. 配布証明書のフルチェーン化
   - 中間CA署名への移行に伴い、リーフ証明書単体の配布では
     クライアント側でチェーン検証が失敗する
   - 配布物を「リーフ証明書 + home_tls_ca.crt を連結したフルチェーン」に
     変更する(全対象サービス: Semaphore / Grafana / pveproxy)
3. 中間CA自体の有効期限監視
   - Playbook実行時に home_tls_ca.crt の残存日数を確認し、
     90日未満の場合は既存の通知経路で警告する
   - 中間CAは有効期間10年であり、失効前の再発行が唯一の能動的更新イベント
4. ドキュメント更新
   - cert_renew Policy を v2.0 に改版し、CA構成変更(中間CA・ルートの
     オフライン保管・復旧手順)を反映する

## 確認項目

- ansy実行時に ansy 自身の証明書配布が除外される既存挙動が、
  改修後も維持されること(現行実装の該当ロジックを特定し報告すること)
- 発行された証明書のチェーン検証が通ること
  (openssl verify -CAfile <ルートCA証明書> -untrusted <中間CA証明書>)
- 各サービスがフルチェーン提示すること(openssl s_client での確認)

## 制約

- 証明書仕様は変更しない(有効期間45日、更新条件 残15日以下 or
  force_renew、EC secp384r1、SAN動的取得)
- 通知方式(Slack通知)は変更しない
- 中間CA秘密鍵の中身を表示・コピー・コミットしない(core.md 禁止事項に準拠)
- CA配置の前提: quory および ansy の /home/yoshi/.cert/ca/ に
  home_tls_ca.{crt,key} を配置済み(手動配置、本roleの管轄外)

## 初回実装で含める範囲

- 上記実施項目 1〜4

## 初回実装では除外する範囲

- CloudKey の証明書管理(内蔵 Let's Encrypt で運用継続)
- authy の EAP-TLS クライアント/サーバー証明書(ルートCA直下30年、別管理)
- 通知の Slack 移行(将来検討)
- Semaphore UI スケジュールへの移行(core.md §18 の将来方針に従う)
