## Findings

### P1: quory 上の CA 署名処理がホスト並列で走り、serial file が競合する可能性がある

- 該当箇所:
  - `playbooks/cert_renew.yml:19`
  - `playbooks/cert_renew.yml:41`
  - `roles/homelab_cert_renew/tasks/issue.yml:36`
  - `roles/homelab_cert_renew/tasks/issue.yml:72`

`issue.yml` の OpenSSL 処理は `delegate_to: "{{ cert_renew_ca_host }}"` により quory 上で実行されるようになった。これは「CA秘密鍵はquoryのみ」という前回指摘への対応として正しい方向。

ただし、Play 2 は `semaphore_servers` の `ansy` / `quory`、Play 3 は `proxmox` の `pve1` / `pve2` を通常の Ansible 並列実行で処理する。つまり、複数ホスト分の `openssl x509 -CAcreateserial` が同じ quory 上の同じ CA ディレクトリに対して同時に走る可能性がある。

`-CAcreateserial` は同じ CA serial file を共有するため、並列実行時に serial の競合、重複、破損、または発行失敗が起きうる。月次自動更新で全ホストが同時に更新対象になる設計なので、通常運用で踏みやすい。

推奨:

- 証明書発行部分を直列化する。
- 例: 対象Playに `serial: 1` を付ける、または CA 署名タスクに `throttle: 1` を付ける。
- もしくは `-set_serial` でホストごとに衝突しない serial を明示生成する。

### P2: `cert_deployed` がサービス再起動成功を表していないため、通知が成功を過大報告する

- 該当箇所:
  - `roles/homelab_cert_renew/tasks/deploy_semaphore.yml:47`
  - `roles/homelab_cert_renew/tasks/deploy_proxmox.yml:64`
  - `playbooks/cert_renew.yml:90`
  - `playbooks/cert_renew.yml:104`

`cert_deployed: true` は証明書・秘密鍵の配置、または `pvenode cert set` 直後に設定される。一方、実際の反映に必要な `restart semaphore` / `restart pveproxy` は handler で後から実行される。

handler のサービス再起動に失敗した場合でも、すでに `cert_deployed: true` が立っているため、最終通知は `renewed and deployed` と表示しうる。実装メモではこの状態を「発行・配布・再起動まで完了」と説明しているが、現在の fact は再起動成功を含んでいない。

要求仕様の反映方法は `systemctl restart semaphore` / `systemctl restart pveproxy` まで含むため、通知が「反映成功」を示すなら handler 成功後に状態を記録する必要がある。

推奨:

- handler を使わず、deploy タスク内で `systemd` restart を明示実行し、その後に `cert_deployed` または `cert_applied` を設定する。
- handler を維持する場合は、通知文面を `renewed and deploy tasks completed` 程度に弱め、再起動成功までは保証しないことを表す。
- 失敗通知を正確に出すなら、`cert_restart_succeeded` のような別 fact を handler 後に記録できる構造にする。

## Notes

- `cert_renew_ca_host: quory` の追加、prepare / cleanup の `hosts: quory` 化、`delegate_to: "{{ cert_renew_ca_host }}"` 化により、CA秘密鍵を ansy に展開しないという前回P1指摘は概ね解消されている。
- quory 上の生成物を `copy src:` で直接参照せず、`slurp delegate_to: "{{ cert_renew_ca_host }}"` で読むようにした点は、ansy から実行するケースにも対応できている。
- PVE の一時ファイル削除は `block` / `always` に移され、失敗経路でも `/tmp/pve_*_{{ inventory_hostname }}.pem` を削除する構造になっている。
- `ansible-playbook -i inventories/homelab/hosts.yml playbooks/cert_renew.yml --syntax-check` は、`ANSIBLE_LOCAL_TEMP=/tmp/ansible-local` と `ANSIBLE_REMOTE_TEMP=/tmp/ansible-remote` を指定して成功した。

## Overall

前回レビューの主要な安全指摘にはかなり手が入っており、CA秘密鍵の配置先固定と PVE 一時秘密鍵 cleanup は改善されている。

一方で、quory に集約した CA 署名処理が並列実行されることで serial file 競合のリスクが新たに目立つ。全ホスト更新が普通に起きる運用なので、マージ前に発行処理の直列化を入れるのがよい。
