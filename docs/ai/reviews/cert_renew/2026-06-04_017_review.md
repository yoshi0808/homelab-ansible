# Review: cert_renew quory local execution 対応

対象:

- `playbooks/cert_renew.yml`
- `roles/homelab_cert_renew/`

参照:

- `docs/ai/prompts/core.md`
- `docs/ai/reviews/cert_renew/2026-06-03_001_requirement.md`
- `docs/ai/reviews/cert_renew/2026-06-04_016_implement.md`
- `docs/ai/prompts/ubuntu_vm_patch_policy.md`

## Findings

### P1: quory 以外からの実行を防ぐガードがなく、ansy 実行時に CA 秘密鍵が ansy へ展開される

- 該当箇所:
  - `playbooks/cert_renew.yml:9`
  - `playbooks/cert_renew.yml:13`
  - `playbooks/cert_renew.yml:17`
  - `playbooks/cert_renew.yml:28`
  - `playbooks/cert_renew.yml:64`
  - `playbooks/cert_renew.yml:77`

今回の修正で、ansy / quory / proxmox / cleanup の各 play に `cert_renew_ca_host: localhost` が揃い、quory から実行した場合の SSH 自己接続は避けられるようになった。前回 P1 の解消として方向性はよい。

ただし、この設計は「Playbook を quory からのみ実行する」ことに強く依存している。`connection: local` と `cert_renew_ca_host: localhost` は Ansible controller のローカルを指すため、誤って ansy から実行した場合、prepare CA play は inventory 上の `quory` ではなく ansy ローカルで実行され、`/run/semaphore-ca/ca.key` も ansy に展開される。

要求仕様では「CA秘密鍵はansyに置かない（quoryのみ）」が明示されているため、コメントだけで運用制約を表すのでは不十分。安全制約として実行時に fail させるべき。

推奨:

- playbook 冒頭に controller が quory であることを確認する preflight play を追加する。
- 例: `hosts: localhost`, `connection: local`, `gather_facts: true` で `ansible_hostname == 'quory'` を assert する。
- facts を使わないなら `lookup('pipe', 'hostname -s')` の結果を確認してもよい。
- 失敗メッセージには「この playbook は quory / Semaphore 上でのみ実行する」と明記する。

例:

```yaml
- name: cert_renew - verify execution host
  hosts: localhost
  gather_facts: true
  connection: local
  tasks:
    - name: Ensure this playbook is running on quory
      ansible.builtin.assert:
        that:
          - ansible_hostname == 'quory'
        fail_msg: "cert_renew.yml must be run on quory only."
```

### P3: cleanup 失敗後の fail メッセージで `cert_renew_ca_host` が未定義になる可能性がある

- 該当箇所:
  - `playbooks/cert_renew.yml:77`
  - `playbooks/cert_renew.yml:89`
  - `playbooks/cert_renew.yml:138`

cleanup play には `vars: cert_renew_ca_host: localhost` があるが、通知 play には同じ変数が定義されていない。`cert_cleanup_status is defined` の場合、通知送信後の fail タスクは `msg` 内で `{{ cert_renew_ca_host }}` を参照する。

play 変数は次の play に引き継がれないため、cleanup 失敗時に意図したメッセージではなく、未定義変数による失敗メッセージになる可能性がある。通知自体は送信後なので実害は小さいが、Semaphore / CLI 上の失敗理由が読みにくくなる。

推奨:

- 通知 play にも `vars: cert_renew_ca_host: localhost` を追加する。
- または fail message を `localhost` 固定にする。

## Notes

- ansy play と proxmox play に `cert_renew_ca_host: localhost` が追加され、前回指摘した `delegate_to: quory` による SSH 自己接続は解消されている。
- prepare play と quory 自身の deploy play は `connection: local` になっており、quory 上で実行した場合は意図通りローカル実行になる。
- proxmox play の `serial: 1` は維持されており、CA serial file 競合対策は残っている。
- cleanup play も `cert_renew_ca_host: localhost` で、quory 上実行前提では `/run/semaphore-ca/` をローカル削除する形になっている。
- `ansible-playbook -i inventories/homelab/hosts.yml playbooks/cert_renew.yml --syntax-check` は、`ANSIBLE_LOCAL_TEMP=/tmp/ansible-local` と `ANSIBLE_REMOTE_TEMP=/tmp/ansible-remote` を指定して成功した。

## Overall

quory から実行する限り、前回の SSH 自己接続問題は解消されている。

ただし、CA 秘密鍵の配置先に関わる安全制約なので、「quory からのみ実行」はコメントではなく playbook の preflight で強制した方がよい。ここを入れれば、設計意図と要求仕様の両方にかなり合った状態になる。
