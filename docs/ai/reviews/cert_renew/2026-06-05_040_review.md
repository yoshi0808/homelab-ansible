# Review: cert_renew monnie (Grafana) 追加実装レビュー指摘対応

日付: 2026-06-05

## 結論

修正必須。`deploy_grafana.yml` に Grafana TLS 設定と疎通検証が追加され、`cert_renew_policy.md` の monnie 行も更新されている。
ただし、公開リポジトリ化を前提に IP アドレスを defaults に直書きしない方針自体は妥当だが、証明書発行時に `/etc/hosts` などの名前解決結果から IP SAN を生成する処理が実装されていない。このままだと次回発行分は IP SAN なしの証明書になる。

## Findings

### High: IP を直書きしない方針に対する代替の名前解決処理がなく、IP SAN が生成されない

- 対象:
  - `roles/homelab_cert_renew/tasks/issue.yml:34`
  - `roles/homelab_cert_renew/defaults/main.yml:20`
  - `roles/homelab_cert_renew/defaults/main.yml:30`
  - `roles/homelab_cert_renew/defaults/main.yml:38`
  - `roles/homelab_cert_renew/defaults/main.yml:46`
  - `roles/homelab_cert_renew/defaults/main.yml:56`
  - `roles/homelab_cert_renew/templates/openssl-san.cnf.j2:23`

今回の差分で `ansy`, `quory`, `pve1`, `pve2`, `monnie` の `san_ip` はすべて `[]` になっている。
このリポジトリを将来パブリックリポジトリにする前提では、内部 IP を `defaults/main.yml` に直接書かない判断は妥当。

しかし、`issue.yml` には `cert_host_config.cn` / `san_dns` / inventory の `ansible_host` を `getent hosts` 等で解決し、その結果を一時的な IP SAN リストとして `openssl-san.cnf.j2` に渡す処理がない。テンプレート側も `cert_host_config.san_ip` だけを見て `IP.N = ...` を生成しているため、現状では IP SAN は一切出力されない。

その結果、次回発行分から IP SAN が消える。IP アドレスで Proxmox / Semaphore / Grafana にアクセスした場合、ブラウザ警告や証明書検証失敗が再発する。

修正案:

- `defaults/main.yml` には IP を直書きしない。
- `issue.yml` で CA ホスト、つまり quory の `/etc/hosts` / DNS を使って `ansy.internal`, `quory.internal`, `pve1.internal`, `pve2.internal`, `monnie.internal` を解決する。
- 解決結果を `resolved_san_ip` のような一時 fact にし、`openssl-san.cnf.j2` では `cert_host_config.san_ip | default([]) + resolved_san_ip | default([])` から `IP.N` を生成する。
- 名前解決できない場合は IP SAN なしで黙って発行せず、fail させる。証明書警告解消が目的の playbook なので、ここは失敗として扱う方が安全。

### Medium: 実装メモの「monnie の san_ip に monnie.internal を追加」は公開 repo 方針とずれており、実体にも反映されていない

- 対象:
  - `roles/homelab_cert_renew/defaults/main.yml:56`
  - `docs/ai/reviews/cert_renew/2026-06-05_039_implement.md:101`

実装メモでは `monnie` の `san_ip` に `monnie.internal` を追加したと説明しているが、実ファイルでは `san_ip: []` のまま。
さらに、公開リポジトリ化を前提にするなら、`monnie.internal` を defaults に直接追加する対応方針そのものが望ましくない。

Phase 2 として monnie/Grafana を正式に `cert_renew.yml` 対象へ入れるなら、monnie も他ホストと同じく、証明書発行時に quory 側の名前解決結果から IP SAN を組み立てる方式に揃えるべき。

### Medium: Grafana の TLS 検証が「期待ホスト名の新証明書」までは確認していない

- 対象:
  - `roles/homelab_cert_renew/tasks/deploy_grafana.yml:73`

追加された検証は `openssl s_client -connect localhost:3000 < /dev/null` の終了コードのみを見ている。
これは「Grafana が TLS で応答する」確認にはなるが、`monnie.internal` 用の期待証明書が返っているか、SAN が期待値か、今回配置した証明書へ切り替わったかまでは確認しない。

たとえば `grafana.ini` に重複した `cert_file` / `cert_key` があり、Grafana が別の既存証明書を読み続けても、TLS ハンドシェイク自体は成功しうる。前回レビューの「更新後の証明書を返すことを検証する」という観点ではまだ弱い。

修正案:

- `openssl s_client -connect localhost:3000 -servername monnie.internal -verify_hostname monnie.internal` を使う。
- 可能なら `-showcerts` や `openssl x509 -noout -subject -issuer -ext subjectAltName` で、返却証明書の SAN に `DNS:monnie.internal` と必要な IP SAN が入っていることを確認する。
- `cert_deployed: true` はこの検証後にのみ set する現在の順序を維持する。

## 確認できたこと

- `docs/ai/prompts/cert_renew_policy.md` の対象サービス表は、monnie を `cert_renew.yml` 対象として更新されている。
- `deploy_grafana.yml` は証明書・秘密鍵配置後に `grafana.ini` の `protocol`, `cert_file`, `cert_key` を設定し、`grafana-server` を再起動する流れになっている。
- 秘密鍵の slurp / copy には `no_log: true` が付いている。
- `ANSIBLE_LOCAL_TEMP=/tmp/ansible-local TMPDIR=/tmp ansible-playbook --syntax-check playbooks/cert_renew.yml` は成功した。

## 補足

構文上は通っているが、SAN の欠落は証明書発行結果そのものを変える。commit 前に、IP 直書きではなく `/etc/hosts` / DNS による名前解決から IP SAN を生成する処理を追加するべき。
