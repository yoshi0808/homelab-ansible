### 2. [高] evacuate / restore のタグおよび HA 動作が実装と一致していない

実装では、VM / CT を次のように分類しています。

- `preferpve*` タグあり、`hacritical` タグなし: non-HA として明示的に migrate
- `hacritical` タグあり: HA 管理対象として扱う
- evacuate 時: `ha-manager crm-command node-maintenance enable <node>` で HA 退避
- restore 時: `ha-manager crm-command node-maintenance disable <node>` で HA 復帰
- restore 時: `hacritical` と `prefer<target_node>` の両方を持つ VM / CT が対象ノードへ戻り、running になるまで待機

参照:

- `roles/proxmox_evacuate_node/tasks/main.yml:95`
- `roles/proxmox_evacuate_node/tasks/main.yml:126`
- `roles/proxmox_evacuate_node/tasks/main.yml:155`
- `roles/proxmox_restore_vm_placement/tasks/main.yml:58`
- `roles/proxmox_restore_vm_placement/tasks/main.yml:161`
- `roles/proxmox_restore_vm_placement/tasks/main.yml:182`

一方、Policy の Section 5.2 は `prefer<node名>` の説明だけで、`hacritical` の意味、HA/non-HA の分岐、maintenance mode による HA 退避・復帰を定義していません。Section 6.4 / 6.7 も一般的な migration の説明に留まっています。

また、`proxmox_vm_home_nodes.yml` はリポジトリ内に存在せず、実装はタグを正本としています。YAML 定義例を追加するのではなく、「外部 home node 定義ファイルは使用しない」こととタグの組み合わせを明記する必要があります。

### 3. [高] non-HA migration と残存 VM / CT の扱いが実装と一致していない

実装の QEMU migration は、running の場合に `--online 1` を付け、設定が有効な場合に `--with-local-disks` を付けます。現在の `inventories/homelab/group_vars/proxmox.yml` では evacuate / restore とも local disk migration が有効です。

参照:

- `roles/proxmox_evacuate_node/tasks/main.yml:126`
- `roles/proxmox_restore_vm_placement/tasks/main.yml:79`
- `inventories/homelab/group_vars/proxmox.yml:7`

Policy は local-only storage 使用時に停止すると記述しており、実運用設定と逆です。また、Policy は migration 可否、リソース余力、Sophos Firewall、local-only storage を事前確認して停止するとしていますが、現実装にその precheck はありません。

さらに実装は、退避後に残った running VM / CT を失敗扱いで停止するのではなく、`qm stop` / `pct stop` で強制停止します。タグなし VM / CT も明示的な skipped 一覧にはならず、running のまま残ればこの強制停止対象です。

参照:

- `roles/proxmox_evacuate_node/tasks/main.yml:200`
- `roles/proxmox_evacuate_node/tasks/main.yml:230`

Section 5.3、5.6、6.4 の実行条件、停止条件、処理概要、出力を実装どおりに修正してください。

### 4. [高] weekly_full の処理順序と出力が実装と一致していない

Policy の Section 6.6 / Section 11 は、pve2 と pve1 の patch 完了後に最後の一括 restore と final healthcheck を行う流れです。

実装は次の順序です。

1. 両ノード healthcheck
2. クラスタ全体 dry-run
3. pve2 evacuate / apply / post-healthcheck / restore
4. pve1 evacuate / apply / post-healthcheck / restore
5. 完了メール

参照:

- `playbooks/proxmox_patch_weekly_full.yml:90`
- `playbooks/proxmox_patch_weekly_full.yml:112`
- `playbooks/proxmox_patch_weekly_full.yml:147`
- `playbooks/proxmox_patch_weekly_full.yml:188`
- `playbooks/proxmox_patch_weekly_full.yml:196`
- `playbooks/proxmox_patch_weekly_full.yml:238`

実装には、両ノードの処理後に独立して行う final healthcheck や weekly_full 専用の full flow report はありません。各 restore role 内の対象ノード healthcheck と各 role の report、最後のメールが実際の出力です。

また `NO_UPDATES` は正常終了として後続処理を skip し、通知は dry-run role に任せます。この分岐も Policy に反映してください。
