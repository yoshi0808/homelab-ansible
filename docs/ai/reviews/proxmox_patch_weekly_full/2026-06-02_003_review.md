## Findings

### 1. dry-run の `_final_status` を localhost で直接参照しているため、weekly_full が常に停止する可能性が高い

Severity: High

`playbooks/proxmox_patch_weekly_full.yml:50`

```yaml
- name: Abort if dry-run status is not PATCH_READY
  hosts: localhost
```

`playbooks/proxmox_patch_weekly_full.yml:59`

```yaml
when: (_final_status | default('')) != 'PATCH_READY'
```

実装メモでは、`proxmox_patch_dryrun` role の `_final_status` は `delegate_to: localhost` により localhost の fact として記録される、と説明されている。

しかし Ansible の `set_fact` は、`delegate_to: localhost` だけでは delegate 先ではなく元の `inventory_hostname` に fact を保存する。`delegate_facts: true` がないため、`roles/proxmox_patch_dryrun/tasks/main.yml` の `run_once: true` タスクで設定された `_final_status` は、通常 `pve1` または `pve2` 側の hostvars に入る。

このため後続の `hosts: localhost` play では `_final_status` が未定義となり、`default('') != 'PATCH_READY'` により、dry-run が実際には `PATCH_READY` でも weekly full が停止する可能性が高い。

ローカルの最小検証でも、`run_once: true` + `delegate_to: localhost` + `set_fact` の fact は `hostvars["localhost"]` ではなく、run_once の元ホスト側に保存されることを確認した。

修正案:

- `proxmox_patch_dryrun` 側で `_final_status` を設定する `set_fact` に `delegate_facts: true` を付ける
- または weekly_full 側で `hostvars[groups['proxmox'][0]]._final_status` のように、dry-run の run_once 元ホストから読む
- もしくは dry-run role が保存する final report JSON を weekly_full 側で読み込んで判定する

### 2. weekly_full 自体が「外部 control node からのみ実行」を開始時に検証していない

Severity: High

`docs/ai/reviews/proxmox_patch_weekly_full/2026-06-02_001_requirement.md` では、制約として以下が明記されている。

```text
control node が Proxmox クラスタ外にあることを確認する
```

また `docs/ai/prompts/proxmox_patch_policy.md` でも、`proxmox_patch_weekly_full.yml` は `quory / 外部 control nodeのみ可` とされている。

現行の `playbooks/proxmox_patch_weekly_full.yml` は、実行冒頭で control node が quory / 外部 control node であることを検証していない。

実装メモでは `proxmox_patch_apply_node` role 内部の control node check に委ねる、とされているが、それでは遅い。weekly full では patch apply 前に `proxmox_evacuate_node.yml` を実行するため、control node が `ansy` など Proxmox 上の VM の場合、apply role の check に到達する前に evacuate が control node VM を migrate する可能性がある。

これは policy の以下の意図と衝突する。

```text
ansy 自身を同一playbook内で migrate して処理を継続する運用はしない。
```

修正案:

- weekly_full の最初に `hosts: localhost` の preflight play を追加し、実行元が `quory` または明示許可された外部 control node であることを assert する
- もしくは `pvesh get /cluster/resources` で `ansy` 等の control node VM が pve1/pve2 上に存在する場合、evacuate 前に fail する
- 少なくとも `proxmox_evacuate_node.yml` を呼ぶ前にこの制約を検証する

## 確認内容

### 要求仕様との整合

以下の大枠フローは requirement と整合している。

- pve2 → pve1 の順に処理する
- pve2 完了後のみ pve1 に進む
- 各ノードで evacuate → patch apply → post-healthcheck → restore の順になっている
- `proxmox_patch_apply_node.yml` は `target_node=pve2/pve1` で呼ばれており、manual apply mode は使っていない
