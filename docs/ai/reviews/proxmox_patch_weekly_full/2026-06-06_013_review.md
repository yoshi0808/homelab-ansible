# Review: proxmox_patch_weekly_full NO_UPDATES 正常終了対応

日付: 2026-06-06
対象: `2026-06-06_012_implement.md` 対応後の git diff

## Findings

### Medium: `NO_UPDATES` 時に通知メールが二重送信される

- 該当:
  - `playbooks/proxmox_patch_weekly_full.yml:112`
  - `playbooks/proxmox_patch_weekly_full.yml:128-157`
  - `roles/proxmox_patch_dryrun/tasks/main.yml:297-365`
- 状況:
  - weekly full は Step 2a で `proxmox_patch_dryrun.yml` を import する。
  - `proxmox_patch_dryrun` role は `_final_status == 'NO_UPDATES'` の場合、`[Proxmox Patch Dry-run] NO_UPDATES - ...` のメール本文を組み立て、`Send email without attachment` で送信する。
  - 今回の変更では、Step 2b でも `_status == 'NO_UPDATES'` の場合に `[Proxmox Patch] NO_UPDATES - 適用すべき更新なし` を追加送信する。
- 影響:
  - 1回の weekly full 実行で `NO_UPDATES` メールが2通届く。
  - 2通の件名・本文が異なるため、運用上「dry-run 通知」と「weekly full の終了通知」が別イベントに見える可能性がある。
  - 実装メモでは `NO_UPDATES` は「メール通知して正常終了」と説明されているが、既存 dry-run 通知との重複整理がされていない。
- 対応案:
  - weekly full 側では `NO_UPDATES` メールを送らず、dry-run role の既存通知を採用して skip flag だけセットする。
  - あるいは `proxmox_patch_dryrun` role に通知抑止変数を追加し、weekly full から dry-run を呼ぶ場合だけ dry-run 側のメールを抑止する。
  - どちらか一方に統一し、policy の「NO_UPDATES はメール通知する」を1通の通知として満たす。

### Low: `NO_UPDATES` 時も inline post-healthcheck play では gather facts が実行される

- 該当:
  - `playbooks/proxmox_patch_weekly_full.yml:186-193`
  - `playbooks/proxmox_patch_weekly_full.yml:233-240`
- 状況:
  - pve2 / pve1 post-healthcheck の inline play は `gather_facts: true` のまま、最初の task で `meta: end_play` している。
  - Ansible の fact gathering は tasks より前に実行されるため、`NO_UPDATES` skip flag が立っていても pve2 / pve1 への fact gathering は行われる。
- 影響:
  - 実装メモの「Steps 3-11 はすべてスキップ」という説明より、実際の NO_UPDATES 経路では Step 5 / Step 9 の接続・fact gathering が残る。
  - 直前の healthcheck / dry-run が成功している前提では実害は小さいが、NO_UPDATES 正常終了経路としては不要な対象ホスト接続になる。
- 対応案:
  - inline play 自体に `when: not (hostvars['localhost']['_weekly_full_no_updates'] | default(false))` を付けることができるか確認する。
  - 難しければ `gather_facts: false` にし、healthcheck role に必要な fact がある場合だけ `setup` を task として skip 後に実行する。

## 確認できたこと

- `NO_UPDATES` を `fail` せず、`PATCH_READY` 以外の手動対応 status だけ fail する方向性は policy の `NO_UPDATES = 更新候補なし / 自動適用不要` と整合している。
- `_final_status` は従来どおり `hostvars[groups['proxmox'][0]]['_final_status']` から参照しており、dry-run role の fact 配置前提と整合している。
- `MAINTENANCE_REQUIRED` / `BLOCKED` / `MAJOR_UPGRADE_DETECTED` / `UNKNOWN` は Step 2b で fail する。
- `PATCH_READY` の場合は skip flag が立たないため、従来どおり pve2 → pve1 の apply flow へ進む。
