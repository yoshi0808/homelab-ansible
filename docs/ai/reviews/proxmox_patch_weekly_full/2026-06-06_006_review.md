## Findings

### Low: allowlist を group_vars で上書きできる、という実装メモの説明と実際の変数優先順位が一致しない

- 該当:
  - `playbooks/proxmox_patch_weekly_full.yml:39-41`
  - `playbooks/proxmox_patch_weekly_full.yml:77-84`
  - `docs/ai/reviews/proxmox_patch_weekly_full/2026-06-06_005_implement.md`
- 状況:
  - `proxmox_patch_weekly_full_allowed_controllers` は preflight play の `vars` に定義されている。
  - Ansible では play vars は inventory / group_vars より優先されるため、実装メモの注意点にある「この変数を `group_vars` や実行時変数で上書きする」という説明のうち、`group_vars` による上書きは通常効かない。
  - `-e proxmox_patch_weekly_full_allowed_controllers=...` のような extra vars であれば play vars より優先されるため、実行時変数による上書きは可能。
- 影響:
  - 現在の要求仕様は「quory からのみ実行する」なので、デフォルトを play vars で `['quory']` に固定すること自体は安全側で問題ない。
  - ただし将来、別 controller を許可するために `group_vars` へ値を追加しても preflight は引き続き `['quory']` として評価され、意図せず abort する。
  - 実行時メッセージの `To override, set proxmox_patch_weekly_full_allowed_controllers.` は間違いではないが、どこで設定できるかが曖昧。
- 対応案:
  - quory 固定を強く維持するなら、実装メモから `group_vars` で上書き可能という説明を削除する。
  - 将来 `group_vars` で許可 controller を管理したいなら、play vars ではなく `default([])` を使ったタスク内 default、または inventory 側の変数定義へ移す。
  - エラーメッセージも `To override, set ... with extra vars.` のように明確化すると運用時の誤解が減る。

## 確認できたこと

- `playbooks/proxmox_patch_weekly_full.yml:51` で `hostname -s` を使うようになり、controller 側 hostname が FQDN を返す環境でも short hostname として比較される。
- `playbooks/proxmox_patch_weekly_full.yml:60-84` により、以下の3条件で preflight が停止する:
  - 実行元 hostname がクラスタ上の QEMU VM 名に含まれる
  - 実行元 hostname が `groups['proxmox']` に含まれる
  - 実行元 hostname が `proxmox_patch_weekly_full_allowed_controllers` に含まれない
- デフォルト allowlist は `quory` のみであり、要求仕様の「quory（外部 control node）からのみ実行する」に沿っている。
- 前回レビュー `2026-06-06_004_review.md` の Medium 指摘である「quory 専用要件が preflight では担保されていない」は、今回のコード差分で解消されている。
