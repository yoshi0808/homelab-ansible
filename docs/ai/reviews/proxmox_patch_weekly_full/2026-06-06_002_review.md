## Findings

### High: `pve1` / `pve2` 自身からの実行を preflight が許可してしまう

- 該当:
  - `playbooks/proxmox_patch_weekly_full.yml:46-63`
  - `docs/ai/reviews/proxmox_patch_weekly_full/2026-06-02_001_requirement.md:5-6`
  - `docs/ai/reviews/proxmox_patch_weekly_full/2026-06-02_001_requirement.md:47-50`
  - `docs/ai/prompts/proxmox_patch_policy.md:695-702`
  - `docs/ai/prompts/proxmox_patch_policy.md:1092-1099`
- 状況:
  - 今回の修正で、preflight は `lookup('pipe', 'hostname')` の結果を `_controller_hostname` とし、クラスタ内 QEMU VM 名一覧 `_cluster_vm_names` に含まれる場合だけ fail するようになった。
  - これにより、quory から実行しているのに ansy VM がクラスタ上に存在するだけで abort する誤判定は解消されている。
  - しかし、`_cluster_vm_names` は `type == qemu` の VM 名だけであり、Proxmox 物理ノード名 `pve1` / `pve2` は含まれない。
  - そのため、誤って `pve1` または `pve2` 自身で `ansible-playbook playbooks/proxmox_patch_weekly_full.yml` を起動した場合、`_controller_hostname` が `pve1` / `pve2` でも preflight を通過する。
- 影響:
  - 要求仕様は「quory（外部 control node）からのみ実行する」と明記している。
  - policy も weekly full の実行条件として「control node が Proxmox クラスタ外にある」「pve1 / pve2 の reboot 影響を受けない」「quory のような物理ノードから実行する」を要求している。
  - `pve1` / `pve2` 自身から実行した場合、後続の patch apply / reboot により Ansible 実行元が停止し得るため、weekly full の安全条件を満たさない。
- 対応案:
  - Step 0 で `_controller_hostname` が `groups['proxmox']` に含まれる場合も fail する。
  - さらに厳密にするなら、weekly full は `quory` 専用と割り切り、`_controller_hostname == 'quory'` または許可リスト変数に含まれる場合のみ続行する。
  - LXC 上の control node を将来許容しないなら、`pvesh` の `type == lxc` も含めてクラスタ内 guest 名を検出対象にする。

## 確認できたこと

- `playbooks/proxmox_patch_weekly_full.yml:48-63` の変更により、quory から実行している場合は ansy VM がクラスタ上にいても abort しない。
- ansy から実行し、かつクラスタ上の QEMU VM 名が `ansy` の場合は `_controller_hostname in _cluster_vm_names` により abort する。
- `roles/proxmox_patch_apply_node/tasks/main.yml` の control node check は、単一ノード apply 時に「対象ノード上に ansy がいるか」を見る別目的のチェックであり、今回の weekly full 冒頭 preflight とは役割が異なる。
