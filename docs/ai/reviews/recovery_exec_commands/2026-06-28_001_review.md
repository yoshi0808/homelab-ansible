## 指摘

### must-fix: HTTP 系調査コマンドに実行時間上限がない

対象:

- `roles/recovery_exec/defaults/main.yml:40`
- `roles/recovery_exec/defaults/main.yml:42`
- `roles/recovery_exec/defaults/main.yml:44`

`prometheus-targets` / `grafana-health` / `loki-health` が `curl -s` のみで、`--max-time` などの上限がありません。
`ssh` 側の `ConnectTimeout=10` は SSH 接続確立だけの上限なので、monnie 側で HTTP/TLS 接続後に応答が止まると、調査コマンドが長時間返らず recovery pipeline 全体を止める可能性があります。

調査コマンドは障害時に使うものなので、HTTP probe は例えば次のように bounded にしてください。

```sh
curl -fsS --max-time 10 http://localhost:9090/api/v1/targets
curl -fsS --max-time 10 https://monnie.internal:3000/api/health
curl -fsS --max-time 10 http://localhost:3100/metrics | head -10
```

失敗時にも調査ログとして読める出力を残したい場合は、`|| echo '...'` を付けるのも妥当です。

### suggestion: `radius-config-check` は sudoers と同じ絶対パスで呼ぶ方が堅い

対象:

- `roles/recovery_exec/defaults/main.yml:29`
- `roles/recovery_exec/templates/sudoers-recovery-exec-target.j2:8`

sudoers は `/usr/sbin/freeradius -XC` を許可していますが、dispatch 側は `sudo freeradius -XC` です。
通常は sudo が PATH 解決して一致しますが、この role は execpolicy / allowlist を堅く作っているので、呼び出し側も `sudo /usr/sbin/freeradius -XC 2>&1` にそろえる方が読みやすく、環境差分にも強いです。

### suggestion: `set -e` と失敗系 probe の出力方針を明示する

対象:

- `roles/recovery_exec/templates/recovery-investigate-dispatch.sh.j2:56`
- `roles/recovery_exec/defaults/main.yml:40`
- `roles/recovery_exec/defaults/main.yml:42`
- `roles/recovery_exec/defaults/main.yml:44`

node-specific check は `extra.cmd` をそのまま shell に展開しており、dispatch script は `set -euo pipefail` です。
そのため、HTTP endpoint が落ちている場合などはコマンドが非 0 で終了し、出力が少ないまま SSH コマンド自体が失敗します。

「probe 失敗を exit non-zero として Codex に見せたい」のか、「調査結果として失敗理由を表示して command は完走させたい」のかをコマンドごとに決めて、`|| echo ...` / `|| true` の有無をそろえると運用時に読みやすくなります。
