# proxmox_healthcheck レビュー

## Findings

1. High: `zpool status` の取得失敗が OK 扱いになる

   - 該当箇所: `roles/proxmox_healthcheck/files/proxmox-healthcheck.sh:14`, `roles/proxmox_healthcheck/files/proxmox-healthcheck.sh:64-76`, `roles/proxmox_healthcheck/tasks/main.yml:40-43`
   - `zpool status` が失敗すると raw は `unavailable` になり、parser は `pools: []` を返す。その後 Ansible 側は DEGRADED/FAULTED の pool だけを critical にするため、ZFS 状態を取得できていないノードが ZFS OK と同じ扱いになる。
   - 要求仕様では ZFS プール健全性を GO/NO-GO 判定に使うため、取得不能は少なくとも CRITICAL、または明示的な collection error として fail すべき。

2. High: root filesystem 使用率の取得失敗が OK 扱いになる

   - 該当箇所: `roles/proxmox_healthcheck/files/proxmox-healthcheck.sh:23-24`, `roles/proxmox_healthcheck/tasks/main.yml:64-76`
   - `df -P /` が失敗すると `root_used_pct=-1` になる。判定では `-1 < 80` なので warning/critical にならず、root FS 使用率を確認できていない状態が OK になる。
   - patch 前の GO/NO-GO では、確認不能を OK に倒すのは危険。`root_used_pct < 0` を CRITICAL にするか、shell 側で collection error として JSON に含め、Ansible 側で fail するのがよい。

3. Medium: ZFS/replication のメッセージ生成で `regex_replace('(.*)', ...)` が二重置換になる可能性がある

   - 該当箇所: `roles/proxmox_healthcheck/tasks/main.yml:50`, `roles/proxmox_healthcheck/tasks/main.yml:67`, `roles/proxmox_healthcheck/tasks/main.yml:70`
   - Python/Jinja の正規表現置換では `(.*)` が末尾の空文字にもマッチする実装があり、`rpool` が `ZFS pool degraded/faulted: rpoolZFS pool degraded/faulted: ` のようなメッセージになる可能性がある。
   - 判定自体は壊れないが、fail メッセージと report の可読性が落ちる。`'^(.*)$'` にするか、Jinja のループで文字列を組み立てる方が安全。

4. Medium: 閾値の境界が要求仕様の「超」と実装でずれている

   - 該当箇所: `docs/ai/reviews/proxmox_healthcheck/2026-05-08_001_requirement.md:31-40`, `roles/proxmox_healthcheck/tasks/main.yml:64-76`
   - 要求仕様は root filesystem `90%超` を CRITICAL、`80%超` を WARNING としているが、実装は `>= 90` と `>= 80` 相当になっている。
   - 運用上 `80%ちょうど` や `90%ちょうど` も止めたいなら実装でよいが、その場合は requirement 側の表現を `80%以上` / `90%以上` に揃えた方がよい。

5. Low: `docs/ai/promps/core.md` は存在しない

   - 実際の正本は `docs/ai/prompts/core.md`。
   - 今回は `docs/ai/prompts/core.md` を読んでレビューした。依頼文または参照パスの typo だけなので、実装上の問題ではない。

## 良い点

- `playbooks/proxmox_healthcheck.yml` は `proxmox` グループ対象で、core.md の管理対象グループ方針に合っている。
- shell は warning/critical/status を生成しておらず、収集と JSON 整形に留まっている。
- critical/warning/fail 制御と report 保存は Ansible tasks 側にある。
- report 保存は `delegate_to: localhost` / `become: false` で、対象ノードではなく実行元に保存する形になっている。
- 秘密情報、鍵、認証情報を読む処理は見当たらない。

## 確認結果

```bash
bash -n roles/proxmox_healthcheck/files/proxmox-healthcheck.sh
ANSIBLE_LOCAL_TEMP=/tmp/ansible-local ANSIBLE_REMOTE_TEMP=/tmp/ansible-remote ansible-playbook --syntax-check -i inventories/homelab/hosts.yml playbooks/proxmox_healthcheck.yml
```

結果:

- shell 構文チェックは成功。
- playbook syntax-check は成功。
- 通常の `ansible-playbook --syntax-check` は、この Codex 環境では `~/.ansible/tmp` が read-only のため失敗した。`ANSIBLE_LOCAL_TEMP` / `ANSIBLE_REMOTE_TEMP` を `/tmp` に向けると成功した。

## 判定

このまま commit するのはまだ早い。

特に `zpool status` と `df -P /` の取得失敗が OK に倒れる点は、patch 前 GO/NO-GO 用の healthcheck として危険。取得不能を CRITICAL にする修正を入れてから再レビューするのがよい。
