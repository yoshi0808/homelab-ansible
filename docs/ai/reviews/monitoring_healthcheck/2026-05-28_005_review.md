# Review: monnie monitoring healthcheck 追加実装

## 結論

修正必須あり。

前回指摘した「reboot 直後に即時確認して誤 CRITICAL になりうる」問題に対して、`retries: 12` / `delay: 5` / `until` が追加され、起動待ち自体は入った。

ただし、現在の書き方では `until` 条件が最後まで満たされなかった場合、その確認タスク自体が failed になり、後続の `nightly_criticals` 作成、レポート保存、CRITICAL 通知まで進まない可能性が高い。これは `ubuntu_vm_patch_policy.md` の「reboot 実行 → サービス確認 NG は CRITICAL 通知」という運用方針に反する。

## Findings

### [High] retry タスクが最終失敗すると、CRITICAL レポート・通知へ進めない

対象:

- `playbooks/ubuntu_nightly.yml:285`
- `playbooks/ubuntu_nightly.yml:295`
- `playbooks/ubuntu_nightly.yml:305`
- `playbooks/ubuntu_nightly.yml:315`
- `playbooks/ubuntu_nightly.yml:325`
- `playbooks/ubuntu_nightly.yml:335`

追加実装では、monitoring_servers 向けの post-reboot 確認に以下が追加されている。

```yaml
retries: 12
delay: 5
until: <正常条件>
failed_when: false
```

しかし、Ansible の `until` は条件が retries 内で成立しなかった場合、そのタスクを failed として扱う。`failed_when: false` は各コマンド実行結果の失敗判定を抑えるためには有効だが、`until` 条件未達による最終失敗を「後続へ進んでよい状態」にする目的では不十分。

そのため、たとえば Grafana が 60 秒以内に `active` にならなかった場合、`Check grafana-server service status` の時点で play が止まり、後続の処理に到達しない。

到達しない可能性がある後続処理:

- `nightly_criticals` の作成
- post-reboot 結果の debug 出力
- `reports/ubuntu-nightly/` への結果保存
- CRITICAL 通知
- 最終的な `Fail if critical`

これは「サービス確認 NG を CRITICAL として通知・レポートする」設計から外れる。

推奨修正:

- retry 付き確認タスクに `ignore_errors: true` を追加して、最終的に条件未達でも register の最終値を使って後続の `nightly_criticals` 判定へ進める。
- もしくは `block` / `rescue` で確認失敗時も `nightly_criticals` を構築し、通知・レポート保存へ必ず進める。

最小修正例:

```yaml
- name: Check grafana-server service status
  ansible.builtin.command:
    cmd: systemctl is-active grafana-server
  register: grafana_active
  changed_when: false
  failed_when: false
  retries: 12
  delay: 5
  until: grafana_active.stdout.strip() == 'active'
  ignore_errors: true
```

port 確認も同様に、`until` 条件未達で play を止めず、最終 stdout を `nightly_criticals` で評価する形にする。

補足:

- `ignore_errors: true` を入れても、最後の `Fail if critical` で CRITICAL として playbook 全体は失敗扱いにできる。
- その前にレポート保存と通知が実行されるため、運用方針に合う。

## 前回指摘への対応状況

### [Medium] reboot 後のサービス起動待ち不足

一部解消。

`retries: 12` / `delay: 5` / `until` により、最大 60 秒の待機が追加された点は妥当。

ただし、条件未達時に CRITICAL 判定へ進めない問題が残っているため、完了とは言えない。

### [Low] read-only 要件と script 配置タスクの扱い

運用解釈として整理済み。

`2026-05-28_004_implement.md` で「read-only はサービス・設定・reboot などの変更操作をしない意味で、healthcheck script の配置は許容する」と明記された。

既存の `radius_healthcheck` と同じ扱いであり、この点は追加修正不要。

## 要件との照合

### OK

- `monitoring-healthcheck.sh` は収集と JSON 整形のみで、判定・通知・保存を行っていない。
- CRITICAL / WARNING 判定は Ansible tasks 側にある。

- `monitoring-healthcheck` timer の追加内容は要求仕様どおり。
- 秘密鍵・認証情報・証明書秘密鍵の追加や読み出しは見当たらない。

- monitoring_servers の post-reboot 確認で、NG 時に CRITICAL 通知・レポート保存へ進めない可能性がある。
