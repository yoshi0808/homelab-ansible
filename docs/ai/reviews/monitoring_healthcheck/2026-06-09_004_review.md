# Review: メモリ使用率チェック追加 再レビュー

Review date: 2026-06-09

## 結論

前回レビューの Medium / Low 指摘は、通常想定される `free -b` の成功・失敗に
対して解消されている。

新たに Low 指摘が 1 件あるが、Ubuntu 標準の `free` が正常な形式で出力する
運用では発生しにくく、commit を止める問題ではない。

## Findings

### Low: `collection_ok` が取得値の妥当性を確認していない

- File: `roles/monitoring_healthcheck/files/monitoring-healthcheck.sh:30`
- File: `roles/radius_healthcheck/files/radius-healthcheck.sh:37`

現在は `free -b` のパイプラインが成功し、`mem_line` が空でなければ
`collection_ok=true` になる。total / used / available が数値であることや、
total が 0 より大きいことは確認していない。

そのため、成功終了した `free` が `Mem: 0 0 ...` を返すケースでは
`collection_ok=true`, `used_percent=0` となり、メモリ収集は正常扱いされる。
また、数値でない値が返ると Python の `int()` で例外になり、JSON、レポート、
Slack 通知を生成する前に healthcheck task が失敗する。

Ubuntu 標準 procps の正常な `free` 出力では通常発生しないため重要度は Low とする。
より堅牢にする場合は、3項目が非負整数であり `mem_total_bytes > 0` であることまで
確認してから `collection_ok=true` とするとよい。

## 前回指摘の確認

### 解消: メモリ収集失敗が 0% として正常扱いされる

`free -b` の失敗または空出力では `collection_ok=false` が JSON に出力され、
Ansible 側で CRITICAL に分類される。

コマンド失敗を模擬した実行でも、以下の出力になることを確認した。

```json
"memory": {
  "collection_ok": false,
  "total_mb": 0,
  "used_mb": 0,
  "available_mb": 0,
  "used_percent": 0
}
```

### 解消: 90%以上で criticals と warnings が重複する

warning 条件は `80 <= used_percent < 90` に限定されている。
Ansible の式を境界値で評価し、以下を確認した。

| used_percent | warning |
|---:|---|
| 79 | なし |
| 80 | あり |
| 89 | あり |
| 90 | なし |

90以上は critical のみに分類される。

## 確認できたこと

- shell は収集と JSON 整形のみを担当し、CRITICAL / WARNING 判定は Ansible 側にある。
- `collection_ok` はレポートとサマリーにも追加されている。
- 通常環境で両 shell が有効な JSON を出力することを確認した。
- 秘密情報や変更操作の追加は見当たらない。
