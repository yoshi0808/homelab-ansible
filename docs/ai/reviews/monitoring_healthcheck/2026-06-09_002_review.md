# Review: メモリ使用率チェック追加

Review date: 2026-06-09

## 結論

修正を推奨する指摘が 2 件ある。

特に、メモリ情報の収集に失敗した場合に使用率 `0%` として正常扱いされるため、
healthcheck が異常を見逃す可能性がある。commit 前に対応したい。

## Findings

### Medium: メモリ情報の収集失敗が使用率 0% の正常状態として扱われる

- File: `roles/monitoring_healthcheck/files/monitoring-healthcheck.sh:30`
- File: `roles/monitoring_healthcheck/tasks/main.yml:40`
- File: `roles/radius_healthcheck/files/radius-healthcheck.sh:37`
- File: `roles/radius_healthcheck/tasks/main.yml:34`

`free -b` の実行失敗時は `mem_line="0 0 0"` にフォールバックし、
Python 側では `mem_total == 0` の場合に `used_percent = 0` を返している。

Ansible 側は `used_percent` だけで判定するため、収集失敗時にも
critical/warning が追加されず、レポートの状態は `OK` になり得る。
監視項目を取得できなかった状態をメモリ使用率 0% と区別できないため、
実際の高負荷やコマンド異常を見逃す。

収集結果に `collection_ok` またはエラー情報を追加し、Ansible 側で収集失敗を
CRITICAL に分類する方法が分かりやすい。少なくとも、total が 0 の場合は
正常な使用率として扱わないようにする必要がある。

### Low: 90%以上では同じメモリ異常が criticals と warnings の両方に入る

- File: `roles/monitoring_healthcheck/tasks/main.yml:40`
- File: `roles/monitoring_healthcheck/tasks/main.yml:44`
- File: `roles/radius_healthcheck/tasks/main.yml:34`
- File: `roles/radius_healthcheck/tasks/main.yml:40`

warning 条件が `used_percent >= 80`、critical 条件が `used_percent >= 90` のため、
90%以上では同じメッセージが両方のリストに追加される。

最終 status は `CRITICAL` になるが、保存レポートと Slack 通知には同じ事象が
Criticals と Warnings の双方に表示されるため、分類が重複して見える。

warning を `80 <= used_percent < 90` に限定すると、分類が排他的になり、
レポートと通知の意味が明確になる。

## 確認できたこと

- shell はメモリ値の収集と JSON 整形のみを担当しており、判定は Ansible 側にある。
- `memory` セクションは両 role のレポートとサマリーに追加されている。
- 80% / 90% の閾値による WARNING / CRITICAL 判定が両 role に追加されている。
- `git diff --check` で whitespace error は検出されなかった。
- 秘密情報や変更操作の追加は見当たらない。
