# Review: メモリ使用率チェック追加 再レビュー

Review date: 2026-06-09

## 結論

前回レビューで指摘した `collection_ok` の妥当性チェック不足は解消されている。

commit を止める問題や、新たな修正必須の指摘は見つからなかった。

## Findings

指摘事項なし。

## 前回指摘の確認

以下をすべて満たす場合だけ `collection_ok=true` になるよう修正されている。

- total / used / available がすべて非負整数
- total が 0 より大きい

コマンド失敗、total が 0、非数値の各ケースでは、3値が 0 にリセットされ、
Python 例外を発生させずに以下の JSON が出力されることを確認した。

```json
"memory": {
  "collection_ok": false,
  "total_mb": 0,
  "used_mb": 0,
  "available_mb": 0,
  "used_percent": 0
}
```

この結果は Ansible 側で CRITICAL に分類されるため、収集失敗が使用率 0% の
正常状態として扱われる問題も再発していない。

## 確認できたこと

- 正常な `free -b` 出力では `collection_ok=true` となり、メモリ値と使用率が出力される。
- `free -b` の失敗では `collection_ok=false` となる。
- total が 0 の場合は `collection_ok=false` となる。
- total / used / available に非数値が含まれる場合も `collection_ok=false` となり、有効な JSON を維持する。
- shell は収集と JSON 整形のみを担当し、CRITICAL / WARNING 判定は Ansible 側にある。
- 秘密情報や変更操作の追加は見当たらない。

## 残留リスク

`collection_ok=false` のメッセージは
`memory collection failed (free -b returned no data)` で統一されているため、
不正値や total=0 の場合も「no data」と表示される。

異常検知と CRITICAL 判定には影響せず、Ubuntu 標準の `free` では通常発生しない
入力であるため、修正必須の指摘とはしない。将来、診断理由を詳しく残す場合は
`collection_error` の追加や、文言を `returned invalid data` を含む表現にするとよい。
