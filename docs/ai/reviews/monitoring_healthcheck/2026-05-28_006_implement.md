# 追加実装内容: monnie monitoring healthcheck（第2回レビュー指摘対応）

## 対応した指摘

### [High] retry タスクが最終失敗すると、CRITICAL レポート・通知へ進めない

`playbooks/ubuntu_nightly.yml` の monitoring_servers 向け post-reboot 確認において、
`retries` / `delay` / `until` を追加していたが、`until` 条件が retries 内で満たされなかった場合、
Ansible はそのタスクを failed として扱い、後続処理（`nightly_criticals` 作成・レポート保存・通知）に進まない問題があった。

`failed_when: false` はコマンド自体の exit code による失敗を抑えるが、`until` 条件未達による
タスク失敗は抑えない。これが抜けていた。

## 変更ファイル

- `playbooks/ubuntu_nightly.yml`（monitoring_servers post-reboot 確認 6 タスクに `ignore_errors: true` 追加）

## 変更内容の詳細

monitoring_servers 向け play の post-reboot 確認 6 タスクに `ignore_errors: true` を追加した。

| タスク名 |
|---|
| Check prometheus service status |
| Check grafana-server service status |
| Check loki service status |
| Check 9090/tcp listening |
| Check 3000/tcp listening |
| Check 3100/tcp listening |

これにより、60 秒（retries: 12 / delay: 5）以内にサービスが起動しなかった場合も、
play は止まらず register の最終値（`inactive` / 空文字）が残る。

後続の `nightly_criticals` 評価で NG として検出され、以下が順に実行される。

1. `nightly_criticals` に該当サービス/ポートのメッセージが入る
2. レポートが `reports/ubuntu-nightly/` に保存される
3. CRITICAL 通知メールが送信される
4. `Fail if critical` で playbook が failed 終了する

`ignore_errors: true` を付けても、最終的な `Fail if critical` タスクは `nightly_criticals` が
空でなければ `ansible.builtin.fail` を実行するため、CRITICAL 時の playbook 失敗は維持される。
