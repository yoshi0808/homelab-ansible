# Review: monnie monitoring healthcheck 追加

## 結論

修正推奨あり。

実装全体は `radius_healthcheck` の既存パターンを踏襲しており、要求仕様の主要項目はおおむね満たしている。

ただし、`ubuntu_nightly.yml` に追加された monitoring_servers 向け post-reboot 確認は、reboot 直後に即時で `systemctl is-active` と `ss` を評価しているため、Prometheus / Grafana / Loki の起動待ちが不足し、実際には数秒後に正常化するケースでも CRITICAL / fail になる可能性がある。ここは運用上の誤検知につながるため、commit 前に修正することを推奨する。

## Findings

### [Medium] reboot 後のサービス起動待ちがなく、誤 CRITICAL になる可能性がある

対象:

- `playbooks/ubuntu_nightly.yml:285`
- `playbooks/ubuntu_nightly.yml:292`
- `playbooks/ubuntu_nightly.yml:299`
- `playbooks/ubuntu_nightly.yml:306`
- `playbooks/ubuntu_nightly.yml:313`
- `playbooks/ubuntu_nightly.yml:320`

monitoring_servers 向け play では、`ansible.builtin.reboot` の完了直後に以下を即時確認している。

- `systemctl is-active prometheus`
- `systemctl is-active grafana-server`
- `systemctl is-active loki`
- `ss -tlnH 'sport = :9090'`
- `ss -tlnH 'sport = :3000'`
- `ss -tlnH 'sport = :3100'`

`ansible.builtin.reboot` は基本的に SSH 接続復旧を待つもので、アプリケーションサービスの listen 完了までは保証しない。

特に Grafana / Loki / Prometheus は、OS 起動後に systemd 上で `activating` または依存処理中になる時間がありうる。その場合、数秒後には正常化するにもかかわらず、今回の実装では即座に `nightly_criticals` に入り、CRITICAL 通知と playbook fail になる。

要求仕様および `ubuntu_vm_patch_policy.md` の目的は「reboot 後のサービス確認」なので、瞬間的な未起動ではなく、一定時間待っても復旧しない場合を CRITICAL とする方が運用意図に合う。

推奨修正:

- 各 service / port 確認に `retries`, `delay`, `until` を追加する。
- もしくは `reboot` に `post_reboot_delay` を設定したうえで、さらに port listen には短い retry を入れる。

例:

```yaml
- name: Check grafana-server service status
  ansible.builtin.command:
    cmd: systemctl is-active grafana-server
  register: grafana_active
  changed_when: false
  failed_when: false
  retries: 12
  delay: 5
  until: grafana_active.stdout.strip() == 'active'
```

port 確認も同様に `until: port_3000_output.stdout | length > 0` のように待つ。

### [Low] read-only 要件と script 配置タスクの扱いを明確にした方がよい

対象:

- `roles/monitoring_healthcheck/tasks/main.yml:8`

要求仕様には `read-only（変更操作なし）` とあるが、healthcheck role は実行時に `/usr/local/sbin/monitoring-healthcheck.sh` へ script を配置する。

これは既存の `radius_healthcheck` と同じパターンであり、サービス状態や設定を変更するものではないため、実運用上は許容できる範囲と考えられる。一方で、厳密には対象ホストのファイルシステムを変更する。

このまま採用する場合は、「read-only」はサービス・設定・reboot などの変更操作をしないという意味であり、healthcheck script の配置は許容する、という運用解釈を明確にしておくとよい。
