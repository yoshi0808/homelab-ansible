# proxmox_hw_check レビュー

## Findings

1. High: NIC 情報の収集失敗が OK 扱いになる

   - 該当箇所: `roles/proxmox_hw_check/files/proxmox-hw-check.sh:34`, `roles/proxmox_hw_check/files/proxmox-hw-check.sh:131-133`, `roles/proxmox_hw_check/tasks/main.yml:25-44`
   - `ip -s link show` が失敗すると raw は `unavailable` になり、parser は `nics: []` を返す。その後 Ansible 側は `nics` の中から DOWN/error/drop を探すだけなので、NIC チェック自体が実行できていない状態が `OK` になる。
   - 要求仕様では NIC リンク状態とエラーカウンタが「判定あり」の項目なので、収集失敗は少なくとも WARNING、patch 前の基準確認に使うなら CRITICAL 相当の collection failure として扱うべき。

2. Medium: ZFS / disk / CPU / memory の収集失敗が空値・0値として保存され、基準値レポートとして紛らわしい

   - 該当箇所: `roles/proxmox_hw_check/files/proxmox-hw-check.sh:9-31`, `roles/proxmox_hw_check/files/proxmox-hw-check.sh:57-101`, `roles/proxmox_hw_check/files/proxmox-hw-check.sh:203-221`
   - `zpool list` / `zpool status` / `df` が失敗しても、report には `pools: []`, `scrub: []`, `filesystems: []` のように保存される。`lscpu` や `/proc/meminfo` が取れない場合も `unknown` / `0` になる。
   - これらは「記録のみ（判定なし）」なので warning/critical 判定しない方針は理解できる。ただし、構成変更後・パッチ前の基準値記録として使うなら、取得失敗と「本当に対象が存在しない」を区別できる `collection_ok` や `raw: unavailable` を report に残した方が安全。

3. Medium: Proxmox firewall peer interface `fwpr*` が除外されていない

   - 該当箇所: `roles/proxmox_hw_check/files/proxmox-hw-check.sh:135`
   - Proxmox の firewall 有効 VM/CT では `fwbr*`, `fwln*`, `fwpr*` 系の仮想インターフェースが作られる。現実装は `fwbr` と `fwln` は除外するが、`fwpr` は収集対象に残る。
   - 要求は物理ハードウェア構成の確認なので、`fwpr*` が DOWN/error/drop warning の原因になるとノイズが出る。`fwpr` も除外対象に入れるのがよい。

4. Medium: patch policy の再構築メモ用途まで含めるなら、現在の requirement / 実装では情報が足りない

   - 該当箇所: `docs/ai/prompts/proxmox_patch_policy_draft_v1.5.md` の「18.4 バックアップ対象」
   - policy では Proxmox ホスト再構築メモとして、NIC 名だけでなく NIC 割当、bridge 設定、VLAN 設定、storage 設定、`/etc/network/interfaces` なども保存すべき情報に含まれている。
   - 今回の requirement は「物理ハードウェア構成」の範囲に絞っているため、実装が要求から外れているわけではない。ただし、この role を将来の再インストール手順・patch policy の基礎資料として使う意図なら、別 role か追加項目として network/storage config の保存も検討した方がよい。

## 良い点

- `playbooks/proxmox_hw_check.yml` は `proxmox` グループ対象で、core.md の方針に合っている。
- shell は warning/status/fail を生成しておらず、収集と JSON 整形に留まっている。
- NIC の warning 判定と report 保存は Ansible tasks 側にあり、責務分離はできている。
- report 保存は `reports/proxmox-hardware/` 配下で、`delegate_to: localhost` / `become: false` になっている。
- patch / reboot / restart / reload などの変更系処理や秘密情報の読み取りは見当たらない。

## 判定

このまま commit するのはまだ早い。
