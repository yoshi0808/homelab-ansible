# proxmox_patch_apply_node 要求仕様

作成日: 2026-05-09

---

## 目的

単一ノードにパッチを適用する。
PATCH_READY / MAINTENANCE_REQUIRED 両方に対応する。

## 対象

- 変数 `target_node` で指定（pve1 または pve2）

## 処理フロー

### Phase 1: 前提チェック

1. control node が target_node 上にいないか確認
   `pvesh get /cluster/resources --type vm` で検出
   → 検出したら即 fail（メール送信して終了）

2. target_node の healthcheck が OK か確認
   → OK でなければ fail（メール送信して終了）

### Phase 2: パッチ適用

3. apt-get update

4. apt-get check
   → 失敗したら fail（メール送信して終了）

5. 再 dry-run（apt-get -s dist-upgrade）
   以下を機械的にチェック（Codex 不要）:
   - 重要コンポーネント含む → MAINTENANCE_REQUIRED
   - remove あり → MAINTENANCE_REQUIRED
   - major upgrade 疑い → MAINTENANCE_REQUIRED

   MAINTENANCE_REQUIRED 検出時:
   → reports/proxmox-dryrun/ の最新 unified_dryrun.json と比較
   → 最新 dry-run から追加された更新候補を抽出
   → 差分を含むメールを送信して終了

   件名: [ABORT] target_node: 自動適用中止 - 更新内容が変化しました
   本文:
     - 中止理由（検出した重要コンポーネント名）
     - 最新 dry-run から追加された更新候補
     - 最新 dry-run から変わらない更新候補
     - 対応方法（dry-run を再実行してください）

6. apt-get dist-upgrade
   → 失敗したら fail（メール送信して終了）

7. /var/run/reboot-required の有無を確認

### Phase 3: reboot

8. reboot-required があれば自動 reboot
9. 最大 15 分待機
   → 15 分で応答なければ fail（メール送信して終了）
10. 起動確認

### Phase 4: 事後確認

11. post-healthcheck
    → NG でもメール送信して終了
12. summary mail（成功・失敗問わず送信）

## エラー時の動作

- Proxmox 操作は即停止
- その時点までの実行内容をレポートに記録
- エラー内容を含むメールを送信
- レポートは reports/proxmox-patch/ に保存

## メール内容（全ケース共通）

- 実行したノード
- 各 Phase の結果
- エラーが発生した場合はエラー内容
- apply 前後の healthcheck 結果

## 制約

- Codex CLI 呼び出しは含めない
- VM 退避・復帰は含めない
- ansy からの単独ノード実行を想定

## 前提

- core.md のルールに従う
- proxmox_patch_policy.md の方針に従う
- shell は収集と JSON 整形のみ、判断は Ansible 側
