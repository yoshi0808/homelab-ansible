# 実装内容: apply後のreboot判定改善

## 実装概要

`proxmox_patch_apply_node` の post-apply reboot 判定を改善した。

従来は `ansible.builtin.stat` で `/var/run/reboot-required` の存在のみを確認していたが、
カーネルパッケージの更新後にこのファイルが作成されない場合があり、reboot スキップが発生した。

新たに `proxmox-patch-apply-collect.sh` を作成し、
`/var/run/reboot-required` チェックに加えてインストール済みカーネルバージョンと
稼働中カーネルのバージョン比較を追加した。

## 変更内容の詳細

### proxmox-patch-apply-collect.sh（新規）

- Check 1: `/var/run/reboot-required` の存在確認
- Check 2: `uname -r` と `dpkg-query` で取得したインストール済みカーネルのバージョン比較
  - 対象パッケージ: `proxmox-kernel-6.17`, `proxmox-kernel-7.0`
  - `dpkg --compare-versions` でインストール済みが稼働中より新しい場合、`reboot_required=true`
- JSON 出力: `reboot_required`, `reboot_trigger`, `running_kernel`, `collected_at`
- `reboot_trigger` フィールドで「どちらの条件で true になったか」を記録（デバッグ用）
- shell 責務: 収集と判定のみ。reboot 実行はしない

### tasks/main.yml の変更点

1. 初期化ファクトに `_apply_collect: {}` を追加

2. predryrun スクリプト copy タスクの直後に post-apply collect スクリプトの copy タスクを追加

3. post-apply reboot チェックの置換:
   - 削除: `ansible.builtin.stat` による `/var/run/reboot-required` チェック
   - 追加: スクリプト実行 → JSON パース → `_reboot_required_before` セット
   - `_apply_collect.reboot_required | bool` を参照することで、両条件を網羅した判定を使用

4. レポートに `reboot_trigger` と `running_kernel` を追加（既存フィールドはそのまま維持）

5. メール本文の Phase 3 セクションに `reboot trigger` 行を追加

## 注意点

- `proxmox-kernel-6.17` / `proxmox-kernel-7.0` は要件で指定されたパッケージ名。
  将来バージョンが追加された場合は、スクリプトの `for pkg in ...` リストへの追記が必要。
- `_apply_collect` は `{}` で初期化しているため、スクリプト実行前に play が停止した場合も
  `_apply_collect.reboot_trigger | default('')` でレポート生成が安全に行える。
