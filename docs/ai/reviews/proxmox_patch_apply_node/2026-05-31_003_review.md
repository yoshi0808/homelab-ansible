## 結論

構文上の問題はない。

ただし、post-healthcheck リトライの待機タイミングが要求仕様・ポリシー文書の説明とずれているため、reboot 直後にサービス起動待ちをしたい目的を十分に満たせない可能性がある。

## 指摘

### 1. 高: CRITICAL 後の最初のリトライ前に待機しない

該当箇所:

- `roles/proxmox_patch_apply_node/tasks/main.yml:386`
- `roles/proxmox_patch_apply_node/tasks/main.yml:392`
- `roles/proxmox_patch_apply_node/tasks/main.yml:393`
- `docs/ai/prompts/proxmox_patch_policy.md:638`

要求仕様では「post-healthcheck が CRITICAL の場合、一定時間待機してから再度 healthcheck を実行する」とされている。

しかし現在の実装は、初回 post-healthcheck が CRITICAL になった直後に、`Retry post-healthcheck if CRITICAL after reboot` タスクの1回目を即時実行する。Ansible の `delay` は最初の実行前ではなく、`until` が満たされなかった後の次試行までの待機なので、最初のリトライ前には待たない。

そのため、デフォルト値 `proxmox_patch_apply_hc_retry_count: 2` / `proxmox_patch_apply_hc_retry_delay: 60` の実際の流れは、ポリシー文書に書かれている以下の流れと異なる。

```text
期待:
post-healthcheck CRITICAL
→ 60秒待機
→ リトライ1回目
→ 60秒待機
→ リトライ2回目

現在:
post-healthcheck CRITICAL
→ 即時リトライ1回目
→ 60秒待機
→ リトライ2回目
```

特に `proxmox_patch_apply_hc_retry_count: 1` にした場合、CRITICAL 後に待機せず即時再実行して終了するため、「待機してからリトライ」という設定にならない。

対応案:

- `Retry post-healthcheck if CRITICAL after reboot` の前に `pause` を入れる。
- または変数の意味を「リトライタスクの総試行回数」として文書を修正し、最初のリトライ前には待機しない仕様であることを明記する。

今回の目的は reboot 直後のサービス起動待ちなので、前者の方が要求に合う。

### 2. 中: リトライ中の healthcheck コマンド失敗を成功条件にしている

該当箇所:

- `roles/proxmox_patch_apply_node/tasks/main.yml:394`
- `roles/proxmox_patch_apply_node/tasks/main.yml:395`

`until` 条件の先頭が以下になっている。

```yaml
(_post_hc_retry_raw.failed | default(false) | bool) or
```

このため、リトライ中に `/usr/local/sbin/proxmox-healthcheck.sh` が失敗すると、`until` は成立した扱いになり、追加リトライせず停止する。

reboot 後の不安定時間帯を吸収するためのリトライであれば、healthcheck スクリプトの一時的な失敗も再試行対象にした方が自然である。現状では、初回 post-healthcheck が CRITICAL になった後、直後のリトライで一度でもコマンド失敗すると、待機せず CRITICAL 確定になりやすい。

対応案:

- コマンド失敗は `until` の成功条件に含めない。
- `until` は「JSON を取得でき、かつ全チェック項目が OK」のみにする。
- 最終的に成功条件を満たせなかった場合に `Re-determine post-healthcheck status after retry` で CRITICAL とする。

### 3. 中: メール通知の「必ず送信」にはまだ未カバーの失敗経路がある

該当箇所:

- `roles/proxmox_patch_apply_node/tasks/main.yml:41`
- `roles/proxmox_patch_apply_node/tasks/main.yml:540`

今回の変更で `_mail_subject` / `_mail_body` のフォールバック値が初期化され、`Set email subject and body` の Jinja2 展開失敗には強くなった。

一方で、`Load mail variables` は main block の外にあり、`ignore_errors` もない。このタスクで失敗すると `always` ブロックへ入らないため、summary mail 送信タスクまで到達しない。

また、`Send summary email` 自体は `ignore_errors: true` のため、SMTP 認証失敗・接続失敗・宛先設定ミスなどで送信されなくても playbook 上は検知しない。

今回の要求が「apply成功時のメール通知が届いていない問題も修正する」であるなら、テンプレート失敗だけでなく、少なくともメール変数読み込み失敗と送信失敗をどう扱うかを明確にした方がよい。

対応案:

- `Load mail variables` を block / always の管理下へ移す、または失敗時の扱いを明示する。
- summary mail 送信失敗を無視し続ける場合でも、レポートに mail status を残す。
- 「パッチ適用成功だがメール送信失敗」は playbook failure にするか、別通知経路で検知するかを決める。

## 確認できたこと

- `proxmox_patch_apply_hc_retry_count` / `proxmox_patch_apply_hc_retry_delay` は defaults に追加されている。
- リトライ条件は `_post_hc_status == 'CRITICAL' and (_apply_reboot_done | bool)` に限定されており、「reboot なしの CRITICAL はリトライしない」という方針に合っている。
- リトライ後に `_post_hc_status` を更新し、後続の final report と summary mail 件名が更新後ステータスを参照する構造になっている。
- `_mail_subject` / `_mail_body` のフォールバック初期値追加により、件名・本文生成タスク失敗時の未定義参照は回避される。
- `core.md` の monnie / monitoring_servers 追加と quory の役割整理は、今回の proxmox_patch_apply_node 実装とは直接衝突しない。

## 推奨修正

最小修正としては、CRITICAL 判定後のリトライタスク前に待機を入れ、`until` から command failed を成功条件として扱う分岐を外すのがよい。

例:

```yaml
- name: Wait before retrying post-healthcheck after reboot
  ansible.builtin.pause:
    seconds: "{{ proxmox_patch_apply_hc_retry_delay }}"
  when: _post_hc_status == 'CRITICAL' and (_apply_reboot_done | bool)

- name: Retry post-healthcheck if CRITICAL after reboot
  ansible.builtin.command:
    cmd: /usr/local/sbin/proxmox-healthcheck.sh
  register: _post_hc_retry_raw
  changed_when: false
  ignore_errors: true
  retries: "{{ proxmox_patch_apply_hc_retry_count }}"
  delay: "{{ proxmox_patch_apply_hc_retry_delay }}"
  until: >-
    ((_post_hc_retry_raw.stdout | default('') | length > 0) and
     ((_post_hc_retry_raw.stdout | from_json).cluster.quorate | default('') == 'Yes') and
     ((_post_hc_retry_raw.stdout | from_json).apt.check_ok | default(false) | bool) and
     ((_post_hc_retry_raw.stdout | from_json).services.pve_cluster | default('') == 'active') and
     ((_post_hc_retry_raw.stdout | from_json).services.corosync | default('') == 'active') and
     ((_post_hc_retry_raw.stdout | from_json).services.pvedaemon | default('') == 'active'))
  when: _post_hc_status == 'CRITICAL' and (_apply_reboot_done | bool)
```

この形なら、初回 CRITICAL 後にまず待機し、その後 `retry_count` 回ぶん再確認する動きになる。
