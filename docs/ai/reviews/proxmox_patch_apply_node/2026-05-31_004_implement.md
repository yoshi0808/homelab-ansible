# 実装内容: post-healthcheck リトライ修正（レビュー指摘対応）

実装日: 2026-05-31

## 対応レビュー

`docs/ai/reviews/proxmox_patch_apply_node/2026-05-31_003_review.md`

## 実施内容

### 指摘1（高）: CRITICAL後の最初のリトライ前に待機しない → 修正済み

`Retry post-healthcheck if CRITICAL after reboot` の直前に `pause` タスクを追加した。

`ansible.builtin.pause` の `delay` は試行と試行の間にしか入らないため、
最初のリトライ前には待機が発生していなかった。
`pause` タスクを別タスクとして追加することで、
CRITICAL確定後に必ず `retry_delay` 秒待機してからリトライを開始する。

```
修正前:
  post-healthcheck CRITICAL → 即時リトライ1回目 → 60秒待機 → リトライ2回目

修正後:
  post-healthcheck CRITICAL → 60秒待機(pause) → リトライ1回目 → 60秒待機 → リトライ2回目
```

### 指摘2（中）: リトライ中の healthcheck コマンド失敗を成功条件にしている → 修正済み

`until` の先頭にあった `(_post_hc_retry_raw.failed | default(false) | bool) or` を削除した。

この条件があると、リトライ中にhealthcheckスクリプトが一時的に失敗した場合、
`until` が満たされた扱いになりリトライが停止していた。
reboot後の不安定時間帯を吸収するためのリトライで、
コマンド失敗も再試行対象とすることで意図通り動作するようになる。

コマンド失敗の場合は `Re-determine post-healthcheck status after retry` タスクで
`_post_hc_retry_raw.failed | bool` を検出して CRITICAL とするため、
最終判定は変わらない。

### 指摘3（中）: メール通知の未カバー失敗経路 → 最小対応を実施

`Load mail variables` タスクに `ignore_errors: true` を追加した。

このタスクは `block/rescue/always` の外側にあり、
失敗すると `always` ブロックに入らずレポートもメールも送信されない。
`ignore_errors: true` を付けることで:
- ファイルが見つからない・権限エラーなど → タスクが失敗してもplay続行、`always` が実行される
- SMTP変数が未定義のままメールタスクへ → `Send summary email` は `ignore_errors: true` のため失敗しても続行
- レポートJSONは保存される（`Build and save final report` も `ignore_errors: true`）

「メール送信失敗の検知経路をどう作るか」はスコープ外として、今回は最小対応とする。

## 変更後の動作フロー（デフォルト: retry_count=2, retry_delay=60）

```text
reboot完了
→ post-healthcheck 1回目（初回）: CRITICAL
→ 60秒待機（pause）
→ post-healthcheck 2回目（リトライ試行1）: CRITICAL
→ 60秒待機（delay）
→ post-healthcheck 3回目（リトライ試行2）: OK → _post_hc_status = OK
→ summary mail 件名: [SUCCESS] ...
```

最大待機時間 = (retry_count + 1) × retry_delay = (2+1) × 60 = 180秒
