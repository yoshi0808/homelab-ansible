## Findings

### 1. 高: target_node 未指定時は pre-play の assert まで到達しません

`playbooks/proxmox_patch_apply_node.yml:17` に localhost pre-play が追加され、`target_node` が `pve1` / `pve2` であることを main play 前に検証する意図になっています。

しかし、`target_node` 未指定時は `playbooks/proxmox_patch_apply_node.yml:29` の main play `hosts: "{{ target_node }}"` 評価で先に落ちます。

実際に `target_node` 未指定で syntax-check すると、pre-play の assert failure ではなく以下のエラーになります。

```text
Error processing keyword 'hosts': 'target_node' is undefined
```

つまり、007_review の「未指定時や gather_facts 前の事故を防ぐ」という観点では、未指定ケースはまだ解消されていません。

対策としては、main play の hosts 側でも default を使い、未指定時に playbook parser が落ちないようにする必要があります。

例:

```yaml
hosts: "{{ target_node | default('__invalid_target__') }}"
```

この形であれば、未指定時は pre-play の assert で意図したエラーメッセージを出せます。

### 2. 低: invalid target は通常実行では pre-play で止まる想定だが、syntax-check では検証されません

`target_node=foo` の syntax-check は成功します。

これは syntax-check が task の assert を実行しないためであり、通常実行では `playbooks/proxmox_patch_apply_node.yml:21` の assert で止まる想定です。

大きな問題ではありませんが、検証記録では「syntax-check OK」は target allowlist の動作確認ではない点を明記しておくと誤解が減ります。

## 改善確認

### 1. VM/CT 残存チェックは改善されています

`roles/proxmox_patch_apply_node/tasks/main.yml:62` が `pvesh get /cluster/resources --output-format json` に変更され、type フィルターなしで cluster resources を取得するようになりました。

そのうえで、`roles/proxmox_patch_apply_node/tasks/main.yml:92` では `type in ['qemu', 'lxc']` を条件にして running VM/CT を検出しています。

007_review の「CT をすり抜ける可能性」は解消されています。

### 2. control node チェックは VM のみに限定されました

`roles/proxmox_patch_apply_node/tasks/main.yml:68` では `type == 'qemu'` を条件にして control node VM を検出しています。

現行要求の「control node VM を検出」には合っています。

ただし、control node 名固定・部分一致の脆さは 006_implement 側でも将来課題として扱われており、今回の修正対象外です。

### 3. メール本文の apply 未実施条件は修正されています

`roles/proxmox_patch_apply_node/tasks/main.yml:466` の apply セクション表示条件に、以下が追加されました。

- `vms_not_evacuated`
- `manual_not_confirmed`

これにより、該当 abort 時に apply セクションへ「適用パッケージ数」「reboot 実施」が表示される問題は解消されています。

## 残っている既知リスク

以下は今回の 008_implement では対象外であり、引き続き既知リスクです。

- 通知保証は best-effort のまま
- control node チェックの name 固定・部分一致の脆弱性
- post-healthcheck 判定が healthcheck role と一致していない
- `MAJOR_UPGRADE_DETECTED` / `BLOCKED` 区別が粗い
- manual confirm 文字列が `MAINTENANCE_REQUIRED` のみで、target node を含まない

## 結論

007_review の指摘 2 と 3 は解消されています。

一方、指摘 1 の `target_node` 検証は、invalid target の通常実行には効く想定ですが、`target_node` 未指定時には main play の `hosts` 評価で pre-play より先に落ちます。

次に直すべき点は、main play の `hosts` に `default` を入れて、未指定時も pre-play の assert で意図したエラーにすることです。
