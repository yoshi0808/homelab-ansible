## 結論

前回レビュー `2026-05-31_009_review.md` の主要指摘は、おおむね解消されている。

- OK 後の後続 retry iteration は、`retry_healthcheck_once.yml` 内の block guard により pause / command / parse / determine がスキップされる。
- retry 判定は rc ではなく JSON 内の health 状態を見る形に戻っている。
- mail vars 不備は apply 前の assert で止まる。
- syntax-check は OK。

残る指摘は、メール変数 assert が「定義済み」だけを見ていて空値を通す点と、ポリシー文書の説明が block guard 方式の実装と少しずれている点。

## 指摘

### 1. 中: mail vars assert が空値を通すため、設定不備のまま apply へ進む可能性がある

該当箇所:

- `roles/proxmox_patch_apply_node/tasks/main.yml:53`
- `roles/proxmox_patch_apply_node/tasks/main.yml:56`
- `roles/proxmox_patch_apply_node/tasks/main.yml:60`
- `roles/proxmox_patch_apply_node/tasks/main.yml:521`
- `roles/proxmox_patch_apply_node/tasks/main.yml:560`

今回 `Assert required mail variables are defined` が追加され、mail vars が未定義なら apply 前に止まるようになった。この方向はよい。

ただし現在の assert は `is defined` のみを確認しているため、以下のような値は通過する。

```yaml
smtp_host: ""
smtp_port: ""
smtp_user: ""
smtp_password: ""
mail_to: ""
```

この場合、`apt-get -y dist-upgrade` まで進んだ後、最後の `Send summary email` で失敗し、report 作成後に `Re-raise failure` で play が失敗する。つまり「メール送信できないことが事前に分かる設定」でも unsafe apply が走りうる。

要求仕様は「apply成功時も必ずsummary mailが送信されることを確認・修正する」なので、mail vars は未定義だけでなく空値も apply 前に弾く方が安全。

対応案:

```yaml
that:
  - smtp_host is defined
  - smtp_host | string | length > 0
  - smtp_port is defined
  - smtp_port | int > 0
  - smtp_user is defined
  - smtp_user | string | length > 0
  - smtp_password is defined
  - smtp_password | string | length > 0
  - mail_to is defined
  - mail_to | string | length > 0
```

`smtp_password` の空値を許容したい運用がある場合は、明示的な変数で例外化するのがよい。

### 2. 低: ポリシー文書の retry 説明に古い `delay` 表現が残っている

該当箇所:

- `docs/ai/prompts/proxmox_patch_policy.md:631`
- `docs/ai/prompts/proxmox_patch_policy.md:638`
- `docs/ai/prompts/proxmox_patch_policy.md:647`
- `docs/ai/prompts/proxmox_patch_policy.md:649`

実装は「各 retry iteration の先頭で pause する」方式になっている。

一方、ポリシー文書の動作例には以下のような古い表現が残っている。

```text
→ post-healthcheck 2回目: CRITICAL   （リトライタスク 試行1、即時実行）
→ 60秒待機                            （delay）
```

現在の実装では Ansible `delay` は使っておらず、各試行の冒頭に `pause` が入る。文書上の「即時実行」「delay」は、読む人に `retries/delay` 実装が残っているような印象を与える。

対応案:

```text
→ 60秒待機
→ post-healthcheck 2回目: CRITICAL   （リトライ試行1）
→ 60秒待機
→ post-healthcheck 3回目: OK         （リトライ試行2）
```

のように、実装に合わせて `pause` ベースの表現へ統一する。

## 確認できたこと

- `retry_healthcheck_once.yml` の block に `when: _post_hc_retry_status != 'OK'` があり、OK 後の後続 iteration では block 内の pause / command / parse / determine が実行されない。
- block 内の先頭で `_post_hc_retry_status: CRITICAL` と `_post_hc_retry_json: {}` を reset しており、試行ごとの前回値残り対策が入っている。
- JSON key 参照に `default()` が追加され、欠損 JSON は CRITICAL に畳み込まれる。
- main 側の include loop から `_post_hc_retry_status != 'OK'` 条件が外され、早期停止の責務が task file 内へ移っている。
- mail vars の存在確認 assert が apply 前に追加されている。
- メール送信結果は final report の `mail_sent` / `mail_error` に保存され、送信失敗時は最後に fail する。
- `core.md` の `monnie` / `monitoring_servers` 追加は、今回の role 差分とは直接衝突しない。

### include_tasks + block guard の挙動確認

確認内容:

- include された task file の block に `when: loop_status != 'OK'` を設定
- item 1 で `loop_status: OK` を set_fact
- item 2 の block 内 task が実行されるか確認

結果:

```text

skipping: [localhost]

TASK [Show iteration]  # item=2
skipping: [localhost]
```

block guard は後続 iteration で効いている。

`sample_json.services.pve_cluster | default('')` の形で、親 key が欠損しても空文字へ落ちることをローカル playbook で確認した。

## 推奨修正

大きなリトライ制御の問題は解消されているため、次の修正は小さくてよい。

- mail vars assert を空値も弾く条件へ強化する。
- `proxmox_patch_policy.md` の retry 動作例から `即時実行` / `delay` 表現を取り除き、各試行冒頭の `pause` ベースに統一する。
