## 結論

syntax-check は通るが、今回の retry 実装には重大な挙動不一致がある。

`include_tasks` に `loop` と `when` を付けても、loop 内で `_post_hc_retry_status` が `OK` になった後に残りの iteration がスキップされる、という実装メモの前提は成り立たない。少なくともローカル確認では、include は先に全 iteration 分展開され、include された task file 側の `set_fact` 結果で後続 include は止まらなかった。

そのため、いったん OK になっても残りのリトライが実行され、最終試行の結果で `_post_hc_status` が上書きされる可能性がある。

## 指摘

### 1. 高: OK 後に残りの retry iteration が止まらず、最終結果が後続試行で上書きされる

該当箇所:

- `roles/proxmox_patch_apply_node/tasks/main.yml:394`
- `roles/proxmox_patch_apply_node/tasks/main.yml:400`
- `roles/proxmox_patch_apply_node/tasks/main.yml:403`
- `roles/proxmox_patch_apply_node/tasks/retry_healthcheck_once.yml:6`
- `roles/proxmox_patch_apply_node/tasks/retry_healthcheck_once.yml:29`
- `roles/proxmox_patch_apply_node/tasks/main.yml:405`
- `docs/ai/reviews/proxmox_patch_apply_node/2026-05-31_008_implement.md`

実装メモでは以下の前提になっている。

```text
_post_hc_retry_status が OK になると以降の iteration は when: false でスキップされる
```

しかし `include_tasks` の loop に付けた `when` は、include の展開時に評価される。include された task file の中で `set_fact` した値により、すでに展開された後続 iteration は止まらない。

ローカルで最小 playbook を作って確認したところ、item=1 で `loop_status: OK` を set_fact しても、item=2 の include は実行された。

今回の実装では各 retry iteration の冒頭で必ず pause し、その後 healthcheck を再実行するため、以下の問題が起きる。

- 2回目で OK になっても、残りの retry iteration が実行される
- OK 後も余計に `retry_delay` 秒待つ
- 後続試行で一時的に CRITICAL / parse 失敗になると、`_post_hc_retry_status` が CRITICAL に戻る
- 最後の `Re-determine post-healthcheck status after retry` で `_post_hc_status` が最終試行の値に上書きされる

これは要求仕様の「再確認でOKになった場合はSUCCESSとして扱う」に反する。

対応案:

- `include_tasks + loop + when` で早期停止しようとしない。
- 早期停止が必要なら、1回分の retry task の先頭に以下のような guard を入れ、OK 済みなら pause / command / parse を実行しない。

```yaml
- name: Skip retry attempt when already OK
  ansible.builtin.meta: noop
  when: _post_hc_retry_status == 'OK'
```

ただし `meta: noop` だけでは後続 task は止まらないので、実際には各 task に `when: _post_hc_retry_status != 'OK'` を付けるか、block に guard を付ける必要がある。

より堅い案:

```yaml
- block:
    - pause: ...
    - command: ...
    - set_fact: ...
  when: _post_hc_retry_status != 'OK'
```

を `retry_healthcheck_once.yml` 内に置く。

### 2. 中: 不正JSONや欠損JSON時に status 判定 task が失敗すると、前回値が残る

該当箇所:

- `roles/proxmox_patch_apply_node/tasks/retry_healthcheck_once.yml:21`
- `roles/proxmox_patch_apply_node/tasks/retry_healthcheck_once.yml:24`
- `roles/proxmox_patch_apply_node/tasks/retry_healthcheck_once.yml:29`
- `roles/proxmox_patch_apply_node/tasks/retry_healthcheck_once.yml:35`
- `roles/proxmox_patch_apply_node/tasks/retry_healthcheck_once.yml:44`

各 iteration の冒頭で `_post_hc_retry_json: {}` に reset している点はよい。

ただし、parse は `ignore_errors: true`、status 判定も `ignore_errors: true` になっている。parse 結果が `{}` なら CRITICAL に落ちるが、例えば非空だが期待 key が欠けた JSON の場合、以下の参照で Jinja 評価が失敗しうる。

```yaml
_post_hc_retry_json.cluster.quorate
_post_hc_retry_json.apt.check_ok
_post_hc_retry_json.services.pve_cluster
```

この status 判定 task が失敗して ignore されると、`_post_hc_retry_status` はその iteration で更新されず、前回値が残る。

特に指摘1の問題と組み合わさると、過去に OK になった値が残ったまま後続 iteration の異常出力を無視する、または過去の CRITICAL が残ったまま最終結果になるなど、最終判定が「今回の試行結果」ではなく「前回の fact」に依存する。

対応案:

- status 判定前に `_post_hc_retry_status: CRITICAL` も毎回 reset する。
- JSON の key 参照は `default` を使い、欠損時も task failure ではなく CRITICAL に畳み込む。

例:

```yaml
(_post_hc_retry_json.cluster.quorate | default('')) == 'Yes'
(_post_hc_retry_json.apt.check_ok | default(false) | bool)
```

### 3. 中: mail vars 読み込み失敗時も unsafe apply へ進む

該当箇所:

- `roles/proxmox_patch_apply_node/tasks/main.yml:46`
- `roles/proxmox_patch_apply_node/tasks/main.yml:51`
- `roles/proxmox_patch_apply_node/tasks/main.yml:506`
- `roles/proxmox_patch_apply_node/tasks/main.yml:519`
- `roles/proxmox_patch_apply_node/tasks/main.yml:545`
- `roles/proxmox_patch_apply_node/tasks/main.yml:554`

今回の変更で、メール送信失敗時は最後に play を fail させるようになった。この点は前回より改善している。

ただし `Load mail variables` は `ignore_errors: true` のまま role の序盤にあり、SMTP設定が読めなくても Phase 2 の `apt-get -y dist-upgrade` まで進む。結果として、「summary mail を送れないことが事前に分かっている状態」で unsafe な patch apply を実行し、最後に mail failed として fail する。

要求仕様が「apply成功時も必ず summary mail が送信されること」であるなら、メール設定不備は apply 前に検出して止める方が安全。

対応案:

- `Load mail variables` の `ignore_errors: true` を外す。
- もしくは読み込み結果を register し、main block に入る前、少なくとも apply 前に `smtp_host` / `smtp_port` / `smtp_user` / `smtp_password` / `mail_to` の定義を assert する。
- メールなし実行を許容するなら、`proxmox_patch_apply_mail_required: true` のような明示変数を持たせる。

## 推奨修正

最優先は `retry_healthcheck_once.yml` 側へ guard を移すこと。

具体的には、pause / command / parse / determine を block に入れ、block 全体に以下を付ける。

```yaml
when: _post_hc_retry_status != 'OK'
```

さらに各試行の冒頭で `_post_hc_retry_status: CRITICAL` と `_post_hc_retry_json: {}` を reset し、JSON key 欠損時も `default` で CRITICAL に落とすと、前回値残りのリスクを抑えられる。

メールについては、送信失敗を最後に fail する方針を維持するなら、メール設定の読み込み・必須変数チェックは apply 前に fail-fast するのがよい。
