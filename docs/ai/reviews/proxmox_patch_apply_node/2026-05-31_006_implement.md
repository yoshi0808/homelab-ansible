## 実施内容

### 指摘1（中）: until に from_json を直接評価しており、不正JSONで再試行されない → 修正済み

**問題**

`until` 条件内で `_post_hc_retry_raw.stdout | from_json` を複数回評価していた。
healthcheck スクリプトが不正JSON・空文字列を返した場合、
Jinja2 テンプレート評価が失敗し、`until` が false ではなく「タスク失敗」として扱われる。
`ignore_errors: true` により play は続くが、残りのリトライが実行されなくなる。

**修正**

`until` を `_post_hc_retry_raw.rc | default(1) | int == 0` に変更した。

- rc != 0 (スクリプト失敗) → リトライ継続
- rc == 0 (スクリプト成功) → リトライ停止、`Re-determine` タスクがJSONを確認

`from_json` は `until` から完全に除外し、JSON パースは `Re-determine post-healthcheck status after retry` タスク（`ignore_errors: true` 付き）のみで行う。

**トレードオフ**

スクリプトが rc=0 を返すが JSON が CRITICAL を示す場合（サービス起動中だが一部チェック失敗）、
リトライは止まる。ただし reboot 直後の最も典型的な失敗パターン（スクリプト自体が失敗）は
引き続きリトライの対象となる。

### 指摘2（中）: Load mail variables の ignore_errors 追加は「必ず届く」修正になっていない → 追加修正

**問題**

`Load mail variables` に `ignore_errors: true` を付けると、
mail.yml の読み込み失敗で SMTP 変数が未定義のままメール送信タスクへ進み、
送信失敗が無音になる。

**修正: 3点**

1. **`_mail_send_result` 初期値を設定**（init set_fact に追加）

```yaml
_mail_send_result:
  failed: true
  msg: "mail send not attempted"
```

送信タスクに到達できなかった場合でも、レポートに `mail_sent: false` が残る。

2. **`always` ブロックの順序を変更**

変更前: Build report → Set subject/body → Send email
変更後: Set subject/body → Send email（register 追加） → Build report（mail_sent 追加）

送信が完了してからレポートを保存することで、レポートに送信結果を含められる。

3. **`Send summary email` に `register: _mail_send_result` を追加**

4. **レポート JSON に `mail_sent` / `mail_error` フィールドを追加**

```json
{
  "mail_sent": true,
  "mail_error": ""
}
```

`mail_sent: false` の場合は `mail_error` に理由が入る。
送信タスクが届かなかった場合（init値）は `mail_sent: false, mail_error: "mail send not attempted"` になる。

### 指摘3（低）: retries 解釈の説明 → 前回実装で対応済み

`proxmox_patch_policy.md` と `defaults/main.yml` コメントはすでに
`retries: N` = 最大 N+1 回実行、最大待機 (N+1)×delay 秒 と記載済み。

```text
1. Run post-healthcheck script         （変更なし）
2. Parse post-healthcheck output       （変更なし）
3. Determine post-healthcheck status   （変更なし）
4. Wait before retrying               （変更なし）
5. Retry post-healthcheck             （until を rc==0 チェックに変更）
6. Re-determine status after retry    （変更なし）
7. Set email subject and body         （順序変更: 旧8番目 → 新7番目）
8. Send summary email                 （順序変更 + register: _mail_send_result 追加）
9. Build and save final report        （順序変更 + mail_sent / mail_error フィールド追加）
10. Re-raise failure                  （変更なし）
```
