## Findings

### 1. 重大: エラー時にメール・レポートが必ず送られる保証がありません

`roles/proxmox_patch_apply_node/tasks/main.yml:24` の mail vars 読み込みと `roles/proxmox_patch_apply_node/tasks/main.yml:30` の report directory 作成は `block` の外なので、ここで失敗すると `rescue/always` に入りません。

さらに `playbooks/proxmox_patch_apply_node.yml:10` の `gather_facts` 失敗や target 到達不能も role 前で止まり、通知されません。

また `roles/proxmox_patch_apply_node/tasks/main.yml:425` のメール送信自体が `ignore_errors: true` なので、送信失敗しても検知されません。

要求仕様の「失敗時もメール送信」「summary mail（成功・失敗問わず送信）」には未達です。

### 2. 重大: reboot 後の到達不能で always 内の通知処理が途切れる可能性があります

`roles/proxmox_patch_apply_node/tasks/main.yml:281` の post-healthcheck は target host 上で実行されます。

reboot timeout や SSH 不通時、`ignore_errors` は通常の failed には効きますが unreachable は別扱いになり得ます。その場合、後続の localhost delegate の report/mail まで進まない可能性があります。

`block/rescue/always` の形自体はありますが、「失敗時も必ず最後に localhost から report/mail」にはなっていません。

### 3. 重大: unsafe playbook として必要な実行条件チェックが不足しています

`proxmox_patch_policy.md` の `proxmox_patch_apply_node.yml` 実行条件は、対象ノード healthcheck だけでなく、以下を求めています。

- 反対側ノードの healthcheck が OK
- 最新 dry-run Status が `PATCH_READY`
- control node が対象ノード上にいない
- 対象ノード上の VM / CT が退避済み
- `MAINTENANCE_REQUIRED` / `BLOCKED` / `MAJOR_UPGRADE_DETECTED` ではない

現状は `roles/proxmox_patch_apply_node/tasks/main.yml:80` で対象ノード healthcheck、`roles/proxmox_patch_apply_node/tasks/main.yml:200` で再 dry-run の簡易判定のみです。

反対側ノードの healthcheck、最新 final/dry-run status の `PATCH_READY` 確認、VM / CT 残存確認がありません。

unsafe playbook としてはガードが薄いです。

### 4. 重大: apply_mode=manual で MAINTENANCE_REQUIRED を続行する実装は policy と衝突します

追加要求 `2026-05-09_004_implement.md` には合っていますが、`proxmox_patch_policy.md` 6.5 は `proxmox_patch_apply_node.yml` について「`MAINTENANCE_REQUIRED` を適用しない」と明記しています。

実装は `roles/proxmox_patch_apply_node/defaults/main.yml:14` で `manual` を許可し、`roles/proxmox_patch_apply_node/tasks/main.yml:209` で warning のみで `apt-get -y dist-upgrade` へ進みます。

`MAINTENANCE_REQUIRED` 手動適用は policy 上は別メンテナンス枠の扱いなので、この playbook の単純な変数切替で続行できるのは危険です。

### 5. 高: control node チェックは要求の最低限は満たしますが、実運用上は脆いです

`roles/proxmox_patch_apply_node/tasks/main.yml:48` で `pvesh get /cluster/resources --type vm` を使い、`roles/proxmox_patch_apply_node/tasks/main.yml:54` で `name` に `ansy` を含む VM を検出しています。

要求仕様の「VM から ansy を検出」には概ね合っています。

ただし実際の Ansible control node を動的に確認しておらず、`proxmox_patch_apply_control_node_name: "ansy"` 固定です。名前の部分一致なので誤検知・見逃しもあります。CT や将来 quory / 別名 control node には対応できません。

### 6. 高: target_node の検証がありません

`playbooks/proxmox_patch_apply_node.yml:9` は `hosts: "{{ target_node }}"` ですが、`pve1` / `pve2` のみ許可する assert がありません。

unsafe playbook なので、未指定・誤指定・グループ指定を早期に止めるべきです。

### 7. 中: post-healthcheck 判定が healthcheck role の判定と一致しません

pre は `roles/proxmox_patch_apply_node/tasks/main.yml:80` で `proxmox_healthcheck` role を使いますが、post は `roles/proxmox_patch_apply_node/tasks/main.yml:281` で script を直接実行し、`roles/proxmox_patch_apply_node/tasks/main.yml:294` で一部項目だけを独自判定しています。

ZFS、replication、failed units、root filesystem、VM / CT 状態など、role 側の判定と差が出る可能性があります。

### 8. 中: MAJOR_UPGRADE_DETECTED / BLOCKED の扱いが policy より粗いです

`roles/proxmox_patch_apply_node/tasks/main.yml:165` では package 数しきい値・重要コンポーネント・remove をすべて `_redryrun_is_mr` に畳み込んでいます。

要求 001 では major upgrade 疑いを `MAINTENANCE_REQUIRED` に寄せていますが、policy では `MAJOR_UPGRADE_DETECTED` と `BLOCKED` は別 status です。

通知件名や report 上も区別されません。

## 観点別レビュー

### core.md / patch_policy.md の方針に違反していないか

`core.md` の shell 責務分離について、`roles/proxmox_patch_apply_node/files/proxmox-patch-predryrun-collect.sh` は収集と JSON 整形のみで、正常 / 異常分類や apply 判断はしていません。この点は適合しています。

一方で `proxmox_patch_policy.md` については、unsafe playbook の実行条件不足と `MAINTENANCE_REQUIRED` 続行が大きな不整合です。

特に policy 6.5 の「`PATCH_READY` の場合のみ」「`MAINTENANCE_REQUIRED` を適用しない」と、`apply_mode=manual` の続行動作が衝突しています。

### block/rescue/always 構造が正しく機能するか

通常の task failure については `rescue` で `_apply_failed: true` を立て、`always` で post-healthcheck / report / email / re-fail へ進む構造になっています。

ただし以下は保証外です。

- role 開始前の `gather_facts` 失敗
- `block` 外の mail vars 読み込み失敗
- `block` 外の report directory 作成失敗
- target host unreachable
- reboot 後 unreachable
- always 内の post-healthcheck unreachable
- localhost 側の report/mail 生成前に変数未定義エラーが起きるケース
- メール送信失敗

したがって「構造はあるが、要求された保証レベルには達していない」という評価です。

### apply_mode=manual 時に MAINTENANCE_REQUIRED で続行する動作は正しいか

追加要求 `2026-05-09_004_implement.md` 単体では、`manual` 時に warning を出して続行する実装になっています。

しかし `proxmox_patch_policy.md` では `proxmox_patch_apply_node.yml` は `PATCH_READY` のみを扱い、`MAINTENANCE_REQUIRED` を適用しないとされています。

そのため、policy 準拠レビューとしては正しくありません。

もし `manual` を維持するなら、通常の `proxmox_patch_apply_node.yml` ではなく、明示的に別 playbook / 別変数名 / 強い確認ゲート / メンテナンス専用 report を持つべきです。

### control node チェックの実装は正しいか

要求仕様の「`pvesh get /cluster/resources --type vm` で検出」という点には沿っています。

ただし検出対象が VM の `name` 部分一致かつ `ansy` 固定なので、以下に弱いです。

- 実行元 control node が ansy 以外の場合
- control node が CT の場合
- VM 名が変わった場合
- `ansy-backup` など部分一致による誤検知
- 実際の Ansible 実行元と Proxmox VM 名の対応が崩れた場合

policy の趣旨である「patch 対象ノードを reboot した場合に control node が止まらないこと」の保証としては不十分です。

### エラー時にメール・レポートが必ず送られる保証があるか

ありません。

通常の task failure では送る意図がありますが、`block` 外の失敗、unreachable、メール送信失敗、localhost 側の準備失敗では保証できません。

さらに `community.general.mail` が `ignore_errors: true` なので、メール送信失敗を playbook 失敗として扱うこともできません。

### unsafe playbook として問題がないか

問題があります。

特に以下が不足しています。

- `target_node` の許可リスト検証
- 単一ノード実行の強制
- 反対側ノード healthcheck OK の確認
- 最新 dry-run final status が `PATCH_READY` であることの確認
- 対象ノード上の running VM / CT が退避済みであることの確認
- `BLOCKED` / `MAJOR_UPGRADE_DETECTED` の明示的停止
- `MAINTENANCE_REQUIRED` 手動適用を通常 apply playbook から分離する設計
- reboot 後 unreachable でも localhost から report/mail を出す構造

---

## 結論

現状の `block/rescue/always` は「通常の task failure を拾って最後に処理する」形にはなっていますが、unreachable、role 前の失敗、always 内の post-healthcheck 不通、メール送信失敗までは保証できません。

また、`apply_mode=manual` は追加要求には沿っていますが、`proxmox_patch_policy.md` の unsafe 分離と `MAINTENANCE_REQUIRED を apply_node で適用しない` 方針には反しています。

unsafe playbook としては、実行前ガードをかなり強くしないと危険な状態です。
