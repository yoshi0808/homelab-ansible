## 結論

syntax-check は通るが、post-healthcheck リトライは要求仕様を満たしていない。

今回 `until` が `rc == 0` に変更されたことで、`from_json` 例外によるリトライ停止は避けられる一方、healthcheck が valid JSON を返し、その中身が CRITICAL のままでもリトライが止まる。

この role が解決したい主問題は「reboot 直後、サービスが起動しきる前に healthcheck が CRITICAL になる」なので、rc ではなく JSON 内の health 判定が OK になるまで待つ必要がある。

## 指摘

### 1. 高: `until: rc == 0` では CRITICAL が続いていてもリトライしない

該当箇所:

- `roles/proxmox_patch_apply_node/tasks/main.yml:396`
- `roles/proxmox_patch_apply_node/tasks/main.yml:404`
- `roles/proxmox_patch_apply_node/tasks/main.yml:407`
- `roles/proxmox_healthcheck/files/proxmox-healthcheck.sh:23`
- `roles/proxmox_healthcheck/files/proxmox-healthcheck.sh:156`
- `docs/ai/prompts/proxmox_patch_policy.md:622`
- `docs/ai/prompts/proxmox_patch_policy.md:648`

`Retry post-healthcheck if CRITICAL after reboot` の成功条件が以下になっている。

```yaml
until: _post_hc_retry_raw.rc | default(1) | int == 0
```

しかし `proxmox-healthcheck.sh` は、`pve-cluster` / `corosync` / `pvedaemon` が inactive でも JSON を生成して終了する。shell は「収集と JSON 整形のみ」を担当しており、health 判定は Ansible 側で行う設計なので、この挙動自体は自然。

つまり、reboot 直後に以下のような CRITICAL 相当の JSON が返るケースでも、スクリプトの rc は 0 になりうる。

```json
{
  "cluster": { "quorate": "unknown" },
  "services": {
    "pve_cluster": "inactive",
    "corosync": "inactive",
    "pvedaemon": "inactive"
  },
  "apt": { "check_ok": true }
}
```

その場合、リトライタスクは1回目で即停止し、`delay` による追加待機も追加試行も行われない。これは要求仕様の「CRITICALの場合に待機してからリトライする」「再確認でOKになった場合はSUCCESSとして扱う」と合わない。

実装メモではトレードオフとして「rc=0 だが JSON が CRITICAL を示す場合、リトライは止まる」と書かれているが、これは今回の主対象そのものに近い。特にサービス起動待ちでは、スクリプト自体は成功しつつ `services.* != active` になる可能性が高い。

対応案:

- `until` は rc ではなく「parse 済み JSON の OK 判定」を見る。
- `from_json` 例外を避けるため、リトライ対象処理を `include_tasks` などへ分け、各試行で以下を fact 化する。

```yaml
_post_hc_retry_parse_ok: true/false
_post_hc_retry_status: OK/CRITICAL
```

そのうえで:

```yaml
until: _post_hc_retry_status == 'OK'
```

のように、JSON不正・空stdout・rc != 0 はすべて `CRITICAL` 扱いで再試行対象にする。

### 2. 中: ポリシー文書のリトライ説明と実装が一致していない

該当箇所:

- `docs/ai/prompts/proxmox_patch_policy.md:622`
- `docs/ai/prompts/proxmox_patch_policy.md:648`
- `docs/ai/prompts/proxmox_patch_policy.md:656`
- `roles/proxmox_patch_apply_node/tasks/main.yml:404`

ポリシー文書では、CRITICAL 後に待機し、リトライタスクの試行1が CRITICAL ならさらに delay 後に試行2へ進む例になっている。

```text
→ post-healthcheck 2回目: CRITICAL
→ 60秒待機
→ post-healthcheck 3回目: OK
```

しかし現在の実装は `rc == 0` で until 成功になるため、post-healthcheck 2回目の JSON が CRITICAL でも rc=0 ならそこで止まる。

文書の「すべての試行でCRITICALのままだった場合は CRITICALとして報告する」という説明も、実装上は「最初の rc=0 の試行が CRITICAL ならそこで報告する」に近い。

対応案:

- 指摘1の修正後、文書は現状の説明を維持できる。
- もし rc ベースで止める仕様にするなら、ポリシー文書側に「CRITICAL JSON は再試行しない」と明記する必要がある。ただし要求仕様には合わないため非推奨。

### 3. 中: メール送信結果は report に残るが、「apply成功時も必ず送信」にはまだ届いていない

該当箇所:

- `roles/proxmox_patch_apply_node/tasks/main.yml:44`
- `roles/proxmox_patch_apply_node/tasks/main.yml:49`
- `roles/proxmox_patch_apply_node/tasks/main.yml:521`
- `roles/proxmox_patch_apply_node/tasks/main.yml:531`
- `roles/proxmox_patch_apply_node/tasks/main.yml:552`
- `roles/proxmox_patch_apply_node/tasks/main.yml:553`

今回の変更で、`Send summary email` の結果を `_mail_send_result` に register し、final report に `mail_sent` / `mail_error` を残すようになった。この点は前回より改善している。

ただし `Load mail variables` と `Send summary email` はどちらも `ignore_errors: true` のままなので、SMTP設定不足・認証失敗・送信失敗があっても playbook は成功方向に進む。これは「送信失敗を可観測にする」修正であって、「apply成功時も必ずsummary mailが送信される」修正ではない。

`mail_sent: false` が report に残るだけでは、メールが届かないという元の問題を利用者がリアルタイムに気づけない可能性がある。

対応案:

- apply が成功し、メール送信が失敗した場合は最後に fail する。
- あるいは運用仕様を「メール送信失敗は report に記録し、playbook 自体は成功扱い」と明文化する。
- 少なくとも `Load mail variables` 失敗時は apply 前に止めるか、`mail_enabled` のような明示的な無効化変数を導入した方がよい。

## 確認できたこと

- `Wait before retrying post-healthcheck after reboot` により、最初のリトライ前の待機は維持されている。
- `until` から `from_json` は除外されており、不正JSONによる until 評価失敗は避けられている。
- `_mail_send_result` の初期値、メール送信結果の register、report への `mail_sent` / `mail_error` 追加は入っている。
- final report はメール送信後に保存されるようになり、送信結果を含められる順序になっている。
- `core.md` の `monnie` / `monitoring_servers` 追加は、今回の role 差分とは直接衝突しない。

## 推奨修正

最優先は、リトライの `until` を `rc == 0` ではなく「post-healthcheck 判定が OK」へ戻すこと。

ただし `from_json` を `until` に直書きすると前回指摘の問題が再発するため、1回分の healthcheck 実行・parse・status 判定を小さな task file に切り出し、その task file を retry する構造がよい。

メールについては、今回の report 記録は有用なので残しつつ、送信失敗時の扱いを「fail する」のか「report のみ」のか、要求仕様またはポリシーに合わせて明確化するのがよい。
