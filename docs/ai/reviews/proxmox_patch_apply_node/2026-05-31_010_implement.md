# 実装内容: include_tasks + loop guard 修正・メール設定 fail-fast

実装日: 2026-05-31

## 対応レビュー

`docs/ai/reviews/proxmox_patch_apply_node/2026-05-31_009_review.md`

## 実施内容

### 指摘1（高）: OK 後に残りの retry iteration が止まらない → 修正済み

**問題**

`include_tasks + loop + when: _post_hc_retry_status != 'OK'` は、
ローカル確認でも後続 iteration が止まらないことが確認された。
Ansible が include の展開を loop 全体分まとめて行うため、
iteration N 内の set_fact 結果で iteration N+1 の when が変わらない。

**修正: guard を retry_healthcheck_once.yml の内側に移動**

`retry_healthcheck_once.yml` 全体を block に変更し、
block 全体に `when: _post_hc_retry_status != 'OK'` を付けた。

```yaml
- block:
    - name: Reset facts for this retry attempt
      set_fact: _post_hc_retry_status = CRITICAL, _post_hc_retry_json = {}
    - name: Wait ...
    - name: Run healthcheck ...
    - name: Parse JSON ...
    - name: Determine status ...
  when: _post_hc_retry_status != 'OK'
```

tasks 内の `when` は task execution time に評価されるため、
iteration N で OK が設定されると、iteration N+1 の block `when` が false になり
pause/run/parse/determine がすべてスキップされる。

**あわせて修正した点**

- reset (`_post_hc_retry_status: CRITICAL`, `_post_hc_retry_json: {}`) を block の先頭に移動。
  block が実行された場合のみリセットされるため、OK が確定した後の iteration では
  reset 自体が実行されず、OK がそのまま保持される。

- 外側 `include_tasks` の `when:` から `_post_hc_retry_status != 'OK'` を削除。
  条件は `_post_hc_status == 'CRITICAL' and _apply_reboot_done` だけにした
  （ループ実行判定として有効な条件のみを残す）。

### 指摘2（中）: 不正 JSON 時に前回値が残る → 修正済み

**問題**

`Determine status` が `ignore_errors` で失敗すると `_post_hc_retry_status` が前回値のまま残り、
さらに JSON key 欠損時に Jinja2 評価エラーが発生することがあった。

**修正**

- block 先頭の reset により、block が実行される iteration では `_post_hc_retry_status` は
  必ず CRITICAL から始まる。Determine が失敗しても CRITICAL が残り、次の iteration で再試行される。

- JSON key アクセスに `| default()` を追加し、key 欠損時も Jinja2 エラーではなく CRITICAL に落ちるようにした。

```yaml
(_post_hc_retry_json.cluster.quorate | default('')) != 'Yes'
not (_post_hc_retry_json.apt.check_ok | default(false) | bool)
(_post_hc_retry_json.services.pve_cluster | default('')) != 'active'
```

### 指摘3（中）: mail vars 不備でも apply へ進む → 修正済み

**修正: `Load mail variables` の後に assert を追加**

```yaml
- name: Assert required mail variables are defined
  ansible.builtin.assert:
    that:
      - smtp_host is defined
      - smtp_port is defined
      - smtp_user is defined
      - smtp_password is defined
      - mail_to is defined
    fail_msg: >-
      Required mail variables are not defined.
      Ensure inventories/vars/mail.yml ...
  delegate_to: localhost
  become: false
```

- `Load mail variables` は `ignore_errors: true` を維持（ファイル不在・YAML不正を捕捉）
- 直後の assert で変数存在を確認、欠損があれば apply 前に明示エラーで停止する
- assert は block の外側にあるため、assertion failure 時は always ブロックに入らない
  （mail 設定不備のまま apply が走ることはなく、apply 後のレポートも不要なため許容）

## 動作フロー（retry_count=2, retry_delay=60）

```text
reboot完了
→ post-healthcheck 初回: CRITICAL

Loop iter 0 (when: CRITICAL → block runs):
  reset: status=CRITICAL, json={}
  pause 60s
  run healthcheck → CRITICAL JSON (services inactive)
  parse → json updated
  determine → CRITICAL

Loop iter 1 (when: CRITICAL → block runs):
  reset: status=CRITICAL, json={}
  pause 60s
  run healthcheck → OK JSON
  parse → json updated
  determine → OK

Loop iter 2 (when: OK → block SKIPPED):
  [pause/run/parse/determine は実行されない]

Re-determine: _post_hc_status = OK
→ report: post_healthcheck_status=OK, mail_sent=true
→ [SUCCESS]
```
