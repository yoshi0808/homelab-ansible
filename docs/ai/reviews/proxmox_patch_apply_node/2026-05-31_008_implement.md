## 実施内容

### 指摘1（高）: `until: rc == 0` では CRITICAL が続いてもリトライしない → 修正済み

**問題**

`proxmox-healthcheck.sh` は「収集と JSON 整形のみ」の設計のため、
サービスが inactive でも rc=0 を返す。
`until: rc == 0` では最初の試行で即停止し、CRITICAL JSON でも再試行されない。

**修正: include_tasks + loop に切り替え**

`include_tasks` は `retries/until` 非対応のため、`loop` + `when` を使う構造に変更した。

新ファイル `roles/proxmox_patch_apply_node/tasks/retry_healthcheck_once.yml`:
- `pause`: 各試行の冒頭で `retry_delay` 秒待機
- healthcheck 実行 (`ignore_errors: true`)
- JSON パース (`ignore_errors: true`、rc失敗・空stdout時はスキップ)
- `_post_hc_retry_status` を OK / CRITICAL に設定

main.yml の retry タスク（loop 使用）:
```yaml
- name: Retry post-healthcheck if CRITICAL after reboot
  ansible.builtin.include_tasks: retry_healthcheck_once.yml
  loop: "{{ range(proxmox_patch_apply_hc_retry_count | int + 1) | list }}"
  loop_control:
    loop_var: _post_hc_retry_idx
    label: "attempt N of M"
  when:
    - _post_hc_status == 'CRITICAL'
    - _apply_reboot_done | bool
    - _post_hc_retry_status != 'OK'
```

- `when:` は Ansible によって各 loop iteration で再評価される
- `_post_hc_retry_status` が OK になると以降の iteration は `when: false` でスキップされる
- コマンド失敗（rc != 0）: JSON パースをスキップ → `_post_hc_retry_json = {}` → `not _post_hc_retry_json` → CRITICAL → 次の iteration で再試行
- 不正JSON: `from_json` の `ignore_errors: true` で吸収 → JSON が `{}` のまま → CRITICAL → 再試行
- サービス inactive: rc=0 だが JSON チェックで CRITICAL → 再試行

**`Re-determine` タスクをシンプル化**

`_post_hc_retry_status` が loop 内で最終決定されているため:
```yaml
- name: Re-determine post-healthcheck status after retry
  ansible.builtin.set_fact:
    _post_hc_status: "{{ _post_hc_retry_status }}"
  when: _post_hc_status == 'CRITICAL' and (_apply_reboot_done | bool)
```

### 指摘2（中）: ポリシー文書の説明が実装と一致しない → 指摘1修正により解消

loop ベースの実装はポリシー文書の動作例と一致する。

```text
デフォルト(retry_count=2, retry_delay=60)での動作:
初回: CRITICAL
→ loop iter 0: 60秒待機 → run → CRITICAL
→ loop iter 1: 60秒待機 → run → OK → done
→ loop iter 2: when=false でスキップ
```

最大待機: (2+1) × 60 = 180秒。ポリシー文書の計算式と一致。

### 指摘3（中）: メール送信失敗を re-raise で検知する → 修正済み（前回実装から維持）

apply 成功 + メール送信失敗の場合、最後に play を fail させる。

```yaml
- name: Re-raise failure to mark play as failed
  ansible.builtin.fail:
    msg: >- (apply failed message or mail failed message)
  when: (_apply_failed | bool) or (_mail_send_result.failed | default(true) | bool)
```

## 動作フロー（retry_count=2, retry_delay=60, 2回目でOKの場合）

```text
reboot完了
→ post-healthcheck (初回): CRITICAL
→ loop iter 0 (when: CRITICAL and reboot and CRITICAL→true):
    60秒待機 → run → parse → CRITICAL
→ loop iter 1 (when: CRITICAL and reboot and CRITICAL→true):
    60秒待機 → run → parse → OK
→ loop iter 2 (when: CRITICAL and reboot and OK→false):
    スキップ
→ Re-determine: _post_hc_status = OK
→ Set email subject: [SUCCESS]
→ Send email: mail_sent=true
→ Build report: post_healthcheck_status=OK, mail_sent=true
→ Re-raise: _apply_failed=false, mail_sent=true → skip
→ play 終了: 成功
```
