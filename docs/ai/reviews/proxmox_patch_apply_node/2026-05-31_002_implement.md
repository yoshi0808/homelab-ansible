# 実装内容: post-healthcheck リトライとメール通知修正

実装日: 2026-05-31

## 実装概要

### 1. post-healthcheck リトライ

reboot直後にCRITICALとなった場合のみ、待機してからリトライする仕組みを追加した。
リトライ対象を「reboot実施済み かつ CRITICAL」に限定することで、
reboot無しのCRITICAL（本物の障害）と区別する。

Ansibleの `retries / delay / until` を使い、OKになったら即停止する。
再確認でOKになった場合は `_post_hc_status` を `OK` に更新し、SUCCESSとして扱う。

### 2. メール通知の確実化

`Set email subject and body` タスクはすでに `ignore_errors: true` を持っていた（既存）。
ただし、このタスクが Jinja2 エラーで失敗した場合、`_mail_subject` / `_mail_body` が
未設定のままメール送信タスクへ進む問題があった。

対策として、初期化ブロックに `_mail_subject` と `_mail_body` のフォールバック値を追加した。
テンプレート展開が失敗してもフォールバック値でメールが送られる。

## 変更内容の要約

### defaults/main.yml

以下を追加:

```yaml
proxmox_patch_apply_hc_retry_count: 2
proxmox_patch_apply_hc_retry_delay: 60
```

- `retry_count`: リトライタスクの最大追加試行回数（初回実行 + retry_count 回 = 最大 retry_count+1 回）
- `retry_delay`: 各リトライの待機秒数

### tasks/main.yml

**変更1: 初期化ブロックにフォールバック値を追加**

```yaml
_mail_subject: "[UNKNOWN] {{ inventory_hostname }}: パッチ適用結果不明"
_mail_body: "パッチ適用の結果を取得できませんでした。ログを確認してください。"
```

`Set email subject and body` の Jinja2 テンプレートが失敗した場合でも、
`ignore_errors: true` により処理は継続し、フォールバック値でメールが送信される。

**変更2: `always` ブロックに2タスクを追加**

`Determine post-healthcheck status` の直後に挿入:

1. `Retry post-healthcheck if CRITICAL after reboot`
   - `when: _post_hc_status == 'CRITICAL' and (_apply_reboot_done | bool)` で条件実行
   - `retries` / `delay` / `until` でOKになるまでリトライ
   - `until` の成功条件: quorum / apt_check / pve_cluster / corosync / pvedaemon すべて正常
   - スクリプト自体が失敗した場合も `until` で検出し、追加リトライを止める

2. `Re-determine post-healthcheck status after retry`
   - 同じ `when:` 条件で実行
   - リトライ結果から `_post_hc_status` を更新（OK / CRITICAL）
   - 以降のレポート・メール件名はこの更新された値を使用

## 動作フロー（reboot後CRITICAL → リトライ → OKの場合）

```text
1. reboot完了
2. post-healthcheck実行 → CRITICAL
3. 条件判定: CRITICAL かつ reboot実施済み → リトライ開始
4. 60秒待機 → 再実行 → OK
5. _post_hc_status = OK に更新
6. レポート保存: post_healthcheck_status = OK
7. メール件名: [SUCCESS] {{ node }}: パッチ適用完了
```

## 注意点

- リトライは reboot実施後のみ。reboot無しでCRITICALの場合はリトライしない（本物の障害として扱う）。
- `retries_count: 2` の場合、リトライタスクは最大 2+1=3 回実行される（デフォルト合計最大待機: 120秒）。
- `_post_hc_json` の更新はしない（リトライタスクの `_post_hc_retry_raw.stdout` から直接判定）。レポートの `post_healthcheck_status` フィールドには更新後の `_post_hc_status` が入る。
