## 変更内容の要約

### playbooks/proxmox_patch_apply_node.yml

- `hosts: "{{ target_node }}"` で単一ノードを対象とする
- `gather_facts: true` / `become: true`
- `proxmox_patch_apply_node` role のみ含む（healthcheck は role 内で呼ぶ）
- Usage コメント付き（`-e target_node=pve2`）

### roles/proxmox_patch_apply_node/defaults/main.yml

- `proxmox_patch_apply_control_node_name: "ansy"` — 検出する control node 名
- `proxmox_patch_apply_reboot_timeout: 900` — 15 分
- `proxmox_patch_apply_major_upgrade_threshold: 50` — package 数でのメジャーアップグレード判定閾値
- 重要コンポーネント一覧（policy section 7 と同一）
- `proxmox_patch_dryrun_report_dir` — latest dry-run JSON 参照先

### roles/proxmox_patch_apply_node/files/proxmox-patch-predryrun-collect.sh

- apt-get update / apt-get check / apt-get -s dist-upgrade を実行し JSON を返す
- changelog 取得なし（機械的判定のみ）
- 出力: `apt_update_ok`, `apt_update_output`, `apt_check_ok`, `apt_check_output`, `sim_ok`, `sim_output`, `reboot_required`
- stdout 汚染なし（全出力を変数に閉じ込め）

### roles/proxmox_patch_apply_node/tasks/main.yml

**初期化:**
- `_apply_run_ts`, `_apply_failed`, `_apply_abort_type`, `_apply_error_msg` など全 result fact を block 前に初期化
- mail vars と report directory を block 前に準備

**block（Phase 1–3）:**

Phase 1:
- `pvesh get /cluster/resources --type vm` で control node 検出 → ABORT
- `proxmox_healthcheck` role を `include_role` で呼び出し（rescue が CRITICAL を捕捉）
- WARNING / CRITICAL の場合 → ABORT（`proxmox_report.result.status != 'OK'`）

Phase 2:
- predryrun-collect.sh をコピー・実行・JSON パース
- apt_update_ok / apt_check_ok / sim_ok 各フェーズで失敗 → ABORT
- sim_output から `Inst` / `Remv` 行を抽出
- 重要コンポーネント照合・remove 検出・major upgrade 閾値（50パッケージ）で MAINTENANCE_REQUIRED 判定
- MAINTENANCE_REQUIRED 検出時: `query('fileglob', ...)` で最新 unified_dryrun.json を読み、新規追加パッケージと既存パッケージを分離して ABORT
- 重要コンポーネント判定: exact match + prefix match（proxmox-kernel-*, libpve-*, firmware-*）

Phase 2 (apply):
- `apt-get -y dist-upgrade` を `DEBIAN_FRONTEND: noninteractive` で実行

Phase 3:
- `/var/run/reboot-required` を確認
- `ansible.builtin.reboot` で自動 reboot（`reboot_timeout: 900`）

**rescue:**
- `_apply_failed: true` + `_apply_error_msg` を設定

**always（Phase 4）:**
- `/usr/local/sbin/proxmox-healthcheck.sh` を直接実行（ignore_errors）
- JSON を簡易パースして `_post_hc_status`（OK / CRITICAL / UNKNOWN）を判定
- レポート JSON を `reports/proxmox-patch/` に保存
- 件名・本文を `_apply_abort_type` に応じて分岐生成
  - MAINTENANCE_REQUIRED: 新規パッケージ差分・既存パッケージを本文に含む
  - その他: Phase 別の結果サマリー
- `community.general.mail` でメール送信（ignore_errors）
- `_apply_failed` が true なら最後に `fail` して play を失敗状態にする

**abort_type 一覧:**
| abort_type | 発生条件 |
|---|---|
| `control_node` | ansy が target_node 上で稼働中 |
| `healthcheck` | pre-healthcheck が WARNING / CRITICAL |
| `apt_update` | apt-get update 失敗 |
| `apt_check` | apt-get check 失敗 |
| `sim_failed` | apt-get -s dist-upgrade 失敗 |
| `maintenance_required` | 重要コンポーネント / remove / 多数 package 検出 |
| `apply_failed` | apt-get dist-upgrade 失敗 |
| `reboot_timeout` | reboot 後 15 分で応答なし |
| `none` | 正常完了 |

## 設計上の注意点

- `include_role: proxmox_healthcheck` は block 内で呼ぶため、rescue が CRITICAL fail を捕捉できる
- post-healthcheck では role ではなく script を直接実行（`ignore_errors: true`）。always ブロック内で role の `fail` を無効化する手段がないため
- `query('fileglob', ...)` はコントロールノードのファイルシステムを参照する。ローカルの reports/ が存在しない場合は `_latest_dryrun_packages: []` として扱う
- `set_fact` は inventory_hostname（target_node）に対して設定し、`delegate_to: localhost` タスクからは通常の変数参照でアクセス可能
