## 指摘

### 1. 中: 新規 task file が untracked のため、commit 漏れで本番実行が壊れる可能性がある

該当箇所:

- `roles/proxmox_patch_apply_node/tasks/main.yml:410`
- `roles/proxmox_patch_apply_node/tasks/main.yml:416`
- `roles/proxmox_patch_apply_node/tasks/retry_healthcheck_once.yml`

`main.yml` は以下で新規 task file を参照している。

```yaml
ansible.builtin.include_tasks: retry_healthcheck_once.yml
```

しかし `git status --short` では、`roles/proxmox_patch_apply_node/tasks/retry_healthcheck_once.yml` は untracked のまま。

```text
?? roles/proxmox_patch_apply_node/tasks/retry_healthcheck_once.yml
```

このファイルが commit に含まれないと、quory など別環境で実行したときに `include_tasks` が対象ファイルを見つけられず、post-healthcheck retry で失敗する。

対応:

- commit 前に `roles/proxmox_patch_apply_node/tasks/retry_healthcheck_once.yml` を必ず git 管理対象に追加する。
- レビューや最終確認では `git diff --cached` / `git status --short` で新規ファイルが含まれていることを確認する。

### 2. 低: 初回 post-healthcheck が UNKNOWN の場合は retry 対象外のまま

該当箇所:

- `roles/proxmox_patch_apply_node/tasks/main.yml:396`
- `roles/proxmox_patch_apply_node/tasks/main.yml:410`
- `roles/proxmox_patch_apply_node/tasks/main.yml:422`

現在の retry 条件は `_post_hc_status == 'CRITICAL'` のみ。

初回 post-healthcheck のコマンド失敗、空 stdout、不正 JSON などで `_post_hc_status` が `UNKNOWN` になった場合は、reboot 済みでも retry されない。

要求仕様は「post-healthcheck が CRITICAL の場合」に限定しているため、仕様違反とは言い切れない。ただし今回の retry task では、リトライ中のコマンド失敗・不正 JSON を CRITICAL 相当に畳み込んで再試行する設計になっている。そこまで堅くするなら、初回 post-healthcheck の `UNKNOWN` も reboot 直後の一時不安定として retry 対象に含めるか、含めない理由を明記しておくと運用上の期待が揃う。

対応案:

```yaml
when:
  - _post_hc_status in ['CRITICAL', 'UNKNOWN']
  - _apply_reboot_done | bool
```

または、現状維持するなら policy / 実装メモに「初回 post-healthcheck が UNKNOWN の場合は retry しない」と明記する。

## 確認できたこと

- `Assert required mail variables are defined and non-empty` で、`smtp_host` / `smtp_user` / `smtp_password` / `mail_to` の空文字と `smtp_port <= 0` を apply 前に弾くようになっている。
- `retry_healthcheck_once.yml` の block guard は前回確認した `include_tasks + loop` の問題に対して有効な構造になっている。
- retry 判定は JSON 内の `cluster.quorate` / `apt.check_ok` / `services.*` を見る形で、`rc == 0` だけで成功扱いする問題はない。
- JSON key 欠損時は `default()` により CRITICAL 側へ落ちる。
- `proxmox_patch_policy.md` の retry 動作例から `即時実行` / `delay` 表現は消えている。
- `core.md` の `monnie` / `monitoring_servers` 追加は、今回の role 差分とは直接衝突しない。

## 総評

post-healthcheck retry の主ロジックは、要求仕様に沿った形まで整理されている。

残る必須対応は、新規 task file の git 管理漏れ防止。`UNKNOWN` を retry 対象に含めるかは運用判断だが、reboot 直後の一時不安定を広く吸収したいなら含める方が自然。
