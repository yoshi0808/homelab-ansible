## 検証結果

`ansible-playbook --syntax-check` は、通常実行では `/home/yoshi/.ansible/tmp` が read-only のため失敗した。

```text
ERROR: Unhandled exception when retrieving 'DEFAULT_LOCAL_TMP': [Errno 30] Read-only file system: '/home/yoshi/.ansible/tmp/...'
```

環境要因と判断し、local temp を `/tmp` に向けて再実行したところ成功した。

```text
ANSIBLE_LOCAL_TEMP=/tmp/ansible-local ansible-playbook -i inventories/homelab/hosts.yml playbooks/proxmox_patch_apply_node.yml -e target_node=pve2 --syntax-check
```

結果:

```text
playbook: playbooks/proxmox_patch_apply_node.yml
```

## Findings

### 1. 高: target_node 検証は role 内なので、未指定時や gather_facts 前の事故を防げません

`roles/proxmox_patch_apply_node/tasks/main.yml:6` に assert は追加されていますが、playbook の `playbooks/proxmox_patch_apply_node.yml:18` の `hosts: "{{ target_node }}"` と `playbooks/proxmox_patch_apply_node.yml:19` の `gather_facts: true` が先に評価されます。

そのため、以下のケースでは role 内 assert に到達する前に失敗します。

- `target_node` 未定義
- `target_node` が inventory に存在しない
- 対象 host が unreachable
- gather facts が失敗

unsafe playbook の「最初の安全ゲート」としては不十分です。

`target_node` の許可リスト検証は、role 内ではなく、localhost の pre-play など role 到達前に実行できる位置に置く必要があります。

### 2. 中: VM/CT 残存チェックの CT 検出は実環境依存に見えます

`roles/proxmox_patch_apply_node/tasks/main.yml:62` は `pvesh get /cluster/resources --type vm --output-format json` の結果だけを使っています。

この API が現行環境で qemu / lxc の両方を返すなら問題ありません。

一方、VM のみを返す場合、CT が target node 上で running のままでも `roles/proxmox_patch_apply_node/tasks/main.yml:90` の退避チェックをすり抜けます。

実装コメントとエラーメッセージは VM/CT を対象にしているため、`--type vm` の取得結果が CT を含むことを実サンプルで保証するか、CT も明示的に取得する実装にする必要があります。

### 3. 中: 新 abort_type が apply セクション表示条件に入っていません

`roles/proxmox_patch_apply_node/tasks/main.yml:390` では `vms_not_evacuated` / `manual_not_confirmed` が前提チェック失敗カテゴリに追加されています。

しかし `roles/proxmox_patch_apply_node/tasks/main.yml:462` の「apply 未実施」判定には、以下の新 abort_type が含まれていません。

- `vms_not_evacuated`
- `manual_not_confirmed`

そのため、この2ケースではメール本文の apply セクションに以下が表示されます。

- `適用パッケージ数`
- `reboot 実施`

実際には apply 未実施なので、メール本文上で誤読しやすいです。

### 4. 低: manual confirm は前進ですが、確認文字列が汎用すぎます

`roles/proxmox_patch_apply_node/defaults/main.yml:19` と `roles/proxmox_patch_apply_node/tasks/main.yml:245` で、`MAINTENANCE_REQUIRED` 検出時の manual 実行には明示確認が必須になりました。

これは `-e proxmox_patch_apply_mode=manual` だけでは続行できないため、前回より安全です。

ただし確認文字列が `MAINTENANCE_REQUIRED` のみで、対象ノードや検出理由を含みません。

将来 policy を見直すなら、例えば以下のように target node を含めた確認値の方が誤実行耐性は上がります。

```text
pve2:MAINTENANCE_REQUIRED
```

## 改善された点

### VM / CT 退避チェックが追加された

`roles/proxmox_patch_apply_node/tasks/main.yml:90` で target node 上の running VM/CT を検出し、残存していれば `vms_not_evacuated` で abort するようになりました。

これは unsafe playbook として重要な改善です。

### manual 実行に明示確認が追加された

`roles/proxmox_patch_apply_node/tasks/main.yml:245` で、manual mode かつ `MAINTENANCE_REQUIRED` 検出時に `proxmox_patch_apply_manual_confirm=MAINTENANCE_REQUIRED` がなければ `manual_not_confirmed` で abort するようになりました。

`manual` の存在自体は policy 見直し前提ですが、単純な指定ミスによる適用リスクは下がっています。

### target_node allowlist が追加された

`roles/proxmox_patch_apply_node/tasks/main.yml:6` に `pve1` / `pve2` の allowlist assert が追加されました。

ただし Findings 1 の通り、現在の位置では role 到達前の失敗には効きません。

## 残っている既知リスク

`2026-05-09_006_implement.md` の通り、以下は未対応です。

- 通知保証は best-effort のまま
- control node チェックの脆さは将来課題
- post-healthcheck 判定差分は将来課題
- `MAJOR_UPGRADE_DETECTED` / `BLOCKED` 区別は将来課題

この整理自体は実装記録と一致しています。

## 結論

再実装により、VM/CT 残存チェックと manual 確認ゲートは改善されています。

ただし、`target_node` 検証は role 内にあるため、未指定や gather_facts 前の失敗を防げません。unsafe playbook の入口ガードとしては、localhost pre-play など role 到達前に移す必要があります。

また、メール本文の apply セクション表示条件に新 abort_type が入っていないため、`vms_not_evacuated` / `manual_not_confirmed` では実際には apply 未実施なのに apply 情報が表示されます。

次に直すなら、以下を優先するのがよいです。

1. `target_node` 検証を role 到達前に効かせる
2. メール本文の `vms_not_evacuated` / `manual_not_confirmed` 表示ズレを修正する
3. `--type vm` が CT を含むか実データで確認し、必要なら CT 取得を明示する
