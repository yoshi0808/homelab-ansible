## 結論

前回レビュー `2026-05-31_003_review.md` の主要指摘2点、つまり「最初のリトライ前に待機しない」と「command failed を until 成功条件にしている」は、おおむね解消されている。

ただし、`until` 条件内で `from_json` を直接評価しているため、healthcheck が一時的に壊れた JSON / 非JSONの標準出力を返した場合は再試行されない。reboot 直後の不安定さを吸収する目的から見ると、まだリトライとして弱い。

また、メール通知の「必ず summary mail が送信される」という要求に対して、今回の `ignore_errors: true` 追加は到達保証にはなっていない。

## 指摘

### 1. 中: リトライ中に不正JSONを返すと再試行されない

該当箇所:

- `roles/proxmox_patch_apply_node/tasks/main.yml:401`
- `roles/proxmox_patch_apply_node/tasks/main.yml:403`
- `roles/proxmox_patch_apply_node/tasks/main.yml:404`
- `roles/proxmox_patch_apply_node/tasks/main.yml:405`
- `roles/proxmox_patch_apply_node/tasks/main.yml:406`
- `roles/proxmox_patch_apply_node/tasks/main.yml:407`

`Retry post-healthcheck if CRITICAL after reboot` の `until` 条件で、`_post_hc_retry_raw.stdout | from_json` を直接複数回評価している。

この形だと、コマンドは成功したが stdout が空ではない非JSON、または途中で壊れた JSON だった場合、`from_json` のテンプレート評価が失敗し、その試行は「until 条件が false」ではなく「タスク失敗」として扱われる。`ignore_errors: true` により play は続くが、リトライは継続されない。

ローカルの最小 playbook で確認したところ、`stdout: not-json` に対して `until: "{{ stdout | from_json ... }}"` を評価すると、`retries: 2` を指定していても再試行されず、その場で ignore された。

reboot 直後の healthcheck では、スクリプト・依存コマンド・Proxmox サービスの一時的な不安定さを吸収したいはずなので、「壊れた出力」も CRITICAL 相当としてリトライ対象にした方が安全。

対応案:

- リトライタスクでは JSON parse を `until` に直接書かず、healthcheck 実行と parse / 判定を分ける。
- もしくは `/usr/local/sbin/proxmox-healthcheck.sh` が失敗時も必ず valid JSON を返す契約を明文化する。
- Ansible 側で吸収するなら、リトライ対象の単位を `block` + `include_tasks` に分け、parse 失敗時は `_post_hc_retry_ok: false` として扱う形がよい。

### 2. 中: `Load mail variables` の `ignore_errors` 追加は「成功時メールが必ず届く」修正にはなっていない

該当箇所:

- `roles/proxmox_patch_apply_node/tasks/main.yml:41`
- `roles/proxmox_patch_apply_node/tasks/main.yml:46`
- `roles/proxmox_patch_apply_node/tasks/main.yml:546`
- `roles/proxmox_patch_apply_node/tasks/main.yml:558`

今回 `Load mail variables` に `ignore_errors: true` が追加されたため、`inventories/vars/mail.yml` の読み込みに失敗しても play は続行する。

しかし、その場合は `smtp_host` / `smtp_port` / `smtp_user` / `smtp_password` / `mail_to` が未定義のまま `Send summary email` に進む。メール送信タスクも `ignore_errors: true` なので、最終的には「play は成功方向に進むが、メールは送られない」状態になる。

要求仕様は「apply成功時も必ずsummary mailが送信されることを確認・修正する」なので、メール変数読み込み失敗を無視するだけでは要求を満たしにくい。

対応案:

- メール変数読み込み失敗時は、apply 前に明示的に fail する。
- ただし always での通知を重視するなら、mail vars 読み込みだけは main block より前に検証し、失敗時は「通知設定不備」として止める。
- `Send summary email` の結果を register し、送信失敗時に report に `mail_sent: false` と理由を残す。
- 「パッチ適用成功だがメール送信失敗」を playbook failure として扱うか、運用ポリシーで明確化する。

少なくとも、メール vars が存在しないケースを握りつぶす変更は、「必ず送信」に対して逆方向の挙動に見える。

### 3. 低: 最大待機時間の説明は Ansible の `retries` 解釈に依存しており、実測値を補足すると安全

該当箇所:

- `roles/proxmox_patch_apply_node/defaults/main.yml:49`
- `roles/proxmox_patch_apply_node/defaults/main.yml:51`
- `docs/ai/prompts/proxmox_patch_policy.md:631`
- `docs/ai/prompts/proxmox_patch_policy.md:637`
- `docs/ai/prompts/proxmox_patch_policy.md:638`

今回の文書では、`retries: N` に対してリトライタスク最大実行回数を `N+1` と説明している。

この理解は実行ログ上も妥当だった。`retries: 2` / `until: false` の最小 playbook では、`2 retries left`、`1 retries left`、最終失敗の流れになり、実行は最大3回になる。

ただし、Ansible のドキュメントや実行結果の `attempts` 表示は読み手に誤解されやすい。運用文書としては以下のように補足するとよい。

```text
注: Ansible の表示上は "2 retries left" のように出るが、
retries: 2 の場合、初回試行 + 再試行2回 = 最大3回 command が実行される。
```

これは動作バグではなく、将来の保守者向けの説明改善。

## 推奨修正

優先度が高いのは、`from_json` 失敗をリトライ対象から外してしまう問題の修正。

最小方針としては、healthcheck スクリプト側が常に valid JSON を返す保証があるかを確認し、保証がないなら Ansible 側で parse 失敗を boolean に畳み込む構造に変えるのがよい。

メール通知については、`ignore_errors` で先へ進めるより、メール設定不備を apply 前に検出して止めるか、送信失敗を report に明示的に残す方が運用上追跡しやすい。
