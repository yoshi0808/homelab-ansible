# Review: apply後のreboot判定改善

## 結論

重大な指摘なし。

`/var/run/reboot-required` だけでは kernel 更新後の reboot 要否を取りこぼす問題に対して、post-apply collect script で稼働中 kernel とインストール済み kernel package version を比較する実装が追加されている。

`proxmox_patch_policy.md` の「reboot 要否は `/var/run/reboot-required` と kernel version 比較を組み合わせて判定する」という方針に沿っており、この内容で commit してよい。

## Findings

なし。

## 要件との照合

### OK

- `roles/proxmox_patch_apply_node/files/proxmox-patch-apply-collect.sh` が追加されている。
- `/var/run/reboot-required` の存在確認が維持されている。
- `uname -r | sed 's/-pve//'` で稼働中 kernel version を取得している。
- `proxmox-kernel-6.17` / `proxmox-kernel-7.0` の installed version を `dpkg-query` で取得している。
- `dpkg --compare-versions "$installed_ver" gt "$running_ver"` で、インストール済み kernel が稼働中 kernel より新しい場合に `reboot_required=true` としている。
- script は reboot 実行や package 操作を行わず、post-apply の収集と `reboot_required` 判定に留まっている。
- Ansible tasks 側は script の JSON を parse し、既存の `_reboot_required_before` / reboot 処理に渡している。
- report に `reboot_trigger` と `running_kernel` が追加され、後から「なぜ reboot したか」を追える。
- 秘密情報・鍵・認証情報の追加や読み出しは見当たらない。
- 対象は既存の `proxmox_patch_apply_node.yml` の対象である `pve1` / `pve2` に限定されている。

## 確認したポイント

### shell

`proxmox-patch-apply-collect.sh` は以下の JSON を返す構造になっている。

- `collected_at`
- `reboot_required`
- `reboot_trigger`
- `running_kernel`

`reboot_trigger` は以下の値になりうる。

- `none`
- `reboot-required-file`
- `kernel-version-mismatch:<pkg> installed=<installed_ver> running=<running_ver>`

これにより、従来の file-based 判定と kernel mismatch 判定のどちらで reboot_required になったかを識別できる。

### Ansible

既存の `stat: /var/run/reboot-required` は、新規 script 実行と JSON parse に置き換えられている。

```yaml
_reboot_required_before: "{{ _apply_collect.reboot_required | bool }}"
```

既存の reboot phase は `_reboot_required_before` を参照しているため、後続の reboot 制御は既存フローを流用できている。

## 注意点

- `git diff --stat` は未追跡ファイルを表示しないため、commit 時は `roles/proxmox_patch_apply_node/files/proxmox-patch-apply-collect.sh` と review / requirement / implement 文書を明示的に stage すること。
- `running_kernel` は post-apply collect 実行時点、つまり reboot 前の kernel 名である。これは今回の用途では妥当だが、report を読むときは「reboot 前の稼働 kernel」として解釈する。
- 将来 Proxmox の kernel 系列が増えた場合は、要件メモどおり script 内の `for pkg in ...` に対象 package を追加する必要がある。
