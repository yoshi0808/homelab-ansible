## 前回指摘の確認

### resolved: `nohup sudo ... &` 失敗時の lock leak

`roles/recovery_trigger/templates/recovery-trigger.sh.j2:111` から `roles/recovery_trigger/templates/recovery-trigger.sh.j2:114` で、background wrapper が `sudo -u "$1" "$2" "$3" "$4"` を実行し、失敗時は `rmdir "$6" 2>/dev/null || true` へ進むようになった。

これにより、sudoers 不備、pipeline script 不在、実行権限不備などで pipeline が起動できない場合でも、wrapper 側が `/var/run/homelab-recovery/<target>.lock` を削除できる。pipeline が起動できた後の通常 cleanup は、引き続き `roles/recovery_trigger/files/recovery-pipeline.sh:25` の EXIT trap が担う。

trigger 側の `trap - EXIT` は `roles/recovery_trigger/templates/recovery-trigger.sh.j2:116` に残っているが、background wrapper が handoff 失敗時の cleanup を持つため、前回指摘した cleanup 所有者不在の経路は解消している。
