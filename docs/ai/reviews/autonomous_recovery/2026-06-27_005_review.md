## 指摘

### must-fix: mute ファイル作成が権限的に失敗する

`roles/recovery_trigger/tasks/main.yml:161` 付近では `/var/lib/homelab-recovery/mute` を `owner: root`, `group: root`, `mode: 0755` で作成している。一方、`roles/recovery_mute/tasks/set.yml:14` と `roles/recovery_mute/tasks/set.yml:40` は `delegate_to: localhost` かつ `become: false` のまま同ディレクトリ作成・mute JSON 書き込みを行う。

この組み合わせだと、通常の quory 実行ユーザーでは `/var/lib/homelab-recovery/mute/<target>.json` を作成できない。結果として、今回 mute を追加した以下の変更系 playbook が、復旧対象を触る前に失敗するか、mute なしで進んで自律復旧の誤検知を止められない。

- `playbooks/proxmox_evacuate_node.yml`
- `playbooks/proxmox_patch_apply_node.yml`
- `playbooks/proxmox_patch_weekly_full.yml`
- `playbooks/proxmox_restore_vm_placement.yml`
- `playbooks/ubuntu_nightly.yml`

`autonomous_recovery_policy.md:239` 以降のミュート/TTL機構は、メンテナンス中の誤復旧を止める安全機構なので、これは must-fix。

対応案:

- `roles/recovery_mute/tasks/set.yml` 側を `become: true` で書く。
- または `mute` ディレクトリを専用 group 書き込み可能にし、実行ユーザーをその group に入れる。
- どちらにしても、実際に対象 playbook の冒頭で mute JSON が作成できることを確認する。

### must-fix: recovery-trigger のロック取得が atomic ではなく、同一 target の並行 pipeline を防げない

`roles/recovery_trigger/templates/recovery-trigger.sh.j2:47` で lock file の存在確認を行い、`roles/recovery_trigger/templates/recovery-trigger.sh.j2:100` で後から `touch "$LOCK_FILE"` している。この check-then-touch は atomic ではないため、ほぼ同時に複数の trigger が来ると、両方が存在確認を通過して複数の `recovery-pipeline.sh` を起動できる。

これは `autonomous_recovery_policy.md:227` の「同targetで前のパイプラインが稼働中ならskip」および `autonomous_recovery_policy.md:288` の実行中ロックの目的に反する。特に monnie は `grafana-server.service`, `prometheus.service`, `loki.service`, `unpoller.service` の複数 OnFailure 経路があり、同一障害で近接 trigger が発生し得るため、実運用で踏む可能性がある。

対応案:

- `mkdir "$LOCK_DIR/${TARGET}.lock"` を atomic lock として使う。
- または `flock -n` を使って lock 取得に失敗したら即 skip する。
- pipeline 側の cleanup も、選んだ lock 方式に合わせて削除する。

### must-fix: Slack 手動 trigger が flapping count に記録されてしまう

`autonomous_recovery_policy.md:219` と `autonomous_recovery_policy.md:295` は、Slack ループの手動依頼では 24 時間 flapping count をスキップすると明記している。しかし `roles/recovery_trigger/templates/recovery-trigger.sh.j2:78` から `roles/recovery_trigger/templates/recovery-trigger.sh.j2:85` で count file へ記録した後、`roles/recovery_trigger/templates/recovery-trigger.sh.j2:87` で `MANUAL == 1` を判定している。

このため、Slack からの手動復旧やテスト実行が `/var/lib/homelab-recovery/counts/<target>.json` に蓄積され、後続の自動 OnFailure trigger が「直近24時間3回以上」と誤判定して ladder をスキップし、即エスカレーションに進む可能性がある。人間の明示依頼を count から除外するという policy の意図と逆の動作になる。

対応案:

- `MANUAL == 1` の場合は count file の読み書き自体をスキップする。
- 自動 trigger の場合だけ recent count 算出、追記、flapping 判定を行う。

### must-fix: setup playbook が quory 上で `git pull` しており、core.md の実行境界に反している

`docs/ai/prompts/core.md:374` 以降は、Ansible playbook 内で `git pull` しないこと、Git 更新は quory 上の Ansible 外スクリプトまたは将来の Semaphore UI Repository 機能に寄せることを明記している。理由も「実行中の playbook が自分自身を更新する自己更新問題」とされている。

しかし `roles/recovery_trigger/tasks/main.yml:218` は setup role 内で `sudo -u yoshi git -C /home/yoshi/homelab-ansible pull` を実行している。さらに `core.md:370` の `--ff-only` と、`core.md:372` の pull 前 clean check も満たしていない。

これは運用境界の明確な逸脱で、setup playbook の再実行時に自分が参照している role/playbook 群を途中で更新し得る。

対応案:

- この task は role から削除する。
- quory の repo 更新は Ansible 外の手順または専用 wrapper に分離し、`git status --porcelain` clean check と `git pull --ff-only` をそこで実施する。
