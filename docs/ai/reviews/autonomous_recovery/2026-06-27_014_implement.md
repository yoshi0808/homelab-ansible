## 対応内容

### 指摘: `nohup sudo ... &` 失敗後に trigger trap が解除済みで lock が残る

**原因:** `nohup sudo -u "$PIPELINE_USER" "$PIPELINE" ... &` の後に `trap - EXIT` していたため、sudo 失敗・exec 失敗（sudoers 不備・script 不在・実行権限なし等）の場合に:
- trigger 側の EXIT trap はすでに解除済み
- pipeline 側の EXIT trap は pipeline が起動していないため未設定

となり、lock cleanup の担い手がいなくなる。

**修正:** `nohup bash -c '... || rmdir ...' -- ...` の wrapper パターンに変更。

```bash
# 変更前
nohup sudo -u "$PIPELINE_USER" "$PIPELINE" "$TARGET" "$ESCALATE_FLAG" \
  >"$LOG_FILE" 2>&1 &
trap - EXIT

# 変更後
nohup bash -c '
  sudo -u "$1" "$2" "$3" "$4" >"$5" 2>&1 \
    || rmdir "$6" 2>/dev/null || true
' -- "$PIPELINE_USER" "$PIPELINE" "$TARGET" "$ESCALATE_FLAG" "$LOG_FILE" "$LOCK_FILE" &
trap - EXIT
```

**効果:**
- sudo/exec 失敗 → wrapper の `||` ブランチが `rmdir "$LOCK_FILE"` を実行
- sudo 成功・pipeline 正常起動 → pipeline.sh の EXIT trap が lock を削除（`rmdir 2>/dev/null || true` で二重削除は無害）
- 変数はすべて `-- "$1" "$2" ...` 経由で渡すため、クォート問題なし
