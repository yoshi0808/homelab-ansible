## 指摘

### must-fix: `target` を未サニタイズのままレポート保存パスに使っている

`playbooks/recovery_service_restart.yml:35` 付近の pre_task は `target is defined` と `target | length > 0` しか検証していない。一方で各 role の report dir は、Proxmox タグ検証に失敗した場合でも `always` で使われる。

該当箇所:

- `roles/recovery_service_restart/defaults/main.yml:7`
- `roles/recovery_vm_reboot/defaults/main.yml:19`
- `roles/recovery_ha_failover/defaults/main.yml:7`
- `roles/recovery_service_restart/tasks/main.yml:121`
- `roles/recovery_vm_reboot/tasks/main.yml:187`
- `roles/recovery_ha_failover/tasks/main.yml:159`

現状だと、許可された `ansible-playbook playbooks/recovery_*.yml -e target=<target>` の範囲内であっても、`target=../../...` のような値が渡ると、タグ検証で失敗したあとに `reports_base_dir/recovery_investigations/{{ target }}` を使ってローカル側にディレクトリ/JSONを書こうとする。policy §5 は「agent から渡された target を信用せず、実行直前にタグを再照会する」設計なので、変更操作だけでなくレポート保存先にも未検証の target を使わない方がよい。

対応案:

- pre_tasks で `target` を `^[A-Za-z0-9_-]+$` 程度に制限する。
- さらに、許可対象が現時点で固定なら playbook ごとに allowed target list を持ち、タグ検証の前に fail させる。
- レポート保存先には `target` ではなく、サニタイズ済みの `recovery_target_safe` のような fact を使う。

### suggestion: `autonomous_recovery_policy.md` を `core.md` のポリシー文書一覧に追加する

`docs/ai/prompts/core.md:330-347` には、複数 playbook が連携する複雑なシステムや変更系で安全度判断が必要なシステムは `docs/ai/prompts/*_policy.md` を正本にし、該当システムでは `core.md` と対応 policy を必ず参照する、とある。

今回の autonomous recovery は複数の変更系 playbook が連携する安全度の高いシステムなので、`docs/ai/prompts/autonomous_recovery_policy.md` もこの表に追加しておくと、次回以降の AI 受け渡しで読み漏れにくくなる。

## 確認結果

### タグ再検証

3つの recovery role はいずれも Phase 1 で `pvesh get /cluster/resources --output-format json` を実行し、`target` 名で VM を探したうえでタグを Ansible tasks 側で判定している。shell 側に判定を逃がしていないため、policy §5 / core.md §7 と整合している。

- `recovery_service_restart`: `hacritical` または `ops`、かつ `sophos-fw` 除外
- `recovery_vm_reboot`: `hacritical` または `ops`
- `recovery_ha_failover`: `hacritical` のみ

### block / rescue / always

3つの role はいずれも `block` / `rescue` / `always` 構造で、失敗時も result/detail を fact に保存し、`always` でレポート保存と Slack 通知を実行する構造になっている。Slack 通知は `roles/common_slack/tasks/notify.yml` 側に rescue があり、通知失敗で本処理を止めない設計になっている。

### recovery_vm_reboot の fallback

authy / monnie は guest agent shutdown を第1選択にし、失敗時は ACPI shutdown にフォールバックしている。停止待機で timeout した場合は `status/stop` に進み、強制停止後に `status/start`、running 待機という流れになっている。要件の「ソフト停止タイムアウト後の強制電源断フォールバック」と整合している。

### recovery_ha_failover

HA 登録確認は `/cluster/ha/resources` の `sid == vm:<vmid>` で実施している。移行先は `/cluster/status` から `online == 1` かつ現在ノード以外を選択し、`ha-manager crm-command relocate vm:<vmid> <target-node>` を使っている。要求仕様と policy §13 の範囲では妥当。

### IPリテラル / 秘密情報

対象ファイルに対して以下を確認し、IPリテラル・秘密鍵本文・Slack Webhook URL・平文 password は検出されなかった。

```text
rg -n "([0-9]{1,3}\.){3}[0-9]{1,3}|BEGIN (RSA|OPENSSH|PRIVATE)|hooks\.slack\.com/services|slack_webhook|ansible_password|password:" ...
```

### read-only / 変更系の分離

既存 healthcheck role の変更は `skip_notifications` guard 追加のみで、read-only 側に restart/reboot/failover 操作は混入していない。変更系操作は `recovery_service_restart.yml` / `recovery_vm_reboot.yml` / `recovery_ha_failover.yml` に分離されている。
