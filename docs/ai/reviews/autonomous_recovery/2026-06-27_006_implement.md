## 対応内容

### 指摘①: mute ファイル作成が権限的に失敗する

**原因:** `/var/lib/homelab-recovery/mute` は `root:root 0755` のため、`delegate_to: localhost, become: false` で実行される yoshi が書き込めなかった。

**修正:** `roles/recovery_mute/tasks/set.yml` のディレクトリ作成タスクとファイル書き込みタスクに `become: true` を追加。

```yaml
# before
  become: false
# after
  become: true  # ← ディレクトリ作成・ファイル書き込みの両タスク
```

### 指摘②: ロック取得が atomic でない

**原因:** `recovery-trigger.sh.j2` で `-e "$LOCK_FILE"` チェックと `touch "$LOCK_FILE"` が分離していた（check-then-touch）。ほぼ同時の trigger で両方がチェックを通過し、複数 pipeline が起動する可能性があった。

**修正:**

```bash
# before
LOCK_FILE="$LOCK_DIR/${TARGET}.lock"
if [[ -e "$LOCK_FILE" ]]; then
  exit 0
fi
# ... (後から)
touch "$LOCK_FILE"

# after
LOCK_FILE="$LOCK_DIR/${TARGET}.lock"
if ! mkdir "$LOCK_FILE" 2>/dev/null; then   # mkdir は atomic
  logger -t "$LOG_TAG" "SKIP $TARGET: pipeline already running (lock exists)"
  exit 0
fi
# touch は不要（mkdir で作成済み）
```

`recovery-pipeline.sh` の cleanup も `rm -f` から `rmdir` に変更（ロックがディレクトリになったため）。

### 指摘③: 手動 Slack trigger が flapping count に記録される

**原因:** count の読み書きを行った後に `MANUAL == 1` を判定していたため、Slack 手動依頼が count に蓄積され、後続の自動 OnFailure= trigger がフラッピング判定（3回以上）を誤検知する可能性があった。

**修正:** `MANUAL == 1` の場合はカウントの読み書きごとスキップ。構造を以下に変更。

```bash
ESCALATE_FLAG=0
if [[ "$MANUAL" == "1" ]]; then
  logger ...  # count 操作なし
else
  # count 読み取り・書き込み・判定をすべてここに収める
  if (( recent_count >= 3 )); then
    ESCALATE_FLAG=1
  fi
fi
```

### 指摘④: Ansible playbook 内で git pull（core.md §11 違反）

**原因:** `roles/recovery_trigger/tasks/main.yml` に `sudo -u yoshi git pull` タスクを追加していた。core.md §11「Ansible playbook 内で git pull しない」に違反。

**修正:** タスクを丸ごと削除。quory のリポジトリ更新は Ansible 外（手動 `git pull` または将来の Semaphore UI）で行う。
