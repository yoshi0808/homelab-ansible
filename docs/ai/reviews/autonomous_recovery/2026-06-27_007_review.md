## 指摘

### must-fix: mute 中に trigger されると lock が残り続ける

`roles/recovery_trigger/templates/recovery-trigger.sh.j2:47` で atomic lock として `mkdir "$LOCK_FILE"` する修正自体は方向性として正しい。ただし、lock 取得後の `roles/recovery_trigger/templates/recovery-trigger.sh.j2:56` 以降で mute 判定に入り、mute 中なら `roles/recovery_trigger/templates/recovery-trigger.sh.j2:60` で logger を出して `exit 0` している。

今回の cleanup は `roles/recovery_trigger/files/recovery-pipeline.sh:25` の EXIT trap に移されているが、mute 中は pipeline を起動しないため、その cleanup は実行されない。結果として `/var/run/homelab-recovery/<target>.lock` ディレクトリが残り、以後その target の trigger が常に `pipeline already running` 扱いで skip される。

これは前回指摘の「同一 target の並行 pipeline を防ぐ」修正の副作用で、mute が一度発動すると自律復旧経路を手動復旧まで塞ぎ得るため must-fix。

対応案:

- `recovery-trigger.sh` 側で lock 取得直後に `trap 'rmdir "$LOCK_FILE" 2>/dev/null || true' EXIT` を張り、pipeline 起動に成功した後だけ trap を解除する。
- または mute/flapping 前提チェックをすべて終えた後、pipeline 起動直前に atomic lock を取得する。
- どちらの場合も、pipeline 側の cleanup と二重削除になっても問題ない形にする。
