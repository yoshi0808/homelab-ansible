# Review: autonomous_recovery (2026-06-28)

対象: 2026-06-28_001_implement.md

## レビュー体制（通常フローからの逸脱について）

本実装は、通常のCodexレビューフロー（実装→Codexレビュー→修正→確定）
ではなく、設計段階からClaude（対話）・ChatGPT（比較検討）・Codex
（自己診断・実機検証協力）・ユーザーによる対話的なマニュアルレビューと
して進行した。実装規模が大きく、アカウント構成・LLM実行基盤の選定を
含む設計判断が実装と並行して何度も見直されたため。理由：CC実装後に
都度Claudeへ報告、Claudeが疑問点・矛盾点を指摘、CCが調査・修正、という
反復が実質的なレビューサイクルとして機能した。本ファイルはその記録。

## 第2ラウンド: アーキテクチャの全面見直し（v0.4 → v0.5）

実装後の振り返りで、以下の構造的な問題が判明し、全面改訂に至った。

- アカウント分離が「アクターが違うから」という基準で進み、単一運用者の
  ホームラボに対して過剰な複雑さ（3アイデンティティ、ノードごとに別鍵）
  を生んでいた（エンタープライズ的発想の誤用）
- `recovery-exec`が`ann`の鍵（forced command保護を持たない唯一の鍵）の
  コピーを保持しており、コードのバグが直接フルシェルへの経路になる
  構造的な弱点があった
- LLM実行基盤の選定（Claude Code継続）が、既に実績のある別の選択肢
  （Codex CLI、patch advisor運用での実績）との比較なしに進んでいた

詳細な議論の経緯はpolicy v0.5 §0, §3, §14.6〜14.8に記録。対応として
アカウント構成を`ann`/`recovery-io`/`recovery-exec`の2+1構成に再設計し、
LLM実行基盤をCodex CLIに切り替えた。

### 2026-06-30: 削除完了報告の不正確さを手動確認で発見、クリーンアップ完了

commit `9e0196b`（2026-06-28）のメッセージは「旧構成（recovery_slack_listener
等）を削除、ansy・quory上の実アーティファクトも削除済み」としていたが、
平日の振り返り作業中に手動確認したところ、実際には以下が残っていた。

| 項目 | ansy | quory |
|---|---|---|
| `recovery-slack` ユーザー | 残存（鍵・プロセス・systemd・sudoersは無し） | 存在なし |
| `recovery-trigger` ユーザー | 存在なし | 存在なし |
| `recovery-runner` ユーザー | 存在なし | 残存 |
| authy/monnie の authorized_keys | ansy向け（orphan）2エントリ + quory向け2エントリ、計4エントリ | 同上（authy/monnie側で共通管理） |

いずれも鍵・プロセス・sudoers等の実害は確認されなかったが、§13禁止事項
（旧アカウント構成への復帰）が技術的に可能な状態が残っていた。全て削除し、
authy/monnieのauthorized_keysはquory向け2エントリ（investigate鍵・action鍵）
のみに整理済み。確認・削除は全て完了。

教訓: 削除・クリーンアップ系のタスクは「削除した」という報告だけでなく、
対象が実際に存在しないことを別の経路（このケースでは`id <user>`）で
確認するまでレビュー完了としない。また、確認依頼に対してCC自身が
「対象を確認せず、文脈から推測して回答してしまった」場面が1度あった
（recovery-execとrecovery-slackの取り違え）。指摘後は即座に対象を確認し
直し、誤りを認めて訂正する対応ができていた。
