## 対応内容

### 指摘: lock skip される重複 trigger が flapping count に記録される

**原因:** 008_implement で lock 取得をすべてのチェック後（手順4）に移したため、並行 trigger が flapping count（手順3）の読み書きを完了してから lock で弾かれるようになっていた。monnie のように複数 unit が近接発火する環境では、実際には pipeline が1本しか起動していないのに count だけが複数増え、後続の自動 trigger が誤フラッピング判定（直近24h 3回以上）でエスカレートしてしまう可能性があった。

**修正:** チェック順序を以下のとおり変更。

```
変更前の順序 (008_implement 後):
  1. validate
  2. mute チェック    ← lock なし
  3. flapping count   ← lock なし（BUG: 並行重複がここで count される）
  4. lock 取得        ← 遅すぎる
  5. pipeline 起動

変更後の順序:
  1. validate
  2. mute チェック    ← lock なし（mute 中は lock 不要）
  3. lock 取得        ← mute 後・count 前に移動（ここを通過するのは真に新規の trigger のみ）
  4. flapping count   ← lock 取得後（並行重複は ③ で skip 済み）
  5. pipeline 起動
```

**効果:**
- mute 中は lock を取らずに exit できる（007 で修正した lock leak は維持）
- 並行重複 trigger は lock で弾かれ、flapping count に入らない
- flapping count は真に新規の pipeline 起動のみを記録する
