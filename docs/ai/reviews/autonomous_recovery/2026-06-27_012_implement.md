## 対応内容

### 指摘: lock 取得後・pipeline 起動前の異常終了で lock が残る

**原因:** `set -euo pipefail` 環境下で lock 取得後（`mkdir "$LOCK_FILE"`）から pipeline 起動前（`nohup ...`）の間に `jq`・一時ファイル書き込み・`mv` などが失敗すると、スクリプトがその場で終了する。この経路では `recovery-pipeline.sh` が起動していないため、pipeline 側の EXIT trap は実行されず lock ディレクトリが残り続ける。

**修正:** lock 取得直後に trigger スクリプト側でも EXIT trap を設定し、pipeline 起動後に解除（`trap - EXIT`）する。

```bash
# lock 取得直後に追加
trap 'rmdir "$LOCK_FILE" 2>/dev/null || true' EXIT

# ... flapping count 処理 ...

# pipeline 起動後に解除（cleanup を pipeline.sh に引き継ぐ）
nohup sudo -u "$PIPELINE_USER" "$PIPELINE" "$TARGET" "$ESCALATE_FLAG" >"$LOG_FILE" 2>&1 &
trap - EXIT
```

**効果:**
- pipeline 起動前の異常終了（jq 失敗・ディスクフル等）→ trigger 側 EXIT trap が lock を削除
- pipeline 起動成功後 → `trap - EXIT` で trigger 側 trap を解除し、pipeline.sh の EXIT trap が cleanup を担う
- 二重削除は `rmdir 2>/dev/null || true` で無害化
