## 各 Playbook の仕様

### A: recovery_service_restart.yml

**目的**: ラダー第1段。対象 VM のサービスを systemctl restart で再起動し、回復を試みる。

**使用方法**: `ansible-playbook playbooks/recovery_service_restart.yml -e target=<target>`

**対象**: authy(freeradius.service), monnie(grafana-server / prometheus / loki / unpoller)
sophos-fw はサービス restart の対象外（VMリブートのみ）

**実行フロー**:
1. pve1 上で pvesh get /cluster/resources → target VM の存在・タグ再検証
   - hacritical または ops タグが必要
   - sophos-fw は対象外 (assert)
   - VM が running 状態であることを確認
2. target VM 上でサービスを restart
3. サービスが active になるまで待機（max 60s）
4. 結果を Slack 通知（best-effort）+ JSON レポート保存（always）

**Fail 条件**: タグ不適格 / VM 不存在 / サービスが timeout 以内に active にならない

---

### B: recovery_vm_reboot.yml

**目的**: ラダー第2段。VM をリブートして回復を試みる。ソフトリブートが応答しない場合は強制電源断＋起動にフォールバック。

**使用方法**: `ansible-playbook playbooks/recovery_vm_reboot.yml -e target=<target>`

**対象**: authy / monnie / sophos-fw

**QEMU guest agent 有無**（実機確認済み）:
- authy / monnie: あり → guest agent shutdown を第1選択
- sophos-fw: なし → ACPI シャットダウン(qm shutdown)のみ

**実行フロー**:
1. pve1 上で pvesh → タグ再検証(hacritical または ops), VMID・現在ノード取得
2. pvesh create /nodes/{node}/qemu/{vmid}/agent/shutdown (authy/monnie)
   または pvesh create /nodes/{node}/qemu/{vmid}/status/shutdown (sophos-fw, ACPI)
3. VM が stopped になるまで待機（max {{ recovery_vm_soft_shutdown_timeout_s }}s）
4. timeout 内に停止しなければ pvesh create .../status/stop（強制電源断）
5. pvesh create .../status/start → VM が running になるまで待機
6. Slack 通知 + JSON レポート（always）

**1回のみ試行**（policy §8: ラダー各段は1回のみ）

---

### C: recovery_ha_failover.yml

**目的**: ラダー第3段（最終手段）。HA 登録済み VM を HA マネージャー経由で別ノードへフェイルオーバーさせる。

**使用方法**: `ansible-playbook playbooks/recovery_ha_failover.yml -e target=<target>`

**対象**: authy / sophos-fw（hacritical タグ持ち、HA 登録済みのみ）
monnie はフェイルオーバー対象外（ops タグのみ、hacritical なし）

**HA 方式**: Proxmox 組み込み HA マネージャー(`ha-manager crm-command relocate`)
- authy / sophos-fw は HA 登録済みであることをユーザーが確認済み

**実行フロー**:
1. pvesh → タグ再検証(hacritical のみ)、VMID・現在ノード取得
2. pvesh get /cluster/ha/resources → HA 登録確認
3. 移行先ノードの決定:
   - 現在のノードとは別の健全なクラスタメンバーを選択
4. ha-manager crm-command relocate vm:{vmid} {target-node}
5. VM が target-node 上で running になるまで待機（max {{ recovery_ha_timeout_s }}s）
6. Slack 通知 + JSON レポート（always）

### ⑤ skip_notifications フラグ

**目的**: healthcheck playbook をアドホック呼び出し（調査フェーズ）したとき、既存の Slack/アラート通知が二重発火しないように制御する。

**対象 role**:
- `roles/proxmox_healthcheck/tasks/main.yml`
- `roles/radius_healthcheck/tasks/main.yml`
- `roles/monitoring_healthcheck/tasks/main.yml`

**実装**: Slack notify タスクの `when:` に `and not (skip_notifications | default(false) | bool)` を追加

**使用方法**: `ansible-playbook playbooks/radius_healthcheck.yml -e skip_notifications=true`

## 共通設計事項

- タグ検証は全 playbook で実行直前に Ansible 側で行う（policy §5: shell 側では判定しない）
- `-e target` は VM 名（authy / monnie / sophos-fw）で渡す
- 構造化レポートは `reports/recovery_investigations/<target>/YYYYMMDDTHHMMSS.json` に保存
- Slack 通知は best-effort（common_slack/tasks/notify.yml、失敗してもメイン処理を止めない）
- ミュートチェック・フラッピングカウント・実行中ロックは recovery-trigger.sh(別スコープ)が担当。playbook 自体はこれらを持たない
- IPリテラルをファイルに書かない(core.md §3)
