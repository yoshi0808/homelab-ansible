# 再レビュー結果: trigger 側 EXIT trap による lock leak 防止

## 指摘

### must-fix: `nohup sudo ... &` が失敗しても trigger 側 trap を解除してしまう

`roles/recovery_trigger/templates/recovery-trigger.sh.j2:61` で lock を取得し、`roles/recovery_trigger/templates/recovery-trigger.sh.j2:67` で trigger 側の EXIT trap を設定したため、前回指摘した flapping count 処理中の異常終了では lock が解放されるようになった。

ただし `roles/recovery_trigger/templates/recovery-trigger.sh.j2:109` の `nohup sudo -u "$PIPELINE_USER" "$PIPELINE" ... &` は、background process を起動した時点で trigger 側の処理が先へ進む。直後の `roles/recovery_trigger/templates/recovery-trigger.sh.j2:113` で `trap - EXIT` しているため、`sudo` が権限不整合、sudoers不備、pipeline script 不在・実行不可などで即失敗した場合、pipeline 側の `roles/recovery_trigger/files/recovery-pipeline.sh:25` の trap は設定されず、trigger 側の trap も解除済みになる。

この場合、lock cleanup の所有者がいなくなり、`/var/run/homelab-recovery/<target>.lock` が残る。今回直した「pipeline 起動前の異常終了で lock が残る」問題が、`sudo`/exec 失敗時にはまだ残っている。

対応案:

- pipeline handoff 失敗時にも lock を削除する wrapper を background で起動する。
- 例: background wrapper 内で `sudo -u "$PIPELINE_USER" "$PIPELINE" ... || rmdir "$LOCK_FILE" 2>/dev/null || true` とし、pipeline 起動に失敗した場合だけ wrapper が cleanup する。
- pipeline が起動して終了した場合は pipeline 側の EXIT trap が cleanup する。二重 `rmdir` は既存どおり無害化できる。
- あるいは、`sudo -n -u "$PIPELINE_USER" test -x "$PIPELINE"` などの同期チェックを trap 解除前に入れ、少なくとも handoff 前提が壊れている場合は trigger 側 trap を残して終了させる。

## 前回指摘の確認

### resolved: flapping count 処理中の異常終了で lock が残る問題

`roles/recovery_trigger/templates/recovery-trigger.sh.j2:67` で lock 取得直後に trigger 側 EXIT trap が設定された。`roles/recovery_trigger/templates/recovery-trigger.sh.j2:76` 以降の flapping count 処理で `jq`、一時ファイル書き込み、`mv` などが失敗しても、trigger 側 trap が `rmdir "$LOCK_FILE"` を実行するため、前回指摘した lock leak は解消している。
