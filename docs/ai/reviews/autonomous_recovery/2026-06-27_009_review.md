## 指摘

### must-fix: lock skip される重複 trigger が flapping count に記録される

`roles/recovery_trigger/templates/recovery-trigger.sh.j2:95` で lock を取得する位置を mute/flapping 後に移したため、前回指摘した「mute 中の lock 残留」は解消されている。ただし、今度は `roles/recovery_trigger/templates/recovery-trigger.sh.j2:67` から `roles/recovery_trigger/templates/recovery-trigger.sh.j2:89` の flapping count 読み書きが、lock skip より前に実行される。

このため、すでに同 target の pipeline が稼働中で最終的に `roles/recovery_trigger/templates/recovery-trigger.sh.j2:95` で skip される trigger でも、先に `/var/lib/homelab-recovery/counts/<target>.json` に記録される。monnie は複数 unit の OnFailure が同時・近接発火し得るため、実際には pipeline が1本しか起動していないのに count だけが複数増え、後続の自動 trigger が「直近24時間3回以上」と誤判定して ladder を飛ばす可能性がある。

`autonomous_recovery_policy.md` の §6.4 は「lock check → mute → flapping count → pipeline起動」の順序で、lock skip されるものは flapping count に入れない前提に読める。今回の lock leak を避けるには、lock をすべての前提チェック後ではなく、mute check の後、flapping count の前に取得するのが安全。

対応案:

- 順序を `validate → mute check → atomic lock → flapping count → pipeline起動` にする。
- これなら mute 中は lock を取らずに exit でき、同時 trigger は count へ入る前に lock skip できる。
- pipeline 側の `rmdir "$LOCK_FILE"` cleanup は現状のままでよい。
