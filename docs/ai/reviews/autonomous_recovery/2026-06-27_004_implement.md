## 3. アーキテクチャ上の主要決定事項

### 3.1 アジェンティックループ（§6.3）

```
Slack mention → run_loop() → Claude -p (JSON決定のみ) → Python実行 → history蓄積 → 次ターン
```

- Claude Code に `deny: [Bash, Write, Edit, Read, Glob, Grep]` を強制。ツール実行ゼロ
- `--setting-sources project` で `/etc/recovery/claude-ws/.claude/settings.json` のみ読む
- `-p` モードでは `allow` が機能しないことを確認。`deny` のみで制御
- MAX_TURNS=6、SSH_TIMEOUT=15

### 3.2 Python側のガード（2種）

**trigger前検証（`_TRIGGER_PREREQ` + `_is_unhealthy()`）:**
- `trigger_recovery_*` は対応する healthcheck が history に存在し、かつ異常確認済みでなければ Python が即拒否
- `ok=False`（SSH失敗 = 不確実）は異常扱いにしない

**重複アクション拒否:**
- 同一 run_loop 内で同一アクションが history にあれば Python が即拒否
- プロンプト指示への依存なし（system prompt の注記は補助的位置づけ）

### 3.3 アカウント構成

- `recovery-slack` (ansy): リスナーサービス実行ユーザー
- `recovery-slack` (authy/monnie): SSH forced command着地・healthcheck実行
- `trigger` (quory): SSH forced command着地・pipeline起動
- `recovery-runner` (quory): ansible-playbook専用実行ユーザー（sudoで起動）
- SSH forced commandユーザーのシェルは `/bin/sh` 必須（nologinは不可）

### 3.4 CLAUDE_CODE_OAUTH_TOKEN

`ANTHROPIC_API_KEY` は使わない。`claude setup-token` で発行した OAuth トークン（`sk-ant-oat01-` 形式）を Vault 管理。Claude Max サブスクリプション内課金。
`ANTHROPIC_API_KEY` が環境変数にある場合は API キーが優先されるため、`recovery-slack` の環境に混入しないよう注意。
