## 指摘

### must-fix: lock 取得後、pipeline 起動前のエラーで lock が残る

`roles/recovery_trigger/templates/recovery-trigger.sh.j2:61` で atomic lock を取得し、`roles/recovery_trigger/templates/recovery-trigger.sh.j2:73` 以降で flapping count の処理を行い、最後に `roles/recovery_trigger/templates/recovery-trigger.sh.j2:106` で pipeline を起動している。

前回指摘した「重複 trigger が flapping count に入る」問題は、この順序変更で解消している。ただし、lock 取得後から pipeline 起動前までの間に `jq`、一時ファイル書き込み、`mv` などが失敗すると、`set -euo pipefail` により `recovery-trigger.sh` がその場で終了する。この経路では `recovery-pipeline.sh` が起動していないため、`roles/recovery_trigger/files/recovery-pipeline.sh:25` の EXIT trap は実行されず、`/var/run/homelab-recovery/<target>.lock` が残る。

つまり、前回の mute 経路の lock leak は直ったが、flapping count 経路で同種の lock leak がまだ残っている。count file の破損、権限不整合、ディスクフル、一時ファイル作成失敗などの異常時に、その target の後続 recovery trigger が永続的に skip され得るため must-fix。

対応案:

- `mkdir "$LOCK_FILE"` 成功直後に、`recovery-trigger.sh` 側でも `trap 'rmdir "$LOCK_FILE" 2>/dev/null || true' EXIT` を設定する。
- `nohup sudo -u "$PIPELINE_USER" "$PIPELINE" ... &` が成功し、pipeline 側に lock cleanup を引き継いだ後で `trap - EXIT` する。
- これにより、pipeline 起動前の異常終了では trigger 側が lock を消し、pipeline 起動後は pipeline 側の EXIT trap が lock を消す。

## 前回指摘の確認

`roles/recovery_trigger/templates/recovery-trigger.sh.j2:49` から `roles/recovery_trigger/templates/recovery-trigger.sh.j2:55` の mute check は lock 取得前に残っており、mute 中は lock を作らず終了する。
