## 概要

レビュー 003 の Medium 指摘 3 件をすべて修正した。修正は `roles/unifi_backup_fetch`
と `playbooks/unifi_backup_fetch.yml`、および要求仕様の CSRF 優先順位追記に限定し、
`ubuntu_vm_patch_policy.md` が定義する Ubuntu の patch / reboot 動作には一切影響しない
（本 playbook は CloudKey バックアップ取得専用で、Ubuntu ノードの reboot 制御とは無関係）。

## 指摘への対応

### [Medium] 同名ファイル時の成功誤判定 → 修正済み

対象: `roles/unifi_backup_fetch/tasks/main.yml`（Phase 4 確定処理）

- 確定タスクの `mv -f` から `creates:` を削除し、`changed_when: true` を付与した。
  これにより CloudKey が同一 filename を返した場合でも、毎回ダウンロードした最新を
  **必ず上書き保存**する。
- 旧実装では `creates` により同名存在時に `mv` がスキップされ、新規一時ファイルは
  `always` で削除されるのに「fetched and saved」と通知され得た。この誤成功経路を解消した。
- 一時ファイルは保存先と同一 FS 上にあり、`mv` は原子的 rename として機能する点は維持。

### [Medium] レスポンスヘッダーからの CSRF 取得未実装 → 修正済み

対象: `roles/unifi_backup_fetch/tasks/main.yml`（Phase 1）

- CSRF を要求仕様どおり**レスポンスヘッダー優先**で取得するよう変更した。
  優先順位: `X-CSRF-Token` → `X-Updated-CSRF-Token` → （両方無い場合のみ）JWT
  ペイロードの `csrfToken` を fallback。
  - uri モジュールはヘッダーを小文字 + ハイフン→アンダースコア化したキーで返すため、
    `unifi_login.x_csrf_token` / `unifi_login.x_updated_csrf_token` で参照する。
  - 選択は `default(..., true)` チェーンで実装（空文字も未供給として扱う）。
- 最終的に CSRF が空でないことを assert（`unifi_csrf | length > 0`）し、fail_msg で
  「ヘッダーにも JWT にも使用可能な CSRF が無い」ことを明示する。
- 使用した供給源を `unifi_csrf_source`（`X-CSRF-Token` / `X-Updated-CSRF-Token` /
  `JWT-csrfToken`）として記録し、Slack 通知サマリに出力する（トークン値は含めない）。
- レビューの要請どおり、JWT を fallback として残す方針と優先順位を
  `2026-06-15_001_requirement.md` の認証方式に追記した。

### [Medium] API 由来 filename の境界検証不足（パストラバーサル）→ 修正済み

対象: `roles/unifi_backup_fetch/tasks/main.yml`（Phase 3 filename 検証）

- filename の検証を allowlist 方式へ厳格化した。
  - 旧: `^unifi_os_backup_[0-9]+_.*\.unifi$`（`.*` が `/`・`..` を許容）
  - 新: `^unifi_os_backup_[0-9]+_[0-9A-Za-z-]+\.unifi$`（UUID 部を英数字とハイフンに限定）。
    この allowlist により `/`・`\`・`..`・`;`・空白は**構造的に通過不可**。
- 多層防御として、確定前に以下も assert する。
  - `unifi_backup_filename == (unifi_backup_filename | basename)`（basename 一致）
  - `'/' not in ...` / `'..' not in ...` / `'\\' not in ...`（区切り・親参照・バックスラッシュ不在）
- 検証は root の `mv` に filename を渡す**前**に行うため、保存先ディレクトリ外への
  書き出しを防止する。

## 未対応事項 / スコープ外

- CloudKey / Synology NFS を使う実機 E2E 実行は変更系操作を伴うため未実施（本番/開発
  ホストでの手動実行が必要）。レビューでも実機 E2E は未実施として整合。
- 固定一時ファイルの並行実行競合は、Semaphore で同一 job の同時実行を認めない運用前提で
  対象外（レビュー確認結果に準拠）。
- systemd timer / Semaphore Schedule への週次登録は運用側設定（本実装スコープ外）。

## 注意点

- CSRF 供給源は `X-CSRF-Token` を最優先とし、JWT は両ヘッダー欠落時のみの安全網。
  `/api/backup/download` は GET だが、要求仕様に沿って認証ヘッダー一式
  （Cookie / X-CSRF-Token / Origin）を付与する方針は維持。
- filename の allowlist は CloudKey の通常出力（`unifi_os_backup_<epoch_ms>_<uuid>.unifi`）に
  整合。CloudKey が想定外の文字を含む名前を返した場合は確定前に fail し、保存・通知しない。
- 秘密情報（認証情報・TOKEN・CSRF・認証ヘッダー）を扱うタスクには `no_log: true` を維持。
  `unifi_csrf_source` はラベルのみでトークン値を含まないため通知に出力してよい。
- IP リテラル不記載・秘密情報の非コミット方針（core.md §3 / §17）を維持。
