# 実装報告: unifi_backup_fetch（UniFi OSバックアップ取得）

対象: `docs/ai/reviews/unifi_backup_fetch/2026-06-15_001_requirement.md`

## 実装概要

CloudKey Gen2 Plus（UniFi OS）のシステムバックアップを取得し、pve1 上の
Synology NFS マウントへ保存する変更系 playbook / role を新規実装した。

`/api/backup/download` は呼び出した瞬間に新規バックアップを生成して返すため、
毎回最新のバックアップが保証される。フルライフサイクルを block / rescue / always
で構成し、ファイル名のタイムスタンプによる鮮度ガード、8世代ローテーション、
Slack 通知（best-effort）を行う。

実行ホストは pve1（`connection: local` は使わず、pve1 上で `become: true`）。
NFS のアクセス権が root に絞られているため root で実行する。CloudKey へは
`cloudkey.internal` で接続する（私設 CA のため `validate_certs: false`）。

## 変更内容の要約

### playbooks/unifi_backup_fetch.yml
- `hosts: pve1` / `become: true` / `gather_facts: true`。
- `vars_files` で `inventories/vars/cloudkey.yml`（Vault）を読み込む。
- `block` で role を `include_role` 実行。
- `rescue` で失敗を捕捉し `unifi_backup_failed` / `unifi_backup_error` を記録。
- `always`:
  - ダウンロード一時ファイルの残骸を削除。
  - サマリを組み立て。
  - `roles/common_slack/tasks/notify.yml` を `include_tasks`（成功=info/ok、失敗=alerts/error）。
  - 失敗時のみ再 `fail`。

### roles/unifi_backup_fetch/tasks/main.yml
- **Phase 1（認証）**: `POST /api/auth/login` → `TOKEN`(JWT) 取得 → JWT ペイロードの
  `csrfToken` を base64url デコードして CSRF を導出（`cloudkey_cert_deploy` と同方式）→
  `Cookie` / `X-CSRF-Token` / `Origin` を持つ認証ヘッダを構築。秘密情報タスクは `no_log: true`。
- **Phase 2（ダウンロード）**: 保存先と同一 FS の一時ファイルへ `GET /api/backup/download`。
  確定（rename）は鮮度ガード通過後に行う。
- **Phase 3（鮮度ガード）**: `Content-Disposition` から filename を抽出（クォート有無の両形式に対応）。
  filename 形式を assert（`unifi_os_backup_<ms>_*.unifi`）。埋め込みタイムスタンプ(ms)と
  実行ホスト（pve1）の現在時刻 `date +%s%3N` の差の絶対値が閾値（既定 60 秒）超なら fail。
- **Phase 4（保存）**: 同一 FS 内 `mv -f`（原子的 rename）で確定し、パーミッションを 0640 に正規化。
- **Phase 5（ローテーション）**: `unifi_os_backup_*.unifi` を列挙し、ファイル名昇順（= タイムスタンプ昇順）で
  ソート。`keep_generations`（既定 8）を超える古いものを削除。

### roles/unifi_backup_fetch/defaults/main.yml
- `cloudkey_host: cloudkey.internal`
- `unifi_backup_login_path` / `unifi_backup_download_path`
- `unifi_backup_dest_dir: /mnt/pve/Synology-nfs/user-backup/unifi`
- `unifi_backup_tmp_path`（保存先と同一 FS の隠し一時ファイル）
- `unifi_backup_keep_generations: 8`
- `unifi_backup_freshness_max_seconds: 60`
- `unifi_backup_download_timeout: 120`

## 未対応事項 / 初回除外スコープ

- 鮮度チェック専用 playbook は作らない（fetch playbook 自身の成否通知で代替）。
- Network アプリバックアップ（`.unf`）取得は対象外（OS バックアップで十分）。
- CloudKey へ実接続しての E2E 実行は未実施（要本番/開発ホストでの手動実行）。
- systemd timer / Semaphore Schedule への週次登録は本実装の対象外（運用側で設定）。

## 注意点

- 鮮度ガードの「現在時刻」は実行ホスト pve1 上で `date +%s%3N` を取得する
  （`lookup('pipe')` の制御ノード時刻ではない）。CloudKey と pve1 の NTP 同期を前提とする。
- CSRF は要求仕様では「`X-CSRF-Token` / `X-Updated-CSRF-Token` ヘッダから取得」と
  記載があるが、実装は実績のある `cloudkey_cert_deploy` と同一の「JWT ペイロードの
  `csrfToken` をデコード」方式を採用した（堅牢性のため）。`/api/backup/download` は
  GET だが、要求仕様に沿って認証ヘッダ一式（Cookie / X-CSRF-Token / Origin）を付与している。
- 一時ファイルは保存先と同一 FS（NFS）上に置き、rename を原子的にしている。
  失敗時は playbook の `always` で残骸を削除する。
- 世代ソートはファイル名のエポックミリ秒（固定 13 桁）に依存しており、辞書順 =
  時刻順が成立する前提（西暦 2286 年頃まで桁数は不変）。
- 秘密情報（認証情報・TOKEN・CSRF・認証ヘッダ）を扱うタスクには `no_log: true` を設定済み。
  秘密情報はリポジトリに保存していない。IP リテラルも記載していない（core.md §3）。
