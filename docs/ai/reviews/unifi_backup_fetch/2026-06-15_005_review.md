## 指摘事項

### [Medium] ヘッダーが有効でも JWT fallback のデコード失敗で処理が停止する

対象: `roles/unifi_backup_fetch/tasks/main.yml:42-67`

要求仕様では、`X-CSRF-Token`、`X-Updated-CSRF-Token` の両方が存在しない場合に
限って JWT ペイロードをデコードする。しかし実装は、レスポンスヘッダーの有無を
判定する前に、常に TOKEN の分割、base64 デコード、JSON 変換、`csrfToken` 抽出を
実行している。

このため、レスポンスヘッダーに使用可能な CSRF トークンが返っていても、JWT の
形式やペイロード構造が変化すると fallback 値を作るタスクで失敗し、ヘッダー値を
選択するタスクまで到達しない。実装報告の「両方無い場合のみ JWT fallback」と
実際の評価順も一致していない。

最初に2つのレスポンスヘッダーを選択し、両方が空の場合だけ `when` 付き block 等で
JWT をデコードすること。その後、ヘッダー値または JWT fallback 値から最終的な
`unifi_csrf` と `unifi_csrf_source` を決定すること。

### [Low] CSRF ヘッダー値を参照する供給源判定タスクに `no_log` がない

対象: `roles/unifi_backup_fetch/tasks/main.yml:69-75`

`Record which CSRF source was used` は結果として供給源ラベルだけを保存するが、
判定式では秘密情報である `unifi_login.x_csrf_token` と
`unifi_login.x_updated_csrf_token` の値を直接参照している。

要求仕様は TOKEN・CSRF・認証情報を扱うタスクに `no_log: true` を求めているため、
このタスクにも `no_log: true` を設定すること。最終 assert も TOKEN / CSRF を
参照するため、同じ基準を厳密に適用するなら併せて設定すること。
