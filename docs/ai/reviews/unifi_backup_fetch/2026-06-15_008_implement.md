## 概要

006（再レビュー 005 反映）後、**pve1 上で実機 E2E テストを実施**した。その過程で
コードの不具合 1 件と、要求仕様の前提（レスポンスヘッダー `Content-Disposition` から
ファイル名取得）と実機挙動の差異 1 件が判明したため、2 点を修正した。修正後、
取得→鮮度ガード→保存→ローテーション→Slack 通知まで E2E で成功している。

いずれの修正も CloudKey バックアップ取得ロジックに閉じており、`ubuntu_vm_patch_policy.md`
が定義する Ubuntu の patch / reboot 動作には影響しない。

## 修正内容

### [Fix-1] 失敗時の always ブロックで role defaults が undefined になる

事象: Phase 1（ログイン）で失敗した実行で、playbook の `always` が
`'cloudkey_host' is undefined` で二次失敗し、サマリ生成・temp 削除・Slack 通知が
動かなかった。

原因: `cloudkey_host` / `unifi_backup_dest_dir` / `unifi_backup_tmp_path` は role の
`defaults` 変数であり、`include_role` 終了後（rescue / always）の play スコープでは
参照できない。

対応:
- role 先頭に `Phase 0 - Initialize fetch facts` を追加し、ログイン前に
  `set_fact`（play スコープに残る）で以下を確定する。
  - `unifi_backup_failed: false`
  - `unifi_backup_cloudkey`（= `cloudkey_host`）
  - `unifi_backup_dest`（= `unifi_backup_dest_dir`）
  - `unifi_backup_tmpfile`（= `unifi_backup_tmp_path`）
- playbook の `always` を、role defaults 直参照から上記の永続 fact 参照へ変更。
  - temp 削除: `path: "{{ unifi_backup_tmpfile }}"` / `when: unifi_backup_tmpfile is defined`
  - サマリ: `cloudkey: {{ unifi_backup_cloudkey | default('n/a') }}` /
    `dest: {{ unifi_backup_dest | default('n/a') }}`
- これにより、Phase 1 で落ちても失敗サマリと #alerts 通知が正しく送られる。
- 既存の cloudkey_cert_deploy が「Initialize facts を先頭に置き、以降は set_fact 値を
  参照する」設計だったのと同じ方針に揃えた。

### [Fix-2] `/api/backup/download` は Content-Disposition を返さず、uri が `filename` に格納する

事象: ログイン・ダウンロードは成功（ファイルは取得済み）するが、Phase 3 で
`object of type 'dict' has no attribute 'content_disposition'` で失敗した。

原因（実機ダンプで確認）: ダウンロード応答に `content_disposition` キーは存在せず、
代わりに `ansible.builtin.uri` が `dest` 利用時にサーバ提示のファイル名を **`filename`**
フィールドへ格納していた。実測の応答キー例:
`['... 'content_type', 'filename', 'path', 'size', 'status', 'url' ...]`、
`filename = unifi_os_backup_1781526859523_<uuid>.unifi`、`content_type = application/octet-stream`。

対応:
- Phase 3 のファイル名取得を、`content_disposition` の正規表現パースから
  `unifi_backup.filename` の直接利用へ変更。
  - `unifi_backup_filename: "{{ unifi_backup.filename | default('') | basename | trim }}"`
  - 念のため `basename` でパス成分を除去。
- 後続の allowlist assert（`^unifi_os_backup_[0-9]+_[0-9A-Za-z-]+\.unifi$`、basename 一致、
  `/`・`..`・`\` 不在）はそのまま維持。実 UUID（hex + ハイフン）は allowlist を通過する。
- 鮮度ガード（filename の ms タイムスタンプ vs 実行ホスト現在時刻、60 秒）も変更なし。

# ログイン probe（実資格・ステータスのみ表示）

  → CSRF はヘッダーから取得され、JWT fallback は発動しない（005 修正の実証）

## 確認できたこと

- ログイン → CSRF（ヘッダー）→ DL → 鮮度ガード通過 → Synology 保存 → ローテーション
  → Slack 通知（#info）まで E2E 成功。
- CSRF 供給源はヘッダー（`X-CSRF-Token`）。両ヘッダーが返るため JWT は一切デコードしない。
- 保存ファイル名はサーバ提示の `unifi_os_backup_<ms>_<uuid>.unifi` で、allowlist 検証を通過。

## 未対応事項 / 注意点

- 世代ローテーションの実削除（8 世代超）は、まだ 1 世代のため今回の実機では未発火。
  ロジックはダミーデータ検証済み（10 件→8 残し 2 削除 / 5 件→0 削除）。複数世代での
  実機確認は今後の運用で蓄積されてから。
- 鮮度ガードは実行ホスト pve1 の `date +%s%3N` を基準にする。pve1 と CloudKey の
  NTP 同期がずれると Phase 3 で fail する。今回のテストでは問題なく通過。
- `/api/backup/download` は変更系（CloudKey 上で新規バックアップ生成 + Synology 書込/削除）。
  実行判断は Yoshinobu。
- IP リテラル不記載・秘密情報の非コミット方針（core.md §3 / §17）を維持。
