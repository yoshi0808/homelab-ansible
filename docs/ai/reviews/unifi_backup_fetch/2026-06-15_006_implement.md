## 指摘への対応

### [Medium] ヘッダー有効時でも JWT デコードを実行していた → 条件実行に修正

対象: `roles/unifi_backup_fetch/tasks/main.yml` Phase 1

- 旧実装は、ヘッダー有無を判定する前に常に「TOKEN 分割 → base64 デコード →
  JSON 変換 → `csrfToken` 抽出」を実行していた。このため、ヘッダーに有効な CSRF が
  あっても JWT 形式・ペイロード構造が変わると fallback 生成タスクで失敗し得た。
- 評価順を要求仕様どおりに作り直した。
  1. **先にレスポンスヘッダーから CSRF を選択**（`X-CSRF-Token` 優先、無ければ
     `X-Updated-CSRF-Token`）。同時に供給源ラベル `unifi_csrf_source` を決定し、
     `_unifi_csrf_jwt` を空文字で初期化する。
  2. **両ヘッダーが空のときだけ** `when: _unifi_csrf_header | length == 0` を付けた
     block で JWT をデコードする（payload 抽出 → base64 パディング → `csrfToken` 抽出）。
     ヘッダーが有効な場合、この block は丸ごとスキップされ JWT を一切触らない。
  3. 最終 `unifi_csrf` を「ヘッダー値があればそれ、無ければ JWT fallback 値」で決定する。
- 実装報告（004）の「両方無い場合のみ JWT fallback」と実際の評価順を一致させた。

### [Low] 秘密値を参照するタスクに `no_log` を補完

対象: `roles/unifi_backup_fetch/tasks/main.yml` Phase 1

- 供給源判定は、ヘッダー値（`x_csrf_token` / `x_updated_csrf_token`）を参照する
  CSRF 選択タスクへ統合し、そのタスクに `no_log: true` を設定した。
  `unifi_csrf_source` はラベルのみで、変数自体は no_log でも保持され通知に使える。
- JWT デコード block 全体に `no_log: true` を付与した。
- 最終 CSRF 決定タスクにも `no_log: true` を設定した。
- 認証材料 assert は TOKEN / CSRF を参照するため、要求仕様 001 の秘密情報方針に
  厳密に合わせて `no_log: true` を追加した。

## 注意点

- 認証材料 assert に `no_log: true` を付けたため、CSRF 導出が全滅した稀なケースでは
  失敗詳細（fail_msg）が伏せられ、rescue 経由の Slack 文面が "unknown error" に
  なり得る。これは既存の no_log タスク（login / download）と同じ挙動で、alerts
  チャンネルへの失敗通知自体は発火する。鮮度ガードや filename 検証など他の失敗点は
  no_log ではないため、引き続き具体的なエラーが通知される。
- `/api/backup/download` は GET だが、要求仕様に沿って認証ヘッダー一式
  （Cookie / X-CSRF-Token / Origin）を付与する方針は維持。
- IP リテラル不記載・秘密情報の非コミット方針（core.md §3 / §17）を維持。
