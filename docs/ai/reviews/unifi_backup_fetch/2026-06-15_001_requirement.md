# 要求仕様: unifi_backup_fetch（UniFi OSバックアップ取得）

## 目的
CloudKey Gen2 Plus の UniFi OS システムバックアップを週次で自動取得し、
Synology NAS に保存する。
`/api/backup/download` は叩いた瞬間に新規バックアップを生成して返すため、
毎回最新のバックアップが保証される。

## 対象
- 取得元: CloudKey Gen2 Plus（`cloudkey.internal`）
- 保存先: `/mnt/pve/Synology-nfs/user-backup/unifi/`（pve1 上の既存 NFS マウント）
- 実行元: quory（本番・Semaphore 週次）/ ansy（開発・CLI）
- 実行ホスト: pve1（connection: local は使わず、pve1 上で become: true で実行）

## 作成・更新ファイル
- playbooks/unifi_backup_fetch.yml（新規・変更系）
- roles/unifi_backup_fetch/（新規）
  - tasks/main.yml
  - defaults/main.yml
- inventories/vars/cloudkey.yml に UniFi 認証情報を追記（既存 Vault ファイル）
  - 変数名: `cloudkey_api_user`（既存）、`cloudkey_api_password`（既存）
  - 既存変数をそのまま流用する。追加不要の場合はそのまま使う。

## 安全度
変更系（CloudKey 上でバックアップを生成する操作を含む。ただし CloudKey の
設定変更は行わない）

## 実施内容（フルライフサイクル / block・rescue・always）

```text
Phase 1  POST /api/auth/login → TOKEN（Cookie）と CSRF トークンを取得する
Phase 2  GET /api/backup/download → バックアップ生成 + ダウンロード（6秒程度）
Phase 3  レスポンスヘッダの filename: から取得時刻タイムスタンプを検証する
         （現在時刻との差が閾値以内であることを確認する。鮮度ガード）
Phase 4  ファイルを Synology の保存先に書き出す
Phase 5  8世代を超える古いファイルを削除してローテーションする
rescue   失敗を捕捉する
always   Slack 通知 → 結果に応じて再 fail
```

## 認証方式
cert_renew_cloudkey と同じ方式。

- ログイン: POST `https://cloudkey.internal/api/auth/login`
  - ボディ: `{"username": "{{ cloudkey_api_user }}", "password": "{{ cloudkey_api_password }}"}`
- TOKEN: レスポンスの `Set-Cookie` の `TOKEN`（JWT）を Cookie として保持
- CSRF: レスポンスヘッダの `X-CSRF-Token` または `X-Updated-CSRF-Token` から取得
  - 優先順位（2026-06-15_003 レビュー反映）: `X-CSRF-Token` を最優先、無ければ
    `X-Updated-CSRF-Token`。両ヘッダとも存在しない場合に限り、JWT ペイロードの
    `csrfToken` をデコードした値を fallback として使用する。最終的に CSRF が
    空でないことを assert する。使用した供給源は通知に記録する。
- 状態変更系リクエストには必ず以下を付与する:
  - `Cookie: TOKEN=<JWT>`
  - `X-CSRF-Token: <csrfToken>`
  - `Origin: https://cloudkey.internal`
- `validate_certs: false`（私設 CA のため）

認証情報は `inventories/vars/cloudkey.yml`（Ansible Vault 暗号化済み）から取得する。

## ファイル命名と世代管理
- ファイル名: レスポンスヘッダの `filename:` をそのまま使用する
  （例: `unifi_os_backup_1781524182600_<uuid>.unifi`）
- 保存先: `/mnt/pve/Synology-nfs/user-backup/unifi/`
- 世代数: 8世代。古いものから削除する（ファイル名のタイムスタンプ昇順でソート）

## 鮮度ガード（Phase 3）
ファイル名に埋め込まれたタイムスタンプ（ミリ秒）と現在時刻を比較し、
差が 60 秒を超える場合は fail する（古いキャッシュや意図しないファイルを検出）。

## 通知
`roles/common_slack/tasks/notify.yml` を include_tasks で呼ぶ（best-effort）。

| 状況 | チャンネル | status |
|---|---|---|
| 取得・保存 OK | info | ok |
| 失敗（認証・ダウンロード・保存・鮮度NG） | alerts | error |

## 制約
- IP リテラルをファイルに書かない（core.md §3）。`cloudkey.internal` で接続する。
- 秘密情報（TOKEN・CSRF・認証情報）を扱うタスクには `no_log: true` を設定する。
- become: true（root）で実行する（NFS のアクセス権が root に絞られているため）。
- 秘密情報をリポジトリに保存しない。

## 初回実装スコープ
- 認証（TOKEN + CSRF 取得）
- バックアップダウンロード
- 鮮度ガード
- Synology 保存
- 8世代ローテーション
- Slack 通知

## 初回除外スコープ
- 鮮度チェック専用 playbook（fetch playbook 自身の成否通知で代替）
- Network アプリバックアップ（`.unf`）の取得（OS バックアップで十分）

## read-only / 変更系の区別
変更系。CloudKey 上でバックアップを生成する操作を含む。
Synology 上のファイル書き込み・削除を行う。
