# 要求仕様: cloudkey_cert_deploy（CloudKey証明書のAPI自動デプロイ）

作成日: 2026-06-13
版: 1.0（要求仕様）
対象: CloudKey Gen2 Plus (UniFi OS) のWeb UI用TLS証明書の自動更新

---

## 1. 目的

CloudKey Gen2 Plus のローカルWeb UI（https://cloudkey.internal）のTLS証明書を、
Home-TLS-CA（中間CA）署名の自動証明書で自動更新する。

これにより以下を達成する。

- CloudKeyをLet's Encrypt運用から離脱させ、homelab全体の証明書を
  Home-TLS-CA配下の私設運用に統一する。
- Let's Encrypt利用に伴う Certificate Transparency ログへの
  ホスト名（ub-unifi.ddnsfree.com）露出を解消する。

### 1.1 リスクの明示（合意済み）

本仕組みは、UniFi OSの**非公開API**（`/api/userCertificates`）を利用する。
公式にサポートされた証明書管理APIではないが、以下を合意したうえで採用する。

- UniFi OSのアップデートによりAPIの仕様変更・廃止が起こり得る。
- その場合、証明書更新が失敗するが、**UIログインや管理機能自体は
  失われない**（証明書が期限切れになってブラウザ警告が出るだけ）。
- 公式の証明書管理APIが提供される場合は、そちらへの移行を再検討する。

この内容判断は意図的なものであり、本仕組みの失敗は homelab の
他のサービス（cert_renew: pve/monnie/ansy/quory）には影響しない。

| 項目 | 内容 |
|---|---|
| デプロイ先 | CloudKey Gen2 Plus（cloudkey.internal） |
| 実行元（本番） | quory（Semaphoreから実行） |

  （APIライフサイクルを実証済み。実装の参考にする。詳細は末尾「参考」）
- Vault暗号化された認証情報: `inventories/vars/cloudkey.yml`
- 署名用CA: quory および ansy の `/home/yoshi/.cert/ca/`

### 3.1 cert_renew とは独立させる理由

CloudKeyの証明書デプロイは以下の点でcert_renewと根本的に異なるため、
同一playbook/roleに混ぜず独立させ、障害を分離する。

- 鍵アルゴリズムがRSA（cert_renewはP-384 ECDSA）
- 配送（ファイル配置・サービス再起動ではなく、HTTP APIの多段呼び出し）
- 認証（SSH（ann鍵 + become）ではなく、APIログイン（yoshi-localのパスワード））

CloudKey対応が壊れても、cert_renew（pve/monnie/ansy/quory）は
独立して動作する。

## 4. 証明書仕様（CloudKey固有の制約・実機検証で確定）

| 項目 | 値 | 備考 |
|---|---|---|
| 署名CA | Home-TLS-CA（中間CA） | cert_renewと同じCAを使う |
| 鍵アルゴリズム | **RSA 2048** | UniFi OSはECDSA証明書を受け付けない |
| 鍵フォーマット | **PKCS#1**（`BEGIN RSA PRIVATE KEY`） | PKCS#8では登録に失敗する |
| 有効期間 | 45日 | cert_renewと統一。force運用（毎回新規発行） |
| SAN | DNS:cloudkey.internal, IP（名前解決から動的取得） | IPリテラルは記載しない（getentで動的取得） |
| keyUsage | critical, digitalSignature, keyEncipherment | |
| extendedKeyUsage | serverAuth | |
| アップロードする証明書 | **フルチェーン**（リーフ + Home-TLS-CA + Home-RADIUS-CA を連結） | 私設CAではルートまで含めないと検証失敗 |

## 5. 実施項目（処理フロー・実機確定）

```
1. リーフ証明書を発行（実行元ローカルで署名。毎回新規＝指紋ユニーク）
2. CloudKeyへログイン（POST /api/auth/login）してTOKENを取得
3. 新証明書をアップロード（POST /api/userCertificates。世代名で登録）
4. アップロードした証明書を有効化（PUT /api/userCertificates/{id}/status）
5. 配信中の証明書を検証（TLS提示証明書の指紋とAPIレスポンスの指紋を照合）
6. 古い自動証明書を削除（DELETE /api/userCertificates/{old_id}）
7. 結果をSlackへ通知
```

### 5.1 API仕様（実機確定）

| 操作 | メソッド + パス | ボディ |
|---|---|---|
| ログイン | POST /api/auth/login | {username, password} |
| 一覧 | GET /api/userCertificates | － |
| アップロード | POST /api/userCertificates | {name, key, cert} |
| 有効化 | PUT /api/userCertificates/{id}/status | {"active": true} |
| 削除 | DELETE /api/userCertificates/{id} | （ボディなし。204応答） |

### 5.2 認証方式（実機確定）

- ログイン応答の `Set-Cookie` に含まれる `TOKEN`（JWT、有効期間2時間）を使う
- CSRFトークンは、JWTペイロードの `csrfToken` クレームに含まれる
- 以後の全リクエスト（特に状態変更系 POST/PUT/DELETE）に以下を付与する:
  - `Cookie: TOKEN=<JWT>`
  - `X-CSRF-Token: <csrfToken>`
  - `Origin: https://cloudkey.internal`

### 5.3 接続に関する必須要件（実機確定）

- **接続はホスト名（cloudkey.internal）で行う。IPアドレス直接 + Origin
  欠如の組み合わせでは DELETE が 403 Forbidden で拒否される。**
- 実行元（quory / ansy）が cloudkey.internal を名前解決できること
  （DNS（Sophos）+ /etc/hosts の二重化で確認済み）

### 5.4 命名と世代管理（delete込みのフルライフサイクル）

- アップロード時の証明書名は世代が分かるユニーク名とする（例: `cloudkey-YYYY-MM`）
- 名前・指紋ともにユニーク制約があり、同名・同指紋は登録できない
  （`USER_CERTIFICATE_DUPLICATE`）。force運用で毎回新リーフを焼けば自然に回避
- 「新規アップロード → 有効化 → 旧削除」の順とし、切り替わる前に旧を消さない
  （ダウンタイム・ロールバック可能性の確保）
- 削除対象は `source=uploaded` かつ非アクティブのものに限定する。
  Let's Encrypt証明書（name: "Yoshi", source: lets_encrypt）は**削除対象から除外**する

---

## 6. 確認項目

- アップロード後、TLSで実際に配信される証明書の指紋が、
  APIレスポンスの fingerprint と一致すること（机上でなく実通信で検証）
- 有効化後、`openssl s_client -connect cloudkey.internal:443` のチェーンが
  リーフ → Home-TLS-CA → Home-RADIUS-CA の3階層になること
- 削除は source=uploaded かつ非アクティブのものに限定され、
  アクティブ証明書とLet's Encrypt証明書を誤って削除しないこと

## 7. 制約と認証情報

- CloudKeyローカル管理者の認証情報はAnsible Vaultで暗号化済み。
  - ファイル: `inventories/vars/cloudkey.yml`（配置・両ホストで復号確認済み）
  - 変数: `cloudkey_api_user`（= yoshi-local）, `cloudkey_api_password`
  - vault_password_file: `/home/yoshi/.ansible/vault/homelab_vault_pass`
    （ansible.cfg経由で自動解決。既存のmail.yml/slack.ymlと同方式）
  - 注: yoshi-local は人間のログインと共用の既存ローカル管理者アカウント。
    将来、自動化専用の最小権限アカウントへの分離を検討する（今回は対象外）
- 秘密鍵の中身を表示・コミットしない
- リーフ秘密鍵は実行元ローカルで生成し、APIボディ（JSON）にのみ含める。
  ファイルとしてGit管理しない
- 破壊的操作（DELETE）を含むため、read-onlyではなく変更系playbookとして扱う。
  playbook名・配置で変更系と分かるようにする（core.md §9 に準拠）

## 8. 実行経路と資材配置

### 8.1 実行経路

- **本番**: quory の Semaphore Task Template から実行。逐次スケジュール。
  force再発行（毎回新指紋）
- **開発**: ansy からの CLI 実行を許可する
  （通信先がCloudKey外部APIであり、ansy自身は再起動しないため、
   cert_renewのようなansy自己除外ロジックは不要）

### 8.2 資材配置（実施済み）

CloudKey対応をquory/ansy双方から実行可能にするため、以下を両ホストに配置済み。

- Home-TLS-CA（中間CA）: 証明書 + 秘密鍵 → `/home/yoshi/.cert/ca/home_tls_ca.{crt,key}`
- Home-RADIUS-CA（ルートCA）: 証明書のみ → `/home/yoshi/.cert/ca/radius_ca.crt`
  （フルチェーン構築に使用）
- Vault暗号化された認証情報: `inventories/vars/cloudkey.yml`

注: cert_renew のレビューでは、assertによるquory限定のためansyにCA不要と
判断しCAをansyから削除したが、本playbookはansyからの実行を許可するため、
ansyへCAを再配置した。用途が異なるため矛盾しない。
