## 実装概要

要求仕様に基づき、CloudKey Gen2 Plus (UniFi OS) のWeb UI用TLS証明書を、
Home-TLS-CA署名のRSA/PKCS#1リーフで自動更新する playbook / role を実装した。

実証済みシェルスクリプトのAPIライフサイクル（発行 → login → upload →
activate → verify → delete）を、リポジトリ流儀に合わせて Ansible ネイティブ
（`community.crypto` + `ansible.builtin.uri`）で再構成した。cert_renew とは
完全に独立している。

責務分離（core.md §7 / §9）:
- 証明書発行・API操作・判定・通知はすべて Ansible tasks 側に置いた。
- 破壊的操作（DELETE）を含むため、変更系 playbook として命名・配置した。
- リーフ秘密鍵は `tempfile` の一時ディレクトリと API ボディ内にのみ存在し、
  `always` で必ず削除する。Git管理しない。

## 設計判断

### 実行モデル
- `hosts: localhost` / `connection: local` / `gather_facts: true`。
  通信先はAnsible管理対象ホストではなく外部CloudKey APIのため、ann/SSH/become
  は使わず、実行元（quory / ansy）の yoshi ローカルで完結する。
- 冒頭で `ansible_hostname in ['quory', 'ansy']` を assert。ansy 実行を許可する
  ため cert_renew のような自己除外ロジックは不要（requirement §8.1）。

### 証明書発行（issue.yml, community.crypto）
- `openssl_privatekey`（type=RSA, size=2048, **format=pkcs1**）で
  `BEGIN RSA PRIVATE KEY` を生成（PKCS#8では登録失敗するため）。
- SANのIPは `getent ahostsv4 cloudkey.internal` で動的解決して付与。
- `openssl_csr` に keyUsage critical / EKU serverAuth を設定。
- `x509_certificate`（provider=ownca, ownca_digest=sha384）で Home-TLS-CA 署名。
- フルチェーン = リーフ + Home-TLS-CA + Home-RADIUS-CA を連結し、
  `BEGIN CERTIFICATE` が**ちょうど3つ**であることを assert。

### APIライフサイクル（deploy.yml, uri）
- `Set-Cookie` の `TOKEN`(JWT) を取得し、ペイロードを base64url パディング
  補正 → `b64decode | from_json` で `csrfToken` を抽出。
- 状態変更系の全リクエストに `Cookie` / `X-CSRF-Token` / `Origin` を付与
  （requirement §5.2 / §5.3）。
- 接続はホスト名固定。私設CA証明書のため `validate_certs: false`。
- 削除対象IDはアップロード**前**に `source==uploaded` のみを採取
  （lets_encrypt 除外。新証明書は集合に入らない）。

### 安全側の差分（スクリプトからの改善）
- スクリプトは指紋不一致でも警告のみで削除を続行するが、本実装は
  **検証成功時のみ旧証明書を削除**する。`community.crypto.get_certificate`
  で実際に配信される指紋を取得し、APIレスポンスの fingerprint と
  正規化（非HEX除去 + 大文字化）して照合。リトライ（既定6回×5秒）。
  不一致なら旧証明書を残したまま fail する（ダウンタイム・ロールバック確保）。

### 通知・失敗ハンドリング
- role 全体を playbook の block/rescue/always で囲み、失敗時も Slack 通知後に
  fail する。成功=info/ok、失敗=alerts/error チャンネル。
- 一時ディレクトリ削除は role 側 `always` で保証。

## 確認結果

- `ansible-playbook --syntax-check`: PASS。
- JWTペイロード→CSRF抽出のJinjaチェーン（base64urlパディング補正含む）を
  サンプルJWTで実行し、`csrfToken` を正しく取り出せることを確認。
- SAN式 `['DNS:'+host] + ['IP:'+ip]` と フルチェーン段数カウント（=3）を確認。
- `community.crypto` での発行を一時CAでエンドツーエンド実行し、生成物が
  `BEGIN RSA PRIVATE KEY` / SAN(DNS+IP) / KeyUsage critical / EKU serverAuth /
  sha384WithRSAEncryption になることを実確認。
- 実機CloudKey APIへの login/upload/activate/delete は、対象機が必要なため
  本実装フェーズでは未実行（参考スクリプトで実証済みのAPI挙動を踏襲）。

## レビュー時の確認観点（提案）

- `validate_certs: false` の妥当性（私設CA + LAN内ホスト名固定）
- 検証失敗時に削除を行わない設計でよいか
- 証明書名のタイムスタンプ採用（`cloudkey-<iso8601_basic_short>`）が、
  requirement §5.4 の例 `cloudkey-YYYY-MM` から逸脱している点の可否
  （同月内の再実行で name 重複(USER_CERTIFICATE_DUPLICATE)を避けるため秒精度に変更）
- `no_log` の付与漏れ（秘密鍵・TOKEN・認証ボディ）がないか
