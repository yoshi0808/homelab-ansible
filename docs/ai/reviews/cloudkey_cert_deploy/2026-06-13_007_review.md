## 前回指摘の確認

### [Medium] 配信チェーンの証明書順序を検証していない

**解消済み。**

`openssl s_client -showcerts` の出力を証明書単位の順序付きリストへ分割し、
以下を位置ごとに完全一致で確認している。

- 0番目: 発行したリーフ証明書
- 1番目: Home-TLS-CA
- 2番目: Home-RADIUS-CA

証明書数が3であることも同時に確認しているため、要求仕様 §6 の
「リーフ → Home-TLS-CA → Home-RADIUS-CA の3階層」を満たす。
順序または内容が異なる場合は `cloudkey_verify_ok` が false となり、
旧証明書を削除せずにfailする。

## 全体確認

- RSA 2048 / PKCS#1、45日、DNS・動的IPv4 SAN、keyUsage critical、
  EKU serverAuthの証明書仕様を満たしている。
- フルチェーンはリーフ、中間CA、ルートCAの3証明書で構成される。
- API認証ではTOKEN、JWT内CSRF、Originを使用し、ホスト名で接続している。
- アップロード、有効化、実配信リーフ指紋照合、配信チェーン照合、
  旧証明書削除、Slack通知の処理順は要求仕様に整合する。
- 削除前にAPI一覧を再取得し、`source == uploaded`、
  `active == false`、`id != cloudkey_new_id` の対象だけを削除している。
- Let's Encrypt証明書、アクティブ証明書、新証明書は削除対象外である。
- 検証失敗時はDELETEへ進まず、旧証明書を残す。
- 秘密鍵、認証情報、TOKEN、CSRF、APIボディを扱う主要タスクには
  `no_log: true` が設定されている。
- 一時秘密鍵は一時ディレクトリに生成され、`always` で削除される。
- 変更系playbookとして独立し、`cert_renew` と分離されている。
- shellはTLSチェーンの収集のみを行い、判定とfail制御は
  Ansible tasks側にあるため、`core.md` の責務分離に整合する。
- 証明書名の秒精度タイムスタンプ化はユーザー了承済みのため問題なし。
- `ubuntu_vm_patch_policy.md` と競合する変更はない。
