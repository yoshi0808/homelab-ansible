## 経緯

ansy から初回実機実行したところ、**API デプロイ本体（login → upload →
activate）は完全に成功**したが、**配信検証タスクが誤判定して fail** した。
安全機構（検証失敗時は旧証明書を削除せず fail）は設計どおり動作し、
旧証明書は保全された。本対応は**検証タスクのバグ修正のみ**で、成功している
API デプロイ本体と DELETE ロジック・assert は変更していない。

---

## 根本原因（特定済み）

`community.crypto.get_certificate` の**戻り値に `fingerprints` キーが存在しない**
ことが原因。`ansible-doc` と実機で確認した get_certificate の戻り値キーは:

```
cert, subject, expired, extensions, issuer, not_after, not_before,
serial_number, signature_algorithm, version, verified_chain, unverified_chain
```

旧実装は `cloudkey_served.fingerprints.sha1` を参照していたため:

- **症状1**: `until: cloudkey_served.fingerprints is defined` が**常に False** →
  証明書取得自体は成功していても `until` を満たせず全リトライ消費 →
  `cloudkey_leaf_fp_ok = False`。
- **症状2**: チェーン取得タスクの `when: cloudkey_leaf_fp_ok | bool` が症状1に
  引きずられて **skip** → 配信チェーンが 0 層 → `served chain layers: 0` で fail。

前工程のオフライン検証では、テスト用に偽の `fingerprints` データを与えて
Jinja 計算のみ確認しており、実モジュールの戻り値構造を検証できていなかった。

---

## 実機で確認した事実（修正方針の裏付け）

- `get_certificate` に `get_certificate_chain: true` を付けると
  `unverified_chain`（サーバー配信順、先頭=リーフ）が長さ3で取得できる。
- 配信リーフの sha1 指紋を `community.crypto.x509_certificate_info` で算出すると
  `85:f7:80:...:8f:14` となり、playbook が表示した Expected fingerprint と一致。
- `unverified_chain[1]` / `[2]` を正規化した値が、実 CA ファイル
  `home_tls_ca.crt` / `radius_ca.crt` の正規化値と完全一致（順序込み）。

## 修正内容（deploy.yml 配信検証セクション）

1. **指紋検証**: `get_certificate`（`fingerprints` 参照、誤）を廃止し、
   `get_certificate`（`get_certificate_chain: true`）で配信リーフ＋チェーンを取得。
   - `until` 条件を「配信リーフ（`cloudkey_served.cert`）の正規化本文が、今回署名した
     リーフ（`cloudkey_expected_cert_bodies[0]`）と一致するまで」に変更。
     有効化直後に旧証明書が配信されていてもリトライで待てる。
   - 配信リーフの sha1 指紋は `x509_certificate_info` で別途算出し、アップロード
     API の fingerprint（`cloudkey_new_fp`）と正規化照合（要求仕様 §6）。
2. **チェーン取得の独立化**: `openssl s_client` の shell タスク（`when: leaf_fp_ok`
   付き、症状2の skip 原因）を**廃止**。配信チェーンは `unverified_chain`
   （証明書単位で分割済み・配信順保持）から直接取得し、**指紋検証の成否に依存せず
   独立して**評価する。
3. **順序検証は維持**: `unverified_chain` の各本文を正規化し、0/1/2 番目が
   リーフ / Home-TLS-CA / Home-RADIUS-CA と**位置一致**すること、段数==3 を確認。
4. 総合判定 `cloudkey_verify_ok = leaf_fp_ok and chain_ok`、検証失敗時は
   DELETE へ進まず旧証明書を残して fail、は従来どおり。

責務分離（core.md §7 / §9）はむしろ強化された（shell を廃し
`community.crypto` ネイティブモジュールで収集、判定は Ansible tasks 側）。
秘密情報を含まない公開証明書のみ扱うため新規 no_log は不要。

## 確認結果

- `ansible-playbook --syntax-check`: PASS。
- 旧参照（`cloudkey_served.fingerprints` / s_client shell / `_cloudkey_served_norm`）が
  deploy.yml から消えていることを grep で確認。
- 実機 CloudKey に対し以下をライブ確認:
  - `get_certificate`（chain）が長さ3の `unverified_chain` を返す。
  - `x509_certificate_info` の sha1 指紋が Expected fingerprint と一致。
  - `unverified_chain[1]==home_tls_ca.crt`、`[2]==radius_ca.crt`、段数==3 を
    assert で実証（順序込み）。
- 検証環境は掃除せず保全（新=active、cloudkey-apitest/Yoshi=非アクティブ）。
  再実行で「アップロード→有効化→配信検証成功→旧 uploaded 削除（DELETE）」の
  本来フローと「Yoshi(lets_encrypt) は削除対象外」を実機検証できる。

---

## レビュー時の確認観点（提案）

- `get_certificate_chain: true` の `unverified_chain` を配信チェーンの正本とする妥当性
  （`verified_chain` ではなく、実際に配信される順序を見る）
- `until` を「配信リーフ == 新リーフ本文」に変更したことで、有効化直後の
  旧証明書配信／再起動を正しく待てるか
- 指紋照合（`x509_certificate_info` の sha1 vs API `cloudkey_new_fp`）が §6 を満たすか
- チェーン検証が指紋検証の成否から独立したこと（skip 連鎖の解消）
