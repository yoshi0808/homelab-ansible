## [High] 削除直前に active=false を確認していない → 対応済み

`deploy.yml`

- アップロード**前**に一覧から `source==uploaded` だけを採取して削除集合に
  していた旧ロジック（`cloudkey_old_ids`）を**廃止**した。
- 有効化＋配信検証の**後**に証明書一覧を**再取得**し、削除対象を以下の
  全条件で再構成するようにした（`cloudkey_delete_targets`）。
  - `source == 'uploaded'`
  - `active` が定義済みかつ false（`selectattr('active','defined') | rejectattr('active')`）
  - `id != cloudkey_new_id`
- 新証明書は有効化済み（active==true）のため `rejectattr('active')` と
  明示的な id ガードの二重で確実に除外。lets_encrypt は `source` で除外。
- DELETE 直前に **assert** を追加し、削除候補に「アクティブ・新証明書・
  uploaded以外」が混入していないことを固定（`item != new`、source限定、active無し）。
- 再取得・候補構成・assert・DELETE・削除ID記録の全ステップを
  `cloudkey_delete_old and cloudkey_verify_ok` で同一ガードした。

## [Medium] 配信チェーンが3階層であることを実通信で確認していない → 対応済み

`issue.yml` / `deploy.yml`

- `issue.yml`: 期待する3証明書（リーフ / Home-TLS-CA / Home-RADIUS-CA）の
  PEMからマーカー・空白を除去した正規化base64本文を
  `cloudkey_expected_cert_bodies` として算出。
- `deploy.yml`: リーフ指紋一致（リトライ付き）の後に、
  `openssl s_client -showcerts` で**実際に配信されるチェーン**を取得
  （収集のみ。判定はAnsible側＝core.md §7準拠）。
  - 配信チェーンの証明書数が **3** であること
  - 正規化した配信チェーンに期待3証明書の各本文が**すべて含まれる**こと
  を `cloudkey_chain_ok` として判定。
- 総合判定 `cloudkey_verify_ok = リーフ指紋一致 and 3階層チェーン一致` とし、
  **検証失敗時は旧証明書を一切削除せず fail** する（リーフのみ配信される
  ケースでも誤って成功通知・削除しない）。

## 確認結果

- `ansible-playbook --syntax-check`: PASS。
- 削除候補フィルタ（`selectattr('source')`→`selectattr('active','defined')`
  →`rejectattr('active')`→`rejectattr('id')`）と削除前assertを、
  active=true / lets_encrypt / active欠落 / 新証明書を混在させた
  サンプルで実行し、`active==false` の uploaded のみが残ることを確認。
- 配信チェーン検証ロジックを `community.crypto` 実発行（リーフ＋中間＋ルート）
  でE2E実行し、フルチェーン段数=3 と 各証明書本文の含有一致を実確認。
  （正規化は実改行を含むPEMに対し `regex_replace('\s+','')` で機能することを確認）
- 実機CloudKey APIの login/upload/activate/delete および s_client 実通信は
  対象機が必要なため本フェーズでは未実行。

## レビュー時の確認観点（提案）

- 配信チェーン照合を「指紋」ではなく「正規化PEM本文の含有 + 段数==3」で
  実装した点の妥当性（subject/issuer文字列照合より厳密かつ順序非依存）
- DELETE 直前 assert の条件（active無し / new無し / uploaded限定）で十分か
- `openssl s_client | awk` を収集専用（changed_when: false）として
  shell に置いた点が core.md §7 / §9 に整合しているか
