## [Medium] 配信チェーンの証明書順序を検証していない → 対応済み

`deploy.yml`（配信チェーン検証ブロック）

**変更前**: 配信チェーン全体を1本の正規化文字列に連結し、期待3証明書の各本文が
「含まれるか（`in`）」を確認していた。これは順序非依存で、
「リーフ → ルートCA → 中間CA」のような誤順でも `cloudkey_chain_ok` が true になり得た。

**変更後**: `openssl s_client -showcerts` の出力を**証明書単位の順序付きリスト**に
分割し、配列の 0/1/2 番目が期待するリーフ / Home-TLS-CA / Home-RADIUS-CA に
**位置一致**することを確認するようにした。

- 分割: `regex_findall('(?s)-----BEGIN CERTIFICATE-----.*?-----END CERTIFICATE-----')`
  で配信順を保ったままPEMブロックを抽出し、各々マーカー・空白を除去して
  正規化base64本文の配列 `cloudkey_served_bodies` を生成。
- 判定 `cloudkey_chain_ok`:
  - `cloudkey_served_bodies | length == 3`（段数確認は維持）
  - `cloudkey_served_bodies[0] == cloudkey_expected_cert_bodies[0]`（リーフ）
  - `cloudkey_served_bodies[1] == cloudkey_expected_cert_bodies[1]`（中間CA）
  - `cloudkey_served_bodies[2] == cloudkey_expected_cert_bodies[2]`（ルートCA）
- 順序非依存の `in` 判定（`_cloudkey_served_norm`）は廃止。
- `cloudkey_served_chain_count` は `cloudkey_served_bodies | length` から算出。

総合判定・削除ガードは従来どおり `cloudkey_verify_ok = リーフ指紋一致 and
3階層順序一致` で、検証失敗時は旧証明書を削除せず fail する。
shell は収集のみ・判定は Ansible 側（core.md §7 / §9）も維持。

---

## 確認結果

- `ansible-playbook --syntax-check`: PASS。
- `community.crypto` 実発行（リーフ＋中間＋ルート）で順序検証ロジックをE2E実行:
  - 正順（リーフ→中間→ルート）の配信チェーン → `cloudkey_chain_ok = true`
  - 誤順（リーフ→ルート→中間）の配信チェーン → `cloudkey_chain_ok = false`
  を実確認（正例true・誤例falseの両方を assert で検証）。
- 実改行を含むPEMに対する `regex_findall('(?s)...')` 分割と
  `regex_replace('\s+','')` 正規化が機能することを確認。
- 実機CloudKey APIの login/upload/activate/delete および s_client 実通信は
  対象機が必要なため本フェーズでは未実行。

---

## レビュー時の確認観点（提案）

- 順序付き位置一致（`served_bodies[n] == expected[n]`）で §6 の
  「リーフ → Home-TLS-CA → Home-RADIUS-CA の3階層」を満たすか
- `regex_findall('(?s)...')` のDOTALL指定が配信チェーン分割として妥当か
- 検証失敗（順序違反含む）時に DELETE へ進まず fail する制御で十分か
