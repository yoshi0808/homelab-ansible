## Findings

### 1. role default が patch policy の停止条件と逆向きになっている

Severity: Medium

`roles/proxmox_evacuate_node/defaults/main.yml:17`

```yaml
proxmox_evacuate_allow_local_disk_migration: true
```

今回の変更で、`proxmox_evacuate_node` は既定で QEMU VM migration に `--with-local-disks` を付与する挙動になった。

これは `2026-06-02_012_implement.md` にある「実環境では全 VM が local-zfs を使っているため、false では退避が常に失敗する」という事情には合っている。

ただし、`docs/ai/prompts/proxmox_patch_policy.md` では local-only storage VM / CT は自動 migration せず停止する条件として定義されている。

該当箇所:

- `5.6 自動 migration の扱い`
  - `対象 VM / CT が local-only storage を使っている`
- `6.4 proxmox_evacuate_node.yml` の実行条件
  - `対象 VM / CT が local-only storage に依存していない`
- `6.4 proxmox_evacuate_node.yml` の停止条件
  - `local-only storage 使用 VM / CT がある`

そのため、現状の diff は repository の正本である patch policy と不整合になっている。特に `defaults/main.yml` は role の最も低い優先順位の既定値なので、playbook 実行者が追加変数を渡さない通常実行すべてで local disk migration が許可される。

この変更を採用するなら、少なくとも以下のどちらかを同時に行う必要がある。

- `proxmox_patch_policy.md` を更新し、この homelab では local-zfs VM の `--with-local-disks` migration を controlled apply の許容範囲に含める
- role default は `false` のまま維持し、homelab 固有の承認済み設定として inventory / group_vars / playbook vars 側で `true` を設定する

実環境に合わせる意図は理解できるが、policy を更新しないまま role default を `true` にすると、前回 `2026-06-02_008_review.md` で指摘した「local-only storage VM を自動 migration しない方針を迂回する」状態が、既定挙動として復活する。

## 補足

`2026-05-31_001_requirement.md` では local-only storage 使用 VM の詳細チェックは初回除外スコープになっている。

そのため、local-zfs を使う VM を移行可能にする判断自体を否定するものではない。ただし、現在共有されている `proxmox_patch_policy.md` を正とするなら、今回の変更は policy 変更または homelab 固有 override とセットで扱うべき。
