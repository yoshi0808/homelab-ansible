## 確認内容

### Fix 1: opt-in 変数の追加

`roles/proxmox_evacuate_node/defaults/main.yml:15`

```yaml
proxmox_evacuate_allow_local_disk_migration: false
```

既定値が `false` なので、通常実行では local disk migration を自動許可しない。

これは `docs/ai/prompts/proxmox_patch_policy.md` の以下の方針と整合している。

- local-only storage を使う VM / CT は自動 migration せず停止する
- local-only storage 使用 VM / CT がある場合は停止条件

### Fix 2: `--with-local-disks` の条件付き付与

`roles/proxmox_evacuate_node/tasks/main.yml:129`

```yaml
qm migrate {{ item.vmid }} {{ destination_node }}
{{ '--online 1' if item.status == 'running' else '' }}
{{ '--with-local-disks' if (proxmox_evacuate_allow_local_disk_migration | bool) else '' }}
```

`proxmox_evacuate_allow_local_disk_migration` が `false` の場合は `--with-local-disks` が付かない。

そのため、local disk を持つ QEMU VM は `qm migrate` 側で失敗し、`rescue` で `_evac_failed: true` となり、最後に `Re-raise failure` で playbook が失敗する。

これは、requirement の「local-only storage 使用 VM の詳細チェックは初回除外スコープ」と、patch policy の「local-only storage 使用 VM / CT がある場合は停止」を、現時点の実装範囲で両立する形になっている。

## 残リスク / 注意点

local-only storage の能動的な事前検出は、今回も未実装。

ただし、これは `2026-05-31_001_requirement.md` の初回除外スコープに含まれているため、今回差分の不備としては扱わない。

既定値 `false` のまま local disk VM に到達した場合、migrate 実行時エラーで停止する。つまり、事前に migration plan 段階で止めるのではなく、`qm migrate` の失敗を停止条件として使う挙動である。この点は実機運用時の表示・レポート確認で注意が必要。
