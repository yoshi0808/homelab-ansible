## Findings

### 1. `ha-manager status` の LRM 行に `:` を期待しているため、maintenance mode 待ちが timeout する可能性が高い

Severity: High

`roles/proxmox_evacuate_node/tasks/main.yml:168`

```yaml
select('match', 'lrm ' ~ inventory_hostname ~ ':.*maintenance mode')
```

`roles/proxmox_restore_vm_placement/tasks/main.yml:172`

```yaml
select('match', 'lrm ' ~ inventory_hostname ~ ':.*idle')
```

今回追加された `until` 条件は、`ha-manager status` の出力に `lrm <node>:` というコロン付き行がある前提になっている。

しかし Proxmox VE の `ha-manager status` 出力例は以下のような形式で、node 名の直後は `:` ではなく `(` になる。

```text
lrm elsa (active, Mon Nov 21 07:23:22 2016)
```

Proxmox forum の実例でも以下の形式になっている。

```text
lrm pve-02 (maintenance mode, Thu Feb 1 17:56:54 2024)
lrm pve-2 (idle, Thu Feb 27 18:25:52 2025)
```

そのため、現行の正規表現では enable / disable のコマンド自体が成功していても `until` が一致せず、120 秒後に timeout して playbook が失敗する可能性が高い。

少なくとも以下のように、実際の出力形式に合わせる必要がある。

```yaml
select('match', '^lrm ' ~ inventory_hostname ~ ' \\(.*maintenance mode')
```

disable 側も同様に `:` ではなく `\\(` を見る必要がある。

### 2. disable 後の LRM 状態を `idle` のみに限定しているため、HA サービスがある環境で timeout する可能性がある

Severity: Medium

`roles/proxmox_restore_vm_placement/tasks/main.yml:172`

```yaml
select('match', 'lrm ' ~ inventory_hostname ~ ':.*idle')
```

Proxmox の LRM は、HA サービスを持つ場合に `active` になり、HA サービスがない場合などに `idle` になる。

今回の目的は「maintenance mode が解除され、LRM が利用可能になるまで待つ」ことなので、`idle` のみを成功条件にすると、HA サービスが存在して `active` へ戻る正常ケースを timeout 扱いする可能性がある。

restore 側は `hacritical` VM の自動復帰を期待する role なので、むしろ HA サービスが存在するケースを想定している。disable 完了待ちは、`idle` 固定ではなく「対象 LRM 行が `maintenance mode` ではない」または `active|idle` のどちらかを許容する条件にする方が安全。

例:

```yaml
until: >-
  _restore_maint_poll.stdout_lines |
  select('match', '^lrm ' ~ inventory_hostname ~ ' \\((active|idle),') |
  list | length > 0
```

## 確認内容

### コマンド変更

`pvecm node-maintenance` から `ha-manager crm-command node-maintenance` への変更自体は、Proxmox VE の現在の `ha-manager(1)` ドキュメントと整合している。

```text
ha-manager crm-command node-maintenance enable NODENAME
ha-manager crm-command node-maintenance disable NODENAME
```

この方向性は妥当。

### defaults 追加

以下の timeout / poll interval 追加は、role defaults として妥当。

```yaml
proxmox_evacuate_maintenance_timeout: 120
proxmox_evacuate_maintenance_poll_interval: 5
proxmox_restore_maintenance_timeout: 120
proxmox_restore_maintenance_poll_interval: 5
```

## 補足

Proxmox 公式ドキュメントにも、LRM status は maintenance mode 要求を LRM が拾った時点で `maintenance mode` と表示され、すべての HA service が移動完了した時点ではない旨が書かれている。

この実装では maintenance mode active 待ちの後に、別途 `pvesh get /cluster/resources` で HA-managed VM が target node から消えるまで待っているため、その点は問題ない。
