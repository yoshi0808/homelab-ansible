## 確認内容

### Fix 1: post-restore healthcheck 非 OK で fail

`roles/proxmox_restore_vm_placement/tasks/main.yml:218` で `Abort if post-restore healthcheck is not OK` が追加されている。

```yaml
when: proxmox_report.result.status != 'OK'
```

これにより、restore 後の healthcheck が `WARNING` のままでも playbook が成功扱いになる前回問題は解消されている。

`proxmox_healthcheck` role 自体は `CRITICAL` の場合に role 内で fail するため、`WARNING` は今回追加された fail task で止まり、`CRITICAL` は healthcheck role 側で止まる。どちらも上位フローへ成功として流れない。

### Fix 2: HA 復帰待ちで running 確認

`roles/proxmox_restore_vm_placement/tasks/main.yml:186` の until 条件が 2 条件に分かれている。

- `hacritical` かつ `prefer<target_node>` の VM/CT が target node 以外にいない
- target node 上にいる同 VM/CT が `running` 以外ではない

これにより、「target node に戻ったが stopped」という状態を復帰完了と誤判定する前回問題は解消されている。

### Fix 3: destination_node コメント

`playbooks/proxmox_evacuate_node.yml:9` は以下になっている。

```text
# destination_node: auto-derived as the opposite node (pve1 <-> pve2)
```

現行実装が pve1/pve2 固定の反対側ノード自動導出であることと一致している。

## 要求仕様・ポリシーとの整合

- OS パッチ適用は含まれておらず、controlled apply の範囲に収まっている
- `proxmox_vm_home_nodes.yml` は使用せず、Proxmox タグで判定している
- target/destination の healthcheck OK を前提に退避する構成になっている
- restore 後に healthcheck 非 OK を成功扱いしないため、pve2 成功後のみ pve1 へ進む patch policy の意図と整合している
