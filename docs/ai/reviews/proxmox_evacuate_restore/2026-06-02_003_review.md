## 結論

構成と責務分離は概ね要求仕様に沿っている。

- `proxmox_vm_home_nodes.yml` を使わず、Proxmox タグで判定している
- 退避・復帰 playbook が OS パッチ適用を含まない controlled apply として分離されている
- `pvesh` / `pvecm` / `qm` / `pct` の実行と Ansible 側の判定・待機が分かれている
- report は `reports_base_dir` 配下に保存される

ただし、復帰 playbook が post-healthcheck 非 OK でも成功扱いになる点は、ローリング更新の安全条件として修正必須。

## Findings

### High: restore 後 healthcheck が非 OK でも playbook が成功する

該当箇所:

- `roles/proxmox_restore_vm_placement/tasks/main.yml:201`
- `roles/proxmox_restore_vm_placement/tasks/main.yml:205`
- `roles/proxmox_restore_vm_placement/tasks/main.yml:209`

`proxmox_restore_vm_placement` は最後に `proxmox_healthcheck` を実行しているが、`proxmox_report.result.status != 'OK'` の場合でも debug warning を出すだけで失敗しない。

要求仕様では restore の Step 4 が `healthcheck OK確認` になっており、patch policy でも pve1 適用は pve2 更新後の healthcheck OK が前提になっている。

このままだと、復帰後の状態が `WARNING` のままでも restore が `SUCCESS` として report され、上位の weekly full flow を作ったときに「pve2 が完全に正常ではないのに pve1 へ進む」事故につながる。

修正案:

```yaml
- name: Abort if post-restore healthcheck is not OK
  ansible.builtin.fail:
    msg: >-
      Post-restore healthcheck on {{ inventory_hostname }} is
      {{ proxmox_report.result.status }}:
      {{ (proxmox_report.result.criticals + proxmox_report.result.warnings) | join(' | ') }}
  when: proxmox_report.result.status != 'OK'
```

### Medium: HA 復帰待ちが running 状態を確認していない

該当箇所:

- `roles/proxmox_restore_vm_placement/tasks/main.yml:179`
- `roles/proxmox_restore_vm_placement/tasks/main.yml:186`

HA VM の復帰待ちは、`hacritical` かつ `prefer<target_node>` の VM/CT が target node 以外に残っていないことだけを見ている。

つまり、HA VM/CT が target node 上に戻ったが `running` ではない場合でも until 条件を満たす可能性がある。

要求仕様の pve1/pve2 フローは「HAで自動復帰」「healthcheck OK確認」を前提にしているため、少なくとも HA 復帰待ちでは `node == inventory_hostname` だけでなく `status == running` も確認した方が安全。

修正案:

- HA 復帰待ち条件を「対象 HA VM/CT のうち、`node != target` または `status != running` が 0 件」にする
- もしくは復帰後に対象 HA VM/CT の状態一覧を作り、非 running があれば fail する

### Low: `destination_node` の optional override が実際には効かない

該当箇所:

- `playbooks/proxmox_evacuate_node.yml:9`
- `playbooks/proxmox_evacuate_node.yml:27`
- `playbooks/proxmox_evacuate_node.yml:47`

コメントでは `destination_node` が optional override 可能と書かれているが、destination healthcheck の `hosts` と target play の `vars.destination_node` はどちらも `target_node` の反対側に固定されている。

現在の要求仕様は pve1/pve2 の2ノード前提なので実害は小さいが、ドキュメントと挙動がズレている。

修正案:

- override を不要とするならコメントから削除する
- override を残すなら `destination_node | default(...)` を使って healthcheck 対象と role 変数の両方を合わせる
