## 指摘事項

### 1. must-fix: `pvesh` 失敗時に stale なしとしてサイレント成功する

該当箇所:
- `roles/proxmox_snapshot_check/files/proxmox-snapshot-collect.sh:26`
- `roles/proxmox_snapshot_check/files/proxmox-snapshot-collect.sh:42`
- `roles/proxmox_snapshot_check/files/proxmox-snapshot-collect.sh:52`
- `roles/proxmox_snapshot_check/tasks/main.yml:16`
- `roles/proxmox_snapshot_check/tasks/main.yml:49`

`pvesh_json()` は `pvesh get` が non-zero、空 stdout、JSON parse error、timeout になった場合に
すべて `[]` を返します。そのため、以下のケースでも Ansible 側では正常な JSON として
処理され、`total_snapshots: 0` / `stale_count: 0` / `status: OK` になり得ます。

- `/cluster/resources` の取得に失敗した
- 特定 VM / CT の snapshot 一覧取得に失敗した
- Proxmox API / pvesh の一時エラーや権限問題が発生した

要求仕様の目的は「放置スナップショットの早期検出」です。収集不能な状態を stale なしとして
扱うと、実際には古いスナップショットが残っていても通知されません。read-only であることとは
別に、監視としては false OK になるため修正が必要です。

修正案:
- shell の JSON に `collection_ok` と `errors` を含める。
- `/cluster/resources` が取れない場合は `collection_ok: false` にする。
- 個別ゲストの snapshot 取得失敗も `errors` に `node/type/vmid/name` と stderr を残す。
- Ansible tasks 側で `collection_ok == false` または `errors | length > 0` を WARNING 以上として扱い、
  Slack #alerts に通知する、または playbook を fail させる。

この修正でも、shell は「収集結果とエラー情報の JSON 整形」に留まり、7日超え判定や通知制御は
Ansible tasks 側に置けます。core.md §7 の責務分離にも反しません。

### 2. suggestion: 7日ちょうどの境界が「以上」ではなく「より古い」になっている

該当箇所:
- `roles/proxmox_snapshot_check/tasks/main.yml:30`

要求仕様は「7日（604800秒）以上前」としている一方、実装は
`snaptime < (now - threshold)` で抽出しています。作成からちょうど 604800 秒の
スナップショットは検出対象外になります。

実運用上は次回実行で検出されるため影響は小さいですが、仕様に合わせるなら
`snaptime <= _snap_threshold_epoch` 相当の判定にするのが自然です。
