## この文書の目的

実装をさらに変更する前に、いったん歩み寄りたい。
003→005→007→009 の4回のレビューは、すべて「同時実行時のロック所有権」を深掘りしている。
各指摘は**技術的事実としては正しい**（pmxcfs の O_EXCL 非原子性、120秒期限、未刻印999の
所有権の穴）。それは認めたうえで、**リスクの重大性と対策の重さが釣り合っているか**を、
リスクベースで一度すり合わせたい。

## 立場

このツールの本質は「コアVMの vzdump バックアップがサイレント破損していないかを月次で
検出する」こと。リストア検証ロジック（ローテーション選定 / agent 期待レベル別判定 /
NIC 切断起動 / 最新バックアップ特定 / 999 の安全な destroy）は、005 review の時点で
「要求仕様に沿っている」と確認済みである。

一方、4回分の指摘が集中している「同時実行の所有権競合」は、以下の性質を持つ。

| 観点 | 実態 |
|---|---|
| 競合の発生条件 | 月次自動（quory・固定時刻）と手動（ansy）が**秒単位で重なった**場合のみ |
| 競合時の被害 | **検証専用の使い捨て VMID 999 が消えるだけ。本番VM・本番データは無傷** |
| 検出性・回復性 | 失敗すれば次回の月次で正常化。月次サイクルなので猶予は十分 |
| 対策の現状 | 分散ロック＋期限更新＋生存監視＋孤児管理まで積み上がっている |

「極めて低頻度 × 被害は使い捨てVM」に対して、分散システム級のロック堅牢性を積み増すのは
ROI が見合わない。core.md §6 の「種別を増やしすぎない（over-engineering を避ける）」、
§14 の「AI は自動決定主体ではなく支援役。リスク受容・運用判断は人間が行う」という本
リポジトリの設計思想とも、方向がずれ始めていると考える。

## 提案（歩み寄り案）

**競合を「技術で完全保証」しに行くのではなく、「運用設計で起こさない」前提に倒し、
残余リスクは受容として明文化する。**

具体的には次の方針を提案する。

1. **同時実行を運用で禁止する**
   - 月次は quory から固定時刻の単一スケジュール。
   - ansy の手動実行は、人が重複しないことを確認して実行する（開発用途）。
   - これにより「秒単位での同時起動」という前提自体を運用で除去する。

2. **ロック機構を最小化する**
   - 公式 `mkdir /etc/pve/priv/lock/<name>`（空dir）で取得し、取れなければ即 fail・通知・
     非ゼロ終了（多重起動の素朴なガード）。
   - refresher / 120秒期限更新 / 生存監視 / 孤児管理は**撤去**する。
   - 既存の「開始前 999 残骸ガード」と「999 への一意トークン刻印（別トークンなら触らない）」
     は残す。これだけでも、本番への危害は構造的にゼロ（destroy 対象は 999 固定 + assert）。

3. **残余リスクを final ドキュメントに受容として明記する**
   - 「ロック保持が 120秒を超える理論的窓で、低頻度の同時実行が重なった場合、最悪で使い捨て
     999 の検証が1サイクル無駄になる。本番影響なし。これを受容する」と記録し、Yoshinobu の
     判断としてクローズする。

この案なら、複雑なロック関連コードを削減でき、本質のリストア検証ロジックが読みやすくなる。

## レビュー担当への問い

1. 「破壊対象が検証専用の使い捨て 999 に限定され、本番影響がない」という前提に、認識の相違は
   あるか。あるなら、本番に波及する具体経路を指摘してほしい。
2. 上記の「運用で同時実行を禁止 + 最小ロック + 残余リスク受容」という線引きに合意できるか。
   合意できない場合、**それは技術的必然なのか、それとも完全性の追求なのか**を明示してほしい。
3. レビュー観点として、今後はロック所有権の深掘りより、**本質（リストア検証の正しさ・
   安全装置が 999 固定であること・通知の妥当性）** に重心を移すことを提案したい。

最終的な採否は Yoshinobu が判断する（core.md §14）。本書は、その判断材料として論点を
そろえるためのものである。
