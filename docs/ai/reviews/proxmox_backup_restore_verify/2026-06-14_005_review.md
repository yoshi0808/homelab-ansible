## 結論

**修正必須。**

003 review の2指摘は修正されている。しかし、追加された所有権判定は同時実行を
考慮しておらず、別実行が作成した VMID 999 を cleanup で削除できる競合が残る。
要求仕様の「自分が作成した 999 のみを destroy 対象とする」を満たすため、月次自動
実行へ投入する前に排他制御または実行単位を識別できる所有権確認が必要。

対象ファイルはすべて未追跡のため、通常の `git diff` には表示されなかった。
新規 playbook / role と追加実装を直接確認した。

## Findings

### [高] 同時実行すると、後発実行が先行実行の 999 を削除する

対象:

- `roles/proxmox_backup_restore_verify/tasks/main.yml:63`
- `roles/proxmox_backup_restore_verify/tasks/main.yml:161`
- `roles/proxmox_backup_restore_verify/tasks/main.yml:276`
- `roles/proxmox_backup_restore_verify/tasks/main.yml:287`
- `roles/proxmox_backup_restore_verify/tasks/main.yml:297`

現在の所有権判定は、開始前に 999 が存在しなかったことと
`_brv_restore_attempted: true` の組み合わせで、その後に存在する 999 をすべて
「自分が作成したもの」と扱う。この判定には Phase 0 の確認から cleanup までを
保護する排他制御がない。

例えば quory の月次実行 A と ansy の手動実行 B が重なると、以下が起こり得る。

1. A と B がどちらも Phase 0 で「999 なし」を確認する。
2. A が `qmrestore` で 999 を作成する。
3. B の `qmrestore` は「999 が既に存在する」ため失敗する。
4. B は `_brv_restore_attempted` が true なので cleanup を実行する。
5. B が再取得した 999 は A の作成物だが、B は `qm destroy 999 --purge 1` を実行する。

これにより、要求仕様の「自分が作成した 999 のみ」を破り、先行実行の検証VMを
後発実行が破壊する。VMID 999 は検証専用で本番VMではないものの、controlled apply の
破壊的操作として所有権判定が成立していない。

Phase 0 の確認から cleanup 完了まで、quory と ansy の実行元をまたいで有効な
排他制御を設ける必要がある。単一 control node のローカル lock だけでは、別 control
node からの手動実行との競合を防げない。排他を取得できなければ変更操作へ進まず、
通知して非ゼロ終了すること。別方式を採る場合も、`restore_attempted` だけではなく、
その実行が作成したことを一意に確認できる根拠が必要。

## 前回指摘の確認

### 1. `qmrestore` 途中失敗時の cleanup

**解消済み。**

`qmrestore` の前に `_brv_restore_attempted` を設定し、`always` で cluster resources を
再取得するため、単独実行では途中生成された 999 を検出して cleanup できる。

### 2. destroy 失敗時の終了コード

**解消済み。**

最後の fail 条件に `not _brv_cleanup_ok` が追加され、destroy または cleanup 状態確認が
失敗した場合は critical 通知後に非ゼロ終了する。
