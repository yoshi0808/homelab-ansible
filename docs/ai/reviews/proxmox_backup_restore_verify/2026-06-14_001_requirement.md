# 要求仕様: proxmox_backup_restore_verify（VMバックアップ リストア検証）

作成日: 2026-06-14
版: 001 (requirement)
参照: docs/ai/prompts/core.md, docs/ai/prompts/proxmox_patch_policy.md

---

## 目的
コアVM（Sophos / authy / monnie）の vzdump バックアップが、実際にリストアして
起動できることを月次で検証する。バックアップが取れているつもりで壊れていた
（サイレント破損）を検出することが目的。

## 対象
- 検証対象VM: `verify` タグを持つVMのうち、月次ローテーションで当たった1台
- リストア先VMID: 999 固定（検証専用）
- 実行ノード: 対象VMの prefer* タグで決まるノード（authy/Sophos → pve1、monnie → pve2）
- 実行元: quory（本番・Semaphore 月次）/ ansy（開発・CLI）
- バックアップ元: vzdump NFS ストレージ（Synology、PVE が dump フォルダを使用）

## 作成・更新ファイル
- playbooks/proxmox_backup_restore_verify.yml（新規・変更系）
- roles/proxmox_backup_restore_verify/ 一式
  - tasks/main.yml
  - defaults/main.yml
  - files/*.sh（生データ収集が必要な場合のみ → 今回は不要と判断）
- inventory / group_vars / host_vars の変更は不要（対象はタグで動的決定）

## 安全度
controlled apply（検証用 999 の作成・起動・削除を伴う。本番VMは reboot/migrate しない）

## 実施内容（フルライフサイクル / block・rescue・always で囲む）
1. 開始前チェック: 999 が既存でないことを確認する。
   残骸がある場合は勝手に消さず、中断して #alerts に通知する。
2. 月次ローテーションで対象VMを決定する。target_vmid が extra vars で
   渡された場合はローテーションをバイパスする（開発/手動用）。
3. 対象VMの `agent` 設定を確認し、期待レベルを決定する（read-only）。
4. 対象VMの prefer* タグからリストア先ノードを決定する。
5. 対象VMの最新 vzdump バックアップを NFS ストレージ内で特定する。
6. qmrestore で 999 へリストアする。
7. NIC を切断してから（IP 重複回避）999 を起動する。
8. 正常性を判定する（期待レベル別）。
9. 999 を停止し destroy する。
10. 結果を Slack へ通知する。

## ローテーション選定ロジック
- cluster resources から `verify` タグを持つVMを列挙する。
- vmid 昇順でソートし、決定的な順序リストを得る。
- (現在の月 - 1) % リスト長 でインデックスを決め、今月の対象VMを確定する。
- これにより対象VMリストを Ansible ファイルに持たない。VMの増減は
  Proxmox 側のタグ付けのみで対応でき、インデックスは自動で振り直される。

## 正常性判定（期待レベルは本番VMの agent 設定で決まる）
| VM種別 | 合格基準 | 手段 |
|---|---|---|
| agent 対応（agent:1） | osinfo 取得成功 | guest agent |
| agent 無し（agent 未設定/0） | 起動後 N 秒経過してもなお running | qm status |

- 期待レベル（どこまで要求するか）= 本番VMの agent 設定
- 実測 = 999 を起動して測った結果
- 判定 = 実測が期待レベルに届いているか
- agent 無しVMは running 継続のみで合格扱いとする（Sophos を特別扱いしない）。
- agent 無しVMの判定部分は、次フェーズで serial0 コンソール文字列マッチに
  差し替えられるよう分離しておく。

## 破壊的操作の安全装置（重要）
- destroy 対象は 999 固定。それ以外の VMID は絶対に destroy しない（assert で固定）。
- 999 が本番運用 VMID と衝突しないこと（cluster resources に 999 が存在しないこと）を
  開始前チェックで確認する。
- リストア/起動が失敗しても、always で必ず 999 のクリーンアップを試みる。
  ただし「自分が作成した 999」のみを destroy 対象とする（残骸は触らない）。
- destroy できず 999 が残った場合は #alerts に critical で通知する（手動対応が必要）。

## shell / Ansible tasks 責務分離
- qmrestore / qm set / qm start / qm stop / qm destroy / guest cmd は
  Ansible tasks 側で実行条件を明示制御する。
- get-osinfo 等の生データ収集も command で行い、OK/NG 判定と fail 制御は
  Ansible tasks 側に置く（core.md §7 / §9）。
- 今回は専用 shell スクリプトを置かず、すべて Ansible tasks で制御する。

## 通知
roles/common_slack/tasks/notify.yml を include_tasks で呼び出す（best-effort）。
| 状況 | チャンネル | status |
|---|---|---|
| 検証 OK（期待レベル達成） | info | ok |
| 検証 NG（リストア失敗 / 正常性未達） | alerts | error |
| 999 残骸検出 or destroy 失敗 | alerts | critical |

## 制約
- 本番VM（既存 VMID）には config 読み取り以外で触らない。
- 秘密情報を扱わない。
- IP リテラルをファイルに書かない（core.md §3）。NIC 切断は net0 等の
  net デバイス削除で行い、IP を指定しない。
- become: true（root）で実行する（NFS のアクセス権が root に絞られているため）。

## 初回実装スコープ
- ローテーション選定
- 最新バックアップ特定 + qmrestore
- NIC 切断起動
- 正常性判定（agent 対応: osinfo / agent 無し: running 継続）
- 999 の destroy + 安全装置
- Slack 通知

## 初回除外スコープ（次フェーズで実装予定）
- agent 無しVM（Sophos）の serial0 コンソール文字列マッチ。
  今回は判定部分を分離しておき、後から差し替え可能にする。
- バックアップ鮮度チェック（別 playbook。取得側整備後）
- 検証履歴の永続化（月番号固定方式のため不要）

## read-only / 変更系の区別
変更系。999 の作成・起動・停止・削除を行う。本番VMは config 読み取りのみ。
