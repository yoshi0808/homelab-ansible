## 概要

003 review の指摘2点（いずれも VMID 999 の残存に関する安全装置の不備）を修正した。
変更は `roles/proxmox_backup_restore_verify/tasks/main.yml` のみ。playbook /
defaults / inventory には変更なし。

## レビュー指摘と対応

### [重大] 1. qmrestore 途中失敗で作成された 999 が cleanup 対象にならない

**指摘要約**: `_brv_restored` は `qmrestore` 成功後にだけ true になるため、
config/ディスク作成後に `qmrestore` が失敗すると 999 が作られたまま cleanup が
丸ごとスキップされ、次回実行が残骸ガードで停止し続ける。「コマンド成功」を
「作成所有権」の唯一の根拠にしないこと、という指摘。

**対応**: 所有権の判定根拠を「コマンドが 0 を返したか」から
「開始前に 999 が無かった事実 ＋ リストアを試行した事実 ＋ 現に 999 が存在するか」へ変更した。

- init facts に `_brv_restore_attempted: false` を追加。
- `qmrestore` の **直前** に `_brv_restore_attempted: true` をセットする task を追加。
  Phase 0 で「開始前に 999 は存在しない」ことを確認済みのため、この時点以降に現れる
  999 はすべて本実行の所有物とみなせる。
- `always` の cleanup を再構成:
  - 実行条件を `_brv_restored` から
    `_brv_restore_attempted and not _brv_preexisting_residue` へ変更。
  - cleanup 内で `pvesh get /cluster/resources` を **再取得** し、999 が現存するかを
    `_brv_999_present` で判定。コマンドの戻り値ではなく実際の状態で判断する。
  - 999 が現存する場合のみ `qm stop`（best-effort）→ 待機 → `qm destroy --purge 1`。
- これにより、qmrestore が途中失敗して 999 を作った場合でも、cleanup が再クエリで
  999 を検出して destroy する。残骸（開始前から存在した 999）には引き続き触れない。
- `qm stop` の実行条件を `_brv_started` から「999 が現存するか」へ変更した
  （途中失敗で起動前でも、現存すれば stop→destroy を試みる。stop は failed_when:false）。

### [高] 2. destroy 失敗でも playbook が成功終了する

**指摘要約**: `qm destroy` 失敗で `_brv_cleanup_ok: false` となり critical 通知は出るが、
最後の再 fail は `_brv_failed` のみを条件としていたため、正常性判定通過後の destroy
失敗で終了コードが成功になる。cleanup 失敗は検証全体の失敗として非ゼロ終了させ、
fail 条件に `not _brv_cleanup_ok` を含め、メッセージにも cleanup 失敗を明示すべき。

**対応**: 最後の "Re-raise failure" task を修正。

- `when` を `(_brv_failed | bool) or (not (_brv_cleanup_ok | bool))` に変更。
- メッセージを、通常の失敗・cleanup 失敗（999 残骸が手動対応必要）のどちらの場合でも
  内容が分かるよう拡張。
- これにより、健全性判定が OK でも 999 の destroy に失敗した実行は非ゼロ終了し、
  Semaphore / timer 側で失敗として記録される。critical 通知（指摘前から実装済み）と
  併せて手動対応へ確実に繋がる。

## 変更点の詳細（task レベル）

- init: `_brv_restore_attempted: false` を追加。
- Phase 2: `qmrestore` 直前に「Mark restore as attempted」task を追加。
  既存の「Mark 999 as created by us」は「Mark 999 restore command succeeded」に改名
  （`_brv_restored` はレポート用途として維持）。
- always / cleanup: 実行条件を所有権ベースへ変更し、内部で cluster resources を
  再取得して 999 の現存を確認してから stop/destroy する二段構成へ再構成。
- report: `restore_attempted` フィールドを追加。
- Re-raise failure: 条件に `not _brv_cleanup_ok` を追加し、メッセージを拡張。

## 注意点

- cleanup の再クエリ（`pvesh get /cluster/resources`）自体が失敗した場合も
  `_brv_cleanup_ok: false` となり非ゼロ終了する。999 の不在を確認できない以上、
  保守的に「要手動確認」として扱う設計。
