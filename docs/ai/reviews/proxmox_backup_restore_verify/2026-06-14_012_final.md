## 3者合意（Yoshinobu / 実装 Claude Code / レビュー Codex）

同時実行時のロック所有権は、このツールの本質ではないと合意した。

- 本ツールの本質はコアVMの vzdump バックアップのサイレント破損検出（月次）。
- 同時実行競合は「月次自動（quory・固定時刻）と手動（ansy）が秒単位で重なった場合のみ」
  発生し、被害は**検証専用の使い捨て VMID 999 の消失に限定**される。本番VM・本番データへ
  の波及経路は、999 固定 assert と開始前残骸ガードが維持される限り存在しない（Codex 確認済み）。

この合意に基づき、ロックは「運用上の同時実行禁止を補助する最小ガード」として扱い、完全な
分散排他は要求しない。残余リスクは運用判断として受容する。

## 確定した方針

1. 同時実行は運用で禁止する（月次は quory 固定スケジュール単一、手動は人が非重複を確認）。
2. ロックは最小化する（公式 pmxcfs の atomic mkdir で取得、取れなければ即 fail・通知・
   非ゼロ終了。空dir なので pmxcfs の 120秒陳腐化回収が効く。refresher / 期限更新 /
   生存監視 / 孤児管理は持たない）。
3. 本番への危害防止は、ロックではなく以下で担保する。
   - destroy 対象を 999 に固定する hard assert（999 以外は絶対に destroy しない）
   - 開始前 999 残骸ガード（既存 999 には触れない）
   - 999 への一意トークン刻印（別実行のトークンなら触れない／cheap defence in depth）
4. 残余リスク（120秒超の保持窓 × 低頻度同時実行で、最悪 使い捨て999 の検証1サイクルが
   無駄になる。本番影響なし）を受容として明記する。

## 今後のレビュー観点（合意）

ロック所有権の深掘りではなく、本質に重心を移す。

- 最新かつ正しい対象VMのバックアップを選べるか
- NIC 切断後に起動して有効な正常性判定ができるか
- destroy 対象が構造的に 999 に固定されているか
- 失敗時に 999 を安全に処理できるか
- Slack 通知 / レポート / 終了コードが実結果と一致するか
- 本番VMへ変更操作が及ばないか
