## 概要

009 review の指摘2点に対応した。いずれもロック機構の堅牢性に関するもの。

1. [重大] ロック期限更新の継続を確認せず「未刻印999＝自分のもの」と判定して destroy
   → 別実行の作成途中999を削除し得る。
2. [高] lock dir 内の `owner` ファイルが pmxcfs 標準の空dir前提の期限切れ回収を妨げる。

変更は `roles/proxmox_backup_restore_verify/` の tasks/main.yml と defaults/main.yml のみ。

## レビュー指摘と対応

### [重大] ロック期限切れ後、別実行の未刻印999を自分のものとして削除し得る

**指摘要約**: refresher は `nohup ... &` するだけで生存・touch成功を確認しない。期限が
切れれば別実行が原子的 mkdir に成功して `qmrestore` へ進み、刻印前の999を作る。先行実行
が cleanup でそれを読み、`brv-owner` 無しを根拠に destroy してしまう。cleanup 時に
「自分が現在もロックを継続保持していること」を確認できない場合、未刻印999を destroy
してはならない。確認不能時は999を残して critical 通知・非ゼロ終了へ倒すこと。

**対応**: 未刻印999の所有判定を「ロックの継続保持を確認できた場合のみ」に厳格化した。

- refresher の **PID を捕捉**（`echo $!`）。`nohup`（exec され fork しない）を使うため
  `$!` は実ループの PID と一致し、後段の `kill -0` / `kill` が確実に効く。
- refresher を **有効期間で上限化**（`verify_lock_max_hold_seconds`、既定1800秒）。
  孤児化した refresher が無期限にロックを保持し続けないようにする（上限到達で touch を
  止め、ロックは ≤120秒で陳腐化し回収可能になる）。
- always の冒頭に **「ロック継続保持の再確認」** を追加し、次の3条件すべてで
  `_brv_lock_still_held` を判定する。
  1. refresher プロセスが生存（`kill -0 <pid>` rc==0）
  2. lock dir が存在（`stat` rc==0）
  3. lock dir の mtime が新しい（`now - mtime < verify_lock_stale_seconds`）
  → プロセス停止・即時終了は (1)、touch 失敗は (3) で検出する（指摘の
  「起動直後の終了 / touch 失敗 / プロセス停止」を網羅）。
- cleanup の所有判定を以下に変更。
  - 刻印トークン == 自分 → 自分の999、destroy。
  - 刻印トークン == 別実行 → destroy しない。
  - 未刻印 **かつ** `_brv_lock_still_held` → 自分の途中失敗（保持中は他実行が999を作れず、
    VMID 999 はクラスタ一意）、destroy。
  - 未刻印 **かつ** 継続保持を確認できない → **destroy せず** `_brv_cleanup_ok: false` を
    立てる → 既存の「`not _brv_cleanup_ok` で再 fail」と critical 通知により、999を残した
    まま非ゼロ終了する。
- この判定を localhost の使い捨て playbook で5ケース検証（下記）。

### [高] lock dir 内の `owner` ファイルが標準の期限切れ回収を妨げる

**指摘要約**: pmxcfs 標準ロックは空ディレクトリを `mkdir`/`rmdir` し、期限切れ回収も空dir
前提。中に `owner` ファイルを作ると、別実行の標準回収がディレクトリを削除できず、プロセス
停止時に手動再帰削除まで全実行が止まる。所有情報は pmxcfs 外で管理すべき。

**対応**: lock dir を **空のまま**扱うようにした。

- Phase -1 から `owner` ファイル作成を削除。所有/更新生存の情報は **pmxcfs 外**
  （refresher の PID = Ansible メモリ上のfact、および lock dir の mtime）で管理する。
- 取得失敗時の診断は `owner` 読み取りをやめ、`stat -c %Y`（lock の mtime epoch）と
  陳腐化秒数を提示する形へ変更。
- 解放は `owner` トークン照合をやめ、**空dirを `rmdir`** する方式へ変更。`rmdir` は
  `_brv_lock_still_held` が真の場合のみ実行（継続保持を確認できた＝確実に自分のdir）。
  確認できない場合は別実行が同じdirを回収済みの可能性があるため `rmdir` せず、pmxcfs の
  標準陳腐化回収に委ねる。
- これにより、Ansible/ノードが cleanup 前に停止しても、空dirは120秒経過後に別実行の
  `mkdir` で回収可能となり、手動削除は不要（標準の期限機構を無効化しない）。

### 解放時の refresher 停止（孤児対策）

- always の解放で、まず **PID を `kill`**、続いて **マーカーで `pkill -f`**（二重化）して
  refresher を確実に停止してから（保持を確認できた場合のみ）`rmdir` する。

## 多層防御の整理（更新後）

| 層 | 役割 |
|---|---|
| Phase -1 公式 mkdir ロック（空dir） | 原子的な同時実行排他 + 標準の陳腐化回収を阻害しない |
| 有界 refresher + 継続保持の再確認 | 期限更新の生存・成功を監視し、確認時のみ未刻印999を自分扱い |
| 999 の description トークン | 一意な所有権の確定的根拠（最優先） |
| 未刻印かつ確認不能 → cleanup失敗 | 別資源を壊さず critical 通知・非ゼロ終了へ escalate |
| Phase 0 残骸ガード | 開始前から存在する999に触れない |
| `not _brv_cleanup_ok` 再 fail | destroy 失敗・確認不能を非ゼロ終了 |

## 注意点・未対応事項

- refresher は `nohup`（fork せず exec、`$!` が実PID）＋全stdioリダイレクトで起動し、
  PID を `kill -0` 監視に使う。`setsid` は fork により `$!` が実ループ PID と乖離し得るため
  採用しない。
- lock dir は**空**を厳守する（pmxcfs 標準の陳腐化回収の前提）。所有/生存情報を dir 内に
  書かない。
- スタールロック（プロセス全停止時）は、空dirのため pmxcfs の120秒回収で自動的に
  `mkdir` 再取得可能になり、手動削除は原則不要。
- 残存し得る前提: 本実装は「refresher 生存＋lock mtime 鮮度」で継続保持を判定する。
  これは Proxmox 公式のロック有効期間（120秒）と更新方式に沿った確認であり、確認不能時は
  破壊せず escalate する安全側に倒している。
- 実機での mkdir ロック競合・期限更新・期限切れ回収・同時実行・途中失敗の確認は未実施。
- 要求仕様の初回除外スコープ（serial0 マッチ / 鮮度チェック / 履歴永続化）は引き続き対象外。
