## 結論

**修正必須。**

007 review の核心だった非原子的な `O_EXCL` は、pmxcfs の原子的な `mkdir` へ
置き換えられている。しかし、ロック期限更新が失敗した場合に「ownerトークンの無い
999は自分のもの」と判定して destroy するため、別実行が作成途中の999を削除する経路が
残っている。要求仕様の「自分が作成した999のみ」を満たすには、未刻印を所有の根拠に
してはならない。

対象ファイルはすべて未追跡のため、通常の `git diff` には表示されなかった。
新規 playbook / role と008追加実装を直接確認した。

## Findings

### [重大] ロック期限切れ後、別実行の未刻印999を自分のものとして削除できる

対象:

- `roles/proxmox_backup_restore_verify/tasks/main.yml:108`
- `roles/proxmox_backup_restore_verify/tasks/main.yml:219`
- `roles/proxmox_backup_restore_verify/tasks/main.yml:239`
- `roles/proxmox_backup_restore_verify/tasks/main.yml:380`
- `roles/proxmox_backup_restore_verify/tasks/main.yml:388`
- `roles/proxmox_backup_restore_verify/tasks/main.yml:402`

refresher は `nohup ... &` を起動するだけで、子プロセスの生存や `touch` 成功を確認しない。
起動直後の終了、`touch` 失敗、プロセス停止などが起きても main lifecycle は継続する。
pmxcfsロックの期限が切れれば、別実行は原子的 `mkdir` に成功して次の `qmrestore` へ
進める。

この状態で以下の競合が成立する。

1. 実行Aがロックを取得するが、refresherが停止または更新に失敗する。
2. Aの処理中に120秒が経過し、実行Bが期限切れロックを回収して取得する。
3. Bが `qmrestore` で999を作成するが、descriptionへのトークン刻印前である。
4. Aがcleanupでその999を読み、`brv-owner` が無いため `_brv_999_owned: true` とする。
5. AがBの999をstop/destroyする。

これは008 implement自身が「残存する極小の競合窓」として認識している経路だが、
controlled applyで破壊対象の所有権を保証する要件に対して無視できない。頻度ではなく、
条件成立時に別実行の資源を削除することが問題。

cleanup時に自分が現在もロックを保持していることを確認できない場合、未刻印999を
destroyしてはならない。少なくとも、lock directoryと所有情報が自分の実行に属し、
期限更新が継続していることを確認できる場合だけ、未刻印999を部分リストアとして
cleanup対象にする必要がある。確認不能時は999を残してcritical通知・非ゼロ終了へ倒すこと。

### [高] lock directory内の`owner`ファイルが標準の期限切れ回収を妨げる

対象:

- `roles/proxmox_backup_restore_verify/tasks/main.yml:99`
- `roles/proxmox_backup_restore_verify/tasks/main.yml:442`
- `roles/proxmox_backup_restore_verify/tasks/main.yml:449`

Proxmox標準のpmxcfsロックは、`/etc/pve/priv/lock/<name>` を空ディレクトリとして
`mkdir`し、解放時は`rmdir`する。期限切れ回収もロックディレクトリの除去を前提とする。
現在はその中へ`owner`ファイルを作るため、別実行の`utime(0, 0, lockdir)`による標準の
期限切れ回収がディレクトリを削除できない可能性が高い。

その結果、Ansibleプロセスや対象ノードがcleanup前に停止すると、公式の120秒期限では
回収されず、手動で再帰削除するまで全実行が停止する。実装報告ではスタールロックを
手動削除する設計としているが、これは公式ロックの期限機構を意図せず無効化している。

ロックディレクトリは空のまま扱い、所有情報が必要ならpmxcfsロックの外側で管理するか、
Proxmoxの`PVE::Cluster`ロックAPIを利用する方がよい。少なくとも実機で、120秒経過後に
別ノードからの取得操作が当該ディレクトリを回収できることを確認できない限り、
自動実行へ投入すべきではない。

## 前回指摘の確認

- pmxcfsの非原子的`O_EXCL`使用: **解消済み**。原子的`mkdir`へ変更されている。
- 120秒を超える処理への更新処理: 実装されたが、更新成功の監視と失敗時の停止がない。
- `qmrestore`途中失敗のcleanup、cleanup失敗時のcritical通知・非ゼロ終了: 対応済み。

## その他の確認事項

- ローテーション、手動VMID指定、preferタグ、agent期待レベル別判定、NIC削除、
  開始前から存在する999の保護は要求仕様に沿っている。
- IPリテラル、秘密情報、inventory / group_vars / host_vars の変更は確認されなかった。
- 実機でのmkdirロック、期限更新、期限切れ回収、同時実行は未確認。
