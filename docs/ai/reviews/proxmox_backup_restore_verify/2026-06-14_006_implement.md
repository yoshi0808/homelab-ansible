## 概要

005 review の指摘1点（同時実行時の VMID 999 所有権競合）に対応するため、
**クラスタ共有のロックによる排他制御**を追加した。変更は
`roles/proxmox_backup_restore_verify/` の tasks/main.yml と defaults/main.yml のみ。
playbook / inventory / group_vars / host_vars には変更なし。

## レビュー指摘と対応

### [高] 同時実行すると、後発実行が先行実行の 999 を削除する

**指摘要約**: 所有権判定が「開始前に 999 なし ＋ `_brv_restore_attempted: true`」の
組み合わせだけで、Phase 0 から cleanup までを保護する排他制御が無い。quory の月次実行 A
と ansy の手動実行 B が重なると、B の `qmrestore` は A が作った 999 で失敗するが、
B は `restore_attempted` が true のため cleanup を実行し、再取得した「A の 999」を
destroy してしまう。Phase 0 から cleanup 完了まで、quory と ansy の実行元をまたいで
有効な排他制御（または一意な所有権確認）を設け、取得できなければ変更操作へ進まず
通知して非ゼロ終了すること。

**対応**: 実行元をまたいで有効な排他を、**Proxmox クラスタファイルシステム
（`/etc/pve`, pmxcfs）上のロックファイル**で実装した。control node ローカルの lock
では別 control node からの実行を防げないという指摘どおり、ロックは control node
（quory / ansy）ではなく**クラスタ側**に置く。これは VMID 999 がクラスタ一意である
ことと整合し、リストア先ノード（pve1 / pve2）の play から `/etc/pve` を介して全ノードが
同一ロックを参照する。

実装の要点:

- defaults に `verify_lock_path: /etc/pve/brv-restore-verify.lock` を追加。
  pmxcfs 上に置く理由をコメントで明示。
- init facts に以下を追加:
  - `_brv_run_token`: 実行を一意に識別するトークン
    （`<node>-<timestamp>-<uuid>`）。
  - `_brv_lock_acquired: false`。
- **Phase -1（block 先頭、Phase 0 残骸チェックより前）でロックをアトミック取得**:
  - `set -C`（O_EXCL 相当 / noclobber）でロックファイルへトークンを書き込む。
    既存なら書き込みは失敗し rc≠0。`failed_when: false` で受けて分岐。
  - 取得失敗時は現ロック保持者を `cat` して診断情報を取得し、
    変更操作前に `fail`（待機せず即時・非ゼロ終了）。
  - 取得成功時のみ `_brv_lock_acquired: true`。
- **always でロックを解放**（cleanup ブロックの後、レポート保存の前）:
  - `_brv_lock_acquired` が true のときのみ実行。
  - 解放前にロック内容を読み、**自分のトークンが含まれる場合のみ** 削除する。
    取得に失敗して中断した実行（`_brv_lock_acquired: false`）はこのブロックを
    丸ごとスキップするため、他実行のロックを誤って削除しない。
  - ロックは cleanup（stop / destroy）を内側に含んだまま保持されるため、destroy も
    ロック保護下で行われる。
- レポートに `run_token` / `lock_path` / `lock_acquired` を追加。

**なぜこれで所有権が一意に確定するか**:
ロックを Phase -1 から cleanup まで保持する間、他実行は同一ロックを取得できず
`qmrestore`（999 作成）に到達できない。したがって、自分の `qmrestore` 後に存在する
999 は構造的に自分の作成物であることが保証される。これは要求仕様「自分が作成した 999
のみを destroy 対象とする」を、実行元（quory / ansy）をまたいで満たす。

**競合シナリオの結果（修正後）**:

1. A・B がともに play2 開始。
2. ロックは一方（例: A）だけが取得成功。
3. B は取得失敗 → 現保持者を診断ログに残し、`qmrestore` 前に fail（非 зеро終了・通知）。
   B は `_brv_restore_attempted` を立てる前に止まるため cleanup は走らず、
   `_brv_lock_acquired` が false なのでロック解放もスキップ。A の 999・ロックは無傷。
4. A は通常どおりリストア検証 → cleanup → ロック解放。

## 既存の安全装置との関係（多層防御）

| 層 | 役割 | 対象 |
|---|---|---|
| Phase -1 クラスタロック | 同時実行の排他（004 で未対応だった競合） | 別実行との競合 |
| Phase 0 残骸ガード | 開始前から存在する 999 に触れない | 過去のクラッシュ残骸 |
| always 再クエリ cleanup | 自実行の途中失敗 999 を確実に destroy | 単独実行の中断 |
| `not _brv_cleanup_ok` での再 fail | destroy 失敗を非ゼロ終了 | cleanup 失敗 |

## 注意点・未対応事項

- ロックは `set -C`（noclobber リダイレクト）で取得する。Proxmox の `/bin/sh`（dash）
  でも `set -C` は有効で、既存ファイルへのリダイレクトは失敗する。
- **スタールロックは自動破棄しない**（自動破棄は競合を再導入するため）。クラッシュ等で
  ロックが残った場合は、取得失敗として通知・非ゼロ終了し、保持者トークンをログに残す。
  運用者が `/etc/pve/brv-restore-verify.lock` を確認して手動削除する。月次ジョブとして
  保守的側に倒す設計。
- ロック解放は「自分のトークンを含む場合のみ削除」。`_brv_lock_acquired: false`（取得失敗）
  の実行は解放処理自体をスキップするため、他実行のロックは削除しない。
- 実機での同時実行・途中失敗・スタールロックの動作確認は未実施。
- 要求仕様の初回除外スコープ（serial0 マッチ / 鮮度チェック / 履歴永続化）は引き続き対象外。
