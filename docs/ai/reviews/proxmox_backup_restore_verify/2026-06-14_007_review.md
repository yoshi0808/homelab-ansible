## 結論

**修正必須。**

005 review で指摘した同時実行競合は未解消。追加されたロックは pmxcfs 上の
`O_EXCL` create（shell の `set -C`）を原子的と仮定しているが、Proxmox 公式の
`pve-cluster` 資料には、pmxcfs の `O_EXCL` create は原子的ではないと明記されている。
したがって2実行が同時にロック取得成功と判断でき、前回指摘した「後発実行が先行実行の
999 を削除する」経路が残る。

対象ファイルはすべて未追跡のため、通常の `git diff` には表示されなかった。
新規 playbook / role と006追加実装を直接確認した。

## Findings

### [重大] pmxcfs 上の `set -C` は原子的なクラスタロックにならない

対象:

- `roles/proxmox_backup_restore_verify/defaults/main.yml:8`
- `roles/proxmox_backup_restore_verify/tasks/main.yml:63`
- `roles/proxmox_backup_restore_verify/tasks/main.yml:72`
- `roles/proxmox_backup_restore_verify/tasks/main.yml:98`

実装は `/etc/pve/brv-restore-verify.lock` への noclobber リダイレクトを
「O_EXCL 相当で原子的、必ず1実行だけが成功する」としている。しかし pmxcfs は通常の
POSIXファイルシステムではなく、公式ソース資料の POSIX Compatibility 節には
`O_EXCL creates were not atomic` と明記されている。

そのため quory と ansy から同時に実行した場合、両方がロック取得成功と判断して
`_brv_lock_acquired: true` へ進む可能性がある。両方が Phase 0 で 999 不在を確認した後、
一方の `qmrestore` が 999 を作成し、もう一方が失敗して cleanup に入ると、後者が前者の
999 を destroy する。これは005 review の競合シナリオと同じであり、要求仕様の
「自分が作成した 999 のみを destroy 対象とする」を満たさない。

Proxmox の同資料が示すクラスタロック方式は、`/etc/pve/priv/lock/<LOCK-NAME>` への
`mkdir` である。pmxcfs ではこの `mkdir` が原子的で、既存ディレクトリがある場合は
失敗する。実装は Proxmox が提供するこのロック機構、または同等の公式APIを使用する
必要がある。

さらに公式方式のロック有効期間は120秒である。`qmrestore` と agent 待機は120秒を
超え得るため、ロック保持中は同じノードから `utime 0, time()` 相当で期限を更新するか、
Proxmox のロックAPIが提供する更新処理を利用する必要がある。更新せずに期限切れとなれば、
別実行がロックを取得して同じ競合が再発する。

公式資料:

- `https://github.com/proxmox/pve-cluster/tree/master/src`
  - POSIX Compatibility: `O_EXCL` create は非原子的
  - File Locking: `/etc/pve/priv/lock/` 配下への原子的な `mkdir` と120秒の期限・更新方法
