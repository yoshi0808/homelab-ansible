## 結論

対象ファイルはすべて未追跡のため通常の `git diff` には表示されなかったが、新規

## Findings

### [重大] 1. `qmrestore` の途中失敗で作成された 999 が cleanup 対象にならない

対象:

- `roles/proxmox_backup_restore_verify/tasks/main.yml:156`
- `roles/proxmox_backup_restore_verify/tasks/main.yml:163`
- `roles/proxmox_backup_restore_verify/tasks/main.yml:262`

`_brv_restored` は `qmrestore` が成功終了した後にだけ `true` になる。`qmrestore` が
VM設定やディスクの作成後に失敗した場合、タスクは直ちに `rescue` へ移るため
`_brv_restored` は `false` のままとなる。その結果、`always` の cleanup block が
丸ごとスキップされ、今回の実行が作成途中の 999 を残す。

これは要求仕様の「リストア/起動が失敗しても、always で必ず 999 のクリーンアップを
試みる」「ただし自分が作成した 999 のみを destroy 対象とする」を満たさない。
次回実行は残骸ガードで停止するため、月次検証が手動復旧まで継続不能になる。

開始前に 999 が存在しないことを確認した事実を保持したうえで、`qmrestore` 失敗後に
cluster resources / config を再確認し、この実行中に出現した 999 を cleanup 対象として
扱う必要がある。少なくとも「コマンド成功」を「作成所有権」の唯一の根拠にしないこと。

### [高] 2. destroy 失敗でも playbook が成功終了する

対象:

- `roles/proxmox_backup_restore_verify/tasks/main.yml:276`
- `roles/proxmox_backup_restore_verify/tasks/main.yml:284`
- `roles/proxmox_backup_restore_verify/tasks/main.yml:349`

`qm destroy` が失敗すると `_brv_cleanup_ok: false` になり critical 通知は送られるが、
`_brv_failed` は更新されない。最後の再 fail は `_brv_failed` だけを条件としているため、
正常性判定まで成功した後の destroy 失敗では playbook の終了コードが成功になる。

Semaphore / timer 側では成功実行として記録される一方、VMID 999 は残存し、次回実行は
残骸ガードで停止する。cleanup 失敗は検証全体の失敗として非ゼロ終了させるべきであり、
最後の fail 条件に `not _brv_cleanup_ok` を含め、エラーメッセージにも cleanup 失敗を
明示する必要がある。

## 確認事項

- ローテーション、`target_vmid` バイパス、prefer タグによるノード選択、agent 設定に
  応じた判定、NIC 削除後の起動、既存 999 を触らないガードは要求仕様に沿っている。
- IP リテラルや秘密情報の追加は確認されなかった。
- inventory / group_vars / host_vars の変更はない。
- 実機での `qmrestore` / 起動 / destroy は実装報告どおり未確認。
