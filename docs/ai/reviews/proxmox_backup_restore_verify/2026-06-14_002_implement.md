## 実装概要

コアVM（`verify` タグ付き）の最新 vzdump バックアップを、検証専用 VMID 999 へ
qmrestore し、NIC を切断して起動し、正常性を判定したうえで 999 を destroy する
月次リストア検証を実装した。フルライフサイクルを block / rescue / always で囲み、
失敗時も 999 を必ずクリーンアップする。本番VMは config 読み取りのみ。

playbook は2プレイ構成。

1. **Play 1（determine, `hosts: proxmox` run_once）**
   cluster resources から対象VM・リストア先ノード・agent 期待レベルを決定し、
   `add_host` で動的グループ `brv_restore_targets` に登録する。
2. **Play 2（verify, `hosts: brv_restore_targets`）**
   決定したノード上で role を実行し、リストア検証のライフサイクルを回す。

`add_host` を採用した理由: `hosts: "{{ ... }}"` を hostvars で動的指定すると
`--syntax-check` がパース時に hostvars 未定義で失敗するため。動的グループなら
実行時に解決され、対象ノード（pve1/pve2）の group_vars/proxmox.yml 接続情報も
そのまま継承される。

## 変更内容の要約

### ローテーション選定（Play 1）
- `pvesh get /cluster/resources` を取得。
- `verify` タグを持つ QEMU VM を `vmid` 昇順でソートし決定的リスト化。
- `(date +%m | int - 1) % リスト長` で今月の対象を確定。
- `-e target_vmid=<id>` が渡された場合はローテーションをバイパスし、その VMID を
  cluster resources から直接特定（存在しなければ fail）。
- 対象VMの `tags` から `preferpve1` / `preferpve2` を判定しリストア先ノードを決定。
  どちらも無ければ fail。
- `pvesh get /nodes/<node>/qemu/<vmid>/config`（API、read-only）で `agent` を読み、
  期待レベルを決定（`agent == '1'` または `enabled=1` を含む → agent 期待）。

### リストア検証ライフサイクル（Play 2 / role）
- **Hard guards（block の外）**: `verify_restore_vmid == 999` を assert。
  `verify_target_vmid` が正かつ 999 でないことを assert。
- **Phase 0 残骸ガード**: cluster resources に VMID 999 が存在したら
  `_brv_preexisting_residue` を立てて fail（残骸には触れない）。
- **Phase 1 バックアップ特定**: `verify_backup_storage` 未指定時は、対象ノードの
  storage 一覧から `type == nfs` かつ `content` に `backup` を含むものを自動検出。
  各ストレージの `content --content backup --vmid <id>` を取得し、`ctime` 昇順で
  ソートして最新（末尾）の volid を選択。0 件なら fail。
- **Phase 2 リストア**: `qmrestore <volid> 999 --storage <verify_restore_storage>`。
  成功で `_brv_restored: true`（= 自分が作成した 999 の印）。
- **Phase 3 NIC 切断**: 999 の config から `net\d+` キーを抽出し、各 `qm set 999
  --delete netN` で削除（IP リテラルは一切参照しない）。
- **Phase 4 起動**: `qm start 999`、`_brv_started: true`。
- **Phase 5 正常性判定**:
  - agent 期待: `qm agent 999 get-osinfo` を retries/delay でポーリングし、rc==0 で合格。
  - agent 無し: `verify_running_settle_seconds` 待機後 `qm status` が running なら合格。
    （次フェーズの serial0 コンソール文字列マッチに差し替えられるよう block で分離）
  - 未達なら fail。
- **rescue**: 失敗情報を捕捉（`_brv_failed` / `_brv_error_msg`）。
- **always**:
  - クリーンアップは `_brv_restored` が真のときのみ（自分が作成した 999 のみ）。
    `qm stop`（started 時のみ best-effort）→ 待機 → `qm destroy 999 --purge 1`。
    destroy 失敗は rescue で `_brv_cleanup_ok: false`（残骸）。
  - JSON レポートを control node 側 `{{ reports_base_dir }}/proxmox-backup-verify/`
    に保存。
  - Slack 通知（best-effort, `common_slack/tasks/notify.yml`）。優先順位:
    残骸検出 / destroy 失敗 → `alerts`/`critical`、検証 NG → `alerts`/`error`、
    OK → `info`/`ok`。
  - `_brv_failed` のとき最後に再 fail して終了コードへ反映。

### 安全装置の対応関係（要求仕様 → 実装）
| 要求 | 実装 |
|---|---|
| destroy 対象は 999 固定 | hard assert + 全 qm コマンドが `verify_restore_vmid` 固定 |
| 999 衝突しないこと | Phase 0 で cluster resources に 999 が無いことを確認 |
| 失敗しても必ずクリーンアップ | `always` で destroy。ただし自作成分のみ |
| 残骸は勝手に消さない | `_brv_restored` ガードで残骸を destroy しない |
| destroy 失敗は critical 通知 | `_brv_cleanup_ok=false` → critical |

## 実行した確認コマンド

```bash
ansible-playbook -i inventories/homelab/hosts.yml \
  playbooks/proxmox_backup_restore_verify.yml --syntax-check
```

結果（抜粋）:

```text
[WARNING]: Could not match supplied host pattern, ignoring: brv_restore_targets
playbook: playbooks/proxmox_backup_restore_verify.yml
```

- syntax-check は成功。`brv_restore_targets` の WARNING は、動的グループが
  実行時（`add_host`）に生成されるためパース時には存在しないことによる想定どおりの警告。
- 併せて playbook / role tasks / defaults を Python `yaml.safe_load_all` でパース確認（全て OK）。

実機（pve1/pve2 上での qmrestore/start/destroy）の実行確認は未実施。

## 注意点

- `become: true`（root）で実行する（NFS のアクセス権が root に絞られているため）。
- `verify_backup_storage` を空のままにすると NFS バックアップストレージを自動検出する。
  複数該当する場合は全ストレージ横断で最新を選ぶ。1 つに固定したい場合はストレージ名
  （IP ではない）を指定する。
- `verify_restore_storage` は既定 `local-zfs`（本 homelab の全 VM が local-zfs）。
- IP リテラルは未記載。NIC は net デバイス削除で切断し IP を指定しない（core.md §3）。
- 実機未検証のため、`qm agent get-osinfo` の待機回数・待機秒や
  `verify_running_settle_seconds` は実環境に合わせて調整が必要な場合がある。
