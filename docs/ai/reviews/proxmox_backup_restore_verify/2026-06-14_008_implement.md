## 概要

007 review の指摘1点（pmxcfs 上の `O_EXCL`/`set -C` ロックは非原子的で、同時実行
競合が未解消）に対応した。ロック機構を **Proxmox 公式の `mkdir /etc/pve/priv/lock/`
方式（原子的）** へ置換し、**120秒期限のバックグラウンド更新（refresher）** を追加し、
さらに **VMID 999 の config に一意トークンを刻印して所有権を判定**するよう変更した。
変更は `roles/proxmox_backup_restore_verify/` の tasks/main.yml と defaults/main.yml のみ。

## レビュー指摘と対応

### [重大] pmxcfs 上の `set -C` は原子的なクラスタロックにならない

**指摘要約**: 実装は `/etc/pve/...lock` への noclobber リダイレクトを原子的と仮定して
いたが、Proxmox 公式 pve-cluster の POSIX Compatibility 節に「pmxcfs の `O_EXCL`
create は非原子的」と明記がある。よって2実行が同時に取得成功と判断し、005 で指摘した
「後発実行が先行実行の 999 を削除する」競合が残る。公式のロック方式は
`/etc/pve/priv/lock/<name>` への原子的 `mkdir`（既存なら失敗）であり、有効期間 120秒の
更新（`utime` 相当）も必要。

**対応（3点）**:

1. **公式の原子的 mkdir ロックへ置換**
   - defaults: `verify_lock_path`（ファイル）を廃止し、
     `verify_lock_dir: /etc/pve/priv/lock/brv-restore-verify` を追加。
   - Phase -1 のロック取得を `mkdir {{ verify_lock_dir }}` に変更。pmxcfs では
     ディレクトリ作成が原子的で、既存なら rc≠0 で失敗する。`failed_when: false` で
     受け、rc≠0 なら保持者トークン（`<dir>/owner`）を読んで診断し、変更操作前に
     即時 `fail`（待機なし）。
   - 取得成功時のみ `<dir>/owner` に自トークンを書き込み、`_brv_lock_acquired: true`。

2. **120秒期限のバックグラウンド更新（refresher）**
   - defaults: `verify_lock_refresh_interval: 60`（< 120s）を追加。
   - Phase -1 で、ロックディレクトリが存在する間 `touch -c` で mtime を更新し続ける
     デタッチ済みループ（`nohup ... &`）を起動。コマンド末尾に
     `# brv-refresh-<token>` のマーカーを埋め込み、`always` で `pkill -f` により確実に
     停止できるようにした。
   - `qmrestore` + agent 待機（最大 retries×delay）は 120秒を超え得るため、保持中に
     ロックが期限切れにならないようにする。

3. **VMID 999 への一意トークン刻印による所有権確認（公式ロックとの二重化）**
   - Phase 2: `qmrestore` 成功直後に
     `qm set 999 --description "brv-owner=<token>"` でトークンを刻印。
   - cleanup: 999 の config（`pvesh .../config`）の description からトークンを抽出し、
     destroy 可否を以下で判定する。
     - トークン == 自分 → 自分が作成した 999、destroy。
     - トークン無し → 自分の途中失敗 999（ロック保持中は他実行が 999 を作成できず、
       VMID 999 はクラスタ一意のため、未刻印の 999 は自分のもの）、destroy。
     - トークン == 別実行 → 他実行の所有物、**destroy しない（競合の核を遮断）**。
   - これにより、万一ロックの取りこぼしがあっても、別実行が作成・刻印した 999 を
     後発実行が削除しない。要求仕様「自分が作成した 999 のみを destroy 対象とする」を
     満たす。

### ロック解放（always）

- まず `pkill -f "brv-refresh-<token>"` で refresher を停止（解放後に再更新されないよう
  destroy 後・rmdir 前に停止）。
- `<dir>/owner` を読み、**自トークンを含む場合のみ** `verify_lock_dir` を削除。
- 取得に失敗して中断した実行（`_brv_lock_acquired: false`）は解放処理ごとスキップし、
  他実行のロックを誤削除しない。

## 多層防御の整理

| 層 | 役割 | 対象 |
|---|---|---|
| Phase -1 公式 mkdir ロック + refresher | 原子的な同時実行排他（007 の核心） | 別実行との競合 |
| 999 の description トークン | 一意な所有権確認（destroy 可否の最終根拠） | 競合・途中失敗の識別 |
| Phase 0 残骸ガード | 開始前から存在する 999 に触れない | 過去のクラッシュ残骸 |
| always 再クエリ cleanup | 自実行の途中失敗 999 を destroy | 単独実行の中断 |
| `not _brv_cleanup_ok` での再 fail | destroy 失敗を非ゼロ終了 | cleanup 失敗 |

## 実行した確認コマンド

```bash
# syntax-check
ANSIBLE_LOCAL_TEMP=/tmp/ansible-local \
ANSIBLE_REMOTE_TEMP=/tmp/ansible-remote \
ansible-playbook -i inventories/homelab/hosts.yml \
  playbooks/proxmox_backup_restore_verify.yml --syntax-check
```

結果: 成功（`brv_restore_targets` の WARNING は add_host 動的グループによる想定どおり）。

加えて、所有権判定の Jinja 式を localhost の使い捨て playbook で 4 ケース検証した。

| 入力 description | 抽出 owner | 判定（owned） |
|---|---|---|
| `... brv-owner=node-123-abc ...`（自トークン） | `node-123-abc` | True（destroy） |
| トークン無し | `` | True（自分の途中失敗として destroy） |
| `brv-owner=other-run-xyz`（別実行） | `other-run-xyz` | False（**destroy しない**） |
| 空文字 | `` | True |

`regex_search('brv-owner=(\S+)', '\1')` は当環境ではリストを返すため、
`regex_search('brv-owner=\S+')`（グループ無し）＋ `regex_replace('^brv-owner=', '')` で
文字列として抽出する実装に修正済み。tasks/main.yml・defaults/main.yml の
`yaml.safe_load_all` パースも確認（OK）。

## 注意点・未対応事項

- ロックは公式の `mkdir /etc/pve/priv/lock/<name>`（原子的）＋ 60秒間隔の mtime 更新で
  保持する。`/etc/pve/priv/lock` は Proxmox が用意する pmxcfs 配下のため、play は
  リストア先ノード（pve1/pve2）上で become root として実行する。
- **スタールロックは自動破棄しない**（自動破棄は競合を再導入するため）。取得失敗時は
  保持者トークンをログに残して通知・非ゼロ終了し、運用者が `verify_lock_dir` を確認して
  手動削除する。
- 残存する極小の競合窓: refresher が停止してロックが期限切れになった瞬間に別実行が
  ロックを奪い、かつ先行実行が 999 を未刻印の状態であった場合に限り、未刻印 999 の
  所有権判定が曖昧になり得る。refresher（60秒間隔 / 120秒期限）により通常運用では発生
  しない。公式方式そのままの設計上の限界であり、運用上は無視できる。
- 実機でのロック競合・refresher 動作・スタールロック・途中失敗の確認は未実施。
- 要求仕様の初回除外スコープ（serial0 マッチ / 鮮度チェック / 履歴永続化）は引き続き対象外。
