# Review: ansy systemd timer setup

レビュー日: 2026-05-10
対象:

- `playbooks/ansy_systemd_timers.yml`
- `roles/ansy_systemd_timers/`

## Findings

### Medium: `RandomizedDelaySec` が未設定

`docs/ai/prompts/ubuntu_vm_patch_policy.md` では、systemd timer 方針として `RandomizedDelaySec` を設定し、実行時刻を若干ばらつかせることになっている。

該当方針:

- `docs/ai/prompts/ubuntu_vm_patch_policy.md:192`

現実装の timer template は以下のみで、`RandomizedDelaySec` がない。

- `roles/ansy_systemd_timers/templates/ansible-timer.timer.j2:4`
- `roles/ansy_systemd_timers/templates/ansible-timer.timer.j2:5`
- `roles/ansy_systemd_timers/templates/ansible-timer.timer.j2:6`

影響:

- 現在の要求仕様には明示されていないため、実装者は要求仕様には従っている。
- ただし、レビュー入力として `ubuntu_vm_patch_policy.md` が含まれている以上、方針との差分として扱うべき。
- 特に `ubuntu_nightly.yml` / `radius_healthcheck.yml` は Ubuntu VM patch policy の systemd timer 方針に直接関係する。

推奨:

- `systemd_ansible_timers` に `randomized_delay_sec` を追加する。
- もしくは role default で共通値を持たせ、timer template に `RandomizedDelaySec={{ ... }}` を出力する。
- あえて設定しない運用にするなら、要求仕様または policy 側に「今回の ansy 暫定 timer では設定しない」と明記する。

## Open Questions

### Ubuntu VM patch policy と今回要求仕様の時刻差分

`ubuntu_vm_patch_policy.md` では以下の時刻になっている。

- `ubuntu_nightly.yml`: 毎日 02:00
- `radius_healthcheck.yml`: 毎日 06:00

一方、今回の要求仕様と実装では以下になっている。

- `authy-reboot-if-required`: 毎日 03:30
- `authy-healthcheck`: 毎日 05:30

今回の要求仕様に従うなら実装は正しい。  
ただし、policy を正本として残すなら、どちらかのドキュメントを合わせた方がよい。

## 確認したこと

- `Proxmox patch apply` は timer 化されていない。
- timer 内で `git pull` はしていない。
- service unit は `flock -n` を使って二重起動を防いでいる。
- `WorkingDirectory=/home/yoshi/homelab-ansible`、`User=yoshi` は要求通り。
- role は `dev_nodes` 対象で、`become: true` により `/etc/systemd/system` 配置と systemd 操作を行う構成。
- `ansible.cfg` に inventory と roles_path があるため、service の `ExecStart` に `-i` がなくても working directory が正しければ inventory は解決される。
- 秘密鍵・認証情報・証明書秘密鍵などの追加はない。
- shell / script の追加はなく、shell 責務分離の問題はない。
- timer schedule は `systemd-analyze calendar` で少なくとも代表値を確認済み。
- `ansible-playbook --syntax-check playbooks/ansy_systemd_timers.yml -i inventories/homelab/hosts.yml` は成功。

## 判定

致命的な実行不能や危険な変更混入は見つからない。

ただし、`RandomizedDelaySec` 未設定は共有ポリシーとの差分であり、commit 前に直すか、今回の要求仕様側で例外として明記することを推奨する。

この点を解消すれば、初期運用版として commit 可能。
