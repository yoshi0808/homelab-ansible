# Review - tester_mode safety gate

## Summary

現状のままでは commit 不可。

`tester_mode=true` の安全弁という方向性は妥当だが、複数 play / `import_playbook`
構成の playbook で `meta: end_play` を使っているため、plan-only 表示後に後続 play が
実行される可能性がある。これは今回の目的である「tester が安全に実行できる」状態を
満たしていない。

## Findings

### must-fix: 複数 play 構成で `meta: end_play` しても後続 play が止まらない

対象:

- `playbooks/proxmox_evacuate_node.yml:27`
- `playbooks/proxmox_patch_apply_node.yml:28`
- `playbooks/proxmox_restore_vm_placement.yml:30`
- `playbooks/proxmox_patch_weekly_full.yml:176`
- `playbooks/cert_renew.yml:35`
- `playbooks/cert_renew_quory.yml:33`

理由:

`meta: end_play` は現在の play だけを終了する。playbook 全体を終了するものではない。

そのため、たとえば `playbooks/proxmox_patch_apply_node.yml` では validate play の
`tester_mode` ブロックで `end_play` した後、次の
`Proxmox patch apply (single node)` play が実行され、`proxmox_patch_apply_node` role に
進み得る。ここには `apt-get -y dist-upgrade` と reboot があるため、tester_mode の安全弁
として成立していない。

同様に `proxmox_evacuate_node.yml` は validate play だけが終了し、後続の healthcheck play と
evacuate play に進み得る。`proxmox_restore_vm_placement.yml` も restore play に進み得る。
`proxmox_patch_weekly_full.yml` は full patch window の mute play だけを終了し、
後続の `import_playbook: proxmox_evacuate_node.yml` / `proxmox_patch_apply_node.yml` へ進み得る。

`cert_renew.yml` / `cert_renew_quory.yml` も prepare CA play のみ終了し、後続の issue/deploy
play は残る。prepare が未実行なので失敗する可能性は高いが、「危険 play に進まない」
という保証にはなっていない。

修正案:

- 複数 play の入口で `tester_mode` 時に playbook 全体へ伝播する skip fact を立てる。
- すべての後続 play に `pre_tasks` で `tester_mode` / skip fact を確認して停止させる。
- もしくは危険 role 側の先頭にも `tester_mode` gate を置き、入口 play の停止漏れがあっても
  実変更 task に到達しない二重防御にする。
- `proxmox_patch_weekly_full.yml` は `tester_mode=true` の時点で後続 import が実行されない
  条件を明示する必要がある。

### must-fix: `.claude/.settings.json` の権限変更が今回の目的外かつ安全性を下げる可能性がある

対象:

- `.claude/.settings.json:37`

理由:

今回の変更範囲一覧に含まれていない `.claude/.settings.json` が変更され、`ask` に入っていた
`git commit` / `git push` / `ssh` / `rm` の明示確認ルールが削除されている。

今回の目的は「Claude Code は Ansible を直接叩かず、tester に安全実行させる」方向の安全弁追加であり、
Claude 側の command permission を緩める変更はスコープ外。特に `ssh` / `rm` / `git push` まわりの
確認ルール削除は、安全性の観点で逆方向に見える。

修正案:

- このファイル変更は今回の実装から除外する。
- もし意図的に必要なら、tester_mode 実装とは別工程で理由を明記してレビュー対象にする。

### suggestion: `core.md` の実装パターンに「複数 play では `end_play` だけで止まらない」を明記する

対象:

- `docs/ai/prompts/core.md:956`

理由:

追加された §18 は `meta: end_play` / `end_host` を実装パターンとして示しているが、
複数 play や `import_playbook` では `end_play` が現在 play のみを止める点が書かれていない。
今回の実装ミスはこの注意不足から再発しやすい。

修正案:

- §18 に以下を追加する。
  - 単一 play では `end_play` / `end_host` でよい。
  - 複数 play / `import_playbook` 構成では、全後続 play に tester gate を置く。
  - 危険 role 側にも gate を置く二重防御を推奨する。

### suggestion: plan-only が早すぎて対象抽出の検証価値が低い playbook がある

対象例:

- `playbooks/recovery_vm_reboot.yml:47`
- `playbooks/recovery_service_restart.yml:45`
- `playbooks/recovery_ha_failover.yml:48`
- `playbooks/proxmox_backup_restore_verify.yml:142`

理由:

現状は `target` の形式検証や monthly target 選定だけで止まるものが多く、実際のタグ再検証、
VMID、node、unit一覧、HA登録、backup volid などの「tester が確認したい計画情報」が十分に出ない。
安全性としては早めに止めるのは妥当だが、依頼要件にある「対象ホスト、対象VM、VMID、対象unit、
実行予定コマンド、判定済み条件を debug で表示する」には不足している。

修正案:

- read-only の `pvesh get` / stat / target selection までは実行する。
- 危険 task 直前で、解決済みの VMID / node / unit / migrate対象 / backup volid を表示して停止する。
- ただし初回は安全優先で、must-fix の停止漏れ修正を先に行う。
