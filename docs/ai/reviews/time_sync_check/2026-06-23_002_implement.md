# 実装内容: time_sync_check（時刻同期チェック）

## 実装概要

quory を基準ノードとして、pve1 / pve2 / ansy / monnie / authy / sophos-fw / CloudKey の
現在時刻を取得し、quory との差分を計算・閾値判定して Slack へ通知する read-only
playbook / role を実装した。

接続方式が異なる3グループ（通常SSH / Sophos Advanced Shell経由 / CloudKeyパスワードSSH）を
単一の playbook・単一の role（`roles/time_sync_check/tasks/main.yml`）に収め、
`inventory_hostname` ベースの `when:` 分岐と `run_once + delegate_to: localhost` の
集計タスクで差分計算・通知まで一本化した（要求仕様の「1 role / 1 playbook」構成に対応）。

あわせて、roles/radius_healthcheck/ の chrony 収集処理（判定なしの死コード）を削除した。

レビュー依頼前のユーザー指摘により、`sophos-fw` の接続変数の置き場所の不整合（後述）も
本セッションで修正した。

## 設計上の判断

### 1. 単一 play + `inventory_hostname` 分岐

`playbooks/time_sync_check.yml` は1つの play で `hosts: quory:pve1:pve2:ansy:monnie:authy:sophos`
を対象にする（CloudKeyは実ホストとしてインベントリに追加していない。後述）。

デフォルトの `linear` strategy では、play 内の全ホストが「同じタスク番号」を実行し終えてから
次のタスック番号に進む。これを利用し、各ホストの時刻取得タスクがすべて完了した後に、
`run_once: true` + `delegate_to: localhost` の集計タスク（Phase 4-6）で
`hostvars[]` 経由で全ホストの結果を読み取り、差分計算・通知をしている
（`proxmox_patch_weekly_full.yml` の `hostvars[groups['proxmox'][0]]['_final_status']` と同種のパターン）。

### 2. CloudKey はインベントリ非登録のまま、expect + delegate_to: localhost で取得

CloudKey は既存の `cloudkey_cert_deploy` / `unifi_backup_fetch` でも常に
HTTPSのuriモジュール経由（delegate_to: localhost）で扱われており、SSH接続可能な
Ansibleインベントリホストとして登録された前例がない。

今回はパスワード認証で `date +%s%3N` を1回叩くだけなので、Sophosと同様に
`ansible.builtin.expect` で `delegate_to: localhost` + `run_once: true` + `delegate_facts: true`
を使い、結果を `hostvars['localhost']` に保存した。インベントリに新規ホストを追加する
よりも変更範囲が小さく、既存のCloudKey運用パターン（API系はすべてlocalhost経由）と一貫する。

### 3. Sophos / CloudKey の時刻取得値の取り出し方

`echo __TIME_SYNC_TS__:$(date +%s%3N)` の1行echoにし、Ansible側で
`regex_search('__TIME_SYNC_TS__:([0-9]+)', '\1')` を使って値を抜き出す。
マーカーが見つからない（接続失敗・コマンド失敗）場合は空文字列にフォールバックし、
`time_sync_observed_ok: false` として「接続/収集エラー」に分類する。

### 4. 通知ステータスの優先順位

要求仕様の表は「閾値超え」と「個別ホスト接続失敗」を別状況として記載しているが、
両方が同時に発生した場合の扱いは明記されていなかった。本実装では1回の通知にまとめ、
接続/収集エラーが1件でもあれば `status: error` を優先し、メッセージ本文に
閾値超えホスト一覧・エラーホスト一覧の両方を含める。

### 5. fail（playbookの異常終了）の扱い

`radius_healthcheck` / `ubuntu_nightly` の既存パターン（critical/エラー系は fail、
warningはfailしない）に合わせた。

- quory NTP異常（critical） → 通知 + fail
- 個別ホスト接続失敗（error） → 通知 + fail
- 閾値超えのみ（warning） → 通知のみ、failしない

## 要求仕様からの逸脱

### `inventories/vars/sophos.yml` を作成しなかった

要求仕様は「Sophos Advanced Shell接続用のSSH鍵パスとユーザー」を新規Vaultファイルに
追記するよう指示していた。しかし `inventories/homelab/hosts.yml` の既存 `sophos` グループに
すでに `ansible_user: admin` / `ansible_ssh_private_key_file: ~/.ssh/id_rsa_sophos` が
平文（非Vault）で設定済みであることを確認した。

鍵パス・ユーザー名は秘密情報ではなく（core.md §5: 「秘密鍵そのものではなく、秘密鍵への
パス参照のみを記載する...これは秘密鍵本体ではないため Git管理してよい」）、`sophos_trim` role も
既にこの値をそのまま使っている。同じ値を別のVaultファイルに複製すると、二重管理になり
将来の鍵パス変更時に更新漏れのリスクが増える。

そのため `roles/time_sync_check/tasks/main.yml` は `sophos_trim` と同様に
`ansible_user` / `ansible_host` / `ansible_ssh_private_key_file` を直接参照する形にし、
`inventories/vars/sophos.yml` は作成しなかった。

### sophos-fw の接続変数の置き場所をレビュー前に修正（ユーザー指摘）

上記の調査中、ユーザーから「pve1等は `group_vars/<group>.yml` に `ansible_user`/鍵パスを
書いているのに、なぜ sophos だけ `hosts.yml` のホストエントリに直接書いているのか」という
指摘があった。

確認すると、proxmox / control_nodes / dev_nodes / monitoring_servers / radius_servers の
5グループは全て `group_vars/<グループ名>.yml` に `ansible_user: ann` +
`ansible_ssh_private_key_file: ~/.ssh/id_ann` を置く一貫した方式だったのに対し、
sophos グループだけ `group_vars/sophos.yml` を作らず `hosts.yml` の `sophos-fw` ホスト
エントリに直接 `ansible_user` / `ansible_ssh_private_key_file` / `ansible_connection` を
書いていた（sophos_trim 実装時 2026-06-07 からの状態で、time_sync_check起因ではない）。

認証情報そのものが他グループと異なるのは正当（Sophosはベンダー製アプライアンスで
core.md §5 の ann 管理ユーザーの対象外）だが、「どこに書くか」のスタイルが揃っていない
のは不整合と判断し、レビュー依頼前に以下の通り修正した。

- `inventories/homelab/hosts.yml`: `sophos-fw` ホストエントリから `ansible_user` /
  `ansible_ssh_private_key_file` / `ansible_connection` を削除（`ansible_host` のみ残す）
- `inventories/homelab/group_vars/sophos.yml`: 新規作成し、削除した3変数を移設

`ansible-inventory --host sophos-fw` で修正前後の解決済み変数が完全に一致することを確認済み
（後述）。`roles/time_sync_check/tasks/main.yml` 側の参照（`ansible_user` /
`ansible_host` / `ansible_ssh_private_key_file` のJinja参照）はホスト変数・グループ変数の
どちらでも同じhostvars名前空間に解決されるため、変更不要だった。

これらの判断の採否はレビューでの確認をお願いしたい（must-fixではないと考えるが、
要求仕様に明記された作成物を作っていない点、および要求仕様の範囲外で既存ファイルを
追加修正した点は明示しておく）。

## 注意点

- CloudKeyへの初回SSH接続時、`StrictHostKeyChecking=accept-new` により known_hosts に
  自動追加される。鍵が変わった場合は接続拒否される（Sophos/sophos_trimと同じ挙動）。
- `cloudkey_ssh_password` の実値はユーザーから直接提供を受けてVaultに保存した
  （要求仕様に値の記載がなかったため）。
- Sophosの `'# '` プロンプトパターンは `sophos_trim` からの流用。実機のAdvanced Shellの
  プロンプト文字列に変更があれば追従が必要（sophos_trimのimplement.mdに同種の注意点あり）。
- quory自体が完全にUNREACHABLEな場合も「NTP同期異常」と同じ扱いでcritical通知+failする
  設計にしている（基準ノードの時刻が取得できない以上、信頼できないため）。
