## 背景

実装過程(Claude Code、ansy上、Yoshinobu承認済み)で、すでに以下を実機(read-only)で
複数回確認している。

- 全7対象ホスト(pve1 / pve2 / ansy / monnie / authy / sophos-fw / cloudkey)の
  正常系が、各must-fix修正のたびに繰り返し成功していること
- quoryのPhase1判定が他chronyホストへの接続より先に完了すること(`-v`出力で確認)
- ローカルのフェイク入力による異常系(chrony offset超過・解析不能、Sophosのコマンド失敗・
  鮮度切れ・未来mtime、CloudKeyのservice inactive・行欠落)が想定どおり分類されること

本テスト計画は、実装中には未確認だった項目に絞る。read-onlyのみで、対象ホストの状態を
変える操作は行わない(safe区分。core.md §16のtester自律境界に従い、すべてtester自律実行可)。

## テスト項目

tester(Codex)はClaude Codeと同じくansy上で動作するため、quory自体から
ansible-playbookを実行することはできない(quoryでの実行確認が必要な場合は
Yoshinobuが手動でgit pull+実行する。本test_planのブロッカーにはしない)。
以下はすべてansy上から実行する。

1. **`--check --diff` での実行**
   `ansible-playbook ... --check --diff` で実行し、エラーなく完了し、対象ホストへの
   変更が一切ないこと(read-onlyであること)を確認する。

2. **連続2回実行での冪等性確認**
   同じ実行を2回連続で行い、結果(各ホストのok/collected/detail)が安定していること
   (大きく変動しないこと)を確認する。chronyのoffset値は実行ごとに多少変動するのが
   正常だが、Leap status・synced判定が毎回安定していることを確認する。

3. **ansible-lint(利用可能であれば)**
   tester の作業環境(ansy)に ansible-lint がインストールされていれば、
   `playbooks/time_sync_check.yml` と `roles/time_sync_check/` に対して実行し、
   重大な指摘がないか確認する(本実装中は未実行)。
