## 指摘事項

### [must-fix] quory の状態収集失敗を NTP 非同期の critical として通知する問題が未修正

対象:

- `roles/time_sync_check/tasks/main.yml` の `Abort and notify if quory NTP is not synchronized`
- `roles/time_sync_check/tasks/check_chrony.yml` の `time_sync_collected` / `time_sync_synced`

今回の incident では `chronyc tracking` の結果を収集できなかったため、
`time_sync_collected=false`、`time_sync_synced=false` となった。それにもかかわらず abort block は
`time_sync_synced=false` だけを条件に、固定の `critical` と「quory NTP not synchronized」を
送信した。つまり、偽 critical の直接原因は自己SSH失敗だけでなく、**収集失敗と確認済みの
NTP非同期を区別していないこと**にもある。

今回追加した動的 local connection は、Semaphore が quory 上でネイティブ実行される現環境の
接続原因には妥当である。しかし今後、local command の実行失敗、権限不足、chronyc 不在・異常、
別コントローラーからのSSH失敗が起きれば、同じ偽の NTP critical を再送する。

requirement の通知分類は以下を区別している。

- quory 自身の NTP 同期異常: `critical`
- 個別ホストへの接続失敗・time_sync_check 実行エラー: `error`

quory gate でも `time_sync_collected` を確認し、以下のように分岐すること。

- collected=false: 「quory の NTP 状態収集失敗」として error 通知 + fail
- collected=true かつ synced=false: 「quory の NTP 同期異常」として critical 通知 + fail

check mode では両通知を引き続き抑止すること。修正後は、少なくとも collected=false と
collected=true/synced=false のフェイク入力で status・title・message が分かれることを確認する
必要がある。

### [suggestion] incident 記録の「quoryを対象にする初めてのplaybook」を修正する

対象:

- `docs/ai/reviews/time_sync_check/2026-06-24_020_implement.md` の「原因」

リポジトリには既に `cert_renew.yml` と `cert_renew_quory.yml` が `hosts: quory` を使用している。
これらは play に `connection: local` を明示しているため、今回の inventory default による
自己SSH問題は起こらない。監査証跡としては「quoryを対象にする初めてのplaybook」ではなく、
「quoryを対象にし、inventory の通常SSH設定をそのまま使う初めてのplaybook」等が正確である。

## 動的 connection 差分の確認

`inventories/homelab/host_vars/quory.yml` の切替方針自体は妥当である。

- controller の `hostname -s` が `quory` なら local connection
- ansy 等の別 controller なら従来どおり SSH connection
- `lookup('pipe', ...)` は controller 側で評価され、既存 preflight と同じ判定方式
- IP リテラルや秘密情報の追加なし

ansy からの実行では従来の実SSH経由で quory 固有の chronyc 出力を取得し、全7対象が正常に
完了したことが 020 implement に記録されている。

この host_var は time_sync_check 以外にも適用されるが、現時点で quory を対象にする既存の
`cert_renew.yml` / `cert_renew_quory.yml` は play 側でも local connection を明示しており、
quory 上での実行方針と競合しない。

## 判定

**要修正。このまま commit 不可。**

動的 local/SSH 切替は incident の接続原因に対する妥当な修正だが、収集失敗を NTP 非同期の
critical と誤分類する根本的な通知問題が残る。error / critical 分岐を修正して再レビューし、
反映後は quory 上の Semaphore または手動実行で local branch を確認すること。
