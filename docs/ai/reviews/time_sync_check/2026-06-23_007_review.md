## 設計転換の評価

Yoshinobu 承認済みの「各ホストの NTP クライアントによる自己報告」への転換は妥当である。
SSH / HTTP 越しの単発時刻測定では、検出したい数百 ms と同程度の測定誤差が入り、補正の
仮定にも依存する。chrony 等が継続的に測定した同期状態を読む方が、「NTP 同期異常の早期検知」
という目的に直接対応する。

元 requirement からの大きな変更だが、調査結果、Yoshinobu の承認、実機結果、既知の制約が
006 implement に記録されており、監査証跡としては十分である。生時刻比較、対話ホスト専用
3000 ms 閾値、`%N` 依存を削除した判断にも問題はない。

ただし、全面改訂した実装には以下の must-fix が残る。

## 指摘事項

### [must-fix] quory の判定前に他の chrony ホストへ接続している

対象:

- `roles/time_sync_check/tasks/main.yml:16-46`
- `roles/time_sync_check/tasks/main.yml:48-71`

Phase 1 の収集タスクは quory だけでなく、pve1 / pve2 / ansy / monnie / authy にも同時に
`chronyc tracking` を実行する。その後で quory の異常を判定しているため、quory が未同期・
到達不能でも、通常 SSH 対象への接続は既に完了している。

これは requirement の「Phase 1 で異常ならそこで中断」と、コード中の「他ホストへの接続を
一切行わず」「他ホストへの接続は行っていません」に反する。quory 専用の収集・判定・通知・
fail を先に完了し、その後に他の chrony ホストを収集するタスクを分離すること。

### [must-fix] Sophos / CloudKey の状態取得コマンド失敗を収集成功として扱う

対象:

- `roles/time_sync_check/tasks/main.yml:82-117`
- `roles/time_sync_check/tasks/main.yml:126-158`

Sophos は `tail ...; echo __NTP_LOG_END__`、CloudKey は
`timedatectl status; echo __NTP_STATUS_END__` を実行し、末尾 marker の存在だけで
`time_sync_collected` を true にする。`tail` または `timedatectl` が失敗しても後続の echo は
成功して marker が出るため、収集成功と判定される。その場合は unsynced warning となり、
collection error の fail 制御へ入らない。

状態取得コマンド直後の rc を marker として収集し、rc=0、完了 marker、必要な出力形式の
すべてを満たす場合だけ collected=true にすること。ファイル欠落、権限不足、timedatectl
失敗は requirement の「個別ホストへの接続失敗・実行エラー」と同じ error / fail 系へ分類する
必要がある。

### [must-fix] Sophos / CloudKey の判定は古い成功状態を現在の正常状態として扱い得る

対象:

- `roles/time_sync_check/tasks/main.yml:78-117`
- `roles/time_sync_check/tasks/main.yml:126-158`
- `docs/ai/reviews/time_sync_check/2026-06-23_006_implement.md` の「未対応事項・既知の制約」

Sophos はログ末尾が `System Clock set to` なら、ログが何日前のものでも正常になる。
ntpclient の daily 実行が停止した場合、最後の成功行を無期限に読み続け、早期検知できない。
ログ本文の日付を解析できなくても、ファイルの mtime 等で「期待する日次周期 + 許容時間」内の
更新かを read-only に確認できる。成功行と鮮度の両方を正常条件にすること。

CloudKey も `System clock synchronized: yes` だけを使い、実機出力に存在する
`NTP service: active` を無視している。最後の同期後に timesyncd が停止した状態を正常扱い
しないよう、少なくとも synchronized=yes と service=active の両方を要求すること。

自己報告方式へ転換した以上、stale な成功履歴ではなく「現在も同期機構が動作している」と
判断できる信号を確認しなければ、設計転換後の目的を満たさない。

### [must-fix] chrony の offset 解析失敗が 0 ms の正常値になる

対象:

- `roles/time_sync_check/tasks/main.yml:29-46`
- `roles/time_sync_check/tasks/main.yml:173-183`

chronyc が rc=0 で `Leap status` を含めば collected=true になる一方、`System time` の数値を
regex で取得できない場合は `_offset_sec` が `0` にフォールバックする。その結果、出力形式の
変更・ローカライズ・部分出力などで offset を解析できなくても、0 ms として 500 ms 閾値を
通過する。

数値 offset の抽出成功を chrony ホストの collected 条件へ含め、解析不能時は collection
error にすること。詳細表示用の default と、判定値用の default=0 を同じ意味で扱っては
ならない。

### [must-fix] Sophos の秘密鍵を使う expect タスクから `no_log: true` が外れている

対象:

- `roles/time_sync_check/tasks/main.yml:82-105`
- `docs/ai/reviews/time_sync_check/2026-06-23_001_requirement.md` の「制約」

requirement は CloudKey パスワードと Sophos SSH 鍵を扱うタスクに `no_log: true` を要求して
いる。CloudKey には設定されているが、Sophos の expect タスクにはない。秘密鍵本体を表示する
処理ではなく、鍵パス自体も秘密情報ではないものの、明示された防御条件であり、初回実装に
あった `no_log` を全面改訂で落とす理由はない。Sophos の expect タスクへ復元すること。

### [suggestion] quory 自身にも chrony offset 閾値を適用する

対象:

- `roles/time_sync_check/tasks/main.yml:29-49`
- `roles/time_sync_check/defaults/main.yml:23-26`

quory は `Leap status` だけでゲートされ、`System time` が 500 ms を超えていても正常として
後続へ進む。他の chrony ホストには同じ offset を閾値判定しているため、基準ノードにも
閾値を適用するか、適用しない運用理由を明示すると判定方針が一貫する。

## 前回指摘の確認

- 通常 SSH の before/after 分離問題: 生時刻比較方式の撤廃により解消。
- 対話ホストの 3000 ms 閾値問題: 数値比較と専用閾値の撤廃により解消。
- embedded target の `%N` 対応断定: epoch 取得コード自体の撤廃により解消。
- CloudKey の register / delegated fact: raw register を論理ホストで参照し、正規化 fact のみ
  localhost へ保存する構成を維持している。
- radius_healthcheck の chrony 削除、Sophos 接続変数の group_vars 移設、Vault の CloudKey
  SSH 変数追加に新たな問題は確認されなかった。

**設計転換は承認。実装は要修正で、このまま commit 不可。**

Sophos の `no_log` 欠落を修正してから再レビューすること。
