## 指摘事項

### [must-fix] 通常 SSH ホストは controller 時刻と remote 時刻を別タスクで取得しており、接続遅延を時刻ずれとして判定する

対象:

- `roles/time_sync_check/tasks/main.yml:17-19`
- `roles/time_sync_check/tasks/main.yml:80-109`
- `docs/ai/reviews/time_sync_check/2026-06-23_004_implement.md` の「must-fix対応 1」

実装報告と role 冒頭コメントは、通常 SSH ホストについて controller 時刻を
「そのホストのタスク自身の変数」「コマンド発行直前」に取得し、別タスクに分割しないと
説明している。しかし実コードでは、controller 時刻を `set_fact` タスクで取得した後、次の
`shell` タスクで remote 時刻を取得している。

`strategy: linear` では全ホストが snapshot タスクを終えてから collection タスクへ進むため、
snapshot 後のバリア待ち、fork のバッチ待ち、SSH 接続・remote module 起動時間が
`time_sync_observed_offset_ms` に含まれる。この値を quory の同様の片側測定と引き算しても、
ホストごとの接続遅延差は相殺されない。

004 implement のローカル検証シナリオ C でも、時計をずらさず通常 SSH ホストの収集だけを
3秒遅らせた結果を `2983ms` の warning としている。これは前回レビューが要求した
「取得処理へ意図的な遅延を入れても、時計が同期していれば閾値超過にならない」を満たさず、
時刻ずれではなく収集遅延を検出している。

controller の before/after と remote の観測値を同じホスト固有の収集区間で取得するなど、
接続時間を時刻オフセットから分離すること。修正後の遅延テストは、remote の時計自体を
ずらしたケースと、時計をずらさず通信・コマンドだけを遅らせたケースを分け、後者を
warning にしてはならない。

### [must-fix] 対話ホスト用 3000 ms 閾値は測定誤差を隠し、要求された検知感度を満たさない

対象:

- `roles/time_sync_check/defaults/main.yml:20-31`
- `roles/time_sync_check/tasks/main.yml:116-158`
- `roles/time_sync_check/tasks/main.yml:172-210`
- `roles/time_sync_check/tasks/main.yml:224-234`

Sophos / CloudKey の `sh -c` ラップ自体は、controller の before/after と対話セッションを
1タスクの標準出力へ収める方法として成立しており、CloudKey の register 保存先修正とも
競合しない。しかし remote 時刻の取得点より前にメニュー遷移が集中し、取得後の exit は短い
ため、区間の中点を remote 時刻と対応付ける推定には既知の非対称バイアスがある。

実装はこのバイアスを除去せず、要求仕様の「既定: 数百ms」に対して対話ホストだけ 3000 ms
へ緩和している。ローカル検証でも、時計が同期したまま3秒の処理遅延を追加した値
約1954 ms を「閾値内」としており、同程度の実際の時計ずれも見逃す。セッション遅延が
変動すれば 3000 ms を超える偽 warning も残るため、この閾値は測定誤差と時計ずれを
区別できない問題を解決していない。

単一の数百 ms 閾値を満たせる観測方式へ修正すること。3000 ms の低感度を運用要件として
採用するなら、実装上の都合だけで追加せず、検出できなくなるずれ幅を明示して Yoshinobu の
承認を得たうえで requirement を改訂する必要がある。現 requirement のままでは commit
可能な変更とは判断しない。

### [suggestion] `%N` が embedded target でも使えるという断定を実機確認前に置かない

対象:

- `roles/time_sync_check/defaults/main.yml:39-46`
- `docs/ai/reviews/time_sync_check/2026-06-23_004_implement.md` の「追加の不具合 a」

`date +%s%3N` をやめた修正は ansy の uutils で確認され、13桁の epoch milliseconds を
返すことも再確認できた。一方、`%N` 自体は POSIX の指定ではなく、BusyBox 等では build や
実装によって未対応の場合がある。「everywhere」「restricted shell でも supported」とは
実機確認前には断定できない。

現在の数値形式チェックにより未対応時は収集エラーへ倒れるため、安全側ではある。コメントを
実証済みの範囲へ修正し、Sophos / CloudKey の実機テストで `%N` とパラメータ展開の出力を
確認することを推奨する。

## 前回指摘の確認

### [must-fix] CloudKey の register 結果は `hostvars['localhost']` に保存されない

**解消済み。**

`time_sync_cloudkey_raw` を run_once の論理ホスト文脈で直接参照し、正規化した
`time_sync_observed_offset_ms` / `time_sync_observed_ok` のみを
`delegate_facts: true` 付き `set_fact` で localhost へ保存している。

### [must-fix] quory NTP 異常時の中断が収集後になる

**解消済み。**

critical 通知・fail block は NTP 判定直後へ移動し、全収集タスクにも NTP 正常時だけ実行する
`when` ガードが追加された。quory が信頼できない場合に Sophos / CloudKey を含む後続接続へ
進まない構造になっている。

### [suggestion] 通常 SSH ホストの epoch 数値検証

**解消済み。**

rc、unreachable に加え、stdout が数字だけであることを検証している。ansy の
`date +%s%3N` 非互換も新しい epoch 生成式で回避され、ローカル確認では13桁の値を返した。

### [suggestion] `.codex/hooks.json` の commit 除外

**対応方針を確認。**

time_sync_check の成果物ではないため個別 add から除外する方針は妥当である。現在も未追跡の
ため、一括 add では混入する点には引き続き注意が必要である。

## その他の確認

- 秒とミリ秒のユニット不一致は、before/after の中点を1000倍してから remote epoch
  milliseconds と引く形に修正されている。
- Sophos 接続変数の group_vars への移設、radius_healthcheck の chrony 削除、Vault の
  CloudKey SSH 変数追加に新たな問題は確認されなかった。

  が無ければ collected=false にするため、失敗を成功扱いする構造にはなっていない。

## 判定

**要修正。このまま commit 不可。**

CloudKey の fact 受け渡し、フェイルファスト、数値検証、単位不一致は修正済みである。
ただし時刻差の中に収集遅延が残る根本問題は通常 SSH ホストで未解消であり、対話ホストでは
3000 ms の別閾値によって隠されている。観測方式を修正するか、検知感度の変更を requirement
と人間判断へ戻したうえで再レビューすること。
