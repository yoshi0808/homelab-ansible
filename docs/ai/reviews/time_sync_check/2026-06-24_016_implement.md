# 追加実装内容: time_sync_check(check modeでの誤通知事故の修正)

対象テスト結果: `docs/ai/reviews/time_sync_check/2026-06-24_015_test_result.md`

## 概要(重要・事故報告)

tester(Codex)による `--check --diff` テストで、**dry-runにもかかわらず実際に
#alertsチャンネルへ緊急(critical)Slack通知が送信される事故**が発生した。

### 原因

`ansible.builtin.command`(`chronyc tracking`)は、Ansibleの既定動作により
`--check`モード下では実行されずスキップされる(モジュールがcheck mode対応を
宣言していないため)。その結果:

1. `Check chrony tracking` タスクが `skipping: [quory]` となり、
   `time_sync_chrony_raw` が未registerのまま。
2. 後続の `Judge chrony sync status`(`set_fact`は常に実行される)が、
   未収集のデフォルト値から `time_sync_synced=false` と判定。
3. 「quoryのNTP異常」を示す `Abort and notify` ブロックが発火し、
   実際にSlack通知タスク(`community.general.slack`)が実行された
   (community.general.slack もcheck mode非対応のため、通知が抑止されず本当に送信された)。
4. `Fail because reference node time is not trustworthy` でplayが中断され、
   他ホストの収集は行われなかった。

quoryの実際のNTP状態は正常だったため、これは完全な偽陽性であり、
かつdry-run中に外部システム(Slack)へ実際の副作用が発生した。

### 修正

1. **診断系タスクへの `check_mode: false` 追加**
   - `roles/time_sync_check/tasks/check_chrony.yml` の `Check chrony tracking`
   - `roles/time_sync_check/tasks/main.yml` の Sophos / CloudKey の expect タスク

   いずれも対象ホストの状態を変更しない読み取り専用コマンドであるため、
   `--check`モード下でも常に実際に実行されるようにした。これにより、
   check modeでも正しい診断結果が収集され、誤判定そのものが起きなくなる。

2. **Slack通知タスクへの `when: not ansible_check_mode` 追加(多層防御)**
   - quory異常時のcritical通知
   - warning/error集約通知

   診断結果が正しくても、dry-run実行で外部システムへの副作用(通知送信)が
   発生しないよう、防御的にcheck mode下では通知自体を送らない。
