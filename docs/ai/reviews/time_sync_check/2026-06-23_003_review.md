## 指摘事項

### [must-fix] 取得時刻の差ではなく、取得処理の経過時間を時刻差として判定してしまう

対象:

- `roles/time_sync_check/tasks/main.yml:29-38`
- `roles/time_sync_check/tasks/main.yml:55-77`
- `roles/time_sync_check/tasks/main.yml:93-113`
- `roles/time_sync_check/tasks/main.yml:165-174`
- `roles/time_sync_check/defaults/main.yml:20-21`

quory の epoch は通常 SSH ホスト用タスクで一度だけ取得する一方、Sophos の epoch は
その後の対話メニュー遷移後、CloudKey の epoch はさらに次の SSH セッション確立後に取得する。
最後の差分計算では、この取得時刻の違いを補正せず、各 epoch から最初に取得した quory の
epoch を直接引いている。

したがって、全ホストの時計が完全に同期していても、Sophos / CloudKey には quory 取得後の
実経過時間がそのまま `diff_ms` として加算される。Advanced Shell のメニュー遷移と別の
パスワード SSH 接続が既定閾値 500 ms 以内に完了する前提は成立しないため、正常時にも
恒常的な warning になる。通常 SSH ホストも Ansible の fork 数や接続時間によって同じ影響を
受ける。

各対象の remote epoch と比較する基準時刻を同じ観測区間で取得し、通信・処理時間を考慮した
比較に変更すること。少なくとも、数秒前に取得した単一の quory epoch と後続の対話処理後の
epoch を直接比較してはならない。修正後は、取得処理へ意図的な遅延を入れても、時計が同期
していれば閾値超過にならないテストが必要である。

### [must-fix] CloudKey の register 結果は `hostvars['localhost']` に保存されない

対象:

- `roles/time_sync_check/tasks/main.yml:93-123`
- `docs/ai/reviews/time_sync_check/2026-06-23_002_implement.md:53-62`

`delegate_facts: true` は `set_fact` 等が作る fact の保存先を delegate 先へ変更するが、
`register` 変数の保存先は変更しない。CloudKey 取得タスクの
`time_sync_cloudkey_raw` は play の論理ホスト側へ登録され、
`hostvars['localhost'].time_sync_cloudkey_raw` には作成されない。

同じ `run_once + delegate_to: localhost + delegate_facts: true + register` を使ったローカル
検証でも、register 値は play 対象ホストに存在し、`hostvars['localhost']` では `missing` だった。
このため現在の normalize は CloudKey の stdout を取得できず、CloudKey を常に収集エラーに
するか、未定義変数で失敗する。implement の「結果を `hostvars['localhost']` に保存した」も
実際の Ansible 挙動と一致しない。

register 値を実際の保存先から参照し、その後に必要な正規化済み fact を明示的に localhost
へ保存する構成へ修正すること。修正後は、取得タスクから normalize、集計までを省略せずに
通すテストが必要である。事前に作った fake hostvars へ値を直接投入するテストだけでは、今回の
保存先不整合を検出できない。

### [must-fix] quory の NTP 異常を判定しても、全対象の収集後まで中断しない

対象:

- `roles/time_sync_check/tasks/main.yml:9-24`
- `roles/time_sync_check/tasks/main.yml:29-123`
- `roles/time_sync_check/tasks/main.yml:126-151`

要求仕様は Phase 1 で quory の同期状態を確認し、異常なら「そこで中断する」と定めている。
しかし abort block は通常 SSH、Sophos、CloudKey の全収集タスクより後にあるため、quory が
未同期または到達不能でも、すべての対象へ接続して時刻取得を試みた後で初めて通知・failする。

基準が信頼できない場合に不要な接続を行うだけでなく、CloudKey のパスワード認証や Sophos の
対話セッションまで開始するため、要求されたフェイルファストになっていない。Phase 1 の判定、
critical 通知、fail を Phase 2 より前に完了させ、異常時に後続の収集タスクが実行されない
ことを確認すること。

### [suggestion] 通常 SSH ホストの epoch を数値として検証していない

対象:

- `roles/time_sync_check/tasks/main.yml:40-49`
- `roles/time_sync_check/tasks/main.yml:170-173`

通常 SSH ホストは stdout が非空で rc=0なら `time_sync_observed_ok: true` になる。
後段の `int` 変換は非数値を 0 として扱うため、想定外出力を収集エラーではなく巨大な時刻差の
warning に誤分類し得る。Sophos / CloudKey と同様に、epoch milliseconds の形式を検証して
から collected 扱いにすると、要求仕様の「個別ホストへの接続失敗・収集エラー」と整合する。
