# 追加実装内容: time_sync_check(レビュー指摘への対応)

対象レビュー: `docs/ai/reviews/time_sync_check/2026-06-23_003_review.md`

## 対応概要

reviewer(Codex)から3件のmust-fixと2件のsuggestionを受け、すべて反映した。
must-fix対応の過程で、ローカル検証によりレビュー指摘以外の追加バグ(ユニット不一致、
`date +%s%3N`の非互換性)も発見し、合わせて修正した。

## must-fix対応

### 1. 取得時刻の差ではなく取得処理の経過時間が時刻差に混入する

`roles/time_sync_check/tasks/main.yml` を再設計した。

- **通常SSHホスト(quory含む)**: 各ホストの「コマンド発行直前のコントローラー時刻」を
  `lookup('pipe', time_sync_check_epoch_ms_cmd)` でそのホストのタスク自身の変数として
  取得し(別タスクに分割しない)、`対象epoch - コントローラー時刻` を「オフセット」とする。
  quoryとの差分は「対象オフセット - quoryオフセット」で計算する。
- **Sophos / CloudKey**: 対話セッション全体を `sh -c 'echo __TS_BEFORE__:...; ssh ...; echo
  __TS_AFTER__:...'` で1つの `expect` タスクとしてラップし、セッション開始直前・終了直後の
  コントローラー時刻を同じ標準出力にマーカー出力する。前後を別タスクに分割しない。

別タスクに分割しなかった理由: Ansible の `strategy: linear`(既定)は、同一play内の
全ホストが同じタスク番号を完了するまで次のタスク番号へ進まない。前後のコントローラー
時刻取得を別タスクにすると、**同じタスク番号を共有する他ホストの処理時間が、無関係な
ホストのオフセット計算に混入する**(後述のローカル検証で実際に再現・確認した)。
1タスク内に前後マーカーを収めることで、このクロスホスト汚染を構造的に排除した。

### 2. CloudKeyのregister結果が `hostvars['localhost']` に保存されない

reviewer指摘の通り、`delegate_facts: true` は `set_fact` 等が生成するfactの保存先のみを
変更し、`expect`/`command`の`register`結果の保存先は変更しない。修正後は、`register`結果
(`time_sync_cloudkey_raw`)を**素の変数名でそのまま参照**し(`run_once`が選んだ代表ホスト側に
乗っているため、後続タスクでも同じ代表ホストの文脈でアクセスできる)、正規化済みfact
(`time_sync_observed_offset_ms`/`time_sync_observed_ok`)だけを `delegate_facts: true` 付きの
`set_fact` で明示的に `localhost` へ書き込む形に修正した。

### 3. quory NTP異常時、全対象の収集後まで中断しない

quoryのNTP判定後、即座に「中断・通知・fail」ブロックを配置した(以前は全収集タスクの
後ろにあった)。さらに、後続のすべての収集タスク(通常SSH / Sophos / CloudKey)の `when`
に `hostvars[time_sync_check_reference_host].time_sync_quory_ntp_ok | default(false)` を
追加条件として付けた。`strategy: linear` では全ホストが同じタスク番号を完了する必要が
あるため、`ansible.builtin.fail` 単体では他ホストの実行を止められない
(failするのは`run_once`の代表ホストだけ)。`when`ガードにより、NTP異常時は
全対象ホストへの接続タスク自体がスキップされ、実接続は一切発生しない。

## ローカル検証で発見した追加の不具合(レビュー指摘外)

3件のmust-fix対応中に、本番roleと同じ変数名・同じwhen式パターンをフェイクホスト
(`add_host` + `ansible_connection: local`)上に再現したローカル検証を実施し、
レビューでは指摘されていなかった2件のバグを追加で発見・修正した。

### a. `date +%s%3N` がansy上で正しく動作しない

`date +%s%3N` の `%3N`(ナノ秒を3桁に切り詰める)はGNU date拡張であり、ansyの
`date`(Rust製 uutils-coreutils 0.8.0)では正しく動作せず、無関係な値を返すことを
確認した。**ansyは比較対象ホストそのもの**(要求仕様の比較対象に含まれる)なので、
これは検証専用の問題ではなく本番でも発生する。

`%N`(桁数指定なし。両実装で正しく動作)とPOSIX shのパラメータ展開(`%`/`#`)のみで
構成する形に変更した(`roles/time_sync_check/defaults/main.yml` の
`time_sync_check_epoch_ms_cmd` / `time_sync_check_epoch_ms_inline`)。`cut`/`awk`等の
外部コマンドにも依存しないため、Sophos Advanced Shell / CloudKey UniFi OSの制限された
シェルでも動作するはずである(実機未検証。後述)。

### b. 秒とミリ秒のユニット不一致

before/afterマーカー(`date +%s.%N`、秒単位+小数)と取得epoch(ミリ秒単位整数)を
そのまま減算しており、10万倍以上ずれた`diff_ms`を生成するバグがあった。
before/afterの中間値を1000倍してミリ秒に変換してから減算するよう修正した。

## 追加で行った設計変更: Sophos/CloudKey専用の閾値

before/after補正を導入した後も、ローカル検証(フェイクの対話メニュー、0.5秒x3回の
sleepを模擬)で、**遅延を一切追加しないベースラインでも約530msの残差バイアス**が
残ることを確認した。原因は、対話セッション内の遅延がdateコマンド実行ポイントの
前後に均等に分布しないため(メニュー遷移は実行ポイントの前に集中し、exit操作は
後に少ないため、before/afterの中間値による補正では完全に相殺できない)。

通常SSHホスト(quory含む)用の閾値(既定500ms)とは別に、対話セッション特有の
オーバーヘッドを許容するための専用閾値
`time_sync_check_threshold_ms_interactive`(既定3000ms。sophos-fw/cloudkeyにのみ適用)
を追加した。これは要求仕様(単一の閾値を想定)からの拡張であり、トリアージ対象として
明示する。

採否の論点:

- 単一閾値のままだと、Sophos/CloudKeyは時刻が完全に同期していても恒常的にwarningに
  なる可能性が高い(ローカル検証で実証済み)。専用閾値はこれを避けるための実装上の
  必然と考える。
- 一方、3000msという具体的な値はフェイクの対話メニュー(0.5秒x3のsleepを模擬した
  もの)による推定であり、実機(本物のSophos Advanced Shell / CloudKey UniFi OS)の
  挙動とは異なる可能性がある。test_planで実機検証し、必要なら
  `defaults/main.yml` の値を調整することを推奨する。

## suggestion対応

### 通常SSHホストのepoch数値検証

`time_sync_observed_ok` の判定に `(stdout | trim) is match('^[0-9]+$')` を追加し、
非数値の想定外出力を「収集成功(巨大なwarning)」ではなく「収集エラー」に正しく
分類するようにした。

### `.codex/hooks.json` のcommit除外

`.codex/hooks.json` は本セッション開始前から存在する未追跡ファイルで、
time_sync_checkの成果物ではない。commit時は `git add` で対象ファイルを個別指定し、
このファイルを含めないようにする(コード変更はしていない)。`.gitignore`への追加は
time_sync_checkのスコープ外と判断し見送ったが、要望があれば別タスクとして対応する。
