# 追加実装内容: time_sync_check(自己報告方式へのmust-fix対応)

対象レビュー: `docs/ai/reviews/time_sync_check/2026-06-23_007_review.md`

## 対応概要

設計転換(自己報告方式)自体は承認されたが、全面改訂した実装に5件のmust-fixと
1件のsuggestionが指摘された。すべて反映した。

## must-fix対応

### 1. quory判定前に他のchronyホストへ接続していた

quoryと他5ホスト(pve1/pve2/ansy/monnie/authy)のchronyc tracking収集を
同一タスク(共通when条件)にまとめていたため、quoryの異常判定より先に他ホストへ
接続してしまっていた。

`roles/time_sync_check/tasks/check_chrony.yml` を新規作成し、収集・判定ロジックを
共有タスクファイル化した。Phase 1(quoryのみ、`include_tasks`)→ Abort判定 →
Phase 2(他5ホスト、`hostvars[reference].time_sync_synced`をwhereガードに追加して
`include_tasks`)の順に明確に分離した。

実機検証(`-v`出力)で、quoryの"Check chrony tracking"タスクが単独で完了し、
その後にのみ他5ホストの同タスクが開始されることを確認した。

### 2. Sophos/CloudKeyの状態取得コマンド失敗を収集成功として扱っていた

両ホストとも、診断コマンド(`tail`/`timedatectl`)の直後に `echo __XXX_RC__:$?` を
追加し、実際の終了コードをマーカーとして収集するようにした。`time_sync_collected`は
このrcマーカーが取得でき、かつrc=0であることを必須条件にした。

ローカル検証(フェイクのstdout、コマンド失敗時のrc=1/127を含む3パターン)で、
コマンド失敗時に`collected=False`(誤ってwarning側ではなくerror側へ正しく分類)
されることを確認した。

### 3. Sophos/CloudKeyの判定が古い成功状態を現在の正常状態として扱い得た

- **Sophos**: `stat -c %Y` でログファイルの mtime を、`date +%s` で現在時刻を
  (両方Sophos自身で)取得し、両者の差が `time_sync_check_sophos_max_age_hours`
  (既定30時間。ntpclientの日次実行サイクルに対する許容バッファ)を超えていれば
  `time_sync_synced=false` とする。「ログ末尾が成功」と「鮮度内」の両方を必須条件にした。
- **CloudKey**: `System clock synchronized: yes` に加え、`NTP service: active` も
  必須条件に追加した(timesyncdサービス自体が停止していても`synchronized: yes`の
  古い状態が残るケースを排除)。

ローカル検証で、鮮度切れ(stale)のケースが正しく`synced=False`に、NTPサービス
停止のケースも正しく`synced=False`になることを確認した。

### 4. chrony offset解析失敗が0msの正常値になっていた

`System time`行の正規表現マッチ自体の成否(`_offset_found`)を `time_sync_collected`
の条件に追加した。解析失敗時はoffsetを0msにフォールバックさせず、収集エラーとして
扱うようにした(`check_chrony.yml`)。

ローカル検証で、`System time`行が無い(想定外フォーマットの)出力に対して
`collected=False`になることを確認した(以前は`offset_ms=0`で閾値判定を素通りしていた)。

### 5. Sophosのexpectタスクからno_logが外れていた

`roles/time_sync_check/tasks/main.yml` のSophos用expectタスクおよびその判定タスクに
`no_log: true` を復元した(要求仕様の「秘密鍵・パスワードを扱うタスクにはno_log」を
再度満たす)。

## suggestion対応

### quory自身にもchrony offset閾値を適用する

`check_chrony.yml` を共有化した結果、quoryも他のchronyホストと完全に同じ判定ロジック
(Leap status + offset閾値判定)を通るようになった。元々の値はミリ秒未満で問題なく
通過することを実機で確認済み。判定方針が全chronyホストで一貫した。
