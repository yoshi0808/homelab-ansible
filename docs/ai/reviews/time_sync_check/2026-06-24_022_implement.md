# 追加実装内容: time_sync_check(quoryゲートのerror/critical誤分類修正)

対象レビュー: `docs/ai/reviews/time_sync_check/2026-06-24_021_review.md`

## 対応概要

正当な指摘。020での修正(quoryの自己SSH接続方式)は今回のincidentの**接続原因**には
対応していたが、より本質的な問題(quoryの収集失敗をNTP同期異常のcriticalとして
誤通知する分類ロジック自体)が残っていた。これを修正した。

## must-fix対応

`roles/time_sync_check/tasks/main.yml` のquory用abortブロックを、`time_sync_synced`
のみを条件にしていたものから、`time_sync_collected`を先に確認する2段階構成に分割した。

- `time_sync_collected=false`(収集失敗: 接続失敗・コマンド失敗・出力解析失敗等)
  → 「quory NTP状態を収集できなかった」として **error** 通知 + fail
- `time_sync_collected=true` かつ `time_sync_synced=false`(収集はできたが、実際に
  NTP非同期)→ 「quory NTP同期異常」として従来通り **critical** 通知 + fail

両方ともcheck mode下では通知を抑止する(`not ansible_check_mode`を維持)。

これにより、自己SSH失敗以外の将来の収集障害(ローカルコマンド実行失敗、権限不足、
chronyc不在、出力形式変化等)が起きても、誤って「NTP同期異常」と通知されることは
なくなる。

## ローカル検証

フェイク入力で3パターンを検証した。

| パターン | collected | synced | 結果 |
|---|---|---|---|
| A: 収集失敗 | false | false | error通知が発火、critical通知は発火しない |
| B: 収集成功・実際に非同期 | true | false | critical通知が発火、error通知は発火しない |
| C: 健全 | true | true | どちらも発火しない |

## suggestion対応

implement(020)の「quoryを対象にする初めてのplaybook」という記述を訂正する。
`cert_renew.yml`/`cert_renew_quory.yml`は既にquoryを対象にしているが、play側で
`connection: local`を明示しているため自己SSH問題が起きていなかった。正確には
「quoryを対象にし、inventoryの通常SSH設定をそのまま使う初めてのplaybook」である。
