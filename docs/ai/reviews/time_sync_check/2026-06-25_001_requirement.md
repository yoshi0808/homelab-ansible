# 要求仕様: time_sync_check（NTPベース時刻同期チェック）

## 目的
quoryを基準として、homelab内の主要ホストの時刻同期状態を定期的に確認する。
SSHで各ホストの`date`を直接比較する方式ではなく、各ホストのNTPクライアントが
報告する同期状態（offset値）を優先して使う。NTPプロトコル自体がネットワーク
遅延を補正する仕組みを持つため、SSH越しの値取得よりも正確に時刻差を測定できる。

## 設計上の前提（重要）
- quoryは外部NTP（Ubuntu pool）に同期し、自宅内の基準点として動作する
  （quory自体にhomelab管理ネットワーク向けのallow設定済み・稼働確認済み）
- 各ノードはquoryを参照先として`chrony.conf`等に追加する
- quory自身が外部NTPからズレている場合、比較全体が無意味になるため、
  Phase 1でquory自身の同期健全性を確認し、異常ならゲートで中断する
- ansyをNTPサーバー化する設計は検討したが、複雑性が増すため採用しない
  （ansyは他ノードと同様、quoryを参照するクライアントの一つとして扱う）

## 対象と取得方式（実機調査済み・確定情報）

| 対象 | NTPクライアント | offset取得方法 |
|---|---|---|
| pve1, pve2, ansy, monnie, authy | chrony | SSH + `chronyc tracking`（`System time`） |
| cloudkey | systemd-timesyncd | SSH + `timedatectl timesync-status`（`Offset`） |
| sophos-fw | 独自ntpclient（BusyBox環境、offset非公開） | SSH → Advanced Shell経由 + `date +%s%3N`（直接時刻比較。自己報告方式は使わない） |

- 基準ノード: quory（chrony、`chronyc tracking`）
- cloudkeyはchronyではなくsystemd-timesyncdである点に注意。`chronyc`コマンドは使えない
- sophos-fwはchrony/ntpd/systemd-timesyncdのいずれも存在せず、独自実装の
  `ntpclient`（`/bin/ntpclient`）が動作している。ログ（`/var/tslog/ntpclient.log`）には
  「System Clock set to」という絶対時刻の記録のみで、offset値は記録されないため、
  ログ解析ではなく`date +%s%3N`によるSSH越し直接比較で代替する
  （この方式はSSH接続の往復遅延が誤差として混入することを実装報告に明記すること）

また、「実行元・対象による除外ルール」内の「ansy実行時はPhase 1をスキップする」という

`chronyc tracking`を取得できるため、**Phase 1は実行元に関わらず常に実行する**。

## 実行元・対象による除外ルール
- 実行元ホスト自身は、比較対象から除外する
  （quoryで実行時はquoryを対象に含めない、ansyで実行時はansyを対象に含めない）
- quory自体は「比較対象」としてではなく「Phase 1の自己健全性確認」として扱う。
  Phase 1（quoryの自己診断）は実行元（quory/ansy）に関わらず常に実行する
  （2026-06-25訂正。旧記述「ansy実行時はPhase 1をスキップする」は撤回）
