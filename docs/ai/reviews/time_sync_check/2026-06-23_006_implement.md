# 追加実装内容: time_sync_check(アーキテクチャ転換 - NTP自己報告方式へ)

対象レビュー: `docs/ai/reviews/time_sync_check/2026-06-23_005_review.md`

## 概要

005_review の2件のmust-fixのうち、1件(通常SSHホストのbefore/afterタスク分割)は
コード修正で対応した。もう1件(対話ホスト用3000ms閾値)について、Yoshinobuに
「対話セッションを避ける他の取得方法がないか調べる」よう指示を受け調査したところ、
**当初想定していた「quoryを基準に各ホストの生時刻を比較する」方式そのものに
構造的な精度限界がある**ことが判明した。Yoshinobuと相談し、**「各ホスト自身の
NTPクライアントに同期状態を自己報告させる」方式へアーキテクチャを転換**した
(Yoshinobu承認済み)。

これは要求仕様(2026-06-23_001_requirement.md)の「quoryを基準として...時刻が
大きくズレていないかを確認する」という前提からの大きな逸脱である。経緯と判断根拠を
本ファイルに記録する。

## 調査で判明したこと(実機検証)

### 1. クロスホスト生時刻比較方式の精度限界

005_reviewのmust-fix#1(通常SSHホストのbefore/after分離)はコードで修正したが、
実機(quory)で測定したところ、**完璧に同期した時計(NTPオフセット0.2ms)に対しても、
SSH接続1回の往復で約755ms、before/after補正後でも約227msの残差バイアス**が
測定された(SSHのTCP+ハンドシェイク+認証コストが計測区間に非対称に乗るため)。

### 2. HTTP Dateヘッダー方式の精度限界

Sophos/CloudKeyの対話SSHセッションを避けるため、認証不要のHTTPS HEADリクエストで
`Date`レスポンスヘッダーを取得する方式を検証した(両ホストとも到達性・ヘッダー
取得を実機で確認)。しかしHTTPの`Date`ヘッダーは秒単位(小数秒なし)であり、
実測で約1秒の見かけ上のズレが生じることを確認した。

### 3. 各ホストが自分自身のNTP状態を自己報告できることが判明

Yoshinobuの提案により、Sophosで`ntpq`/`chronyc`/`timedatectl`等の有無を実機調査した
ところ、いずれも存在しなかったが、代わりに **`/bin/ntpclient`** と
**`/var/tslog/ntpclient.log`** が存在することを発見した。ログには実際に
**DNS解決失敗によるNTP同期エラー**が記録されていた(起動直後の一時的なもので、
その後正常に同期していた)。

さらに調査を広げ、以下を実機で確認した。

| ホスト | 自己診断手段 | 実機確認結果 |
|---|---|---|
| quory / pve1 / pve2 / ansy / monnie / authy | `chronyc tracking` | 全ホストで稼働。`Leap status: Normal`、オフセットはミリ秒未満 |
| CloudKey | `timedatectl status`(systemd-timesyncd) | `System clock synchronized: yes`、`NTP service: active` |
| Sophos | `/var/tslog/ntpclient.log`(`ntpclient`) | 日次で同期。末尾行が`System Clock set to`なら直近の同期は成功 |

各ホスト自身のNTPクライアントは、長期間の統計処理によりミリ秒未満の精度で
同期状態を把握している。これに対し、こちらから一発で時刻を取りに行く方式は、
接続オーバーヘッドそのものが検出したい時刻ズレと同程度になってしまう。
「NTP同期の異常を早期に検知する」という目的そのものには、各ホストの自己報告を
読む方が、クロスホスト比較より直接的かつ高精度である。

## 新しいアーキテクチャ

```text
Phase 1  quory自身のchronyc trackingを確認する。Leap status異常なら中断
         (基準ノードの健全性ゲートとしての位置づけは要求仕様どおり維持)
Phase 2  pve1/pve2/ansy/monnie/authyのchronyc trackingを確認する
         (Leap status + chrony自身が測定したSystem timeオフセットを閾値判定)
Phase 3  Sophosの/var/tslog/ntpclient.log末尾を確認する
         (最終行が "System Clock set to" なら正常)
Phase 4  CloudKeyのtimedatectl statusを確認する
         ("System clock synchronized: yes" なら正常)
Phase 5  結果を集約し、Slack通知する
```

quoryとの「生時刻比較」「オフセット補正」「対話ホスト専用閾値」は撤廃した。
通常SSHホスト(chronyベース)のみ、chrony自身の測定値に対する数値閾値判定
(既定500ms。要求仕様の「数百ms」をそのまま使える、自己測定値のため十分な精度がある)
を残した。Sophos/CloudKeyは真偽値の自己報告のみで判定する(数値閾値は使わない)。

## 未対応事項・既知の制約

- **Sophosの鮮度チェック未実装**: 現状は「ログ末尾が成功で終わっているか」のみ判定し、
  「最後の成功がいつか(daily cronが長期間停止していないか)」は確認していない。
  ログの日付フォーマットに年が含まれず、Sophosのシェルでの日時パース可否も未検証
  のため、初回実装では見送った。将来的な拡張候補。
- **CloudKeyのNTPサーバ到達性は未確認**: `timedatectl status`の`NTP service: active`
  までは確認したが、実際のサーバとの直近の同期成功時刻までは見ていない
  (`System clock synchronized: yes` のみで判定)。
- 上記2点は「日次/定期的にNTPデーモンが完全に停止して、古い成功状態のまま
  固定されている」ケースを見逃す可能性がある。test_planでの実運用確認、または
  将来のイテレーションでの改善を推奨する。
