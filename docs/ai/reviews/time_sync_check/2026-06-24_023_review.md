## 前回指摘の確認

### quory の収集失敗を NTP 非同期の critical として通知する

**解消済み。**

quory gate は以下の相互排他的な2段階へ分離されている。

- `time_sync_collected=false`: error 通知後に fail
- `time_sync_collected=true` かつ `time_sync_synced=false`: critical 通知後に fail

収集失敗を「NTP not synchronized」と通知する経路はなくなった。両通知には
`not ansible_check_mode` があり、dry-run の外部副作用抑止も維持している。健全時は両 block を
スキップし、Phase 2 以降へ進む。

022 implement には収集失敗、収集成功・非同期、健全の3パターンについて、error / critical の
分岐が期待どおりであるローカル検証結果が記録されている。

### incident 記録の事実関係

**訂正済み。**

022 implement で、既存の `cert_renew.yml` / `cert_renew_quory.yml` は quory を対象にするが
play 側で local connection を明示していること、今回が inventory の通常SSH設定をそのまま使う
初めてのケースであることを記録している。

## 動的 connection の確認

`inventories/homelab/host_vars/quory.yml` は controller の `hostname -s` に応じて以下へ解決する。

- quory: `local`
- ansy 等の別 controller: `ssh`

reviewer が ansy 上で `ansible ... quory -m debug -a var=ansible_connection` を実行し、`ssh` に
解決されることを独立確認した。020 / 022 implement の ansy 実機実行でも、quory を含む全7対象が
正常終了している。

既存の quory 対象 playbook は local connection を明示しており、host_var の方針と競合しない。
IP リテラル、秘密情報、対象ホストへの変更操作は追加されていない。

## Verification

- `ansible-playbook -i inventories/homelab/hosts.yml playbooks/time_sync_check.yml --syntax-check`: PASS
- ansy controller で quory の `ansible_connection=ssh` を確認
- 022 implement に記録された3分岐のローカル検証結果を確認
- 022 implement に記録された ansy からの全7対象 read-only 正常実行を確認

## 判定

**承認。修正内容は commit してよい。**

自己SSHを避ける動的 connection と、収集失敗を error、実際の NTP 非同期を critical とする
通知分類の両方が整合している。

quory / Semaphore での local branch は未確認のため、反映後に手動実行または次回スケジュールで
`ansible_connection=local` により quory の chronyc が収集されることを確認し、その結果を
監査証跡へ追記すること。確認完了までは本番 incident の解消を確定扱いにしない。
