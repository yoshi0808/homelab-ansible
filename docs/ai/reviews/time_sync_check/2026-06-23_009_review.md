## 指摘事項

### [must-fix] Sophos の未来 mtime を fresh と判定する

対象:

- `roles/time_sync_check/tasks/main.yml` の `Judge ntpclient sync status (Sophos)`

`_fresh` は `_age_sec <= max_age` だけを確認しており、`_age_sec` が負数でも true になる。
Sophos の現在時刻が最後の成功ログの mtime より前へ戻った場合、例えば時刻異常や RTC 復帰で
clock が巻き戻った場合に、未来の mtime を fresh として正常判定する。この playbook が検出
すべき時刻異常そのもので発生し得る false healthy である。

鮮度条件を `0 <= age_sec <= max_age` とし、mtime が現在時刻より未来の場合も synced=false
へ倒すこと。修正後は、正常、上限超過、未来 mtime の3パターンを分けて検証すること。

### [must-fix] CloudKey の必須ステータス行が無くても collected=true になる

対象:

- `roles/time_sync_check/tasks/main.yml` の `Judge NTP sync status (CloudKey)`

`time_sync_collected` は `timedatectl` の rc marker が 0 であることだけを条件にしている。
`System clock synchronized:` または `NTP service:` の行が欠落・変更・解析不能でも
collected=true となり、collection error ではなく unsynced warning に分類される。

007 review は、rc=0 に加えて必要な出力形式を取得できた場合だけ collected=true とするよう
求めていたため、この指摘は一部未解消である。両ステータス行を解析できたことを collected
条件へ含めること。値が `no` / `inactive` 等なら collected=true・synced=false、行自体を
解析できなければ collected=false と区別する必要がある。修正後は rc=0 かつ片方または両方の
行が欠落するケースも検証すること。

`System clock synchronized: yes` と `NTP service: active` の両方を正常条件にしている。

**解消済み。**

### Sophos の `no_log` 欠落

**解消済み。**
