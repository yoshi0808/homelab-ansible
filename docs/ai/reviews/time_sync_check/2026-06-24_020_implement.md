# 追加実装内容: time_sync_check(本番incident対応 - quoryの自己SSH失敗)

対象: commit済み(`a5216f4`)後、本番Semaphore(quory上でネイティブ実行)での実行で発生したincident

## 概要(本番incident)

commit後、Semaphoreで本playbook(テンプレート「SAFE: Time sync check」)が実行され、
quoryが自分自身(`quory.internal`)へ`ann`鍵でSSHしようとして
`Permission denied (publickey)`で失敗した。

```text
TASK [time_sync_check : Check chrony tracking]
fatal: [quory]: UNREACHABLE! => ann@quory.internal: Permission denied (publickey)
```

roleの判定ロジック自体は正しく動作した(接続失敗 → `time_sync_synced=false` →
critical通知 → fail)。しかし、これによりquoryのNTPが実際には正常であるにもかかわらず、
「quory NTP異常」という偽のcritical Slack通知が本番の#alertsへ送信された。

## 原因

quoryは `control_nodes` グループに属し、`ansible_user: ann` /
`ansible_ssh_private_key_file: ~/.ssh/id_ann` で通常SSH接続する設定になっている。
この設定は、ansyのような**別の**実行元からquoryへ接続する場合は正しく機能する
(本実装の検証中、ansyから繰り返し確認済み)。

しかし、Semaphoreはquory上でネイティブに実行されるため、**quory自身がAnsibleの
実行元(コントローラー)であり、かつtime_sync_checkの対象ホストでもある**という
状況が発生する。time_sync_checkは、quoryをターゲットに含める初めてのplaybookであり
(他のhealthcheck系はpve1/pve2/authy/monnieのみが対象)、この「コントローラーが
自分自身を対象にする」ケースがこれまで存在しなかった。

quory上で実行された場合、`ann@quory.internal`への接続に使われる鍵
(Semaphoreの実行コンテキストでの`~/.ssh/id_ann`)が、quory自身の`ann`ユーザーの
`authorized_keys`に対して正しく認証されない状態だった。

## 修正

`inventories/homelab/host_vars/quory.yml` に、実行元ホスト名を動的に判定して
接続方式を切り替える設定を追加した。

```yaml
ansible_connection: "{{ 'local' if lookup('pipe', 'hostname -s') == 'quory' else 'ssh' }}"
```

- quory上で実行された場合(`hostname -s` が `quory`)→ `ansible_connection: local`
  (自分自身へのSSHを介さず、ローカルでコマンドを実行する)
- 他の実行元(ansy等)から実行された場合 → 通常の`ssh`接続(既存動作を維持)

この判定方式は、`playbooks/proxmox_patch_weekly_full.yml` のpreflightチェックで
既に使われている `lookup('pipe', 'hostname -s')` による実行元ホスト名判定と
同じパターンである。Semaphoreはquory上でコンテナ化せずネイティブに動作している
ことをユーザーに確認済みのため、`hostname -s`は確実に`quory`を返す。

`ansible_connection: local`をquoryに無条件で固定する案は採用しなかった
(ansyからquoryを対象に実行する開発フローが、誤ってansy上でローカル実行されて
しまう回帰を引き起こすため)。


ansyから実行し、quoryへの接続が引き続き実SSH経由で行われ、実際のquoryのchronyc
出力(他ホストとは異なる固有のRef time等)を取得できていることを確認した
(回帰なし)。全7対象ホストが`collected: true, ok: true`で正常終了した。

Semaphore環境での問題を解決するかは、次回のSemaphoreスケジュール実行、または
