# 要求仕様: time_sync_check（時刻同期チェック）

参照:

- docs/ai/prompts/core.md
- roles/common_slack/tasks/notify.yml（Slack通知の呼び出し方式）
- roles/radius_healthcheck/（chrony収集処理の削除対象。tasks/main.ymlとfiles/*.shの該当箇所）
- 既存のexpect利用箇所（Sophos SSD Trim関連 role。Advanced Shell遷移の参考）
- inventories/vars/cloudkey.yml（Vault。SSH用認証情報を追記する）

## 目的

quory を基準として、homelab内の主要ホストの時刻が大きくズレていないかを
定期的に確認する。NTP同期の異常を早期に検知する。

## 対象

- 基準ノード: quory
- 比較対象: pve1, pve2, ansy, monnie, authy, Sophos（sophos-fw）, CloudKey
- 実行元: quory（本番）/ ansy（開発・CLI）
- 安全度: safe（read-only。時刻取得のみ、設定変更は行わない）

## 作成・更新ファイル

- playbooks/time_sync_check.yml（新規・read-only）
- roles/time_sync_check/（新規）
  - tasks/main.yml
  - defaults/main.yml
- inventories/vars/cloudkey.yml に以下を追記（既存Vaultファイル）
  - `cloudkey_ssh_user`（値: root）
  - `cloudkey_ssh_password`（CloudKeyのrootパスワード。既存の`cloudkey_api_user`/`cloudkey_api_password`とは別物・別変数として扱う。混同しないこと）
- inventories/vars/sophos.yml（新規 or 既存があれば追記、Vault暗号化）
  - Sophos Advanced Shell接続用のSSH鍵パス（`~/.ssh/id_rsa_sophos`想定）とユーザー（admin）

## 既存実装の整理（重要・先に実施）

roles/radius_healthcheck/ から chrony 関連の収集処理を削除する。

- files/radius-healthcheck.sh 内の `chronyc tracking` 収集部分
- tasks/main.yml 内の `radius_report.chrony.tracking` をレポートに転記する部分

理由: 現状は収集のみで判定（warning/critical化）が無く、「時刻チェックをやっているように見えて実際は何もしていない」状態になっている。これが今回、既存実装の有無を誤解する原因になった。時刻同期の判定は本Playbookに一本化し、責務を分離する。

## 実施内容

```text
Phase 1  quory自身のNTP同期状態を確認する（chronyc tracking等）
         同期できていない場合はそこで中断する（基準が信頼できないため）
Phase 2  quoryの現在時刻を取得する（date +%s%3N、エポックミリ秒）
Phase 3  各対象ホストの時刻を取得する（下記「取得方法」参照）
Phase 4  quoryとの差分を計算する
Phase 5  差分が閾値（既定: 数百ms。defaults/main.ymlで変更可能）を超えるホストがあれば
         一覧化する
Phase 6  結果を判定し、Slack通知する
```

## 取得方法（対象ごとに異なる）

| 対象 | 接続方式 | 認証 | 取得コマンド |
|---|---|---|---|
| pve1 / pve2 / ansy / monnie / authy | 通常SSH | ann鍵（既存） | `date +%s%3N` |
| Sophos（sophos-fw） | SSH → Advanced Shell経由 | `id_rsa_sophos`鍵 | メニュー操作後（"5"→"3"）に `date +%s%3N`。抜ける時は `exit`→"0"→"0" |
| CloudKey | 通常SSH | `cloudkey_ssh_user`/`cloudkey_ssh_password`（Vault、新規追加） | `date +%s%3N` |

Sophosの Advanced Shell 遷移は `ansible.builtin.expect` を使用する。既存の SSD Trim
関連roleで使われているexpectパターン（メニュー選択 "5"→"3"、終了時 exit→"0"→"0"）を
参考にし、同一の遷移ロジックを流用する。

## shell / Ansible tasks 責務分離（core.md §7）

- 各ホストでの時刻取得（`date +%s%3N`の実行）は command/expect で行う
- 差分計算・閾値判定・fail制御はAnsible tasks側に置く
- shell/expectに正常/異常判定ロジックを持たせない

## 通知

`roles/common_slack/tasks/notify.yml` を include_tasks で呼ぶ（best-effort）。

| 状況 | チャンネル | status |
|---|---|---|
| 全ホスト閾値内 | 通知なし | — |
| 閾値超えホストあり | #alerts | warning |
| quory自身のNTP同期異常（Phase 1で中断） | #alerts | critical |
| 個別ホストへの接続失敗（time_sync_check自体の実行エラー） | #alerts | error |

## 制約

- read-only。対象ホストの設定変更は行わない（NTP設定の修正等は本Playbookの範囲外）
- 秘密情報（CloudKey SSHパスワード、Sophos SSH鍵）を扱うタスクには `no_log: true` を設定する
- IPリテラルをファイルに書かない（core.md §3）。ホスト名で接続する
- 秘密鍵・パスワードをリポジトリに保存しない

## 初回実装スコープ

- quory自身のNTP同期確認
- 全対象ホストの時刻取得（方式は対象ごとに上記の通り）
- 差分計算・閾値判定
- Slack通知
- radius_healthcheckからのchrony収集処理削除

## 初回除外スコープ

- NTP設定そのものの修正・自動補正
- 時刻ズレの履歴管理・トレンド分析

## read-only / 変更系の区別

read-only。状態収集と通知のみ。radius_healthcheckの既存コード削除は「整理」であり、
対象ホストへの変更操作ではない。
