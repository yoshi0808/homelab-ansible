# 追加実装内容: time_sync_check(境界値2件のmust-fix対応)

対象レビュー: `docs/ai/reviews/time_sync_check/2026-06-23_009_review.md`

## 対応概要

2件のmust-fixはいずれも正当な指摘で、両方とも反映した。

### 1. Sophosの未来mtimeをfreshと誤判定する

`_fresh` の条件が `_age_sec <= max_age` のみで、`_age_sec` が負数(mtimeが現在時刻より
未来。clockの巻き戻り等で発生し得る)でも真になっていた。`(_age_sec | int) >= 0` を
条件に追加し、`0 <= age_sec <= max_age` の範囲のみをfreshとするよう修正した。

ローカル検証で、mtimeが現在時刻より未来のケースが正しく`synced=False`になることを
確認した(`detail`には負の経過時間が表示され、異常を示唆する)。

### 2. CloudKeyの必須ステータス行欠落時もcollected=trueになる

`time_sync_collected` がrc markerのみを条件にしており、`System clock synchronized:`/
`NTP service:` 行自体が解析できないケースでもcollected=trueになっていた
(007_reviewで一度指摘され、008では`time_sync_synced`側にのみ反映していたが、
`time_sync_collected`側への反映が漏れていた)。

両行の正規表現マッチの成否(`_status_found`)をcollected条件に追加した。
ローカル検証で、`timedatectl`自体は成功(rc=0)するが期待する行が出力にない
ケースが正しく`collected=False`になり、かつ`'yes' in None`のような例外が
発生しない(Jinja2の`and`短絡評価により後続条件が評価されない)ことを確認した。
