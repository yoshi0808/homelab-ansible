# 実装内容: ansible_facts 移行（INJECT_FACTS_AS_VARS deprecation 対応）

作成日: 2026-06-14
版: 1.0（初回実装）
参照: 2026-06-14_001_requirement.md

---

## 1. 実装概要

fact 由来の `ansible_*` トップレベル注入変数を `ansible_facts['...']` 辞書参照へ
移行した。ansible-core 2.24 で `INJECT_FACTS_AS_VARS` のデフォルトが `False` に
なった場合でも、参照が壊れず DEPRECATION WARNING も出ない状態にした。

実際に移行が必要だった fact 注入変数は 2 種類のみ:

- `ansible_hostname`  → `ansible_facts['hostname']`（実行ホスト制限 assert と通知文）
- `ansible_date_time` → `ansible_facts['date_time']`（証明書名のタイムスタンプ）

---

## 2. 調査結果（移行対象の棚卸し）

リポジトリ全 `*.yml` / `*.j2` / `*.cfg` を対象に `ansible_` を grep し、
参照経路・`gather_facts` 状態・ロール呼出元を確認した。

### 2.1 移行した箇所（計 9 箇所 / 4 ファイル）

| ファイル | 行 | 文脈 | 変換前 → 変換後 | fact 収集 |
|---|---|---|---|---|
| playbooks/cert_renew.yml | 21 | assert that | `ansible_hostname == 'quory'` → `ansible_facts['hostname'] == 'quory'` | play1 localhost `gather_facts: true` ✓ |
| playbooks/cert_renew.yml | 24,25 | fail_msg | `{{ ansible_hostname }}` → `{{ ansible_facts['hostname'] }}` | 同上 |
| playbooks/cert_renew_quory.yml | 20 | assert that | 同型 | play1 localhost `gather_facts: true` ✓ |
| playbooks/cert_renew_quory.yml | 23 | fail_msg | 同型 | 同上 |
| playbooks/cloudkey_cert_deploy.yml | 24 | assert that | `ansible_hostname in [...]` → `ansible_facts['hostname'] in [...]` | play1 localhost `gather_facts: true` ✓ |
| playbooks/cloudkey_cert_deploy.yml | 27 | fail_msg | 同型 | 同上 |
| playbooks/cloudkey_cert_deploy.yml | 68 | Slack 通知文 | 同型 | 同上 |
| roles/cloudkey_cert_deploy/tasks/main.yml | 15 | set_fact (cert 名) | `ansible_date_time.iso8601_basic_short` → `ansible_facts['date_time']['iso8601_basic_short']` | 唯一の呼出元 playbooks/cloudkey_cert_deploy.yml が localhost `gather_facts: true` ✓ |

- `comparison` の右辺（`== 'quory'` / `in ['quory','ansy']`）は変更していない。
- cloudkey_cert_deploy role の呼出元は playbooks/cloudkey_cert_deploy.yml のみ
  （`include_role`）。当該 play は `gather_facts: true` のため、`ansible_facts['date_time']`
  は収集済みで安全。

### 2.2 対象外として除外した箇所（仕様 §2-4 / §4 に準拠）

| 種別 | 変数 | 該当箇所 | 除外理由 |
|---|---|---|---|
| 接続予約変数 | `ansible_user` / `ansible_ssh_private_key_file` | group_vars/*.yml, hosts.yml, roles/sophos_trim/tasks/main.yml | 接続設定であり fact ではない |
| 接続予約変数 | `ansible_host` / `ansible_connection` | hosts.yml, roles/sophos_trim/tasks/main.yml | 同上 |
| マジック変数 | `ansible_failed_result` | cloudkey_cert_deploy.yml:38, proxmox_evacuate_node/main.yml:256, proxmox_patch_apply_node/main.yml:345, proxmox_restore_vm_placement/main.yml:269 | fact ではなくマジック変数。INJECT_FACTS_AS_VARS の影響を受けない |
| マジック変数 | `ansible_play_hosts` | proxmox_healthcheck/main.yml:138,161, proxmox_hw_check/main.yml:109,125 | 同上 |
| ユーザー定義変数 | `systemd_ansible_timers` | roles/systemd_timers/defaults/main.yml:6, tasks/main.yml:9,21,35 | ユーザー定義。`ansible_` 接頭辞だが fact 注入変数ではない |
| 既に移行済み | `ansible_facts['date_time']` | ubuntu_nightly.yml, monitoring_healthcheck, proxmox_healthcheck, proxmox_hw_check, radius_healthcheck | 既に辞書参照形式 |

### 2.3 意図的に変更しなかった箇所

- `playbooks/proxmox_patch_weekly_full.yml:30` の `ansible_hostname` は
  **コメント文中の言及**であり、実行時の変数参照ではない。DEPRECATION WARNING
  にも実行挙動にも無関係のため、変換ルール（実参照のみ対象）に従い変更しない。

## 4. 実行した確認コマンドと結果

### 4.1 全 playbook の syntax-check（PASS）

```
for p in playbooks/*.yml; do ansible-playbook --syntax-check -i inventories/homelab/hosts.yml "$p"; done
```

- 移行対象を含む全 playbook が PASS。
- `proxmox_evacuate_node.yml` と `proxmox_restore_vm_placement.yml` は
  `hosts: "{{ target_node }}"` のため `-e target_node=...` 未指定だと
  `'target_node' is undefined` で失敗するが、これは**本移行とは無関係の
  既存仕様**。`-e target_node=pve2` を渡すと両者とも PASS することを確認済み。
  （両ファイルとも本移行では未変更）

### 4.2 INJECT_FACTS_AS_VARS=False（2.24 将来デフォルト）での動作検証

> 【訂正 2026-06-14 / 2026-06-14_003_review.md Low 指摘を反映】
> 当初この箇所には「`INJECT_FACTS_AS_VARS` には環境変数バインドが無い」と
> 記載していたが、これは事実誤認である。正しい env バインドは
> `ANSIBLE_INJECT_FACT_VARS`（config 名 `INJECT_FACTS_AS_VARS` とは綴りが異なり、
> env 名は **FACT** 単数形）。当初は誤って `ANSIBLE_INJECT_FACTS_AS_VARS`（FACTS
> 複数形）を試し、反映されなかったため誤った断定をした。一時 ansible.cfg を使った
> 当初の検証結果（注入 OFF 時の挙動）自体は有効。

`INJECT_FACTS_AS_VARS` を `False` にして検証した。env バインド
`ANSIBLE_INJECT_FACT_VARS=False`（推奨）、または一時 ansible.cfg
（`inject_facts_as_vars = False`）を `ANSIBLE_CONFIG` で適用、いずれでも同じ結果。

- 注入 OFF の確認: 旧形式 `{{ ansible_hostname | default('<<UNDEFINED>>') }}` が
  `<<UNDEFINED>>` を返す（= 旧コードなら 2.24 で壊れることを再現）。
- 新形式 `ansible_facts['hostname']` は `ansy`、
  `ansible_facts['date_time']['iso8601_basic_short']` は `20260614T...` を返し正常。

### 4.3 安全装置 assert の挙動（従来通り）

検証ホストは ansy（開発機）。

- `cert_renew_quory.yml` play1（quory 限定）を INJECT_FACTS_AS_VARS=False で実行:
  ```
  TASK [Ensure this playbook is running on quory]
  fatal: [localhost]: FAILED! =>
    "assertion": "ansible_facts['hostname'] == 'quory'",
    "evaluated_to": false,
    "msg": "cert_renew_quory.yml must be run on quory only (current host: ansy)."
  ```
  → quory 以外で fail し、fail_msg の `{{ ansible_facts['hostname'] }}` が `ansy` に
  正しく解決。安全装置が従来通り機能。
- cloudkey 形式 assert（quory/ansy 許可）の正常系を同条件のスタンドアロン play で確認:
  `assert passed on ansy`（ansy は許可ホストのため通過）。
  ※ cloudkey_cert_deploy.yml 本体は実 API デプロイを伴う変更系のため全実行はしていない。

## 6. 未対応事項・注意点

- 本変更は read-only な式の書き換えのみで、証明書配置・サービス再起動等の
  変更系処理・分岐条件・しきい値には一切手を入れていない。
- 安全装置 assert の比較対象（quory / ansy）は不変。
- 実機 quory 上での実行確認は Yoshinobu の運用判断に委ねる
  （ansy 開発機では quory 限定 assert は設計通り fail するため）。
