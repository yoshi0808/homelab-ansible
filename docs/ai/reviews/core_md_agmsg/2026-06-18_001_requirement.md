# 要求仕様: core.md への agmsg 受け渡し運用の追記

## 目的
従来 VS Code 拡張で手動コピペしていた implement / review の工程間受け渡しを、
agmsg（エージェント間メッセージング）に置き換える運用方針を `core.md` に明文化する。
ファイルベースの監査証跡は維持したまま、伝達手段だけを agmsg に移す
「監査証跡=ファイル / 伝達=agmsg」のハイブリッドとする。

## 対象
- 更新ファイル: `docs/ai/prompts/core.md`（ドキュメントのみ。コード変更なし）

## 方針（ユーザー確定済み）
- implement / review の**本文はファイルに書く**。ファイルが正本かつ
  git にコミットされる監査証跡。
- agmsg のメッセージには本文ではなく**ファイルパス**を載せる。
  両エージェントは同一作業ディレクトリ（ansy 上のリポジトリ）を共有するため、
  受け取った側はパスから実体ファイルを読む。
- ロール: `claude`（Claude Code, 実装）/ `reviewer`（Codex, レビュー）、team `homelab`。
- Codex（`reviewer`）は Monitor 非対応で自動受信しないため、Claude Code 側から
  agmsg をキックして受信・処理させる。Claude Code は monitor モードで自動受信。
- 運用判断・本番実行判断・commit 判断は §14 の通り Yoshinobu が行う。
  agmsg ループは Yoshinobu の判断待ちで止まる。AI が agmsg 経由で自律的に
  commit や危険操作を行う構成は採用しない。

## 確認項目（レビュー観点）
- 既存の core.md の思想（§14 人間ゲート、§15 ファイル命名規則・恒久記録）を
  壊していないか。
- 「メッセージにはパスのみ、本文はファイル」という役割分担が明確に書けているか。
- §15 の新節と §16 の相互参照に矛盾・重複がないか。
- IP リテラル混入など §3 違反がないか（ドキュメントも対象）。

## 制約
- ドキュメントのみの変更。playbook / role / shell には触れない。
- IP リテラルを書かない（§3。ホスト名は `*.internal`）。

## 安全度
read-only 相当（ドキュメント追記のみ。実行・変更操作なし）。
